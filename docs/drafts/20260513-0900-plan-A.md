### PR-A1 — Codegen producer NodeRef registry (LINCHPIN, retires L0 cascade)

**Scope.** Generalise `resolveCodegenDepRefs` from EN/PB/EV to the
full set of seven kinds (AR `.pyc.inc`, BI, R5, CF, JV, PR, PY)
plus the objcopy-chunk and PR-residue inline cases. Every missing-
dep symptom in the L0 table above resolves through this one PR.

**Files touched.**
- `gen.go` (`genCtx` map fields + `resolveCodegenDepRefs`)
- `m3_misc.go` (populate AR/BI/R5/CF/JV/PR/PY-objcopy producer maps)
- `cc.go` (call-site already feeds inputs; pass extended ctx)
- `resource.go` (objcopy-chunk yapyc3 dep wiring at L1042-1048)
- one new tiny producer table in `codegen_registry.go` if header-
  closure consumers need symmetry

**Mechanism.** Each producer emitter (AR for `.pyc.inc`, BI for
`buildinfo_data.h`, R5 for `.rl.tmp`, CF for `CONFIGURE_FILE`'d
`.cpp`, JV for ANTLR4 `.cpp`/`.h`, PR for RUN_PROGRAM `.cpp`, PY for
`.yapyc3` per-source) already knows its `(absolute output path,
platform) → NodeRef` mapping at emission time. We collect these
into per-kind `map[codegenOutputKey]NodeRef` on `genCtx`,
exactly mirroring the existing `enOutputs`/`pbOutputs`/`evOutputs`.
`resolveCodegenDepRefs` then iterates the consumer's
`includeInputs` and consults all seven maps, threading the
NodeRefs into the consumer's `DepRefs`. The objcopy chunk path
(`resource.go:1043-1048`) is a parallel inlining of the same
walk against the chunk's `ch.pathInps`. The PR-residue is the
same pattern at `m3_misc.go:810` against its `inputs[]`.

**Risk.** Medium. Risk vectors: (a) over-emission — adding a dep
edge to a producer that should be a TRANSITIVE dep, not a direct
one. The 53 leaves we measured are all symmetric to the EN/PB/EV
case we already shipped, so the precedent is solid. (b) Cherry-
pick from stale-base commit `5fdd741` had pb.go conflicts;
re-implement against current master fresh — do **not** apply the
old patch as a hunk.

**Expected lift.**
- L0: **+599 / 8 750 → 100.00 %** (best case, the prior probe
  measured 100 % cascade coverage by these 53 leaves).
- L1: unchanged.
- L2: unchanged (these CC/PY consumers already had correct
  `inputs[]`; only the dep edge was missing).
- L3: unchanged.

**Dependencies.** None — operates on the existing EN/PB/EV
registry pattern; the producers all exist already.

### PR-A2 — JV ANTLR-output system-header inputs (retires 4 of 8 L2)

**Scope.** When a JV node's generated `.cpp` becomes input to a CC
closure (the `.g4.cpp.o` consumer), the CC's `inputs[]` should
include the libcxx + glibc system headers that the generated C++
pulls in. The 69 missing system headers in `TConfLexer.g4.cpp.o`
inputs[] are the same set we already include on every other
hand-written CC consumer — they just don't propagate through the
JV→CC path because the JV's "synthetic generated source" hasn't
been fed through the system-header scanner.

**Files touched.**
- `m3_misc.go` (JV emitter — when emitting the generated `.cpp` /
  `.h` outputs, add the standard system-header set to the JV's
  `EmitsIncludes`)
- possibly `scanner.go` (if JV outputs aren't currently visited
  by the system-include closure)

**Mechanism.** Mirror what PR-M3-F-7-D did for protobuf-runtime
on PB/EV: the JV producer has known generated source content
that always pulls in `<cstdint>`, `<inttypes.h>`, etc. Register
that closure on the JV's `EmitsIncludes`; the CC consumer then
inherits it via the existing scanner machinery. The 69-header
set is fixed (libcxx `inttypes.h`, glibc `bits/{stdint-intn,
stdint-uintn,libc-header-start}.h`, etc.) and identical to what
every other `.cpp.o` already inherits.

**Risk.** Low. The 69-header set is byte-identical to what
neighbouring CC nodes already have; we're routing it through
one new edge, not inventing headers.

**Expected lift.**
- L0: 0 (these 4 nodes are already paired at L0 — they're L2-
  only divergences).
- L2: **+4 / 8 → only ev.pb.cc.o self-include + python3-Lib AR
  + struct2fieldcalc.rl.tmp remain**.
- L3: 0.

**Dependencies.** Independent of PR-A1.

### PR-A3 — EV self-include removal + python3-Lib AR closure clamp + ev/event2cpp protobuf include + abseil py3 ADDINCL (retires the remaining L2 + 3 L3)

5. `main.cpp.o` and `print.cpp.o` need `-I$(BUILD_ROOT)/library/cpp/

**Files touched.**

- `cc.go` or AR composer (python3-Lib AR closure clamp) — small.
- `m3_misc.go` (event2cpp/PB consumer protobuf -I slot) — ≈5 lines.
- `cc.go` / py3 module composer (abseil ADDINCL on py3) — ≈5 lines.
- `m3_misc.go` (BI's own-dir ADDINCL propagation to peers) — ≈5 lines.

**Mechanism.** All five are slot-population fixes in the existing

one PR because they share `m3_misc.go` / `cc.go` and are all
"one slot missing, sibling already does it" patterns.

**Risk.** Medium. Slot-position bugs are the highest M3-sensitive
risk class; verify byte-exact M1/M2 (`p`/`kv` field-presence)
holds after each sub-edit. Edits 3 + 4 + 5 are include-path

### PR-A4 — AR member-order + objcopy input-order (LINCHPIN — retires the final 2 L3)

**Scope.** Two ordering tweaks:

1. `liblibrary-cpp-build_info.a` member-order: REF orders
   sandbox.cpp.o (10), build_info.cpp.o (11),
   build_info_static.cpp.o (12); we currently order build_info,
   sandbox, build_info_static (or some other permutation). The
   AR member list is order-significant for L3 (cmd_args).
2. `objcopy_c325f0009e9625395005936d90.o` cmd_args: REF lists
   `__init__.py.yapyc3` THEN `__init__.py`; we list them swapped.
   This is the PY3 single-source module case in `resource.go`'s
   `--inputs` composer; we sort one way, REF sorts the other.

**Files touched.**
- `m3_misc.go` AR-member ordering — likely a stable-sort tweak in
  `composeAR` or an `OrderMembers` helper. The expected order is
  declaration-order-in-`build/conf/` for `build_info.cpp` / `sandbox.cpp`
  / `build_info_static.cpp`; verify by reading
  `yatool_orig/build/conf/`.
- `resource.go` (`emitPySrcObjcopy` — yapyc3 input first, then `.py`).

**Mechanism.** Both are sort-key inversions — fix the comparator.
For AR, drive ordering off the `build/conf/` declaration sequence

config). For objcopy, the input order in upstream `link_dyn_lib.py`
(or `resource.py`) lists the bytecode-compiled output before the
source — match that.

**Risk.** Low. Ordering changes are localized and don't affect

re-running the topology comparator after.

**Expected lift.**
- L0: 0.
- L2: 0.
- L3: **+2 / 2 → 100.00 %**.

**Dependencies.** Independent of PR-A1/A2/A3. **The single

### PR-A5 — `stats_uid` emission + closure-rolled `self_uid != uid` (M3 closeout, optional for L0-L3 100 %)

**Scope.** Match upstream's two-tier identity scheme:

1. Emit `stats_uid = md5(str([platform, sorted(tags), kv.p,
   sorted(outputs)]))` on every node. Upstream formula at
   `yatool_orig/devtools/ya/build/gen_plan/__init__.py:632-642`.
2. Make `uid` a closure-rolled hash:
   `uid = md5(self_uid ‖ for each build-dep d: d.uid)`. Upstream
   algorithm at `yatool_orig/devtools/ymake/export_json.cpp:360-421`
   (`ComputeFullUID`). Keep `self_uid` as the present 22-char
   content hash; rename current `uid` → `self_uid` and synthesize
   a new closure-rolled `uid` in a post-emission Merkle pass.

**Files touched.**
- `uid.go` (new `selfUID` + `fullUID` separation; the Merkle pass
  already exists in `Finalize`).
- `node.go` (`StatsUID` becomes `json:"stats_uid"` instead of
  `json:"-"`).
- `gjson_write.go` (emit `stats_uid` between `sandbox` and
  `self_uid` per upstream key order).
- Test fixtures: `node_test.go`, `gjson_test.go`.

**Mechanism.** Pure post-processing. Both fields are derived
from data we already have. The order of writes already matches
upstream's alphabetical field order (sandbox, self_uid,
stats_uid? — verify against `sg2.json` byte layout).

**Risk.** Low for stats_uid (additive field). Medium for the
self/full uid split — every test that pins `uid:` byte-for-byte
needs to be updated, and the Merkle pass that resolves DepRefs
must use the new `uid` (not `self_uid`) for child references.
Order operations carefully:

  1. Compute `self_uid` for every node (current `computeUID`).
  2. Topologically order build-deps.
  3. Roll `uid = md5(self_uid ‖ children.uid)`.
  4. Substitute child uids into Deps using the rolled values.

**Expected lift.**
- L0: 0 (topology comparator is identity-independent).
- L1: 0.
- L2: 0.
- L3: 0.
- New axis ("identity fields paired"): 0 → 100 % on stats_uid;
  closes 1 282 / 8 750 unpaired self_uid on the closure axis.

**Dependencies.** Land after PR-A1–A4 so byte-exact byte
comparators don't get polluted by simultaneous identity churn.
Strictly optional for the literal L0=L1=L2=L3=100 % criterion;
include only if the milestone definition also requires identity-

## Verification recipe per PR

After each PR, run the three-line acceptance check above + the
topology fp re-compute below to ensure L0 hasn't regressed:

```
python3 -c '
import json, hashlib
def build_fps(g, by):
    memo = {}
    def calc(u):
        if u in memo: return memo[u]
        n = by.get(u);
        if not n: memo[u]="0"; return "0"
        h = hashlib.sha1()
        h.update(n.get("kv",{}).get("p","").encode())
        for c in sorted(calc(d) for d in n["deps"]): h.update(c.encode())
        memo[u] = h.hexdigest()
        return memo[u]
```

## What this plan deliberately does NOT do

- No new module-system refactoring. The existing producer maps
  already cover every kv.p we need; we just thread NodeRefs
  through them.
- No host-platform axis work. M1/M2 carry through unchanged.
- No sysincl/sysconfig-yaml work. The L2 system-header gap on
  JV outputs (PR-A2) is solved by registering the closure on the
  JV producer, NOT by re-running the system-header scanner over
  generated `.cpp`.
- No `gen.go` / `m3_misc.go` mass rewrite. Each PR is additive
  to the existing emitter — no churn on M1/M2 byte-exact code.
