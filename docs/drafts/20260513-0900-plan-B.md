## 5. Systemic root causes (mechanism)

### 5.1 Root cause A — `resolveCodegenDepRefs` only consults 3 hand-coded maps

**Where:** `gen.go:344-385`. The function that decides which producer
NodeRefs to thread into a CC consumer's `deps[]` (`srcIn.ExtraDepRefs`)
consults *only* `ctx.enOutputs`, `ctx.pbOutputs[k]`, `ctx.evOutputs[k]`.

```go
if r, found := ctx.enOutputs[p]; found { ref, ok = r, true }
else if r, found := ctx.pbOutputs[codegenOutputKey{...}]; found { ref, ok = r, true }
else if r, found := ctx.evOutputs[codegenOutputKey{...}]; found { ref, ok = r, true }
```

Producers of type `AR / CF / BI / JV / PR / R5 / PY` never enter this
lookup, so no CC consumer that #includes their generated output ever
gets the corresponding ref-edge.

**The CodegenRegistry (`codegen_registry.go`) is the right place for
this generalisation but is currently used only by the scanner's
existence-tier (`codegenLocator.Exists`)**, not by the producer-ref
emitter. It already registers EN/R6/EV/R5/CF (and could be extended to
PB/BI/JV/PR/AR/PY). The whole `enOutputs/pbOutputs/evOutputs` triple
is functionally a redundant, type-narrow shadow of CodegenRegistry.

### 5.2 Root cause B — uneven registry coverage

Compared against the upstream node-kind set that the C++ ymake emits
typed `EDT_BuildFrom` / `EDT_BuildCommand` edges for (per
`json_visitor.cpp`), the OUR-side codegen registry has gaps:

| Producer | Registered in CodegenRegistry? | In producer-ref-resolver? |
|---------:|:------------------------------:|:-------------------------:|
| EN       | yes (gen.go:5386, 5406)        | yes (enOutputs)           |
| EV       | yes (gen.go:5962, 5973)        | yes (evOutputs)           |
| PB       | **no**                         | yes (pbOutputs — but map is never written; latent) |
| R6       | yes (gen.go:5809)              | no                         |
| R5       | yes (gen.go:6065, 6070)        | no                         |
| CF       | yes (gen.go:6138)              | no                         |
| BI       | **no**                         | no                         |
| JV       | **no**                         | no                         |
| PR       | **no**                         | no                         |
| AR       | **no** (different kind: linkage product, not codegen output) | no |
| PY       | **no** (same — bytecode product) | no                       |

**`pbOutputs` exists but is never `Register`-ed** — gen.go:693 allocates
the map, gen.go:366 reads it, no site writes to it. Either dead code or
the closure under test never exercises a CC consumer that #includes a
`.pb.h`. Latent defect either way.

### 5.3 Root cause C — `AR` and `PY` are not "codegen output" producers but `deps[]` still needs ref-edges from `#include`-bearing inputs

For `AR/PY` producer-ref edges (e.g.
`__res.cpp.o` `deps[]` contains an `AR` ref to `__res.pyc.inc`), the
mechanism in upstream ymake is *not* an #include-driven scan — it's
the `pyc.inc` being a generated input listed in the CC node's
`inputs[]` rather than a `#include` discovered through the scanner.
Verification:

- `__res.cpp.o` is missing `AR ref → __res.pyc.inc`. The `.pyc.inc` is
  generated by the AR step (RESOURCE objcopy variants).
- Each of the 41 PY-leaf `objcopy_*.o` files needs a `PY ref` to a
  `*.yapyc3` bytecode — the bytecode is an *input* to the objcopy
  step, not a `#include` target.

So the architectural lift is broader than "expand
`resolveCodegenDepRefs`": it must cover both
**#include-driven** producer refs (the current EN/PB/EV path) AND
**input-driven** producer refs (RESOURCE inputs, PY_SRCS bytecodes,
INDUCED_DEPS, CF outputs included as headers, etc).

### 5.4 Root cause D — Generator-ref filtering already exists; reuse it

The `exclude` parameter of `resolveCodegenDepRefs` (gen.go:344) shows
the pattern is in place: when a CC compiles a producer's own .cc, the
producer is excluded so it isn't double-listed. The same filter
should apply after generalisation — only the lookup side has to grow.

## 6. Architectural PRs ordered by cascade-yield

### PR-1 — Generalise the producer-ref path so AR & friends thread refs (cascade-yield: 543 + 11 + 1 = 555 mismatches close)

**Mechanism**

Replace the three-map shadow (`enOutputs/pbOutputs/evOutputs`) with a
single producer-NodeRef map keyed by `(platform, OutputPath)` that
*also* carries `ProducerKvP`. Wire every emitter (EN/PB/EV/R6/R5/CF/BI/
JV/PR/AR/PY) to write into it at registration time. Replace
`resolveCodegenDepRefs`'s body with a single map probe.

For `AR` / `PY` (input-driven, not include-driven), extend the lookup
trigger: in addition to scanning `includeInputs` for $(BUILD_ROOT) paths,
also scan the CC node's `inputs[]` for $(BUILD_ROOT) entries and probe
the same map. Inputs that originate from a registered producer become
ref-edges.

**Why it has the highest yield**

- The 11 CC leaves (AR=4, CF=2, BI=1, PR=1, JV=2, R5=1) all become matched.
- AR `runtime_py3.a` cascader closes.
- LD `py3cc/slow` closes (its only divergent child was that AR).
- LD `ymake` closes its `runtime_py3.a` child.
- All 536 PY cascaders close (their only divergent child was py3cc/slow LD).
- 6 other AR cascaders close (their children are the 6 non-AR CC leaf
  clusters).
- LD `struct2fieldcalc` closes (R5 leaf below it closes).
- PR `dep_types.h_dumper.cpp` cascader closes (its EN ref also needed).

Yield: **555/599** (everything except the 41 PY leaves, see PR-2).

**Risk:** the input-driven branch must filter out non-producer inputs
(real source files in $(BUILD_ROOT) via earlier codegen) carefully.
The existing `exclude` mechanism handles "this CC compiles the
producer's own .cc"; the new branch needs a similar
"don't add a self-ref when the input IS the consumer's own output".
M2 byte-exact stays intact because no `inputs[]` change is required,
only `deps[]` is enriched.

### PR-2 — PY-leaf producer-ref: thread PY (yapyc3) refs into RESOURCE objcopy CC nodes (cascade-yield: +44 mismatches close)

**Mechanism**

The 41 PY leaves are objcopy_-named `.o` files generated by
`library/python/symbols/module` and `contrib/tools/python3/Lib` (the
RESOURCE-via-objcopy producer at `gen.go:emitResourceObjcopy`). Each

emits a `PY` ref-edge per `.yapyc3` input; our emitter does not.

Wire `emitResourceObjcopy` (gen.go:~4325, ~2668, ~4102) to: for each
`*.yapyc3` path in its inputs, look up the producer PY NodeRef via the

**Yield:** 41 PY leaves + 2 trivial ancestors (`libpy3python-symbols-
module.global.a`, `libpy3tools-python3-Lib.global.a` — these are 2 of
the 7 AR cascaders, so really +43 if not already credited; +44 total
includes the next-level `ymake LD` re-check).

Closes the remaining 44 mismatches → **L0 599 → 0 (100%)**.

## 10. Risks and caveats

- **M2 invariant.** Producer-ref enrichment touches `deps[]` only. The
  RESOURCE objcopy inputs that drive PR-2 are already in our
  `inputs[]` (M2 byte-exact verified) — we just don't lift them into
  `deps[]`. No M2 disturbance expected, but every PR must re-verify.

- **The `exclude` filter.** When extending to AR/PY input-driven
  lookups, the self-ref guard becomes: "do not thread a ref to a
  producer whose output path equals (or is downstream of) the
  consumer's own producer output." Existing `exclude` covers
  cc-compiles-own-producer.cc; the new path must extend coverage.

- **Platform keying.** AR/PY producers are platform-specific
  (target vs host). The current per-platform `codegenOutputKey` works
  for PB/EV; the generalised registry must adopt the same key shape
  for AR/PY. Cross-platform leakage would mis-pair a target consumer
  with a host AR.

- **Latent `pbOutputs` writer.** If a future closure (i.e. a target
  that #includes a `.pb.h`) is added to M3 scope, the missing
  pbOutputs writer will surface as new L0 mismatches. PR-3 (or PR-5)
  removes this trap.

- **Cycle detection in registry.** With AR/PY consumers also being
  registry-lookup candidates, a producer whose own #includes lead
  back to itself transitively could cycle. The current `seen` map in
  `resolveCodegenDepRefs` prevents duplicate refs; whether it
  prevents cycles depends on whether AR/PY consumers can re-enter the
  same producer through a different path. Mitigation: the current PR
  loop catches such regressions in the adversarial review.

- **CodegenRegistry duplicate-throw.** `Register` throws on duplicate
  OutputPath (mirrors upstream DupSrc). AR-product registration must
  avoid colliding with per-source AR variants (e.g. `.a` vs
  `.global.a`); these have distinct OutputPaths so no collision is
  expected, but PR-1 must verify on first registration sweep.

## 11. Anti-recommendations (what NOT to do)

- **Do not** hand-thread AR ref edges in `emitResourceObjcopy` only.
  That fixes the symptom (the 4 AR leaves) without addressing the
  systemic seam in `resolveCodegenDepRefs`. The next missing
  producer-ref kind that appears in a different closure will require
  another bespoke patch.

- **Do not** add a fourth/fifth/sixth hand-coded map alongside
  `enOutputs/pbOutputs/evOutputs`. PR-1 collapses them; do not extend
  them.

- **Do not** "fix" the PY leaves first. Their cascade contribution is
  43 of 599 (7%); the cascade is dominated by AR.

- **Do not** touch `deps[]` ordering or sort semantics in the
  generalised path — the M3 L1=100% byte-exact metric depends on
  exact ordering. The current `exclude`+seen pattern preserves
  scan-order dedup; the generalised path must match.
