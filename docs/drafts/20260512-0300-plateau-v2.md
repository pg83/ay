## 1. L3 plateau — top 5 clusters

### L3-A. CC peer-include ordering inside `devtools/ymake` — 249 nodes

The dominant L3 cluster. 249 of 255 CC TUs under `devtools/ymake` differ
from REF in pure ordering of `-I…` peer-include paths. Set membership of
`-I` flags is identical except in 6 outliers (mostly the `*.pb.cc.o`
files which still differ by 7 missing protobuf/abseil/cxxsupp `-I` — see
L3-D below).

**First-divergence histogram** (CC ordering-only across whole graph):

```
  idx   9..10:   6  (early-block reordering)
  idx  21:     141  (lzma vs python/Include rotation)
  idx  26:      71  (sparsehash vs brotli rotation)
  idx  68:      29  (-DASIO_STANDALONE vs -DOPENSSL_RENAME_SYMBOLS rot.)
```

Concrete example, `devtools/ymake/dump_owners.cpp.o`:

```
OUR @21:  lzma → openssl → libffi(2) → python/Include → runtime_py3 → abseil-cpp
REF @21:  python/Include → abseil-cpp → runtime_py3 → lzma → openssl → libffi(2)
OUR @34:  sparsehash → brotli → snappy
REF @34:  brotli → snappy → sparsehash
```

Same 32 `-I` directories; REF emits them in a **DFS-post-order on the
peer graph** while OURS emits in a different (likely declaration-order /
BFS) traversal. The third reorder at `@68` is the same bug applied to
`-D` macros emitted from peers (asio vs openssl-rename).

**Root cause hypothesis**: OUR's CCFlags-aggregating walker traverses
the peer DAG in a different order than ymake's actual `gen-plan` walker
(probably visiting children before their grand-children, or sorting by
something other than topo-post-order).

**Scope**: traversal-order fix in the peer-flags accumulator. Single
file change in `gen.go`/`scanner.go`. Risk: low — set membership is
already correct; only emission order changes.

**Estimated lift**:
- L3: +249 paired (96.08 → ~98.93).
- Knock-on: 5 AR `ordering-only` and small additional improvements on
  related contrib/libs CC TUs that share the same bug.
- Net L3: 96.08 → ~99.0.

**Effort**: 1 small PR. ~½ day; deterministic verification.

### L3-C. AR `.o` input-list ordering — 5 nodes

Same root cause as L3-A: AR collects `.o` files in a different order
than REF. Five nodes: `devtools/ymake/diag/libdevtools-ymake-diag.a`,
`library/cpp/json/fast_sax/libcpp-json-fast_sax.a`,
`devtools/ymake/lang/libdevtools-ymake-lang.a`, etc. Sample:

```
@14: OUR='…/trace.ev.pb.cc.o' REF='…/trace.cpp.o'
```

Sorts/orderings of `*.cpp.o`, `*.pb.cc.o`, `*.ev.pb.cc.o`, `*.rl6.cpp.o`
within one library differ. Almost certainly the same emit-order issue —
codegen-derived `.o`s emitted before the hand-written `.cpp.o`s instead
of the REF order which interleaves them.

**Lift**: L3 +5 (negligible). **Effort**: piggy-backs on L3-A fix.

## 2. L2 plateau — top 5 clusters

### L2-A. PY contrib/tools/python3 — 82 nodes, single missing .py each

`contrib/tools/python3/lib2/py/objcopy_<hash>.o` nodes. OUR input-count
distribution is **17 inputs** while REF is **18-33**. Each affected node
is missing exactly one .py from a `.pyc → objcopy` bundle, except a
handful with 14–15 inputs vs REF 29–33 (those miss many).

Distinct missing .py files: **556**. Top-frequency .py files are missed
in 3–4 nodes each (no concentration on one file → the bug is in the
batcher, not in one file's metadata):

```
  4× hp_roman8.py
  4× secrets.py
  3× forkserver.py, bz2.py, wintypes.py, …
```

**Root cause hypothesis**: the .py → objcopy batching uses a different
ordering or `len()` cutoff than REF, splitting the .py list into bundles
with a slightly different boundary on each platform.

**Lift**: L2 +82 (97.84 → ~98.78). **Effort**: medium — needs to read
the upstream `_split_to_chunks` (or whatever the batcher is in ya build
ulib) and match it byte-for-byte.

## 3. L0 cascade analysis

### REF-only nodes (14 total)

Two genuine topology gaps drive the count:

```
  [AR]  certs/libcerts.global.a                       (aarch64)
  [AR]  library/cpp/malloc/jemalloc/libcpp-malloc-jemalloc.a    (x86_64)
  [CC]  library/cpp/malloc/jemalloc/malloc-info.cpp.pic.o       (x86_64)
  [AR]  tools/enum_parser/enum_serialization_runtime/lib*.a     (aarch64)
  [CC]  …/dispatch_methods.cpp.o, enum_runtime.cpp.o,
        ordered_pairs.cpp.o                                     (aarch64)
  [AR]  contrib/tools/python3/Lib/libpy3tools-python3-Lib.global.a  (aarch64)
  [AR]  contrib/tools/python3/lib2/py/libpy3python3-lib2-py.global.a (both)
  [AR]  library/python/runtime_py3/__res.pyc.inc                (both)
  [AR]  library/python/runtime_py3/sitecustomize.pyc.inc        (both)
```

This is **5 independent root-cause clusters**, not 14:

1. **certs.global.a (aarch64-only)** — missing PEERDIR/SRCS path on
   that platform axis. 1 node.
2. **jemalloc on x86_64** — platform-conditional library not emitted
   on x86_64. 2 nodes (LD + AR). Investigate the `arch` switch in the
   malloc selector.
3. **enum_serialization_runtime on aarch64** — 4 nodes (3 CC + 1 AR)
   not emitted on aarch64. Likely the same kind of platform-axis bug:
   x86_64 has them, aarch64 misses them.
4. **python3 .pyc.inc / global.a chain** — 6 nodes. The .pyc.inc
   include files and their containing global.a's are missing entirely.
   Same root cause as L3-D's `runtime_py3.global.a` sub-bug — the
   resource/pyc emission pipeline is incomplete.
5. **None of these are cascade-effects of leaf fixes** — they are
   genuine topology divergences.

**Lift**: each of clusters 1–4 also resolves a small bundle of
downstream L2/L3 diffs (when the AR is created, anything that PEERs to
it picks up its bits). Total +14 paired L0 directly, plus ~5-10 L2/L3
ripples.

**Effort per cluster**:
- Cluster 1 (certs.global.a aarch64): small — single PEERDIR
  conditional.
- Cluster 2 (jemalloc x86_64): small — flip the arch selector.
- Cluster 3 (enum_serialization_runtime aarch64): small — arch
  conditional in the generator.
- Cluster 4 (.pyc.inc emission): medium — needs the resource-pipeline
  closure described in `20260511-2200-pr-resource-objcopy-research.md`.
