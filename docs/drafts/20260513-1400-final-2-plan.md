## 1. Node 1 diagnosis — ymake aarch64 peer-archive ordering

### Surface

```
slots 18-21 (--whole-archive .global.a block):
  REF: [tcmalloc/no_percpu_cache, ZSTD, symbols/module, symbols/libc, symbols/python]
  OUR: [tcmalloc/no_percpu_cache, symbols/module, symbols/libc, symbols/python, ZSTD]

slots 70-76 (--start-group regular .a block):
  REF: [absl_flat_hash, blockcodecs/core, xxhash, zstd, resource, symbols/module, symbols/registry]
  OUR: [symbols/module, symbols/registry, absl_flat_hash, blockcodecs/core, xxhash, zstd, resource]
```

The same 7 modules appear in both, with the same 5-item "resource
cluster" contiguously, and the same 2-item "symbols cluster"
contiguously. The defect is the **relative order**: REF places the
resource cluster BEFORE the symbols cluster; OUR does the opposite.

### Closure paths

Both clusters reach ymake/bin via the USE_PYTHON3 chain:
- `contrib/libs/python` PEERs `[library/python/symbols/module, symbols/libc,
  symbols/python, library/python/runtime_py3, contrib/tools/python3, ...]`
  (`/home/pg/monorepo/yatool_orig/contrib/libs/python/ya.make:23-25, 49`).
- `library/python/runtime_py3` PEERs `library/cpp/resource`
  (`/home/pg/monorepo/yatool_orig/library/python/runtime_py3/ya.make:5-7`).
- `library/cpp/resource` PEERs `[absl_flat_hash, blockcodecs/core,
  blockcodecs/codecs/zstd]`
  (`/home/pg/monorepo/yatool_orig/library/cpp/resource/ya.make:3-7`).

`library/cpp/blockcodecs/codecs/zstd` is GLOBAL-only (`SRCS(GLOBAL zstd.cpp)`,
`/home/pg/monorepo/yatool_orig/library/cpp/blockcodecs/codecs/zstd/ya.make:8`)
— it contributes only `.global.a`, no plain `.a`. That is why slot 18
of the whole-archive list is its `.global.a` and not a plain `.a` further
down.

### Upstream link cmdline shape

The ld template
(`/home/pg/monorepo/yatool_orig/build/conf/linkers/ld.conf:216-232`):
```
REAL_LINK_EXE_CMDLINE = ...
    ${pre=--whole-archive-peers :WHOLE_ARCHIVE_PEERS} \
    ${pre=--whole-archive-libs :_WHOLE_ARCHIVE_LIBS_VALUE_GLOBAL} \
    ...
    $_LD_SRCS_GLOBALS \
    $VCS_C_OBJ $_EXTRA_OBJS $AUTO_INPUT -o $TARGET \
    $_EXE_FLAGS ...
```
where `_LD_SRCS_GLOBALS` (ld.conf:80) expands to the whole-archive
block:
```
-Wl,--whole-archive --ya-start-command-file ${rootrel;ext=.a:SRCS_GLOBAL}
--ya-end-command-file -Wl,--no-whole-archive
${rootrel;ext=.o:SRCS_GLOBAL} ${rootrel;ext=.supp:SRCS_GLOBAL}
```
and `_EXE_FLAGS` (ld.conf:170-200) expands to the start-group..end-group
peer-archive block:
```
... $_START_GROUP ${rootrel:PEERS} $_END_GROUP ...
```

Both blocks (`SRCS_GLOBAL` and `PEERS`) are populated from the
module's `UniqPeers` list — see
`/home/pg/monorepo/yatool_orig/devtools/ymake/module_restorer.cpp:502-530`.
`UniqPeers` is built in `TTransitivePeersCollector::Collect`
(module_restorer.cpp:70-93): on `Left` from a peer, `parent.UniqPeers
+= MergeLists(peer.UniqPeers) + [peer]`. Order is **postfix DFS over
direct-peerdir edges** (`compact_graph/peer_collector.h:16, 38`).

### Why upstream slots resource-cluster before symbols-cluster

Multimodule resolution. `library/python/symbols/module` is declared
`PY23_LIBRARY()` (`/home/pg/monorepo/yatool_orig/library/python/symbols/module/ya.make:1`)
— a multimodule. `library/python/runtime_py3` is `PY3_LIBRARY()`
— also a multimodule. Their `.a` outputs receive the `libpy3` prefix
once `PEERDIR_TAGS=PY3` resolution picks the PY3 variant.

By contrast `library/cpp/resource` and `library/python/symbols/registry`
are plain `LIBRARY()` — single-variant — and resolve immediately at
parse time. `library/python/runtime_py3/main` is also a plain `LIBRARY()`
(`/home/pg/monorepo/yatool_orig/library/python/runtime_py3/main/ya.make:1`)
— that is why REF slots `main.a` BEFORE `runtime_py3.a` in py3cc/slow
(REF slots 65-66 vs OUR 65-66 inverted, same defect).

The empirical rule observable in the reference graph: **within a
contiguous segment of UniqPeers contributed by one DFS branch,
multimodule-resolved peers (those whose archive has the `libpy3`
prefix from PEERDIR_TAGS=PY3 selection) slot AFTER single-LIBRARY
peers.** Three independent data points support this:

| Node | Position | Single peers | Multi peers | Order in REF |
|---|---|---|---|---|
| ymake aarch64 | regular .a 70-76 | [absl_flat_hash, blockcodecs/core, xxhash, zstd, resource] | [symbols/module, symbols/registry] (registry is single but is pulled via symbols/module) | single first |
| ymake aarch64 | .global.a 18-21 | [zstd.global.a] | [symbols/{module,libc,python}.global.a] | single first |
| py3cc/slow x86_64 | regular .a 65-66 | [runtime_py3/main.a] | [runtime_py3.a] | single first |

OUR's walker (`gen.go:2900-3030`) iterates `allPeers` in declaration
order without distinguishing multimodule peers; postfix DFS over the
peer list of `contrib/libs/python` enumerates `[symbols/module,
symbols/libc, symbols/python, runtime_py3, ...]` in textual order,
giving `[symbols/{module,registry}, then resource cluster]`. REF
inverts this.

### Fix sketch

Partition the peer-archive aggregator output into two ordered
sub-lists: `singleLIBPeers` and `multiLIBPeers`, where the
discriminator is "this peer's archive name has the `libpy3`
prefix" (== "this peer was emitted from a multimodule selection").
Concatenate `singleLIBPeers ++ multiLIBPeers` preserving relative
order within each group.

Apply the same partition to **both**:
- the `--whole-archive` `.global.a` aggregator (the `peerGlobalPaths`
  slice in gen.go:3032).
- the `--start-group` `.a` aggregator (the `peerArchivePaths` slice in
  gen.go:3031).

Discriminator implementation: `moduleEmitResult` already carries
`ARPath` (gen.go ~line 245); detect multimodule by
`strings.Contains(ARPath, "/libpy3")` (or a structural flag set by
the emitter that observed the PY3 variant). The partition must
preserve **two segment boundaries**: language-defaults precede
program-defaults precede user-peers (this is the existing
gen.go:3027 ordering, untouched). The single/multi split applies
*within* each segment, **and** within the transitive closure of
each direct user-peer.

LOC estimate: ~30 lines in gen.go's `peerArchiveAddPath` /
`peerGlobalAddPath` helpers + ~10 lines threading a flag through
`moduleEmitResult` (if structural). Single PR.

### Risk

- **M1/M2 byte-exact**: M1 has no multimodule peers; trivially safe.
  M2 (`tools/archiver`) also has no `libpy3`-prefixed archives in
  its closure (verified by spot-check: `tools/archiver/archiver`
  cmd[2] in M2 has zero `libpy3-` substrings). Safe.
- **L0**: topology fingerprint depends on UID, not on cmd_args slot
  order. Safe.
- **L1**: outputs/tags/kv unchanged. Safe.
- **L2**: inputs may shift if input-aggregator also folds peers in
  order. Inputs are *set* equality in L2 (sg.json's `inputs` lists
  are checked as multisets); unaffected.
- **L3**: the change CAN only shift slot positions inside cmd[2] of
  PROGRAM LD nodes. Spot-check the 13 host PROGRAMs in M3 to ensure
  none regress (protoc x86_64 currently matches — its closure has
  no `libpy3` peers, so the partition is a no-op).

## 2. Node 2 diagnosis — py3cc/slow x86_64

### 2a. objcopy paths emitted with `$(BUILD_ROOT)/` prefix instead of bare

OUR slot 24-26:
```
$(BUILD_ROOT)/tools/py3cc/slow/objcopy_5950979f44ce74c40c102f9c54.o
$(BUILD_ROOT)/tools/py3cc/slow/objcopy_4b1c18d0dc6973976969ad23be.o
$(BUILD_ROOT)/tools/py3cc/slow/objcopy_c3a5182796bc68c054c676bcc0.o
```
REF slot 23-25:
```
tools/py3cc/slow/objcopy_4b1c18d0dc6973976969ad23be.o
tools/py3cc/slow/objcopy_5950979f44ce74c40c102f9c54.o
tools/py3cc/slow/objcopy_c3a5182796bc68c054c676bcc0.o
```

**Upstream rule**: the `.o ext` SRCS_GLOBAL aggregator in ld.conf:80
uses the `rootrel` modifier
(`${rootrel;ext=.o:SRCS_GLOBAL}`) which strips `$(BUILD_ROOT)/` from
each path. The `${rootrel;ext=.a:SRCS_GLOBAL}` modifier inside
`-Wl,--whole-archive` does the same — see the whole-archive
`.global.a` slots in this same node, which are all bare
(e.g. slot 17 `contrib/libs/tcmalloc/no_percpu_cache/...`, not
`$(BUILD_ROOT)/contrib/libs/...`).

**Fix sketch**: in `ld.go::composeLDCmdLinkExe` (line ~595), when
appending the objcopy-derived elements of `ccPaths` to cmd_args,
strip the `$(BUILD_ROOT)/` prefix. The `__vcs_version__.c.pic.o`
itself (== `vcsOPath`) is NOT a SRCS_GLOBAL output — it is
`$VCS_C_OBJ` (ld.conf:230) — so it KEEPS the `$(BUILD_ROOT)/` prefix.
This separation forces splitting `ccPaths` into `objcopyPaths` (bare)
and `regularCCPaths` (keep `$(BUILD_ROOT)/`).

The caller (`gen.go:4234-4236`) currently appends `objcopyOutputs`
into `ccPaths`. Change it to pass `objcopyOutputs` as a separate
argument to `EmitLD`, or fold the prefix-stripping into
`composeLDCmdLinkExe` when it detects `.../objcopy_<hex>.o`.

LOC: ~15 lines (gen.go: pass separate slice; ld.go: emit before
vcsOPath, strip prefix).

### 2b. objcopy block slots BEFORE `$VCS_C_OBJ`, not AFTER

OUR slots 23-26:
```
23: $(BUILD_ROOT)/.../__vcs_version__.c.pic.o    ← OUR's vcsOPath
24-26: objcopy_*.o                                ← OUR's objcopyOutputs
```
REF slots 23-26:
```
23-25: objcopy_*.o                                ← part of _LD_SRCS_GLOBALS
26: $(BUILD_ROOT)/.../__vcs_version__.c.pic.o    ← $VCS_C_OBJ
```

**Upstream rule**: ld.conf:228-230 places `$_LD_SRCS_GLOBALS` BEFORE
`$VCS_C_OBJ`. `$_LD_SRCS_GLOBALS` includes `${rootrel;ext=.o:SRCS_GLOBAL}`
— the objcopy outputs (which are OUT_GLOBAL `.o` files). So objcopy
outputs MUST appear before `__vcs_version__.c.pic.o` in cmd[2].

**Fix sketch**: in `ld.go::composeLDCmdLinkExe` (line ~593-597),
change the emission order from:
```go
cmdArgs = append(cmdArgs, "-Wl,--no-whole-archive", vcsOPath)
cmdArgs = append(cmdArgs, ccPaths...)
```
to:
```go
cmdArgs = append(cmdArgs, "-Wl,--no-whole-archive")
cmdArgs = append(cmdArgs, objcopyPathsBare...)   // bare paths
cmdArgs = append(cmdArgs, vcsOPath)              // keep $(BUILD_ROOT)/
cmdArgs = append(cmdArgs, regularCCPaths...)     // regular main.cpp.o etc.
```

Required input separation: `ccPaths` (today the union) becomes split
into the SRCS_GLOBAL `.o` set (from emitResourceObjcopy) and the
regular `.cpp.o` set (member-CC outputs). The split exists at the
caller (gen.go:4232 `emitResourceObjcopy` returns `objcopyOutputs`
distinctly from `ccOutputs`); just stop appending them into the same
`ldCCOutputs` slice.

LOC: ~10 lines (gen.go: separate slice; ld.go: separate cmd-args slot).
This is the same edit as 2a (combined PR).

### 2c. objcopy emission order: PY_MAIN before namespace

OUR emission order (`resource.go:212-225`):
```
1. emitPyNamespaceObjcopy   → 5950979f...
2. emitPyMainObjcopy        → 4b1c18d0...
3. emitNoCheckImportsObjcopy
4. emitPySrcObjcopy         → c3a51827...
```

REF order (extracted from `probe_objcopy_kv.py`):
```
1. 4b1c18d0 = PY_MAIN
2. 5950979f = py/namespace
3. c3a51827 = resfs/src/resfs/file (PY_SRCS path-flush)
```

**Upstream rule**: in `pybuild.py` (line ~395), `py_main(unit, ...)`
is called INSIDE the source-iteration loop, immediately when the
MAIN-flagged source is seen. `py_main` invokes
`unit.onresource(['DONT_COMPRESS', '-', 'PY_MAIN={}'.format(arg)])`
(pybuild.py:759) which emits one objcopy node. The py_namespaces
loop (pybuild.py:588-594) runs AFTER the source-iteration loop and
calls `unit.onresource(ns_res)` with the namespace map — emitting
the namespace objcopy AFTER PY_MAIN. The `_split_macro_call(
unit.onresource_files, ...)` for the resfs/src/resfs/file entries
runs LAST (pybuild.py:597). So upstream order is PY_MAIN, namespace,
resfs.

**Fix sketch**: swap the order of `emitPyMainObjcopy` and
`emitPyNamespaceObjcopy` calls in `resource.go:emitResourceObjcopy`.
The kv-only sub-emitters are mutually independent (no inter-emitter
state); only the emission order changes.

LOC: ~5 lines (literal block-swap).

### 2e. Missing `-Wl,--strip-all` trailing flag

OUR slot 81 = `-Wl,--gc-sections`.
REF slot 81-82 = `-Wl,--strip-all`, `-Wl,--gc-sections`.

**Upstream rule**: `STRIP_FLAG` is empty by default
(`build/conf/linkers/ld.conf:80`). The
`when ($STRIP == "yes" && $NO_STRIP != "yes")` clause in
`build/ymake.core.conf:2668-2672` sets `STRIP_FLAG=$LD_STRIP_FLAG`
which is `-Wl,--strip-all` on Linux/Android (ld.conf:22). `STRIP=yes`
is enabled by the `STRIP()` macro (`build/ymake.core.conf:4311`).
`_BASE_PY3_PROGRAM` calls `STRIP()` in its module body
(`build/conf/python.conf:884`).

The slot position: `STRIP_FLAG` is in `_EXE_FLAGS` (ld.conf:170-180)
between `$C_SYSTEM_LIBRARIES` and `$DCE_FLAG`. `DCE_FLAG` is
`-Wl,--gc-sections`. So `-Wl,--strip-all` slots immediately before
`-Wl,--gc-sections` (matches REF slots 81-82).

**Fix sketch**: in `ld.go`, OUR has three trailing-flags tables:
- `ldStaticMuslTrailingFlags` (target, 12 args)
- `ldHostStaticTrailingFlags` (host without libc_compat, 12 args)
- `ldHostStaticRTTrailingFlags` (host with libc_compat, 14 args)

PY3_PROGRAM_BIN goes through the host code path with libc_compat
(`tools/py3cc/slow` PEERDIRs `library/python/runtime_py3/main` which
transitively peers `contrib/libs/libc_compat`) so it uses
`ldHostStaticRTTrailingFlags`. The fix: introduce a fourth variant
(or a `stripFlag bool` parameter) that inserts `-Wl,--strip-all`
between `-lm` and `-Wl,--gc-sections`. Trigger condition: the
module type is PY3_PROGRAM_BIN AND `NO_STRIP` is not set (none of
the M3 PY3_PROGRAM_BINs declare NO_STRIP — verified by checking
the gen.go module data for `tools/py3cc/slow`).

`STRIP()` is also invoked by other macros: `STRIP()` itself,
`SET(STRIP yes)`, `_BASE_PY_PROGRAM` (python.conf), etc. The minimal
correct trigger: emit `--strip-all` when `instance.Path` is a
PY3_PROGRAM_BIN module. Broader correctness (M4+) requires reading
the upstream `STRIP` variable through the env evaluator.

LOC: ~20 lines (gen.go: pass `wantsStrip bool` to EmitLD; ld.go:
inject before `--gc-sections`).

## 4. Implementation plan

Three PRs, sequenced. PR-1 and PR-2 are independent and can land in
parallel. PR-3 depends on PR-2's `wantsStrip` plumbing if implemented
that way, otherwise also independent.

### PR-A: Multi-LIBRARY peer-archive late-slot (closes Node 1, half of Node 2)

**Scope.** In `gen.go::genModule`'s peer-aggregation pass (lines
~3030-3210), partition the `peerArchivePaths` and `peerGlobalPaths`
output into `[single-LIBRARY peers, multimodule peers]` segments,
preserving relative order within each. Apply within each of the
three peer-category groups (lang-defaults, prog-defaults, user-peers)
independently. Discriminator: archive name contains `/libpy3` (this
is the stable marker upstream uses for PEERDIR_TAGS=PY3 selection).

**Files.** `gen.go` (~30 LOC in two helper closures + one partition
pass). `module.go` (optional: explicit `IsMultimodule bool` on
`moduleEmitResult` for clarity vs name-sniffing). `gen_test.go`
(unit test: a synthetic 4-peer module with 2 single + 2 multi
peers → assert single-first ordering).

**Risk.** Medium. The partition fires only on PY3-tagged closures
(M3 only; M1/M2 untouched). The partition's correctness depends on
the empirical "single-first within UniqPeers segment" rule — verified
on three independent cluster pairs in the data; no counter-examples
seen.

**Acceptance.**
- L3 → 100.00% for `devtools/ymake/bin/ymake` aarch64.
- L3 → 99.99% (~99.98% + 1 pair) for `tools/py3cc/slow/py3cc` x86_64
  if PR-B has not yet landed; → 100.00% if all PRs are merged.
- M1 sha `bf30a6...` preserved; M2 sha `c1c75021...` preserved.
- All other L3 pairs preserved.
- `gen ≤ 5 s` (no measurable perf delta — the partition is one
  extra pass over a typically-small slice).

### PR-B: PY3_PROGRAM_BIN LD cmd[2] slot fixes (closes Node 2 sub-defects a + b + c)

**Scope.**
1. Split `ccPaths` into `objcopyPaths` (bare) and `regularCCPaths`
   ($(BUILD_ROOT)/ kept) at the EmitLD call site (gen.go:4234).
2. In `ld.go::composeLDCmdLinkExe`, emit:
   `-Wl,--no-whole-archive`, `objcopyPaths...`, `vcsOPath`,
   `regularCCPaths...`, `-o`, `outputPath`.
3. In `resource.go::emitResourceObjcopy`, swap the order of
   `emitPyMainObjcopy` and `emitPyNamespaceObjcopy` calls.

**Files.** `gen.go` (~10 LOC), `ld.go` (~15 LOC, replace one
append line with three), `resource.go` (~5 LOC, block-swap).
`ld_test.go` (existing M2 test must still pass — M2 has no objcopy
outputs; trivially preserved). New test: `TestEmitLD_PY3ProgramBin`
