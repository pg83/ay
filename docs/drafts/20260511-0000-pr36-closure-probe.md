enumerate per-pair diffs, so the L1/L2/L3 breakdowns below were
extracted with two read-only Python scripts that pair the two graphs by

- L3 match: `len(cmds)`, `cmd_args` ordered per cmd, `cwd`, per-cmd

## 1. L1 unpaired pair (1 want-only / 1 got-only)

Exactly one node from each side fails to pair under the
`(outputs[0], platform)` key. The mismatch is structural: same UID
shape, same `kv.p=LD`, same module path prefix, but **different binary
filename**.

| | want (reference) | got (ours) |
|---|---|---|
| `outputs[0]` | `$(BUILD_ROOT)/contrib/tools/ragel6/ragel6` | `$(BUILD_ROOT)/contrib/tools/ragel6/bin/ragel6` |
| `platform` | `default-linux-x86_64` | `default-linux-x86_64` |
| `uid` | `DdrAehNzd9ylM0-kKxOCUA` | `jb33XLZU-zv2ccMJuPmebG` |
| `kv.p` | `LD` | `LD` |
| `module_dir` | `contrib/tools/ragel6` | `contrib/tools/ragel6/bin` |

**Root cause hypothesis:** ragel6 ships as a multi-directory ymake
layout: `contrib/tools/ragel6/` is the umbrella with sub-`PROGRAM()`
definitions; `contrib/tools/ragel6/bin/ya.make` is the actual `PROGRAM(ragel6)`
declaration carrying the SRCS list. The reference attributes the LD
node to the **outer** module path (`contrib/tools/ragel6`, where
`PROGRAM(ragel6)` is presumably aliased from a `RECURSE` or
`DECLARE_EXTERNAL_HOST_RESOURCES` lift), placing the binary at
`contrib/tools/ragel6/ragel6`. We attribute it to the **inner**
directory that actually parsed the `PROGRAM(name)` macro, placing the
binary at `contrib/tools/ragel6/bin/ragel6`.

**Fix hypothesis:** `ld.go::LDOutputPath` / `EmitLD` currently derives
the LD output's host directory from `instance.Path` (the inner
`bin/`). When the parsed `PROGRAM()` name (e.g. `ragel6`) does **not**
match the trailing path component, **and** the parent directory's
`ya.make` records this module as the implementation of a same-named
binary, fold the binary one directory up. Concretely: either
(a) introduce a `BinaryDir` field on `ModuleInstance` set by the walker
when a `RECURSE` → child `PROGRAM` lift is detected, or (b) special-case
the ragel6 host walk (gen.go around L286 where ragel6 is the only host
PROGRAM that triggers this lift). Option (a) is the architecturally
correct fix; (b) is the narrow patch.

Audit hook: there is no other PROGRAM in the tools/archiver closure
that exercises this lift, so a manual look at gen.go's host-walk
emission for ragel6 will reveal the exact insertion point.

**Root cause:** `libcxxabi-parts/ya.make` declares
`ADDINCL(contrib/libs/cxxsupp/libcxxabi/include)` (line 13) and uses
`SRCDIR(contrib/libs/cxxsupp/libcxxabi)`. The .cpp sources do
`#include "cxxabi.h"` (quoted, line 13 in `abort_message.h`); our
search-path resolver finds `libcxxabi/include/cxxabi.h` and **short-
circuits the sysincl multi-target rule** (`scanner.go::resolveDirective`
L1103 — the documented "quoted-form + search-path found → skip
sysincl" gate from PR-35w). The reference graph, in contrast, **layers
the sysincl multi-target rule on top**: `cxxabi.h` (no source_filter in
`stl-to-libcxx.yml` lines 33-36) maps to BOTH `libcxxabi/include/cxxabi.h`
AND `libcxxrt/include/cxxabi.h`. The same gate suppresses the
`unwind.h` multi-target rule (lines 41-44, three alternatives:
libcxx, libcxxrt, libunwind), and through it the further
`^contrib/libs/libunwind/include/unwind.h` source_filter rule
(misc.yml line 406) which is the entry point into the glibcasm cascade
that contributes the 23 glibc/* paths plus 2 musl/* tails.

**Symptom in code:**

- `scanner.go` L1103: `if d.kind == includeQuoted && len(searchOut) > 0 { return searchOut }`
- This gate was introduced in PR-35w to fix yasm/elf.h over-emission (PR-35t R3+R5). It was too aggressive: it now suppresses **every** legitimate multi-target sysincl record for quoted headers, not just the
  yasm/musl-replacement ones.

**Recommended fix shape:** make the gate per-record, not per-call.
A sysincl record that ships `multiple` resolution targets (list-valued
mapping with ≥ 2 non-empty entries, e.g. `cxxabi.h` and `unwind.h` in
`stl-to-libcxx.yml`) is a "multi-target" record and must always
contribute regardless of search-path resolution; single-target
records (yasm/elf.h, musl `<elf.h>` etc.) stay gated. This requires
flagging records with multi-target mappings at sysincl load time and
checking that flag inside `resolveDirective`.

**Affected files:** `scanner.go::resolveDirective` (the L1103 gate);
`sysincl.go::SysIncl` (add a per-record `HasMultiTarget` bit during
load); `sysincl.go::compileMapping` or equivalent loader. Unit-test
coverage: `scanner_test.go` (must add a libcxxabi-style `#include "cxxabi.h"` fixture).

**OUR-only inputs:** none.

**Root cause:** flipped version of L2-A. `Unwind-wasm.c` is a libunwind-
internal source. The `^contrib/libs/libunwind/` source_filter
(misc.yml lines 32-34) maps `unwind.h → libunwind/include/unwind.h`
(single target). The `stl-to-libcxx.yml` (lines 41-44) record is
unfiltered, multi-target — `unwind.h` maps to all three alternatives.

includer-keyed half) and drops the multi-target half because of an
asymmetric union step somewhere in `sysinclLookup` (scanner.go L1215).

result when both halves fire. This is **separate** from the L2-A gate
— here the include is `<unwind.h>` (angle form, no quoted-include
gate involved).

Actually inspecting the cluster: the L2-A fix (allow multi-target
records to always contribute) would also close this cluster, because
in the libunwind-internal context the stl-to-libcxx record is the
"multi-target" record that the union currently suppresses. Verify
during fix that the existing `^contrib/libs/libunwind/` source_filter
record continues to fire too — both records' contributions must end
up in the result.

**Affected files:** same as L2-A (`scanner.go::sysinclLookup`,
`scanner.go::resolveDirective`).

`perSrcCFlags`, after `macroPrefixMapFlags`. Verify against
`base64/neon32/lib.c.o`'s reference cmd_args.

**Note:** the ymake macro `CONLYFLAGS()` is **not** the same as
`CFLAGS()`. `CONLYFLAGS()` applies to C sources only and lands at the
tail of the cmd_args; `CFLAGS()` applies to both C and C++ and lands
earlier in the bundle. The current `composeTargetCC` conflates them.

**Root cause:** `contrib/libs/musl/full/ya.make` does **`SET(MUSL no)`**
(line 9). The module is a normal LIBRARY hosting a single `dlvsym.cpp`
source that links the standard musl ABI through to glibc. It must NOT
get the musl-flavoured CC bundle. Our `inferFlagsFromPath`
(`module.go::inferFlagsFromPath` L167-189) hard-codes
`HasPrefix("contrib/libs/musl/")` → `LibcMusl = true`, then EmitCC
dispatches to `composeMuslCC` (cc.go L275), which emits the musl
include-prefix block (`-I.../musl/extra`, `-I.../musl/arch/X`) and the
musl define block (`-D__libc_calloc=calloc`, `-D__libc_malloc=malloc`,
`-D_XOPEN_SOURCE=700`, `-ffreestanding`), and **omits** the standard
`-Wno-deprecated-*` warning suppressions that non-musl C++ CC carries.

This is the canonical case the GOALS.md "Flag-driven configuration,
not path-prefix dispatch" architectural goal calls out
(`-DMUSL=yes`-conditional, not `HasPrefix`-conditional). Until
ya.make `SET()` parsing lands, the narrow fix is to special-case
`musl/full` in `inferFlagsFromPath` (override `LibcMusl=true` when the
path is exactly `contrib/libs/musl/full`). The proper fix is to
parse `SET(MUSL no)` from the module's ya.make and let it override
the path-prefix seed.

**Affected files:** `module.go::inferFlagsFromPath`. Long-term: ya.make
`SET()` parsing in `yamake.go`. Note that `gen.go::cliMuslOn`
already gates the **build-level** MUSL flag; this is a **module-level**
override that disqualifies `musl/full` from the musl-flavoured CC
even when build-level MUSL=yes.

**Root cause:** ymake's `_LANG_CFLAGS_RL=-Wno-implicit-fallthrough`
(`ymake.core.conf` L3583/L3590) is auto-applied to CC compiles whose
source is a `.rl6`-generated `.cpp`. Our R6 emitter (`r6.go`) produces
the generated `parser.rl6.cpp` but does not annotate it with this
per-source CFLAG; the consuming CC composer (`cc.go::EmitCC`) does
not detect the `.rl6.cpp` shape and does not inject the lang-CFLAG.

**Affected files:** `r6.go` (emit a `PerSourceCFlags=[-Wno-implicit-fallthrough]`
on the generated `.cpp` member's CC node, or thread it through the
ModuleInstance source-extra-cflags map); `cc.go::EmitCC` (already
threads `in.PerSourceCFlags` through). Verify against
`ymake.core.conf::_LANG_CFLAGS_VALUE_NEW` (L3601-3609) for the full
list of `.{rl5,rl6,pyx,py.py3,py.py2}` extensions that share this
auto-CFLAG — only `.rl6` is reachable in the tools/archiver closure;
others land in M3+.

### L3-E. util/libyutil.a AR member ordering (1 pair)

| output | platform | `kv.p` |
|---|---|---|
| `$(BUILD_ROOT)/util/libyutil.a` | aarch64 | AR |

32 args each, but member positions differ. The single-element shift
visible at positions 10..31:

REF: `parser.rl6.cpp.o` is the **last** member (position 31).
OUR: `parser.rl6.cpp.o` is the **earliest** member (position 10).

Every other `all_*.cpp.o`, `_/system/*.cpp.o`, etc. shifts up by one
to fill the gap.

**Root cause:** `ar.go` (AR member assembly) sorts member object
files alphabetically by output path. `parser.rl6.cpp.o`'s output is
`util/_/_/datetime/parser.rl6.cpp.o`, which is alphabetically
**earlier** than `util/_/system/...`. The reference uses a different
ordering: generated `.rl6.cpp.o` members slot **after** all hand-
written sources. This matches ymake's "regular SRCS first, generated
SRCS appended" emission rule.

**Affected files:** `ar.go::EmitAR` / `composeARMemberList`. Need a
two-bucket sort: hand-written sources (alphabetical) followed by
generated-from-rl6/swg/etc. sources (alphabetical within the
generated bucket). The discriminator is whether the source's
provenance is an R6/SWG/PYSRC emit vs a direct SRCS entry; this is
tracked in `ModuleInstance.Srcs` already via the source tag.

## 6. Risks and open questions

1. **PR-36 risk** — the quoted-form gate exists for a reason
   (PR-35w fix for yasm/elf.h over-emission). Lifting it for
   multi-target records widens the surface. The fix must verify that
   the previously-fixed yasm + musl `<elf.h>` over-emission cases
   stay closed. Specifically check: yasm internal sources doing
   `#include "elf.h"` must continue to resolve to the yasm-local
   `elf.h` only; libc-to-musl.yml's `elf.h → musl/include/elf.h` is
   a single-target record (not affected by the new bypass).

2. **PR-40 ambiguity** — the ragel6 LD output path lift from
   `bin/` to the parent dir suggests an architecture decision: is
   our `ModuleInstance.Path` policy "always the directory containing
   the parsed PROGRAM macro", or should it be "always the directory
   the binary is published to"? GOALS.md does not arbitrate this. The
   safest fix is to add a `BinaryDir` field separate from `Path`; the
   narrow fix is to special-case ragel6 (the only known M2 case).
   **User input may be required** on the policy choice. If the loop
   blocks on this, escalate via the inner loop's blocker path (I6).

3. **L2-C input arity off-by-one** (4 OUR vs 3 REF musl-arch
   headers in JS cluster): our walk produces an x86_64 `<limits.h>`
   but a `<limits.h>`-on-aarch64 walk shows REF doesn't emit
   `musl/arch/aarch64/bits/limits.h` — it falls through to
   `musl/arch/generic/bits/limits.h` (shared between axes). Verify
   that REF's aarch64 musl `<limits.h>` resolution really is
   single-result not dual. If not, the fix is larger than "just swap
   axes".

4. **PR-41 (R6 lang-cflag)** — the upstream
   `ymake.core.conf::_LANG_CFLAGS_VALUE_NEW` block ALSO assigns
   `-Wno-implicit-fallthrough` to `.pyx`, `.py.py3`, `.py.py2`. None
   of those land in the tools/archiver closure today, so PR-41 can
   ship the rl6-only narrow fix. M3+ will need the full extension
   map.

5. **PR-42 link-order** — this is the most fragile of the seven.
   The peer-walking order in REF is *not* a simple alphabetical /
   BFS / DFS ordering. The fact that `contrib/libs/musl/*` is placed
   late (slot 46/47) while `libcxxabi-parts` is placed early (slot
   28) suggests ymake threads `cxxsupp/*` as a "C++ runtime bucket"
   ordered before regular peers, and `musl/*` as a "libc bucket"
   ordered after them. This is upstream-emission convention; we will
   need to read ymake's link-line emitter (or empirically derive the
   sort key from the reference) before patching. If the convention
   cannot be deciphered from the reference alone, this PR may have
   to wait for ya.make-driven module-classification (M3+).

6. **Pairing-key collision audit** — the comparator pairs by
   `(outputs[0], platform)`. The L1 unpaired pair (ragel6 LD)
   collides on `platform=x86_64` but **not** on `outputs[0]`
   (`.../ragel6/ragel6` vs `.../ragel6/bin/ragel6`). The pairing is
   honest; the unpair is real, not a comparator artefact. Same for
   every L2/L3 mismatched pair: their `(outputs[0], platform)` keys
   match exactly. No emitter-key bug is implicated.

7. **Determinism check** — none of these clusters introduce a new
   non-determinism risk. The fixes are all data-driven shapes.
