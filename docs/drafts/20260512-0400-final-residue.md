## 1. L3 plateau — what cmd_args still differ

### 1a. Order-only (260 nodes — the dominant L3 class)

These nodes have identical *sets* of cmd_args but differ in order. Bucket by
the first-differing argument:

| Kind            | Nodes | First-diff position bucket |
|-----------------|------:|----------------------------|
| `-I` reorder    |   150 | pos 20-29 (142) and 60-69 (a few) |
| `-D` reorder    |   102 | pos 60-69 (85), 70-79 (17) |
| obj/lib reorder |     7 | pos 10-19 |
| other           |     1 | one BUILD/SOURCE confusion in a symbols .o |

`-I` reorder, canonical sample
`$(BUILD_ROOT)/devtools/ymake/_/plugins/ymake_module_adapter.cpp.o` (aarch64):

```
OUR slot 21: -I.../lzma/liblzma/api      REF slot 21: -I.../python/Include (the SECOND copy)
OUR slot 22: -I.../openssl/include       REF slot 22: -I.../abseil-cpp
OUR slot 23: -I.../libffi/include        REF slot 23: -I$(BUILD_ROOT)/library/python/runtime_py3
OUR slot 24: -I.../libffi/configs/...    REF slot 24: -I.../lzma/liblzma/api
OUR slot 25: -I.../python/Include (2nd)  REF slot 25: -I.../openssl/include
OUR slot 26: -I$(BUILD_ROOT)/library/python/runtime_py3
OUR slot 27: -I.../abseil-cpp
```

Same set, identical *libcxx/musl/zlib/double-conversion* prefix (slots 7-20),
identical asio/yaml-cpp/fmt/re2/protobuf/abseil-tstring/brotli/snappy tail.
The discrepancy is purely the order in which the PEERDIR walker visits the
direct peers `lzma` vs `python` vs `openssl` vs `libffi` vs `abseil-cpp` vs
`runtime_py3`.

`-D` reorder, canonical sample
`$(BUILD_ROOT)/devtools/ymake/resolver/path_resolver.cpp.o` (aarch64):

```
OUR slot 77: -DFMT_EXPORT, -DASIO_STANDALONE, -DASIO_SEPARATE_COMPILATION,
             -DOPENSSL_RENAME_SYMBOLS=1, -DPCRE_STATIC, ...
REF slot 77: -DFMT_EXPORT, -DOPENSSL_RENAME_SYMBOLS=1, -DASIO_STANDALONE,
             -DASIO_SEPARATE_COMPILATION, -DPCRE_STATIC, ...
```

Same root cause: ASIO contributes `ASIO_STANDALONE/ASIO_SEPARATE_COMPILATION`
via CFLAGS-as-peer-export; OpenSSL contributes `OPENSSL_RENAME_SYMBOLS=1`.
OUR walks asio's CFLAG block before openssl's; REF the opposite. Both blocks
are emitted in PEERDIR-visit order, and the visit order in REF matches the
declaration order in the upstream module's `PEERDIR(...)` list while OUR's
visit order is alphabetised or otherwise normalised somewhere in the
composer/walker.

The 7 "obj/lib reorder" nodes are global.a archives where OUR emits
constituent `.o` inputs in a different order than REF — same input set, same
files, different walk order. Same architectural source as above (the linker
input list is built from the same peer walk).

**Proposed PR — "PR-M3-peer-walk-order": preserve direct-peer declaration
order through the composer.** Find the place where OUR walker sorts or
re-orders direct PEERDIR results (likely in `composers.go`/walker glue —
not in scanner). Change it to iterate in the same order as the AST records
PEERDIRs (i.e. source order). Verify on the three canonical samples above.
Lift: **fixes all 260 order-only nodes → +260 L3 pairs**, which on its own
moves L3 from 8417/8744 (96.26 %) to 8677/8744 (99.23 %). It will also tidy
up the obj/lib order on the 7 global.a nodes (free).

### 1b. ref_superset (45 nodes — REF has args we lack)

| Count | Missing arg | Sample target |
|------:|-------------|---------------|
| 22 | `-DARCADIA_PCRE_ENABLE_JIT` | `$(BUILD_ROOT)/contrib/libs/pcre/pcre_get.c.o` |
| 11 | `-I$(SOURCE_ROOT)/contrib/restricted/abseil-cpp` | `$(BUILD_ROOT)/library/python/symbols/module/library.python.symbols.module.syms.reg3.cpp.py3.o` |
|  9 | `-I$(SOURCE_ROOT)/contrib/libs/protobuf/src` | `$(BUILD_ROOT)/library/cpp/eventlog/proto/events_extension.pb.cc.o` |
|  9 | `-I$(SOURCE_ROOT)/contrib/restricted/abseil-cpp-tstring` | same as above |
|  5 | `-I$(BUILD_ROOT)/library/cpp/build_info` | `$(BUILD_ROOT)/library/cpp/build_info/build_info_static.cpp.o` |
|  4 | `-Wno-unused-variable` | `$(BUILD_ROOT)/devtools/ymake/lang/CmdParser.g4.cpp.o` |
|  2 | `--plugin` / `$(SOURCE_ROOT)/contrib/libs/openssl/ar.pyplugin` | openssl `.a` |
|  1 | `-DPyInit_syms=…` and `-Dinit_module_syms=…` | symbols/module .o |
|  1 | `-I$(SOURCE_ROOT)/contrib/libs/protobuf` | event2cpp proto_events.cpp.pic.o |

These are five distinct narrow bugs:

1. **pcre JIT flag (22 nodes).** `ARCADIA_PCRE_ENABLE_JIT` is a CFLAG declared
   by `contrib/libs/pcre/ya.make` for its own translation units (i.e. a
   private CFLAG, not exported). OUR scanner is dropping CFLAGs that are
   declared as `CFLAGS(-DX)` (un-globbed). Translate the pcre block.
2. **abseil-cpp -I (11 nodes).** Same pattern as -I reorder, but here the
   path is missing entirely — likely a transitive PEERDIR fan-out where
   abseil-cpp comes in through `contrib/libs/protobuf` only when the
   reg3-bound translation unit is in the python-bind path. Probably the
   `EmitsIncludes` set for `protobuf/runtime` (touched in PR-M3-F-7-D) is
   missing abseil-cpp for the python-bind subset.
3. **protobuf/abseil-cpp-tstring/build_info -I (9+9+5 nodes).** Same family
   as (2), narrower targets — `eventlog/proto/*.pb.cc.o`, `build_info_static`.
   Almost certainly all three are the same fix as (2): when a `.pb.cc` is
   compiled, our scanner forgets to inject the protobuf-runtime -I closure
   into the translation unit's CFLAGS even though we already emit the
   protobuf-runtime inputs (per PR-M3-F-7-D).
4. **`-Wno-unused-variable` (4 nodes).** `CmdParser.g4.cpp.o` etc — ANTLR4
   generated C++. Comes from `g4-generated.macro` or similar; one missing
   flag in the codegen recipe.
5. **`--plugin` + `ar.pyplugin` for openssl libcontrib-libs-openssl.a.** The
   openssl module overrides the archiver invocation with a Python plugin
   (`ar.pyplugin`). OUR archiver-invocation builder does not honour
   `LIB_PLUGIN` (or whatever the ya.make symbol is). Two args, one fix.

The `-DPyInit_syms` and protobuf `-I` (count 1 each) are the tail of the same
families — fold them into (2) and (3).

**Proposed PR — "PR-M3-pcre-jit-and-archiver-plugin": four micro-fixes.**
Translate pcre's private CFLAGS, route `abseil-cpp`/`protobuf-src`/`abseil-tstring`
into protobuf-runtime EmitsIncludes for the python-bind subset, add the
ANTLR4 `-Wno-unused-variable` to the g4 codegen recipe, and honour openssl's
`ar.pyplugin`. Lift: **45 L3 pairs**, plus typically the same nodes also
flip into pair on L2 (the python-bind .py3.o and the .pb.cc.o nodes
overlap with the L2 abseil-tstring cluster).

### 2a. The "deep header closure on .pb.cc and other generated TUs" family

14 protobuf-touching `.o`/`.a` nodes each miss the exact same 145
`abseil-cpp-tstring` headers, plus a similar count of protobuf/cxxsupp
headers. The 4 `library/cpp/build_info` .o nodes miss ~700 libcxx headers
each. `library/cpp/retry/retry.cpp.o` misses 145 tstring + 96 libcxx + 41
protobuf headers. **These are not nodes "missing a peer" — these are nodes
where our scanner walks the *direct* preproc closure but stops short of
following the preproc closure of *included* protobuf-runtime headers.**

Sample evidence: `library/cpp/eventlog/proto/events_extension.pb.cc.pic.o`
is missing 145 distinct `y_absl/...` headers — all of them transitively
included by `google/protobuf/port_def.inc` → `y_absl/strings/string_view.h`
→ `y_absl/strings/internal/...`. We resolve the *first* hop (port_def.inc
is in our inputs) but not the second.

**Proposed PR — "PR-M3-protobuf-runtime-deep-closure":** extend the
EmitsIncludes registry hookup (touched in PR-M3-F-7-D) to recursively
chase the include closure of every codegen-emitted header, not just the
direct hop. Apply to protobuf, abseil-cpp-tstring, and to the protobuf
codepaths in `library/cpp/build_info`. Lift, conservatively:

- Closes all 14 abseil-cpp-tstring nodes on L2 (each currently misses
  ~145+96+41 ≈ ~280 edges, all from this family). **14 L2 pairs.**
- Closes the 4 `library/cpp/build_info` nodes (~700 each). **4 L2 pairs.**
- Closes the 8 `library/cpp/eventlog/proto` and 5 `library/cpp/protobuf`
  nodes if not already in the 14 above. **≤8 more L2 pairs.**

Total: ~26 L2 pairs from one PR. Plus the cmd_args `-I protobuf/abseil-tstring`
ref_superset entries (§1b items 2–3) close automatically because the same
walker that emits the inputs also emits the -I.

### 2b. The "python3 stdlib global.a" family

The two huge nodes `contrib/tools/python3/Lib/libpy3tools-python3-Lib.global.a`
and `contrib/tools/python3/lib2/py/libpy3python3-lib2-py.global.a` each miss
535 inputs (the entire `.pyc` stdlib). Then ~60 sibling `objcopy_*.o` nodes
each miss ~15 inputs — these are the `objcopy --redefine-syms`-wrapped
.pyc.o files. The total is 2227 edges concentrated in ~65 nodes.

This is the `RESOURCE` / `PY_REGISTER` / py3-stdlib pipeline: when we
build `libpy3tools-python3-Lib.global.a` we glob `Lib/*.py`, compile each
to .pyc, archive each via `archiver`, then objcopy + .o. We miss the
upstream `.py` and `.pyc` inputs on the global.a. Likely the same fix that
handles the runtime_py3 pyc.inc unpaired-want nodes (§3).

**Proposed PR — "PR-M3-pystdlib-resource-closure":** treat the
`contrib/tools/python3/Lib` archive as the `tools/archiver`-driven pipeline
it is (the same pipeline that produces `runtime_py3/__res.pyc.inc`). One
generator entry produces the .pyc, objcopy_*.o and the global.a. Lift:
~67 L2 pairs (the global.a's plus 65 objcopy_*.o), and also resolves the
5 unpaired-want pyc.inc nodes from §3 (the runtime_py3 cousins of the
same pipeline).

## 3. The 6 unpaired-want nodes — recommendation

All six belong to one family: the **`tools/archiver` "compile, archive,
inc" pipeline** for `library/python/runtime_py3`.

| # | Output | Platform |
|---|--------|----------|
| 1 | `library/cpp/malloc/jemalloc/libcpp-malloc-jemalloc.a` | linux-x86_64 |
| 2 | `library/cpp/malloc/jemalloc/malloc-info.cpp.pic.o`    | linux-x86_64 |
| 3 | `library/python/runtime_py3/__res.pyc.inc`             | linux-aarch64 |
| 4 | `library/python/runtime_py3/__res.pyc.inc`             | linux-x86_64 |
| 5 | `library/python/runtime_py3/sitecustomize.pyc.inc`     | linux-aarch64 |
| 6 | `library/python/runtime_py3/sitecustomize.pyc.inc`     | linux-x86_64 |

Nodes 3–6 are pure `tools/archiver` outputs (`archiver -q -x -p .pyc: -o .pyc.inc`).
Nodes 1–2 are the `libcpp-malloc-jemalloc.a` and its single member
`malloc-info.cpp.pic.o` — REF emits a one-source `.a` for jemalloc that
OUR considers a leaf .a with no members (or omits entirely because we
gate jemalloc behind a flag).

The five python-runtime nodes are the same `tools/archiver` pipeline that
appears in the mixed-class cmd_args (§1c): OUR emits the .pyc.inc using
`mod=${MODDIR}/...` and `__res.pyc` as bare names; REF emits them as full
`library/python/runtime_py3/...` paths. The path difference plus the
extra `__res.pyc.inc` archiver output means the *outputs* set differs and
the keying loses the match. There's no missing logic — just a path/output
mismatch.

**Recommendation: roll this in with §2b ("PR-M3-pystdlib-resource-closure")**.
Both clusters are the same archiver pipeline, just at different scopes
(stdlib vs runtime_py3). A single PR that:

1. canonicalises the archiver-output keying for `*.pyc.inc` to use the
   absolute build path (no `${MODDIR}` expansion in the keying);
2. emits the explicit `__res.pyc.inc` and `sitecustomize.pyc.inc` nodes
   for runtime_py3;
3. emits the .pyc and objcopy_*.o pipeline for python3/Lib;

closes all six unpaired-want and the §2b cluster. The jemalloc pair
(nodes 1–2) is a separate one-line CONFIG gate — fold in opportunistically
or split out.

Lift: **6 unpaired-want closed (pure pair gain)** plus the 67-node L2
cluster from §2b plus the 5-node mixed L3 cluster from §1c.
