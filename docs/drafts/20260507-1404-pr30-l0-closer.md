# PR-30 — L0 closer (closure widening + AR dep cleanup + Generator wiring)

Plan author: PR-30 planning subagent, 2026-05-07. Empirical probes archived
at `./debug/20260507-135500-pr30-probe[2-4]-output.txt`. The companion
brief that produced this plan is on the orchestrator side; this document
is the executor-facing plan and the L0 acceptance lever map.

## Problem

PR-29 closed M2's L3 acceptance gate (80.05% ≥ 50%). The single open M2
gate is **L0 ≥ 95%**. Current L0 = 88.34% (3295 / 3730 fingerprints
matched). Empirical probes attribute the gap to four mostly-orthogonal
mechanisms: (a) ~423 reference nodes are entirely missing from our emit
because their reaching paths are not modelled (musl/full closure for
PROGRAM, ALLOCATOR_IMPL routing, abseil-cpp pulled transitively via
tcmalloc, host yasm closure pulled via host asmlib); (b) ~24 paired CC
nodes are byte-identical at L1/L2/L3 but their L0 fingerprint diverges
because we emit them with no `Generator` dep (PR-29-D07 was deferred);
(c) ~24 paired AR nodes diverge because we wrongly stuff peer-archive
ARs into AR.deps where the reference graph has zero AR-on-AR edges; (d)
~14 spurious nodes (libcxxabi-parts SRCDIR + ragel6 platform-mismatch)
inflate our count and depress L0's denominator. Mechanism (c) is the
cheapest single lever and was not in the original brief — it dominates
the "AR diverged" subset of the 50 paired-but-diverged pairs and was
discovered during the planning probe round.

### F3. ALLOCATOR_IMPL closure

PROGRAM `_LINK_UNIT` declares `DEFAULT(ALLOCATOR $DEFAULT_ALLOCATOR)`.
For `MUSL=yes` (default) AND `OS_LINUX=yes` (default) →
`DEFAULT_ALLOCATOR=TCMALLOC_TC` (overrides earlier SYSTEM/J defaults).
The `select($ALLOCATOR)` block at `build/ymake.core.conf:961-1035` for
`TCMALLOC_TC` → `PEERDIR += library/cpp/malloc/tcmalloc` and
`PEERDIR += contrib/libs/tcmalloc/no_percpu_cache`.

`tcmalloc/no_percpu_cache/ya.make` uses `INCLUDE(../common.inc)` (parser
already supports), `SRCDIR(contrib/libs/tcmalloc)`, `GLOBAL_SRCS(...)`,
`SRCS(aligned_alloc.c)`, plus `PEERDIR(contrib/restricted/abseil-cpp,
contrib/libs/tcmalloc/malloc_extension)` (from common.inc) — **all
parser-supported**. abseil ya.make uses `ELSEIF` (parser-supported) and
`OS_DARWIN OR OS_IOS` (Environment-supported).

**Closure once ALLOCATOR_IMPL routes to tcmalloc**: 57 (no_percpu_cache)
+ 2 (malloc_extension) + 2 (library/cpp/malloc/tcmalloc) + 157 (abseil)
= 218 net new target-aarch64 nodes.

L0 is fingerprint-topology only (`kv.p` + sorted child fingerprints).

### F6. Generator → DepRefs revisit

Probe (e): every JS-derived CC and R6-derived CC in the reference has
exactly 1 dep — the JS or R6 UID. Our walker emits these CC nodes with
0 deps (PR-29-D07 noted "wiring deferred because adding it cost 2 L0
matches"). Empirical recount today: 23 JS + 1 R6 = **24 paired CC nodes
fingerprint-diverge** because of this. The PR-29 cost was 2 L0 matches
(libcxx-side fingerprint collisions that go away when wider closure
adds new fingerprints to the multiset). With PR-30 widening the
closure by ~400 matched nodes, the 2-cost is recovered many times over.
**Wire it in PR-30; the L0 swing is +24 matches − ~2 collisions =
+22 net.**

### F7. AR over-deps regression (NEW; surfaced by planning probe)

Probe `debug/20260507-135500-pr30-ar-divergence.py` scanned every AR
node in the reference graph: **0 reference AR nodes have any AR dep**.
Every AR's `deps` is exclusively its own CC/AS objects. Our `genModule`
passes `peerArchiveRefs` to `EmitAR`, which appends them to `DepRefs`
(ar.go:155-157). All 24 paired-but-fp-diverged AR pairs in our output
diverge for exactly this reason: spurious peer-AR child fingerprints
shift the parent AR's fingerprint. Removing the peer-AR threading from
EmitAR is a one-line surgical change that closes 24 L0 divergences AND
removes ~50 LOC of dead infrastructure from the AR emission path.

Caveat: `EmitAR`'s `peerArchiveRefs` parameter signature must stay (or
be removed entirely) — call-sites must be audited. PROGRAM LD's
`peerArchiveRefs` are correctly used; only the LIBRARY AR call site
should drop them.

### F8. SRCDIR module_dir convention (NEW; surfaced by planning probe)

PR-28-D02 rebases module_dir to SRCDIR's target. Empirical: this is
correct for ragel6/bin (SRCDIR points at parent of module path; ref
puts CC at `contrib/tools/ragel6`) but wrong for libcxxabi-parts,
musl_extra, tcmalloc/no_percpu_cache where reference keeps module_dir
at the LIBRARY's own path and places source outputs under
`<libpath>/__/<srcdir-relative>/<src>` (note the **double-underscore**
`__/`, not `_/`).

| Module | SRCDIR | ref module_dir | ref output infix |
|---|---|---|---|
| `contrib/tools/ragel6/bin` | `contrib/tools/ragel6` | `contrib/tools/ragel6` | none (same dir) |
| `contrib/libs/cxxsupp/libcxxabi-parts` | `contrib/libs/cxxsupp/libcxxabi` | `contrib/libs/cxxsupp/libcxxabi-parts` | `__/libcxxabi/src/...` |
| `contrib/libs/musl_extra` | `contrib/libs/musl` | `contrib/libs/musl_extra` | `__/all.c` ish |
| `contrib/libs/tcmalloc/no_percpu_cache` | `contrib/libs/tcmalloc` | `contrib/libs/tcmalloc/no_percpu_cache` | `__/tcmalloc/...` |

Pattern: rebase **only** when SRCDIR points to an ancestor of the
module path (the "include-from-parent" case typified by `<x>/bin`
PROGRAMs). Otherwise: keep module_dir at instance.Path; use `__/` infix
in output paths and reference the source under
`$(SOURCE_ROOT)/<srcdir>/<src>`. Closes the 6 spurious libcxxabi-parts
nodes AND unblocks the 6 missing libcxxabi-parts CC pairs at correct
paths (net L0 swing: +6 paired matches, −6 spurious = +6 effective
matched + 6 fewer denominator inflators).

## Design decisions

### Q1. musl/full closure mechanism — option (a) extended `defaultPeerdirsFor`

Pick **option (a)**: extend `defaultPeerdirsFor` to add
`contrib/libs/musl/full` for **PROGRAM modules only** when `MUSL=yes
&& !MUSL_LITE`. Rationale:

* The upstream rule lives in `_BASE_PROGRAM` (ymake.core.conf:1238-1245)
  inside the PROGRAM module's own implicit peerdirs — semantically
  identical to our existing per-language defaults. Hardcoding the
  closure (option b) would lose generality for future PROGRAMs that
  flip MUSL_LITE.
* musl/full's own ya.make has a clean `IF (ARCH_X86_64) PEERDIR(asmlib,
  asmglibc, linux-headers)` block — our parser handles this; the host
  walk naturally visits asmlib via the standard PEERDIR pipeline.
* The `isProgram` bit must be threaded into `defaultPeerdirsFor`. The
  current signature `defaultPeerdirsFor(ctx *genCtx, instance
  ModuleInstance)` does not carry the module type because peer
  resolution happens **before** the peer's own ya.make is parsed. Two
  options:
  - **(α)** Make `genModule` call `defaultPeerdirsFor` with a
    `programOnly` second slice and merge after parsing the module to
    determine `d.moduleStmt.Name`. Cleanest.
  - **(β)** Parse the ya.make inside `defaultPeerdirsFor` to discover
    the module type. Eager double-parse; slower; rejected.
* Pick **(α)**. Add a sibling helper `defaultProgramPeerdirsFor(ctx,
  instance)` that returns musl/full + asmlib (x86_64) + cpuid_check.
  Call it in `genModule` after parsing, before the peer walk.

The existing `runtimeAncestorPaths` exclusion list still applies —
musl/full IS a runtime ancestor; its peers are NOT
default-peerdir-extended (it manually declares `PEERDIR(musl)` which is
respected). Add `contrib/libs/musl/full` to `runtimeAncestorPaths`.

### Q2. ALLOCATOR_IMPL routing — extend `defaultProgramPeerdirsFor`

PROGRAM gets `DEFAULT(ALLOCATOR $DEFAULT_ALLOCATOR)` per
`_LINK_UNIT`. Default for our environment (MUSL=yes, OS_LINUX=yes) is
`TCMALLOC_TC` (line 410 wins over earlier `SYSTEM`). `select($ALLOCATOR)`
at line 996 maps `TCMALLOC_TC → library/cpp/malloc/tcmalloc +
contrib/libs/tcmalloc/no_percpu_cache`.

Implementation: `defaultProgramPeerdirsFor` returns the TCMALLOC_TC
peers when (a) the parsed module is PROGRAM AND (b) it has NO explicit
`ALLOCATOR(...)` macro AND (c) MUSL=yes (we don't model GCC/sanitizer
overrides). PROGRAMs that explicitly type `ALLOCATOR(MIM)` etc. still
go through the existing `applyAllocatorStmt` table — keep that path.

PROGRAM-mode default-allocator detection lives **after** `collectModule`
because the macro evaluator sets `d.moduleStmt` and we need to know
whether `ALLOCATOR(...)` was declared. Implementation point: after
`collectModule`, compute `wantsDefaultAllocator := d.moduleStmt.Name
== "PROGRAM" && !d.hadAllocator` and merge those peers into the
deduped `allPeers` set.

`d.hadAllocator` is a new boolean on `moduleData` set by
`applyAllocatorStmt`.

Add a regression test pinning that a synthetic JS+CC pair produces a

### Q6. AR peer-archive deps — DROP IN PR-30 (NEW; F7)

Drop `peerArchiveRefs` from `EmitAR`'s caller (`genModule`'s LIBRARY
branch), at gen.go:1004. Three options for the function signature:

* **(α)** Keep `peerArchiveRefs` parameter on `EmitAR` but pass `nil`
  from genModule. Cheapest; preserves test API.
* **(β)** Remove the parameter entirely; update `EmitAR` callers; emit
  warning in `emitARNode` if `peerArchiveRefs != nil`.
* **(γ)** Remove only from `EmitAR`'s public signature; keep
  `emitARNode` flexibility for future use.

Pick **(α)**. The signature is consumed by tests; refactoring to (β)
risks chasing test churn that distracts from the L0 lift. Fix is one
line in gen.go + a regression test "ref AR nodes have no AR deps;
verify ours match".

The PROGRAM LD's `peerArchiveRefs` slot stays — LD correctly links
peer archives.

### Q7. SRCDIR module_dir semantics — REFINE IN PR-30 (NEW; F8)

Per F8: rebase module_dir to SRCDIR target ONLY when SRCDIR is an
ancestor of `instance.Path` (the include-from-parent pattern,
ragel6/bin only in current closure). Otherwise: keep
`module_dir = instance.Path`; emit output path as
`$(BUILD_ROOT)/<instance.Path>/__/<srcRel>` (or similar — match
reference exactly via the `__/` infix); read source from
`$(SOURCE_ROOT)/<srcdir>/<srcRel>`.

Implementation lives in `emitOneSource` (gen.go:1119+). Today it
unconditionally rebases:

```go
srcInstance := instance
if srcDir != "" {
    srcInstance.Path = srcDir
}
```

Replace with:

```go
srcInstance := instance
useRebasedInstance := srcDir != "" && isAncestorPath(srcDir, instance.Path)
if useRebasedInstance {
    srcInstance.Path = srcDir
}
// when !useRebasedInstance, EmitCC/EmitAS/EmitR6 must compose
// outputPath as <instance.Path>/__/<srcRel> AND read input from
// <srcdir>/<srcRel>. Pass srcDir through ModuleCCInputs.
```

EmitCC needs a new field `ModuleCCInputs.SrcDir` so the composer can
choose the right input path and infix. EmitAS and EmitR6 need
analogous parameters. JOIN_SRCS branch (gen.go:912-948) gets the same
treatment.

The exact reference output infix is `__/` (double underscore) when the
module declares SRCDIR pointing at a sibling. Verify byte-exact pin
test against reference for libcxxabi-parts before declaring done.

## Implementation plan

### D01 — MUSL_LITE binding (and yasm-walk verification)

**Scope.** Add `MUSL_LITE: false` to `macros.go` `DefaultIfEnv.bools`
(default M2 is full musl, not lite). Confirm `USE_ASMLIB` does not
need binding (yasm uses `DISABLE(USE_ASMLIB)` which is metadata-only,
not consulted by our `IF` evaluator — yasm's `IF (MUSL)` enables
MUSL_LITE inside its own scope, but our walker doesn't track SET/ENABLE
state across IF branches because each genModule starts from a fresh
Environment).

**Caveat.** yasm's `ENABLE(MUSL_LITE)` would only matter if yasm
itself parsed conditional branches that read MUSL_LITE; it doesn't —
yasm only uses MUSL itself. The MUSL_LITE binding is consulted in
`_BASE_PROGRAM`'s implicit-peer logic, which lives in our new
`defaultProgramPeerdirsFor` (D02), not in yasm's own walk. So
`MUSL_LITE: false` is read by D02's logic to decide musl/full vs musl.

**Acceptance.** `./yatool gen --target tools/archiver` does not throw
"unknown IF identifier MUSL_LITE" anywhere in the closure; existing
emit count unchanged (D01 alone adds nothing — it only enables D02 to
pick the musl/full branch).

**Files.** `macros.go`.

### D02 — `defaultProgramPeerdirsFor` (musl/full + asmlib + cpuid_check)

**Scope.** New helper `defaultProgramPeerdirsFor(ctx *genCtx, instance
ModuleInstance, hadAllocator bool, hadAsmlib bool) []string` returns
the implicit peerdirs that `_BASE_PROGRAM` adds. Logic mirrors
ymake.core.conf:1219-1253 for our M2 environment (MUSL=yes,
OS_LINUX=yes, CLANG=yes, no sanitizer, no PIC-mode caveats):

* `MUSL=yes && !MUSL_LITE` → `contrib/libs/musl/full`
* `ARCH_X86_64=yes && USE_ASMLIB!=no && !MUSL && !SANITIZER` →
  `contrib/libs/asmlib` (host PROGRAMs that AREN'T `MUSL=yes`; today
  none in our closure — leave conditional in place)
* `CPU_CHECK=yes` (always, except when `USE_SSE4!=yes || NOUTIL=yes ||
  ALLOCATOR=FAKE`) → `library/cpp/cpuid_check` — already emitted via
  another path; verify duplicate-suppression handles it.

Call site: `genModule` after `collectModule`, **before** the peer-walk
loop. Append the program-only peers to the existing `defaults` slice
the dedup pipeline already handles.

Add `contrib/libs/musl/full`, `contrib/libs/asmglibc` to
`runtimeAncestorPaths` so they don't recursively pull defaults
(they're inside the runtime stack).

**Acceptance.** `./yatool gen --target tools/archiver` increases node
count by ~33 (target musl/full 4 + host musl/full chain). Pinning
test: synthetic PROGRAM walk emits `contrib/libs/musl/full` AR.

**Files.** `gen.go`.

### D03 — PROGRAM default ALLOCATOR routing (TCMALLOC_TC for our env)

`contrib/libs/tcmalloc/malloc_extension` (2) +
`library/cpp/malloc/tcmalloc` (2) + `contrib/restricted/abseil-cpp`
(157). Pinning test: synthetic PROGRAM (no ALLOCATOR macro) → walker
emits AR with module_dir=tcmalloc/no_percpu_cache.

**Files.** `gen.go`.

### D06 — SRCDIR module_dir rebase guard (ancestor-only)

**Scope.** Replace PR-28-D02's unconditional rebase in `emitOneSource`

Implementation steps:

1. Add `isAncestorPath(srcDir, instancePath string) bool` helper:

## Risks and open questions

### R1. The exact reference SRCDIR path infix (F8 / D06)

Probe found `__/` (double underscore) for libcxxabi-parts. Need to
confirm the same convention applies to musl_extra, tcmalloc, and any
other in-closure SRCDIR-sibling. Executor MUST byte-exact pin one
representative CC per shape against the reference before generalising.

### R2. Default ALLOCATOR detection across SET-overrides

Our walker does not evaluate `SET(...)` macros. If a PROGRAM in the
M2 closure types `SET(ALLOCATOR FAKE)` or similar before its module
declaration, our `hadAllocator` boolean will be wrong. Empirical: no
M2 closure PROGRAM does this (verified by inspection — only ragel6/bin,
yasm, tools/archiver are PROGRAMs in our closure; only ragel6
declares `ALLOCATOR(MIM)`, others elide). Risk is real for future
closures; flag as TODO in `defaultProgramPeerdirsFor` and ship.

### R3. Cycle handler may fire more often

PROGRAM-mode default peers (musl/full + asmlib + tcmalloc) introduce
new edges into the runtime/allocator stack. Existing 7 `util` cycles
should be unaffected. New cycle risk: tcmalloc/no_percpu_cache →
abseil-cpp → sanitizer/include (header-only) → ... If reviewer sees
cycle count > 14, investigate by adding the new participants to
`runtimeAncestorPaths`.

### R4. PR-29 deferred D04 (peer-propagated GLOBAL CXXFLAGS) interaction

PR-30 explicitly does NOT touch GLOBAL CXXFLAGS routing. PR-29's
silent-drop discipline (gen.go:350-367) stays. PR-31 will pick up
peer-propagated CXXFLAGS / `-nostdinc++` / `-DLIBCXXRT` work, after
PR-30 lands the L0 lift.

### R5. Spurious-node count after D06

Today: 14 spurious (7 ragel6 JS + 6 libcxxabi CC + 1 ragel6 LD). D06
fixes the 6 libcxxabi. The 7 ragel6 JS may need further investigation
— probe (a) of `probe-output.txt` flagged them at platform
`default-linux-x86_64` while reference puts them at the same platform
but with different module_dir or paths. Executor verifies
post-D02-D06: if ragel6 JS spurious persists, file a follow-up
defect.

### R6. EXPORT_CMAKE-gated CXXFLAGS in libcxxabi-parts

`IF (EXPORT_CMAKE) CXXFLAGS(GLOBAL -nostdinc++) ENDIF()` — our
`EXPORT_CMAKE=false` binding skips this branch. Confirm this matches
reference (which is presumably built without EXPORT_CMAKE). If
reference DID build with EXPORT_CMAKE, the CXXFLAGS GLOBAL routing
becomes a PR-30 concern. Empirical check: probe the reference's
libcxxabi-parts CC cmd_args for `-nostdinc++` — if absent, our
`EXPORT_CMAKE=false` is correct.

### R7. F9 (SRC_C_SSE41 / builtins truncsfbf2) deferred

util/charset's missing 1 + builtins' missing 5 sources are real but
small (6 nodes). Investigate during D07 if cheap; otherwise defer to
PR-31. Both are likely macro-vocabulary gaps that fit later.
