# PR-28 — Dual-platform emission

## Problem

The walker today emits 1696 target nodes / 0 host nodes for `tools/archiver`.
Reference: 1933 target + 1797 host = 3730. The host gap is total — not a
single `host_platform=true` node ships from `Gen(...)` against any module,
which puts L0 at the 45% ceiling reported in PR-27 regardless of how many
target gaps we close. Closing the host gap is the dominant lever for the
M2 acceptance gate (L0 ≥ 95%, L1 ≥ 80%, L2 ≥ 70%, L3 ≥ 50% on the FULL
graph per D40 / PR-26 spec).

## Empirical findings

All probes against `/home/pg/monorepo/yatool_orig/g.json` (3730 nodes,
1797 host, 1933 target).

### F1. The host closure is exactly two host PROGRAMs.

There are exactly two host LD nodes in the reference graph:

- `contrib/tools/ragel6` host LD, UID `RbpxjH5a-mL6dYBfQTu16g`,
  output `$(BUILD_ROOT)/contrib/tools/ragel6/ragel6`
- `contrib/tools/yasm`   host LD, UID `OsvsEu9xnOqLNXi3INZ9CQ`,
  output `$(BUILD_ROOT)/contrib/tools/yasm/yasm`

Doing BFS over `deps` ∪ `foreign_deps` from each LD:

- ragel6 closure = 1805 nodes (1797 host, 8 target — the 8 are M1's
  build/cow/on target {CC, AR} pair plus 6 from where target and host
  closures intersect at e.g. asmlib).
- yasm closure   = 1736 nodes (1735 host, 1 cross-edge into target).
- ragel6 ∪ yasm  = 1805 nodes; **all 1797 host nodes are in this union;
  zero host nodes lie outside it.**

The host closure equals the union of the host PROGRAM closures. Period.

### F2. Cross-platform edges are sparse and one-directional.

Counting target↔host edges across `deps` and `foreign_deps`:

- target → host via `deps`: **1 edge** (`util/_/datetime/parser.rl6.cpp`
  R6 node → ragel6 host LD).
- target → host via `foreign_deps`: 0 (the R6's `foreign_deps.tool`
  carries a placeholder UID `XO1d8CLk3qDKv0XQTlDKmQ` that is **not in
  the graph at all** — verified: `XO1d... in graph = False`. This is a
  ymake-internal sentinel for a host-tool, not a real edge.)
- host   → host via `deps`: 1827 edges (the host closure is internally
  cohesive).
- host AS → yasm host LD via `foreign_deps.tool` ([the real cross-edge
  inside the host closure]): 50 edges (25 nodes × 2 paths each).

So:

- Only ONE target node references a host node. That node is the R6
  generator node for `util/datetime/parser.rl6.cpp`.
- The yasm dependency lives entirely *inside* the host closure (host
  asmlib `.S` files need yasm to build, and asmlib itself is a host AR
  pulled in by ragel6's transitive musl/full PEERDIR — see F4).

`graph.result = ["yiNI6FkM2GVJuViuMOS3HQ"]` — single UID, the target
archiver LD. Host LDs are reachable via dep traversal but never marked

`contrib/libs/musl/full/ya.make` is critical:

`contrib/tools/yasm/ya.make`:

asmlib AS nodes pull yasm via `foreign_deps.tool` (verified). asmlib's
host AR depends on those 25 AS nodes. Hence yasm host closure is
reached via ragel6 → musl/full(@x86_64) → asmlib → asmlib's AS nodes →
yasm LD. **No direct PEERDIR or `deps` from ragel6 to yasm exists; yasm
is pulled in by asmlib AS's foreign_deps.tool wiring at AS-emit time.**

### F5. Tools/archiver itself does NOT have a host closure of its own.

The archiver target LD has 35 deps; **0 of them are host nodes**.
Archiver does not link against any host artifact. The host nodes only
appear in the graph because of the ragel6 + yasm tool dependencies that
get pulled in via parser.rl6.cpp generation and asmlib's yasm AS
foreign-dep wiring respectively.

### F6. ALLOCATOR / DEFAULT_ALLOCATOR is in scope only insofar as
ragel6's `ALLOCATOR(MIM)` introduces `library/cpp/malloc/mimalloc` as a
host-side peer. The default-allocator-resolution macro
`when ($ALLOCATOR == "MIM") { PEERDIR+=library/cpp/malloc/mimalloc }`
lives in `build/ymake.core.conf:961`. This is structurally identical to
the explicit-PEERDIR path once we resolve `ALLOCATOR(...)` to a
peer-list addition.

For the target archiver LD's tcmalloc deps (`library/cpp/malloc/tcmalloc`
and `contrib/libs/tcmalloc/no_percpu_cache`): these come from
`DEFAULT_ALLOCATOR=TCMALLOC_TC` triggered by `MUSL=yes` (line 410 of
ymake.core.conf), then resolved at the PROGRAM module via
`DEFAULT(ALLOCATOR $DEFAULT_ALLOCATOR)`. **This is target-side
ALLOCATOR resolution, unrelated to dual-platform emission proper.**
Out of scope for PR-28 (deferred).

## Design decision

**Model (c): host emission is rooted at host PROGRAMs.**

The walker, after completing its target walk, additionally walks each
host PROGRAM in the canonical host-tool set
{`contrib/tools/ragel6/bin`, `contrib/tools/yasm`} as an instance with
`Target=cfg.Host.ID, Flags.PIC=true` — the SAME mechanism PR-25 already
implements for R6's per-source ragel6 recursion, lifted from "lazy
per-source" to "eager-after-target". The host walk reuses
`ctx.memo`/`ctx.walking` since the keys are full ModuleInstances and
target-vs-host hash distinct (D34). All recursion edges
(PEERDIR → genModule, host PEERDIR → genModule, asmlib AS → yasm via
`WithHost` recursion) thread through the same `ctx.emit`, which already
accumulates both target and host nodes in one Graph (D32).

**Why not (a) tool-driven**: A pure tool-driven model means we only walk
host instances when target sources demand them. That mostly works
because PR-25 already does this for R6's ragel6. But the asmlib host AS
nodes — the 50 yasm cross-edges — are NOT triggered by any target
source. They are triggered by the host walk hitting asmlib (via
`musl/full` PEERDIRs in host flavour). Tool-driven model never hits
asmlib because it never enters host flavour for musl/full at all.

**Why not (b) mirror-walk**: A mirror-walk re-walks every target module
under host instance. But the reference does NOT host-build every target
module — only ~17 of the 42 target module_dirs have host emission. Mirror
walking would over-emit: tools/archiver itself, library/cpp/archive,
util, abseil-cpp etc. would all gain spurious host nodes. The reference
makes them target-only.

Model (c) matches the reference exactly because the reference is
literally "what's reachable from the target PROGRAM root + what's
reachable from each host-PROGRAM root that any tool dependency has
demanded."

The set of host PROGRAMs to walk is the union of:

1. Host tool dependencies named directly by source-emitters
   (`contrib/tools/ragel6/bin` for `.rl6`, `contrib/tools/yasm` for
   asmlib's `.S` outputs via foreign_deps).
2. Transitively, host tool dependencies named inside the host closure.
   For tools/archiver the closure has none beyond ragel6 and yasm
   themselves.

For the M2 archiver scope, the canonical pair `{ragel6/bin, yasm}` is
sufficient. We hardcode it as `hostToolPrograms` (alongside the existing
hand-translated config like `runtimeAncestorPaths`); a closure reachable
from a non-archiver target may extend it, deferred to M4/M5.

## Implementation plan

The plan is a single linear sequence in `gen.go` plus three small
support changes (toolchain, r6, gen-test). No interface or addressing
changes; all the addressing surface (D30..D40) lands as-is.

### D01 — Hardcode `hostToolPrograms` in gen.go

`gen.go`: add a package-level slice or map naming the host PROGRAM
ya.make paths the walker drives in addition to the target seed. Initial
contents:

```go
var hostToolPrograms = []string{
    "contrib/tools/ragel6/bin",
    "contrib/tools/yasm",
}
```

Rationale: M2 closure is bounded. Future PRs widen this when a non-
archiver target requires another host tool. Comment: "INVARIANT — every
entry MUST be a directory with a `PROGRAM(...)` ya.make whose host
closure (via PEERDIR + DEFAULT_PEERDIR + ALLOCATOR + asmlib's yasm
recursion) is genuinely host-only. Mistakenly listing a target-only
PROGRAM here will pollute the graph with spurious host_platform=true
nodes."

Acceptance: `go vet` clean.

### D02 — Add a host-walk pass at the end of `Gen(...)` in gen.go:175-196

After the target walk (`root := genModule(ctx, seed)` /
`ctx.emit.Result(root.LDRef)`), iterate `hostToolPrograms`. For each
entry, build a host ModuleInstance:

```go
for _, p := range hostToolPrograms {
    hostInstance := ModuleInstance{
        Path:     p,
        Language: LangCPP,
        Target:   cfg.Host.ID,
        Flags:    inferFlagsFromPath(p, true),
    }
    genModule(ctx, hostInstance)
    // No emit.Result() — host LDs are reachable via deps but not
    // marked as result roots in the reference (F3).
}
```

Acceptance: `Gen(...)` host pass runs to completion against
`tools/archiver` without panicking. `go test -count=1 ./...`
succeeds (M1 regression preserved).

### D04 — Wire the R6 → ragel6 LD dep through `DepRefs`, not just
`ForeignDepRefs["tool"]`.

Per F2: the reference R6 node has the ragel6 host LD in `deps`, not in
`foreign_deps`. `r6.go:69-71` currently puts `ragel6LD` in
`ForeignDepRefs["tool"]` only. Reference shape:
- `deps = [ragel6 host LD UID]`
- `foreign_deps = {"tool": ["XO1d8CLk3qDKv0XQTlDKmQ"]}` (a different,
  dangling UID — ymake-internal placeholder, NOT the real ragel6 UID).

PR-28 plan: add `ragel6LD` as a `DepRef` in the R6 node. Continue to
populate `foreign_deps["tool"]` if the reference comparator demands it,
but the foreign_deps UID is the ymake-internal placeholder which we
cannot synthesize byte-exact (it's not in the real graph either, so
neither comparator pairing nor L3 will fail on it — the L3 comparator
only checks UIDs as part of the topology fingerprint at L0/L2).

For PR-28 simplicity: drop the `ForeignDepRefs["tool"]` wiring in r6.go
entirely (the placeholder UID is unreachable, so dropping it lowers
divergence) and put the real ref in `DepRefs`. Add regression test
`TestEmitR6_RagelHostRecursion_DepRefShape` to pin the shape.

Acceptance: R6 emitted node has `DepRefs = [ragel6LD]`, `ForeignDepRefs
= nil`. The reference R6 node has `deps=[ragel6 LD UID]` and
`foreign_deps={tool:[<dangling>]}`; we structurally match deps
byte-exact and intentionally divert the dangling foreign_deps slot. The
walker passes byte-exact at L1/L2 on the R6 node; L3 differs by exactly
the foreign_deps slot which we accept as a known divergence.

### D05 — Wire the AS → yasm LD dep through `DepRefs` AND
`ForeignDepRefs["tool"]` to match reference shape.

Per F2: reference asmlib host AS nodes have yasm in `foreign_deps.tool`
(NOT in deps; yasm UID OsvsEu9xnOqLNXi3INZ9CQ is real and resolvable).
`as.go:123-127` already wires `yasmLD` into `ForeignDepRefs["tool"]`
when non-nil. **No change to as.go required.** The gap is at the call
site: `gen.go:950` currently passes `nil` for `yasmLD` always.

Plan: when the source extension is `.S`/`.s` AND the parent instance's
flags say PIC (host) AND the parent's path is in a known set
(`asmlib_hostuses_yasm = {"contrib/libs/asmlib"}`), recurse into yasm
host instance via `WithHost`, get its LDRef, and pass `&yasmRef` to
EmitAS. Otherwise pass `nil` (preserves the chkstk target path).

The "known set" gate is needed because not every host AS uses yasm —
only asmlib's. `cxxsupp/builtins/chkstk_aarch64.S` for example uses
clang's built-in assembler (no foreign_deps). Check the reference:
asmlib's 25 host AS have foreign_deps.tool=yasm; ALL OTHER 58 host AS
(the chkstk shim + builtins/libcxx) have NO foreign_deps. So:

```go
var asmlibYasmModules = map[string]bool{"contrib/libs/asmlib": true}

// in emitOneSource for `.S`/`.s`:
var yasmRef *NodeRef
if instance.Flags.PIC && asmlibYasmModules[instance.Path] {
    yasmInstance := instance.WithHost(ctx.cfg)
    yasmInstance.Path = "contrib/tools/ragel6/bin"  // wrong; fix to yasm
    yasmInstance.Path = "contrib/tools/yasm"
    yasmInstance.Flags = inferFlagsFromPath("contrib/tools/yasm", true)
    yasmResult := genModule(ctx, yasmInstance)
    yasmRef = &yasmResult.LDRef
}
ref, outPath := EmitAS(instance, srcRel, nil, yasmRef, ctx.emit)
```

Acceptance: TestGen_HostWalk_AsmlibYasmWired (new) — synthetic asmlib
fixture, host walk emits 25 AS nodes each with `ForeignDepRefs["tool"]
= [yasmLD]`. Reference comparator pairs them at L0/L1/L2.

identically). Rendering the host walk reuses the existing
`defaultPeerdirsFor` machinery without modification.

`runtimeAncestorPaths` is platform-agnostic — also reused as-is.

**Acceptance: this D-task is verification only**: confirm by inspection

### D07 — `inferFlagsFromPath` extended for host-tool program paths.

`module.go:158-174` only special-cases `build/cow/on` and `musl`. Host
walk seeds need NO_PLATFORM-equivalent flags for ragel6/bin and yasm.
Look at their ya.makes:
- ragel6/bin: `NO_UTIL()`, no NO_LIBC/NO_RUNTIME → seed FlagSet{PIC: true}
  is fine; `NO_UTIL` overlay happens during `collectModule`.
- yasm:      `NO_RUNTIME()`, `NO_PLATFORM()` (when MUSL is true the IF
  branch sets it implicitly via `NO_PLATFORM()` at the else; here MUSL
  is true so the IF branch fires PEERDIR(musl_extra, jemalloc) and
  DOESN'T set NO_PLATFORM). yasm's host build under MUSL needs the
  default peers.

Decision: leave `inferFlagsFromPath` alone (defaults to PIC-only).
Macro overlay will fix flags during `collectModule`. The path heuristic
exists only as a M1-vintage seed; D31's mechanism is the macro
overlay.

Acceptance: ya.make macro overlay correctly sets NO_UTIL on ragel6/bin
host, NO_RUNTIME on yasm host. Verify via debug print or test.

### D08 — Cycle handler narrowed to "tolerate per-platform cycle"
diagnostic.

## L0 projection

**Honest projection: L0 ≈ 75-85% after PR-28 alone.**

Math: 1696 (target) + ~1500-1797 (host) = 3196-3493 nodes emitted
out of 3730 reference. Topology fingerprint match rate depends on
whether each host flag bundle (host-musl CC, host-asmlib AS, host
libcxx CC, etc.) hits byte-exact — **not all do today.** PR-25 only
ships 3 CC bundles (target-default, host-PIC, target-musl); host-musl
and host-libcxx will need tuning in M5 PRs.

For L0 ≥ 95% acceptance per PR-26 spec: PR-28 plus follow-up host-
bundle tuning PRs are needed. PR-28 closes the **structural** gap —
the 0-vs-1797 host-emission cliff — but full L0 byte-exact requires
host CC composers for every host module class. Two scenarios:

- **Single-PR L0 ≥ 95%**: Not realistic. Even if every host node
  emits in the right shape, byte-exact L1 (kv.p / outputs / target_props)
  generally holds (most host nodes are CC nodes whose only difference
  from target is `--target=x86_64` and `-fPIC` / `.pic.o`, both already
  encoded), but some host modules (libcxx, libcxxabi-parts, jemalloc)
  use bundles we have not authored.
- **Multi-PR path to L0 ≥ 95%**: PR-28 (structural host walk) → PR-29
  (host-musl CC bundle) → PR-30 (host-libcxx/libcxxrt CC bundles) →
  PR-31 (jemalloc/mimalloc CC bundles, ALLOCATOR resolution). Each ~1
  bundle's worth of work, gated on real divergence reports from the
  comparator after PR-28 lands.

**Recommended L0 target for PR-28: ≥ 75%** (on the FULL graph). This
encodes "we close the structural host gap and accept that per-host-
bundle tuning is follow-up work." If this lands at 75-85%, follow-ups
move it to 95%+.

## Risks and open questions

### R1. `ALLOCATOR(MIM)` macro is not yet typed; whitelisted as no-op.

Today `whitelistedMetadataMacros` includes `ALLOCATOR_IMPL` but **not
`ALLOCATOR`**. Wait — let me re-check `applyUnknownStmt` (gen.go:298-319):
ALLOCATOR not in switch, not in whitelist → would throw
`PR-25 does not yet support macro "ALLOCATOR"`. **This is a confirmed
blocker for ragel6 host walk** because `contrib/tools/ragel6/bin/ya.make`
declares `ALLOCATOR(MIM)`.

**Resolution path**: PR-28 must EITHER (a) add `ALLOCATOR` to
`whitelistedMetadataMacros` (drops the implicit mimalloc PEERDIR; the
host closure misses 17 mimalloc nodes — acceptable for D10 ≥1500
threshold), OR (b) implement minimal ALLOCATOR resolution: parse the
arg, switch on name (MIM → `library/cpp/malloc/mimalloc`, FAKE →
none), append to peerdirs. (b) is preferred but adds scope. Sub-task:

- **D12** (recommended): `applyUnknownStmt` recognizes ALLOCATOR; maps
  arg → peer (`MIM`/`MIM_SDC`/`J`/etc.) and appends to `d.peerdirs`.
  FAKE / SYSTEM resolve to no-peer additions. Unknown allocator values
  throw with a diagnostic naming the unknown allocator name. Driving
  reference: `build/ymake.core.conf:961-1035`.

Open question for orchestrator: **OK to include D12 in PR-28, or split
to PR-29?** Recommendation: include — it's <50 lines and unblocks the
ragel6 walk fully.

Wait — D02's host walk fires unconditionally, not gated on whether
the target tree references a host tool. That means `Gen(target=
build/cow/on)` would now emit ragel6+yasm host closures (1797 extra
nodes) even though build/cow/on doesn't need them. **This breaks M1
acceptance** (TestGen_BuildCowOn_TwoNodeSubgraph asserts exactly 2
nodes).

**Open question for orchestrator (BLOCKER for D02 design)**: should
the host walk be (i) unconditional (every Gen call walks ragel6+yasm
host closures) or (ii) demand-driven (walk a host tool only when the
target walk's source enumeration finds an `.rl6`/`.S` source needing
it)?

(i) keeps the architecture simple; M1 test must update from "exactly
2" → "≥2 + (host closure if target needs it)".
(ii) preserves M1 but adds gating logic in `Gen()` (scan target deps
for `.rl6`/`.S` first, then conditionally walk).

Reference behaviour favours (ii): for an M1-style minimal target
without any tool deps, `g.json` would only contain that target's
nodes — but in practice the reference IS for tools/archiver which
DOES have a parser.rl6.cpp transitively. We have no reference data
for "pure target with zero tool deps."

**Recommendation: (ii) demand-driven**. Implement by scanning the
target's transitive sources (after walk completion) for any source
extension/foreign-tool dep that requires a host PROGRAM, then walk
each required host PROGRAM exactly once. The set of "discovered host
tools" is built up during the target walk via `ctx.requiredHostTools
map[string]bool` keyed on PROGRAM path; after the target walk, iterate
that set and call `genModule` on each. M1 still emits 2 nodes for
build/cow/on (zero tool deps detected).

Concrete D02 update:

```go
// During target walk, when emitOneSource sees `.rl6`:
ctx.requiredHostTools["contrib/tools/ragel6/bin"] = true
// During target walk, when emitOneSource sees `.S` AND parent module
// path is in asmlibYasmModules:
ctx.requiredHostTools["contrib/tools/yasm"] = true

// After Gen's target genModule returns, before Finalize:
for hostPath := range ctx.requiredHostTools {  // (sorted)
    ...
    genModule(ctx, hostInstance)
}
```

Note: ctx.requiredHostTools must be populated as soon as any node in
the target *or host* closures (transitively) discovers a tool need.
Including host walks themselves — when ragel6 host walk hits asmlib's
host AS sources, asmlib needs yasm, so asmlib's emitOneSource adds
yasm to `requiredHostTools` mid-host-walk. The post-walk drainer must
loop until no new host tools are added: a fixed-point iteration.

Practical M2 reality: the closure-discovery is bounded — ragel6 walk
discovers yasm; nothing further. A simple two-pass (target walk → walk
each discovered host tool → if any host walk discovered NEW tools,
walk those too) is sufficient. Hardcode an iteration limit (e.g. 5) as
defensive ceiling.

This sub-decision is **the biggest open question for orchestrator**;
the rest of the plan adapts cleanly to either choice.
