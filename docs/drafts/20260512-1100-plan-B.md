## 2. Systemic root causes

Three causes carry the residue. Each is one architectural defect, not
a per-node defect.

### S-1 — aarch64 CC peer-CFLAGS aggregation re-orders three GLOBAL defines

**Signature.** Diff of cmd_args[69..71] on aarch64 `.cpp.o` and similar:

```
[69]  OUR=-DASIO_STANDALONE              REF=-DOPENSSL_RENAME_SYMBOLS=1
[70]  OUR=-DASIO_SEPARATE_COMPILATION    REF=-DASIO_STANDALONE
[71]  OUR=-DOPENSSL_RENAME_SYMBOLS=1     REF=-DASIO_SEPARATE_COMPILATION
```

OUR and REF carry identical *sets* of tokens; only the cyclic position
of `-DOPENSSL_RENAME_SYMBOLS=1` differs (it sits *after* both ASIO
defines for us, *before* both for the reference). The slot is the
peer-propagated GLOBAL CFLAGS slot composed in
`composeOwnAndPeerCFlagsAtOwnSlot` (`cc.go:717`). The order at which
peer CFLAGS GLOBAL are aggregated is **not** the same on aarch64 as
in REF: REF preserves a peer-traversal order in which OpenSSL appears
before Asio; OUR currently produces an order in which Asio appears
before OpenSSL. The composer doc-comment already names the right
empirical anchor ("idx 73-78: …peer-GLOBAL `-DLZMA_API_STATIC`,
`-DOPENSSL_RENAME_SYMBOLS=1`, `-DFFI_STATIC_BUILD`") — but the input
list reaching `composeOwnAndPeerCFlagsAtOwnSlot` is already in the
wrong order before slot-assembly, so the slot logic itself is innocent.

**Hits.** 252 CC + 7 AR (member-list reorder is the same upstream
under different exit point) = **259 L3 leaves on one fix**.

**Why aarch64 only.** Aarch64 enables `mno-outline-atomics` and is the
"target" arch in the host-tool split, so its peer-CFLAGS bundle is
larger and the ordering-bug manifests; x86 host tools either don't
PEERDIR Asio or have a smaller bundle where the swap collapses to no
visible diff. (Six x86 CC mismatches are unrelated — see S-2 / S-3.)

### S-2 — PY-objcopy bundle under-collects `.py` (and `.yapyc3`) inputs

**Signature.** Every PY L2 mismatch (82/82, both arches) is the same
shape: REF has additional `.py` / `.yapyc3` source files in the
objcopy node's `inputs`, OUR has *zero* extra entries. The cascade
into AR is the 13 "AR-missing-py-bundle-inputs" entries — the AR
node aggregates the same missing `.py` files into its own input list
(plus `build/scripts/objcopy.py`). No L3 (cmd) divergence — the
command lines are identical; only the recorded `inputs` differ. The
defect is therefore in the **input-edge emitter** for PY objcopy
nodes — sources collected from `PY_REGISTER` / `PY_SRCS` /
`PY_SUBDIR` are not all being attached as inputs (probably a closure
walk missing a `.py.yapyc3` companion, or only the *primary* source
of a multi-source objcopy bundle being emitted).

**Hits.** 82 PY L2 + 13 AR-py-bundle L2 = **95 L2 leaves on one fix**.

The four `*.pyc.inc` archiver nodes in § 1.4 are a *related* PY
emission gap: those `runtime_py3/{__res,sitecustomize}.pyc.inc`
archiver-AR nodes themselves are missing — a sibling of the same
pipeline. Closing S-2 should be designed to also emit those four
nodes (the archiver bundle); confirm post-fix.

### S-3 — host-LD warning-flag bundle collapsed to `-Wno-everything`

**Signature.** Nine of eleven LD-x86 L3 mismatches have OUR's `cmd[1]`
(the link step) carrying the single token `-Wno-everything`, where
REF carries `-Wall -Werror -Wextra -Wno-implicit-const-int-float-conversion
-Wno-parentheses -Wno-unknown-warning-option`. The token
`-Wno-everything` is *not* a Clang flag; it appears to be an internal
placeholder/sentinel that has not been expanded into the real
warning bundle in the host-LD command-composer. Identical pattern
across `tools/archiver`, `tools/rescompressor`, `tools/event2cpp`,
`tools/rescompiler`, `tools/enum_parser`, `tools/struct2fieldcalc`,
`tools/py3cc/*`, `tools/py3cc/slow/*`,
`library/python/runtime_py3/stage0pycc`.

Two of the eleven LD mismatches are independent (missing protoc
peer-link `contrib/libs/protobuf/libcontrib-libs-protobuf.global.a` +
`-ldl -lrt` on `protoc` and `cpp_styleguide`).

**Hits.** 9 LD L3 (and the 7 LD L2 mismatches mostly piggyback on
the same nodes). One fix.

### Cascade interpretation

- **S-1 alone closes 259/327 = 79 % of L3 mismatches.** It does not
  touch L2 or L1 (inputs are identical; only cmd-arg positions
  change). With S-1 the L3-mismatched set drops to ~68 leaves, of
  which ~50 are the S-3 (LD warning bundle) and the cluster-4 sub-fixes.
- **S-2 alone closes 95/167 = 57 % of L2 mismatches** and yields the
  4 unpaired AR-archiver-bundle nodes. L3 is unaffected (no command
  divergence). With S-2 the L2 set drops to ~72 entries dominated by
  AR-libcxx-include diffs and one openssl-S asm path diff.
- **S-3** is a single composer fix (host-LD warning-bundle expansion);
  the 9 LD nodes are *all* the host-tool LDs in the residue.
- **Cluster 4** is *not* one bug — it is several small codegen / peer
  consumer-include registry gaps (PCRE_JIT define on PCRE consumers,
  protobuf+abseil include on `.pb.cc.o`, build_info include on its
  three direct consumers, antlr4 `-Wno-unused-variable`). They share
  the *shape* "consumer-include / consumer-define missing", which
  argues for one architectural pass on the consumer-include registry
  (extending PR-M3-F-7-D's pattern to a handful of new emitters).

Combined S-1 + S-2 + S-3 closes:

- **L3** : 259 + 9 ≈ **268 / 327 (82 %)** in three architectural PRs.
- **L2** : 95 / 167 (57 %).
- **L1** : +4 unpaired AR nodes; remaining gap (jemalloc x86_64) is
  one independent PR.

Cluster-4 then closes another ~40 L3 (≈ 12 % of L3) in one
consumer-include-registry sweep. After all four PRs L3 sits around
~20 residual leaves, the L2 residue at ~70.

## 4. Plan B — proposed architectural PRs (ordered by cascade yield)

The order below is by **cascade yield per PR**, not by code-locality.
Every PR is one architectural change with a single hypothesis and a
single verification command.

### PR-B-1 — Stabilise aarch64 peer-CFLAGS-GLOBAL aggregation order

Hypothesis: the *upstream* peer-walk that feeds
`composeOwnAndPeerCFlagsAtOwnSlot` produces a different traversal order
on aarch64 than REF expects. Confirm by capturing the
`PeerCFlagsGlobal` slice contents at one representative aarch64 CC
node (`devtools/ymake/include_parsers/ydl_parser.cpp.o`) before
slot-assembly, comparing to the REF cmd-args slice. Diagnose whether
the divergence is (a) peer-walk traversal order, (b) per-peer flag
ordering inside one peer's GLOBAL CFLAGS, or (c) a final stable-sort
that is keyed differently from REF.

**Yield:** 252 CC + 7 AR L3 leaves; 0 L2 / L1 effect.

**Verification:** `./yatool make -j 0 -G devtools/ymake/bin` followed
by `python3 diff-cmdargs.py` showing zero CC L3 mismatches in
`cc-aarch64-cflags-reordering` bucket.

### PR-B-2 — Restore PY-objcopy `.py`/`.yapyc3` input aggregation

Hypothesis: the PY-objcopy emitter only records the *primary* source
of each objcopy bundle and drops the companion `.py` / `.yapyc3`
files that REF carries. Either (i) the closure walk over
`PY_SRCS` / `PY_REGISTER` stops at the first source, or (ii) the
companion files are produced but not threaded into the node's
`inputs` slot. Identify which, fix the aggregation, and verify that
the 4 currently-unpaired `*.pyc.inc` archiver-AR nodes also emit
(they are the cascade consequence of the bundle being complete).

**Yield:** 82 PY + 13 AR L2 leaves; +4 unpaired AR nodes (L1
recovery).

**Verification:** `./yatool make -j 0 -G devtools/ymake/bin` then
`paired-count` script returns 8 748 / 8 750 (only jemalloc remaining
unpaired).

### PR-B-3 — Expand host-LD warning sentinel into the real flag bundle

Hypothesis: an internal placeholder `-Wno-everything` is reaching the
emitted command in `ld.go` / host-LD composer instead of the
expanded `-Wall -Werror -Wextra -Wno-implicit-const-int-float-conversion
-Wno-parentheses -Wno-unknown-warning-option` bundle. The placeholder
exists because the bundle is being *deferred* but never realised.

**Yield:** 9 LD L3 leaves.

**Verification:** zero LD-x86 mismatches in `ld-warning-bundle` bucket.

### PR-B-4 — Consumer-include / consumer-define registry sweep

Group of four small, **systemically-similar** misses, all of the shape
"REF emits flag/include on the consumer of a peer, OUR does not":

1. `-DARCADIA_PCRE_ENABLE_JIT` on all consumers of `contrib/libs/pcre`
   (22 nodes — actually internal to pcre's own ya.make, but the define
   is missing for every pcre source on aarch64).
2. `-I…protobuf/src` + `-I…abseil-cpp` + `-I…abseil-cpp-tstring` on
   `.pb.cc` consumers (9 nodes — extends the registry that
   PR-M3-F-7-D started for protobuf-runtime headers, to compile flags).
3. `-I$(BUILD_ROOT)/library/cpp/build_info` on its 3 direct consumers
   plus 2 cascaded (5 nodes).
4. `-Wno-unused-variable` on antlr4-generated `g4.cpp.o` (4 nodes).

All four arguably live in the same `codegen_registry.go` / consumer
include-flag map. One architectural extension covers them.

**Yield:** ~40 CC L3 leaves.

**Verification:** `./yatool make -j 0 -G devtools/ymake/bin` followed
by cluster-tally showing each of the four labels at zero.

### PR-B-5 — Round-up: jemalloc/malloc-info x86_64 emission + ymakeyaml/rapidjson over-emission cleanup

Two unrelated tiny PRs grouped only because each is small:

- emit the missing `library/cpp/malloc/jemalloc` CC + AR pair on x86_64
  (the only L1 residue after PR-B-2);
- drop the spurious `-I…brotli/c/include`, `-I…snappy/include`,

  yaml/rapidjson contrib closures.

**Yield:** 2 L1 + 4 L3 leaves.

**Verification:** L0 / L1 / L2 / L3 = 100 % on `devtools/ymake/bin`.

## 5. How Plan B differs in spirit from a node-by-node plan

A node-by-node plan would (re-)enumerate the 327 L3 + 167 L2 + 6
unpaired residue defects, group them by *output prefix* (e.g. "devtools/
ymake/include_parsers/* — 27 nodes"), and propose one PR per cluster
with anchor outputs. That worked through M3-A and M3-B but is
already showing diminishing returns: each cluster's "fix" is
empirically the same upstream symptom expressing through a different
consumer.

**Plan B's deltas:**

1. **Groups by upstream architectural defect, not by output prefix.**
   The 133 leaves under `libdevtools-ymake.a` are *not* a
   `libdevtools-ymake`-specific defect — they are the
   `peer-CFLAGS-GLOBAL ordering` defect manifesting through every CC
   whose ya.make happens to PEERDIR a module that PEERDIRs Asio and
   OpenSSL. Fixing 133 nodes by walking the AR-input list would
   diverge wildly from fixing 1 emitter once.
2. **Names two cross-arch symmetries.** PY-objcopy under-emission
   manifests on both x86_64 (21 leaves) and aarch64 (61 leaves) with
   the *same* diff shape; the host-LD warning-bundle defect is only
   x86_64 because it's a host-tool-only emitter. A node-by-node plan
   would split each into per-arch PRs and lose the dual-arch yield.
3. **Treats over-emission and under-emission as one defect class.**
   The 4 "spurious brotli/snappy/re2 includes" on ymakeyaml/rapidjson
   are the same registry, expressed in the wrong direction; clustering
   them with PR-B-4 (under-emission consumer-include misses) reveals
   the single registry pass that should touch both.
4. **Cascade-orders PRs by *leaf yield per PR*, not by topological
   depth.** PR-B-1 is the highest-yield PR by a factor of ~8 over the
   next-best — schedule it first even though it's a one-line ordering
   change, because every other PR can land its verification on a
   cleaner residue.
5. **Names the *single most cascade-yielding move* explicitly:** see
   the summary below.

## Summary (≤ 400 words)

**Three systemic causes carry the M3 residue, accounting for ~93 % of
the L1+L2+L3 gaps:**

- **S-1 — aarch64 peer-CFLAGS-GLOBAL aggregation order.** A single
  upstream ordering defect that places `-DOPENSSL_RENAME_SYMBOLS=1`
  *after* the two `-DASIO_*` defines on aarch64. The flag *set* is
  identical to REF on 252 / 299 CC L3 mismatches; only the position
  differs. The bug lives upstream of
  `composeOwnAndPeerCFlagsAtOwnSlot` (cc.go:717) — in the peer-walk
  or peer-CFLAGS-GLOBAL accumulator that hands it the input slice.

- **S-2 — PY-objcopy input aggregation under-collects.** Every PY L2
  mismatch (82/82) is pure under-emission of `.py` / `.yapyc3`
  inputs into objcopy nodes; the same omission cascades into 13 AR
  globals and into the 4 unpaired `*.pyc.inc` archiver-AR nodes.

- **S-3 — host-LD warning sentinel never expanded.** OUR's host-LD
  cmd[1] carries the single token `-Wno-everything` where REF
  carries the real `-Wall -Werror -Wextra -Wno-*` bundle. Hits 9 of
  11 LD-x86 mismatches.

A small fourth cluster (consumer-include / consumer-define registry
gaps: PCRE_JIT, protobuf+abseil, build_info, antlr4) accounts for
~40 more CC L3 leaves and behaves as one architectural extension of
the codegen/consumer-include registry, not as four independent fixes.

**Proposed shortlist of 5 architectural PRs**, ranked by leaf
yield-per-PR:

1. **PR-B-1** — Stabilise aarch64 peer-CFLAGS-GLOBAL ordering
   (~259 L3 leaves; **highest yield**).
2. **PR-B-2** — Restore PY-objcopy `.py`/`.yapyc3` input aggregation
   (~95 L2 leaves; 4 L1 recovery).
3. **PR-B-3** — Expand host-LD warning sentinel into real bundle
   (~9 LD L3).
4. **PR-B-4** — Consumer-include / consumer-define registry sweep
   (PCRE_JIT + protobuf+abseil + build_info + antlr4-`-Wno-unused-variable`
   + ymakeyaml/rapidjson over-emission cleanup), ~44 leaves.
5. **PR-B-5** — Jemalloc/malloc-info x86_64 emission round-up (2 L1).

**Single most cascade-yielding move:** **PR-B-1** — one upstream
ordering fix takes L3 mismatches from 327 to ~68 in one PR, dropping
L3 from 96.19 % to ~99.2 % paired-correct and leaving the remaining
work in three small, well-bounded PRs.
