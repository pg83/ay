## Cluster A — link-libs residue on host (x86_64) LD nodes [9 nodes]

Every LD node on `default-linux-x86_64` is missing **`-lrt -ldl`** in
`cmds[2].cmd_args` (the link command):

- `contrib/tools/protoc/plugins/cpp_styleguide/cpp_styleguide`
- `contrib/tools/protoc/protoc`
- `tools/archiver/archiver`
- `tools/enum_parser/enum_parser/enum_parser`
- `tools/event2cpp/event2cpp`
- `tools/py3cc/py3cc` (also `cmd[1]` `-DLZMA_API_STATIC` etc., see B)
- `tools/rescompiler/rescompiler`
- `tools/rescompressor/rescompressor`
- `tools/struct2fieldcalc/struct2fieldcalc`

Root cause: x86_64 host linker missing the default system-libs trailer
(`-lrt -ldl`). Look at `LDFLAGS_GLOBAL`/`LINK_DYN_LIBRARIES` composition
in the host-platform branch in `gen.go`. Same flags are emitted correctly
for aarch64 nodes; only the host LD path is short.

**Proposed fix:** PR-RES-A — append `-lrt -ldl` to the host-linux x86_64
default link-libs slot. Single composer site.

## Cluster B — host-CFLAGS missing macros on Python/musl link tools [3 nodes]

LD nodes whose `cmd[1]` (link-time CFLAGS aggregation) drops 3-5 defines:

- `tools/py3cc/py3cc`: missing `-DLZMA_API_STATIC`, `-DOPENSSL_RENAME_SYMBOLS=1`, `-DFFI_STATIC_BUILD`
- `tools/py3cc/slow/py3cc`: same + `-D_musl_=1`, plus a 3-objcopy ordering diff in `cmd[2]`
- `library/python/runtime_py3/stage0pycc/stage0pycc`: same trio as `py3cc`
- `devtools/ymake/bin/ymake` aarch64 (cmd[1]): same trio

All four share **PEERDIR transit through python3+contrib(lzma/openssl/ffi)**.
The `-D_musl_=1` only on `py3cc/slow` is the musl variant suffix.

**Proposed fix:** PR-RES-B — propagate `LZMA_API_STATIC`,
`OPENSSL_RENAME_SYMBOLS=1`, `FFI_STATIC_BUILD` from `contrib/libs/{lzma,
openssl, ffi}` PEERDIR exports. These are `CFLAGS GLOBAL -D…` in upstream
`ya.make`. Investigate why our PEERDIR closure drops them on the link
node but keeps them on compile nodes (cluster passes through link
aggregator).

## Cluster E — ymake aarch64 link object ordering [1 node, part of D14]

`devtools/ymake/bin/ymake` (aarch64) `cmd[2]` has the same objects as
REF but at positions 18-21 + 70 OUR places
`library/python/symbols/{module,libc,python}` **before**
`library/cpp/blockcodecs/codecs/zstd`; REF places zstd first. Same
contents, deterministic order mismatch.

**Proposed fix:** PR-RES-E — review GLOBAL-libs linker ordering: REF
appears to sort by PEERDIR closure order (zstd via `library/cpp/...`
comes from the c++ side before `library/python/symbols`); OUR appends
python first. Likely a stable-sort key off-by-one in the link-args
composer.

## Cluster F — inputs ordering: libcxx ordering vs project-local first [7952 nodes]

Same content set, different sequence. REF places one project-local
header (e.g. `antlr4_cpp_runtime/src/support/Declarations.h`) between
the cpp's own headers and the libcxx tail; OUR puts libcxx headers
contiguously after the cpp's own headers. The diff appears at indices
4-7 of nearly every CC node.

Diagnosis: include-scanner emits parsed-include `inputs` in scanner
discovery order; REF emits them in **first-mention order in the
preprocessor traversal**, which interleaves a project header that the
cpp transitively pulls.

**Proposed fix:** PR-RES-F — replicate upstream `inputs` ordering. In
the include-resolver path (`scanner.go` post-PR-M3-vfs-paths), insert
discovered project headers at the position they're first referenced,
not at the libcxx-tail. Single ordering-policy change. Verify with
`sorted(OUR) == sorted(REF)` invariant (currently holds) plus
position-by-position equality on a sampled CC node.

## Cluster G — foreign_deps: OUR-empty on PY objcopy nodes [131 nodes]

All 131 are `kv.p=PY` `objcopy_*.o` (Python frozen module objcopy
outputs). REF carries `{'tool': [2-uid-list]}`; OUR carries `None`.
The two REF UIDs resolve to objcopy-tool + ld-tool host instances.

**Proposed fix:** PR-RES-G — when generating PY objcopy nodes,
populate `foreign_deps.tool` with the host UIDs of the resolved
`objcopy` and `ld` tools (consistent with how non-PY nodes already
populate `foreign_deps.tool`).

## Cluster H — host-platform tool flagging for codegen tool-only nodes [4 nodes]

REF marks these as host-only (`host_platform=True`, `tags=['tool']`),
OUR omits both:

- `library/cpp/eventlog/proto/events_extension.pb.h` (x86_64)
- `library/cpp/eventlog/proto/internal.pb.h` (x86_64)
- `util/_/datetime/parser.rl6.cpp` (x86_64)

Pattern: a `.h`/`.cpp` codegen output that participates **only** in
host-side tool builds. The propagation of `tool` taint through
codegen outputs is incomplete in OUR.

**Proposed fix:** PR-RES-H — propagate `tags=['tool']` and
`host_platform=True` to codegen outputs (PB/R6) that are reachable
solely from host-tool LD nodes. Single tag-propagation pass over
the closure.

## Cluster I — foreign_deps UID-only diffs [608 nodes]

All 608 have identical `tool: [N uids]` shape but different uid
strings. Spot-check: OUR's `4wOlHFgxh-bjIPbepDvw92` and REF's
`p3NN5_pMd3D3E_gJOdfp4g` both resolve to the same yasm tool node
(same outputs, platform, tags). **Not a defect** — these are
consequences of the global uid-naming hash differing between
implementations. They will close trivially when `self_uid` / `uid`
hashing aligns (Cluster J).

## Cluster J — self_uid/stats_uid mismatch across all 8750 nodes

OUR emits `self_uid=None, stats_uid=None`; REF emits hex strings.
Universal — not a per-node defect.

**Proposed fix:** PR-RES-J — implement node `self_uid` / `stats_uid`
hashing per upstream algorithm (`devtools/ymake/.../uid.cpp`). Closes
this row entirely and incidentally clears Cluster I.
