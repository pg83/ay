# Plan: 2× speedup for `yatool make -G -j0` graph generation

Scope: single-threaded graph-generation phase of *this* Go repository's
`yatool make`. The upstream `ya make -G -j0` semantics are the contract
the output must keep matching (sg.json), but no upstream code is touched.

Out of scope: executor runtime, distributed/parallel build, parallelisation
of the gen-path. Local micro-optimisation is allowed only as a secondary
source of speedup; the bulk of the plan targets work that should not run
at all under a better-organised pipeline.

`dev/normalize.py`) — except in `--digest` mode, where the canonical
hash of sg.json is the equality witness.

`tools/archiver` retains the existing project-wide ≤ 5 s hard gate;
this plan targets a more ambitious ≤ 0.4 s point for it without
parallelism.

## 2. Counter-derived structural facts (target scanner, `ymake/bin`)

- 108 distinct `scanCtx` instances allocated, peak 64 live.
- 4,631,217 `resolve` calls; hit rate 90.4%; only 443,828 unique
  `(includer, target, flags)` keys actually populated. Per-call cost
  is dominated by the cache lookup, not by miss-path computation.
- 1,217,081 subgraph-cache hits / 56,311 misses — 95.6% of every
  per-includer subgraph lookup is a hit; each hit still pays a
  `subgraphInnerKey{abs, sourceClass}` map probe in per-scanCtx maps.
- Sysincl lookups: 4.6 M source-keyed + 4.6 M includer-keyed = 9.2 M
  probes for 70,952 unique misses (target only); 99.8% / 98.6%
  hit rate.
- Source-keyed misses = 7,366; includer-keyed misses = 63,586. Each
  miss creates exactly one cache entry, so the unique-entry counts
  per probed total are: source-keyed ≈ 4.63 M probes / 7,366 entries
  ≈ 628 probes/entry; includer-keyed ≈ 4.63 M probes / 63,586 entries
  ≈ 73 probes/entry. The source-keyed axis is ~8.6× denser in reuse
  per entry; the includer-keyed axis carries ~8.6× more unique values
  per probed total. The two halves factorise reuse along the two
  axes; merging would multiply cardinality, not collapse it.
- `parsedHits` 957,517 / `parsedMisses` 10,351: parsing is effectively
  a constant — every closure walk asks 90+× per parse.
- `existsMisses` 97,379 / `existsHits` 865,904: ~10% of stat-cache
  traffic is a real `os.Stat` for paths we then mostly throw away.

The headline: at hit rates this high, the bottleneck is *probing the
cache*, not *filling the cache*. Any plan whose primary lever is
"raise the hit rate of an existing cache" is misdirected — the levers
worth pulling are (a) ask fewer questions, (b) make each question
cheaper than a Go-map probe.

Caveat noted in framing: the `walkClosureCache` counter is 0 in
`scanCtx`-stats output because the gen-path enters via
`genCtx.getScanCtx(…).WalkClosure(…)`, which bypasses
`IncludeScanner.walkClosureCache` (that field caches scanCtxs for the
test-facing `IncludeScanner.WalkClosure`). The number is not a proof
of missing reuse on the production path.

## 3. Stable identifiers for any cross-run reuse

`R`-class items below depend on cross-run keys that are not tied to
the in-process interner. Runtime IDs in `scannerInterner` (scanner.go:58)
and `subgraphInnerKey.abs` (scanner.go:223) are assigned by visit order
and are not stable across processes. They cannot appear in any
on-disk key.

The cross-run key space is built from three primitive types of
fingerprint, all derivable without running the scanner:

| stable id              | derivation                                                         | cost to compute |
|------------------------|--------------------------------------------------------------------|-----------------|
| `path-key`             | SOURCE_ROOT-relative path as a string (or `$(B)/…`)                | free            |
| `file-stamp`           | `(size, mtime-ns)` of the file                                     | one `stat`      |
| `content-digest`       | SHA-256 of the file bytes                                          | one read + hash |
| `ctx-fingerprint`      | FNV-1a of the **ordered triple** `(OwnAddIncl, PeerAddInclSet, BaseSearchPaths)`. Each tier hashed in its declaration order, tiers separated by a sentinel byte; equal fingerprint iff each tier's ordered path list is byte-equal | O(addincl size) |
| `sysincl-version`      | SHA-256 of the concatenated contents of every loaded `*.yml`, plus tool semver | one-shot, < 1 ms |
| `sourceClass-fingerprint` | sorted set of canonical record-indices into the on-disk sysincl set that match this source's filter; serialised as a uvarint list | O(sysincl records) |

`file-stamp` is **not** a content address. It is a quick-check hint:
equal stamp is strong evidence the bytes are unchanged but is not a
proof (mtime can be preserved across edits on some filesystems; size
can collide on small edits). Two validation policies cover the
spectrum:

1. **Stamp-trust** (default): if the recorded `file-stamp` matches
   the current `stat`, the entry is reused. Cheap (one `stat` per
   unique input) but admits a small false-hit rate on filesystems
   that compress mtime.
2. **Content-strict** (`--cache-verify-strict`): in addition to
   `stat`, the file bytes are re-hashed and compared against the
   recorded `content-digest`. Eliminates false hits; pays one read +
   hash per unique input on every run. Used by CI.

A shard-level **bulk gate** runs first: each shard records a top-level
`(tool-version, sysincl-version, ctx-fingerprint)` triple. If any
differs from the current process the shard is skipped without
per-entry checking.

Validation cost (default stamp-trust) is bounded by O(unique inputs
in shard) `stat` calls — at most ~10–20 K stats on `ymake/bin`. At
~3 µs/`stat` from page cache this is < 60 ms. Content-strict mode
costs O(file bytes) and is opt-in.

## 4. Initiative classes

Initiatives are tagged with three orthogonal axes:

- **Class** — where the change lands:
  - `L` = local in-process;
  - `R` = cross-run persistent reuse;
  - `C` = protocol / storage / contract change.
- **Bucket** — what category of work it eliminates or compresses:
  - `B1` inter-run reuse;
  - `B2` intra-run laziness / reachability culling;
  - `B3` format change as work-elimination;
  - `B4` algorithmic;
  - `B5` allocation / GC pressure;
  - `B6` output/contract elimination;
  - `B7` invalidation model.

Each initiative declares: target buckets, cold-run effect, warm-run
effect, risk, invalidation key, dependencies.

`B7-0` (invalidation manifest) is the prerequisite for every `R`-class
item — no free-standing speedup, but unblocks the rest.

---

### B7-0. Invalidation manifest (prerequisite for any R-class work)

**Class:** L. **Cut:** structural; no direct CPU saving.
**Cold:** 0. **Warm:** unlocks all `R-*` savings.

Define a per-shard manifest using only the stable identifiers from §3:

| artifact                        | shard freshness key (top-level)                         | per-entry key |
|---------------------------------|---------------------------------------------------------|---------------|
| parsed include set per file     | `tool-version`                                          | `path-key + file-stamp` |
| `exists`-cache entries          | `tool-version`                                          | `path-key + file-stamp` of containing dir |
| parsed `ya.make` AST            | `(tool-version, yamake-parser-version)`                 | `path-key + file-stamp` |
| per-ctx resolve set             | `(tool-version, ctx-fingerprint, sysincl-version)`      | `(path-key of includer, target string, flags)` |
| per-ctx subgraph                | `(tool-version, ctx-fingerprint, sysincl-version, sourceClass-fingerprint)` | `path-key of includer` |
| sg.json output                  | `(tool-version, target, platform, conf-digest)`         | shard-level only |

Storage layout: one binary file per shard under
`.out/yatool-cache/<shard-id>.bin`. A shard with an outdated top-level
key is dropped on load without inspection. Mismatching per-entry
`file-stamp` drops only that entry.

Risk: low; this is a spec, not yet enforcement. The artefact is
"shard files exist, top-level keys validate, per-entry stats agree";
anything that fails is just discarded silently.

Acceptance: a `--debug-cache` flag prints per-shard load/skip
decisions; touching any input file flips state correctly.

### B2-1. Lift the resolve cache to (ctx-fingerprint, includer, target) keying (`L, B4, B7`)

Cuts: 4.6 M resolve calls (target). At 90% hit rate the dominant cost
is the per-scanCtx probe; collapsing 108 disjoint caches into one
shared cache with a richer key removes the duplicate populations and
the per-scanCtx probe stack.

**Cold:** target ~6-9% wall (lower than v1's ~11%, after accounting
for the codegen-epoch invalidation overhead below).
**Warm:** same.

Refactor:

1. Move `resolveCache` off `scanCtx` onto `IncludeScanner`. New key:
   `(ownAddInclID, peerAddInclID, baseSearchID, includer-id, target-id, flags)`.
   Each of the three tier IDs is the content-intern ID of an **ordered
   slice**: two contexts collapse to the same tier ID iff their tier
   slice is byte-equal *in declaration order*. Tier order matters:
   `resolveContextSearchTier` (scanner.go:1279-1358) processes
   `OwnAddIncl` → `PeerAddInclSet` → `BaseSearchPaths` first-match-wins,
   so re-ordering tiers can change which path wins.

   Note on tier representation: three separate IDs are equivalent in
   *cardinality* to one combined ID over the ordered triple — both
   key forms have `unique(triple)` distinct values. The reason to
   keep three fields rather than one combined ID is **not** a
   cardinality argument, it is preservation of tier identity for
   downstream uses: per-tier hit/miss counters, a future hierarchical
   cache where a `BaseSearchPaths`-only resolution can be stored once
   and reused across distinct (Own, Peer) overlays, and human-
   readable debug dumps. Today's first-cut implementation uses the
   wider struct as a flat key with `unique(triple)` cardinality;
   hierarchical cache (if needed later) is a separate refactor that
   the three-field representation does not preclude.

   The cross-`scanCtx` collapse benefit depends on how often the
   three tier-IDs are equal across the 108 currently-distinct
   scanCtxs. This is not measured in the current profile — see §8.
2. Decouple from runtime codegen mutation. The codegen registry
   (`IncludeScanner.codegen`) mutates during gen — see the
   producer-before-consumer ordering hoist at `gen.go:1997-2012`.
   A cached miss whose lookup *would have* hit a producer that
   registered after caching is the existing stale-miss bug class.

   Mitigation: every resolve-cache *miss* entry carries a
   `codegenEpoch` counter. The scanner maintains a global
   `codegenEpoch` field, bumped by `CodegenRegistry.Register` (and
   any future `fallbackLocators` mutator). On lookup, an entry
   with `entry.epoch < scanner.codegenEpoch` AND `len(entry.out) == 0`
   is treated as a miss and re-resolved. Hits (`len > 0`) never
   stale-invalidate because once a path resolves to a file, that
   file remains resolvable for the rest of the gen.

   Cost: the epoch field is uint32; eviction is implicit; the only
   per-resolve overhead is the epoch comparison on miss-cached
   entries.

3. The existing emission hoists (EN/JV/PR/AR before CC at
   `gen.go:1997-2068`) remain in force. The epoch mitigation is
   defence-in-depth, not the primary correctness mechanism — it
   protects against any future ordering regression and lets us safely
   share cache state across modules.

Risk: medium. Two concerns:

- Codegen epoch is the new invariant; it must be wired through every
  registry mutation site. Acceptance: an assertion `epoch ==
  scanner.codegenEpoch` is checked on every Lookup that returns a
  cached miss in debug builds; CI runs with assertions on.
- Cache-cardinality blow-up: collapsing 108 caches into one shared
  cache must not regress entry count. From v1 counters, 443,828
  populated entries across 108 caches imply on average ~4,100 unique
  keys per cache, with high overlap. Expect the unified cache to
  land at 50-80 K entries (~10× collapse).

Dependency: B7-0 manifest (only for the `R`-extension; the L change
stands alone).

### B3-1. Replace per-call `VFS.String` materialisation with interner IDs (`L, B5`)

Cuts: 243 MB / 21.5% of allocations, plus a slice of the 20% GC.

**Cold:** ~5-9% wall. **Warm:** same.

`VFS.String` is called pervasively in resolve / sysincl machinery to
key maps and compare paths. The scanner-internal interner (already
present, scanner.go:58) is the natural target. Concretely:

1. Audit every `map[string]` and `map[VFS]` reached from the hot path
   in `scanner.go` / `parser_manager.go` / `sysincl.go`. Replace
   string keys with interner `uint32` IDs; the string form
   materialises only at the serializer boundary (`gjson_write.go`).
2. `resolveSearchPath` builds its candidate-path slice as `[]VFS`;
   downstream `subgraph` already interns each one. Build the
   candidate slice as `[]uint32` directly and skip the round-trip.
3. CI check: count `VFS.String` calls per run; regression fails CI.

Risk: low — interner is proven; the lift is mechanical. Medium-risk
piece is the audit: missing one materialisation site leaves the
allocation cost unchanged.

Dependency: none. Sequenced after B2-1 because the lifted cache is the
biggest beneficiary.

---

### B4-2. Replace closure `idSet` map with a dense epoch array (`L, B5`)

Cuts: a portion of the 20% GC cost.

**Cold:** ~2-4% wall. **Warm:** same.

`walkSubgraphID` / `plainDfsID` take pooled `idSet` (Go `map[uint32]
struct{}`) and `[]uint32` buffers from `sync.Pool`. Clear-on-return
costs O(entries). Replace with a single shared `[]uint32` epoch array
indexed by interned VFS id; membership = `epoch[id] == currentRunID`.
Increment `currentRunID` per walk; `idSet` is gone. Order stays a
slice.

Risk: low. Standard pattern; widely used in compilers.

Dependency: B3-1 (dense interned IDs).

---

### B3-3. Drop `existsMisses` traffic via directory-listing snapshots (`L, B3, B4`)

Cuts: 97 K `os.Stat` calls per `ymake/bin` gen (existsMisses), plus a
slice of the 3.1% syscall cost.

**Cold:** ~1-3% wall.

`fileExistsByRel` does an `os.Stat` per missed key, then memoises.
Most existsMisses cluster in dirs that exist; the syscall storm is
per-file not per-dir. Replace with: on first miss under dir `D`,
`os.ReadDir(D)` once and populate every entry of `D` in `exists`.
Subsequent probes hit memory.

Invalidation: dir `file-stamp` (mtime); on warm restart from a
persistent cache, re-stamp.

Risk: low — read-dir is cheap and the workload only consults the
cache for paths it's about to lay claim to. Edge case: directories
with thousands of entries; bound the eager-populate to dirs the
scanner has already missed > 1 entry into.

### B1-1. Persistent parser cache (`R, B1, B5`)

Cuts: `parsedHits`/`parsedMisses` map traffic + 118 MB / 10% of cold
allocations.

**Cold:** 0. **Warm:** ~3-5% wall (parsing + readfile bypassed on
unchanged files).

`sharedParseCache.parsed` dies with the process. Persist as a single
binary file (`.out/yatool-cache/parsed.bin`) keyed by `path-key`,
validated by `file-stamp`. Layout: a packed `(key → directives)`
table addressed by a perfect hash built at write time, mmap'd on
read. Lookup becomes a stamp comparison + an mmap-bounded slice copy.

Invalidation: per-file `file-stamp`; on stamp mismatch, parse, append
to the in-memory delta, rewrite the file at process exit.

Risk: low under `stamp-trust`, lower still under `content-strict`.
The cache is **stamp-addressed by default, content-addressed under
`--cache-verify-strict`** (see §3). Stamp-trust admits a small
false-hit rate on filesystems with coarse mtime; CI runs strict and
will catch any divergence; the user-facing default is the fast path.

Dependency: B7-0.

### B1-2. Persistent subgraph cache (`R, B1`)

Cuts: 56 K cold misses of `subgraph()` per `ymake/bin` (each miss a
recursive DFS through `forEachResolvedChild`). On a warm rerun every
hit becomes an mmap-table lookup.

**Cold:** 0. **Warm:** ~10-18% wall (less than v1's ~15-25% after
costing the per-entry `stat` validation explicitly).

Currently `scanCtx.subgraphCache` is allocated per `scanCtx`, lifetime
= one Gen. Persist into shards keyed by `(tool-version,
ctx-fingerprint, sysincl-version, sourceClass-fingerprint)`. Per-entry
key: `path-key` of the includer. Per-entry value: the canonical
DFS-discovery-order list of `path-key`s in the closure plus a
recorded `file-stamp` for every input that contributed to that walk
(includer + every header expanded under it).

Validation on load:

1. Top-level shard key must match → else skip the whole shard.
2. Walk the union of recorded input `file-stamp`s once via `stat`;
   any mismatch drops every subgraph entry whose key set touched
   that input.
3. Surviving entries are exposed as if they had been computed.

Tainted subgraphs (`subgraphTaintedKnown`) carry a single boolean
"this key is on a cycle"; persisted as a separate per-shard slice so
we don't re-pay the speculative walk on warm load.

Risk: medium. Two failure modes:

- Per-entry recorded input set must be complete; missing a file
  produces a *false hit* that doesn't see a change. Mitigation: the
  scanner emits the set as a side product of the walk (it already
  visits every input); the recorder is a sink in the same DFS, not a
  separate analysis.
- `sourceClass-fingerprint` must be stable across runs. Compute it as
  the sorted record-index list (a canonical permutation of the
  on-disk sysincl set) — independent of allocation order.

Dependency: B7-0, B1-1.

### B3-2. Sysincl as a precompiled binary index (`L+R, B3, B5`)

Cuts: YAML parse on cold start + every lookup's machinery cost
(9.2 M sysincl probes on ymake/bin).

**Cold:** ~3-5% wall. **Warm:** modest additional gain if mmap'd.

Today: `*.yml` loaded once into `SysInclSet`, served at runtime.
Alternative:

1. One-shot `yatool sysincl compile` (or implicit first-run) writes
   `.out/yatool-cache/sysincl-<sha>.bin`:
   - flat array of records;
   - radix / perfect-hash table over `target` keys;
   - source-filter regexes pre-compiled and serialised as a packed
     DFA, not `*regexp.Regexp` re-built per startup.
2. Runtime: mmap, no parse. Lookup is a hash → record-index → bitmap
   test. The current two-half cache split (sysinclSourceCache /
   sysinclIncluderCache) is preserved verbatim — it factorises reuse
   along two independent axes (sourceClass × includer) and merging
   would multiply cardinality. Compilation only changes the *lower*
   layer (record storage and matching), not the upper cache shape.
3. The runtime-parsed-YAML exception in scratch memory still holds —
   the YAMLs remain the source of truth; this is a compile step,
   not a hand-translation.

Invalidation: `sysincl-version` = sha of every input YAML + tool
semver (B7-0).

Risk: medium. Source-filter regex semantics must be preserved
bit-exactly (today's `tryBacktrack` callers, 1.9% flat, sit here).
Mitigation: characterisation tests on a snapshot of every distinct
source-filter pattern observed in `tools/archiver` +
`devtools/ymake/bin` before the compile path lands.

Dependency: B7-0.

### B6-1. Skip whole-graph emission when the caller wants only verification (`L+C, B6`)

Cuts: 4.4% flat `appendStringEscapedBody` + ~7% cum in `writeGraph`;
reduces output to a digest + per-node manifest.

**Cold:** target ~5-8% wall when caller wants digest only.
**Warm:** same.

Many `-G` callers consume only:

- the set of inputs/outputs per node (no command strings);
- or the full graph once to compute its hash.

New flag: `--digest` emits a content hash of the canonical sg.json
inline as nodes are produced (single-pass hash; no full materialisation).
`--filter <glob>` emits only nodes matching the glob.

Risk: low — additive flag; default behaviour unchanged.

---

### B6-2. Demand-driven gen: emit a node only when its key is requested (`C, B2, B6`)

Cuts: every node an offline consumer doesn't reach. For an IDE asking
for one TU's closure, ~5-15% of `ymake/bin`'s eager walk.

**Cold:** target-dependent; not load-bearing for the headline `-G`
case but unlocks a class of consumers.

`genModule` walks the whole transitive peer-closure eagerly. Split
the API: caller asks for a node by key, the gen-path lazily expands
only what's reachable from the query. Today's eager mode becomes
"emit all reachable from target" on top of the lazy core.

Risk: high. Multi-week contract change at the boundary between
`yatool make -G` and consumers; not on the critical path to 2×.
Sequence after B1-2 because the persistent subgraph cache is the
backing store.

Dependency: B1-2.

---

### B6-3. Streaming binary IR alongside sg.json (`C, B3, B6`)

Cuts: 4.4% flat JSON-escape + most of the emitter's allocation share
when the consumer is `yatool` itself.

**Cold:** ~3-5% wall when caller is in-process executor (not the case
for `-G -j0`, where JSON is the contract).

Add an emitter mode that streams typed nodes through a channel and
elides text serialisation. JSON remains the contract for offline `-G`
use.

Risk: medium — byte-equality on `-G` mode must survive the refactor.

Dependency: none; best after B2-1 / B3-1 so the typed in-memory
representation is already clean.

### Local micro-opts (`L, B5`) — opportunistic fill-in

Not load-bearing individually (~1-2% each):

- precompiled regex paths cached as DFAs (`tryBacktrack` 1.9% flat);
- buffer reuse for `stripComments` / `eachLine` (2.5% flat each);
- replace `crypto/sha1` content hashing of output paths (2.3% flat)
  with a fast 64-bit hash where collision resistance isn't required
  (output-path uniqueness is achieved by including a per-module path
  component);
- bigger pre-size for `resolveCache` after B2-1.

### Removed from this revision

The following items were in the previous version of this plan and are
removed:

- **B2-2 (union-walk per (scanCtx, sourceClass) with bitmask
  attribution)** — bitmask intersection over a union walk produces the
  union's first-visit order intersected with each source's reach
  set, not each source's own first-visit DFS order. `WalkClosure`'s
  contract (scanner.go:430) requires the latter. The proposed mechanism
  cannot produce the bytes the contract demands; an honest
  replacement (per-source walks against shared subgraph hits) is what
  the current subgraphCache already does, so there is nothing new to
  add. Removed.

- **B4-1 (merge sysincl source-keyed and includer-keyed caches into one
  `(sourceClass, includer, target)` key)** — current counters show the
  two halves factorise reuse along independent axes: 7,366 source-side
  misses vs 63,586 includer-side misses (target scanner). The cache
  comment at scanner.go:1156 confirms each half is reused along its
  own axis and the storage gain is exactly the factorisation. A merged
  cache would carry cardinality on the order of `sourceClass-count ×
  includer-count`, almost certainly larger than the current sum.
  Without a positive estimate of unique combined keys this item is
  speculative and was net-negative on inspection. Removed.

Target acceptance: `devtools/ymake/bin` ≤ 2.00 s wall (currently 3.98 s).

| B2-1 (lift+epoch)   |  3.98 s     |  3.70 s    |  -7%   | medium (epoch risk) |

**L-only lands ~3.05 s, i.e. ~1.30×.** The previous version's 1.56×
relied on two removed items (B2-2, B4-1); the honest L-only ceiling
is closer to 1.30-1.35×.

For the scoped task — `yatool make -G -j0` writing canonical sg.json
to stdout — **no listed in-process, in-contract combination reaches
2× on `devtools/ymake/bin` cold.** The L-only ceiling is the L-only
ceiling. This is the honest framing the previous revision blurred:

- **Cold, JSON contract preserved**: L-only ≈ 3.05 s ≈ 1.30×. Cannot
  cross 2.00 s with any combination of items above without violating
  some contract dimension (output bytes, persistence, or eagerness).
- **B6-3 (streaming IR) is not in play for `-G -j0`**: the JSON
  emission *is* the contract for this caller. B6-3 helps the
  in-process executor path (`-j N`, `N > 0`) and is listed only for
  completeness; it does not contribute to the scoped target.

Two contract-loosening options are available; **neither alone reaches
2× on the scoped benchmark set**, but their combination does. Each
loosens one dimension of the contract:

**Option A — opt-in output contract (`-G --digest -j0`).** B6-1 adds
a flag that emits the canonical sg.json hash inline as nodes are
produced, without materialising the bytes. Existing callers using
plain `-G -j0` see no change. The `--digest` mode targets callers
whose only consumed value is the equality decision: CI "did the
graph change?" gates, content-addressed build-cache lookup keys,
and any other reduction-to-a-witness consumer. It is **not** a
drop-in for `dev/normalize.py`, which canonicalises full JSON and
byte-compares the canonicalised forms (see `dev/normalize.py:1-22`)
— that consumer still requires the full sg.json. A separate
hash-only variant of the diff harness, or a CI gate that compares
two `--digest` outputs against each other, are the realistic
consumers; both are additive new pieces, not free reuses of
existing tooling.

| target              | cold L-only | cold + `--digest` | × vs. today |
|---------------------|-------------|-------------------|-------------|
| `tools/archiver`    |  0.62 s     |  ~0.50 s          | 1.62×       |
| `devtools/ymake/bin`|  3.05 s     |  ~2.75 s          | 1.45×       |

`--digest` alone **does not clear 2× on either target on cold**. It
narrows the gap (emission is a larger share of `archiver`'s smaller
wall, so the relative gain is bigger there) but neither row crosses
the 2.00 s / 0.40 s bars. Its role in this plan is as a multiplier on
top of Option B's warm savings; on its own it is not sufficient.

**Option B — opt-in inter-run reuse (cache files on disk).** B7-0 +
B1-1 + B1-2 turn second-and-subsequent invocations against the same
source tree into mostly-cache-hit runs. The first invocation pays
full L-only cost; the warm bar applies to the development workflow
where the tree is edited incrementally.

| target              | cold (= first run) | warm  | warm × |
|---------------------|--------------------|-------|--------|
| `tools/archiver`    |  0.62 s            | ~0.42 s | 1.93×  |
| `devtools/ymake/bin`|  3.05 s            | ~2.10 s | 1.90×  |

Borderline on both targets; the headroom is consumed by B1-2's
per-entry `stat` validation (priced at 50-80 ms).

**Option A ∪ B — opt into both.** This is the only combination on
this profile that clears 2× **with margin on both targets** and is
the recommended target for an aggressive ramp:

| target              | cold A only | warm B only | warm A+B  | warm A+B × |
|---------------------|-------------|-------------|-----------|-----------|
| `tools/archiver`    |  ~0.50 s    |  ~0.42 s    |  ~0.34 s  |  2.4×      |
| `devtools/ymake/bin`|  ~2.75 s    |  ~2.10 s    |  ~1.85 s  |  2.15×     |

**Recommendation for the scoped task** (`-G -j0`, no parallelism,
graph generation):

- Default callers stay on the JSON contract. Cold ceiling ≈ 1.30×;
  warm (Option B) ≈ 1.90×. **Neither row crosses 2× — this must be
  communicated clearly to anyone treating "2× without contract
  change" as feasible.**
- Aggressive callers opt into `--digest` (Option A). Cold reaches
  ≈ 1.45× on `ymake/bin` and ≈ 1.62× on `archiver` — **still below
  2× on both targets**. Combined with Option B (Option A∪B warm)
  the same callers reach ≈ 2.15× / 2.4×, clearing 2× on the warm
  path.

**Cold 2× is not achievable on either listed target from any
combination of items in this plan** without one of: re-opening
parallelism, accepting a deeper contract change than `--digest`
(e.g. dropping eager whole-graph emission), or finding a structural
saving the current profile does not surface.

If the user's requirement is *strictly* 2× on cold runs of
`-G -j0` with the unchanged sg.json contract, this plan does not
deliver it. The honest framings are: (a) accept "2×" as a warm-path
metric (matches normal dev workflow) — Option A∪B clears it with
margin on both targets, Option B alone lands at 1.90-1.93× (borderline,
contingent on the `stat`-validation cost estimate); (b) accept
`--digest` as the output contract and pair with persistence (Option
A∪B) for 2× on the warm path; (c) re-open parallelism, excluded by
the problem statement; (d) reframe the scoped task as a weighted
benchmark mixing cold first-runs and warm reruns — under a dev loop
dominated by warm reruns, Option A∪B clears 2× on the weighted
average.

**Recommended order of work**:

1. B7-0 (manifest spec, cheap, unblocks R-items).
2. B3-1 (interner audit). Cleanest correctness story, biggest single
   L step. Doubles as prep for B2-1 by making the lifted cache key
   compact.
3. B4-2 (epoch idSet). Low-risk pair with B3-1 (both attack GC).
4. B2-1 (resolve cache lift + codegen epoch). Higher correctness
   surface; ship behind a `--unified-resolve` flag, default off,
   flip to default after one week of CI confirmation. Acceptance: a
   debug-build assertion guarantees no cached miss is returned with a
   stale epoch.
5. B3-3, B3-2, micro-opts (commutative; ship in any order).
6. B1-1 (parser persist) — first `R` step; gives a small warm win and
   exercises the manifest path with low correctness risk.
7. B1-2 (subgraph persist) — the headline warm step; only land after
   B1-1 is steady-state in CI.
8. B6-1 (`--digest`) — cleanest cold-path lever; behaviour is opt-in.
9. B6-3 (streaming IR), B6-2 (demand-driven gen) — optional;
   bigger architectural moves, on the table only if 2× is missed
   after items 1-8.

## 6. What this plan deliberately does NOT do

- **No new goroutines or parallelism.** Every change is single-threaded.
- **No upstream-ymake features added speculatively.** This is about
  doing less, not matching upstream coverage.
- **No backwards-compat shim layer.** Cache files are rebuilt on
  schema changes; pre-1.0.
- **No refactor of the rule-emit path (`emit_*.go`)** beyond what
  B6-3 strictly needs. The hot path is the scanner.
- **No change to `parsers.go` raw-scanning contract.** DFA-replacement
  of regex backtracking is opt-in micro-opt; recognised directives
  stay bit-equal.

## 7. Measurement & non-regression

Every initiative carries:

1. A perf-counter delta on `tools/archiver` and `devtools/ymake/bin`
   (`YATOOL_PERF_STATS=1`).
2. A wall-clock delta, median of 5 runs.
3. A byte-equality check on sg.json output (`dev/normalize.py`,
   then diff).
4. For `R`-class items: a forced-miss path (`--no-cache`) producing
   the same bytes.
5. For B2-1 specifically: a debug-build assertion validates the
   codegen-epoch invariant on every cache hit.

Regress on (3) and the initiative is rolled back regardless of
speedup.

## 8. Honesty about the estimates

Numbers in §5 are model-based, derived from cum-bucket shares in the
profile. They are not measured per-step. Confidence:

- B3-1, B4-2, B3-3: high — they attack well-isolated buckets (alloc/
  GC, syscalls) and have analogous prior implementations to anchor
  the savings.
- B2-1: medium — savings depend on how much overlap exists across
  the 108 scanCtxs once unified; the upper bound is the per-scanCtx
  resolve-probe cost (cum ~31%), the realistic floor is the share
  of probes that today miss in one scanCtx and would hit in another.
- B3-2: medium — savings hinge on regex-DFA vs `tryBacktrack` speed,
  which depends on the actual filter shapes. Worth measuring on a
  representative sample before committing.
- B1-1, B1-2: medium — net win depends on the `stat` validation cost
  being well under the saved compute. Priced explicitly above.
- B6-1: high — the saved work is a known fraction of the emission
  bucket and the contract change is local.

Per-step actual deltas must be re-measured before the aggregate
stack is treated as committed.
