### Cluster L2-A — missing `enum_serialization_runtime/*.h` set (≥222 CC nodes)

```
222× $(SOURCE_ROOT)/tools/enum_parser/enum_serialization_runtime/dispatch_methods.h
222× $(SOURCE_ROOT)/tools/enum_parser/enum_serialization_runtime/enum_runtime.h
222× $(SOURCE_ROOT)/tools/enum_parser/enum_serialization_runtime/ordered_pairs.h

197× $(BUILD_ROOT)/devtools/ymake/diag/stats_enums.h_serialized.cpp
```

Every CC that compiles a TU consuming `Y_ENUM_*` macros (~all of
`devtools/ymake` and its peer-graph closure) should transitively pull
these four `.h` files plus the generated `stats_enums.h_serialized.cpp`
(the EN node's emitted source, used downstream as an `inputs`-only
witness, not compiled directly by the dependent TUs but tracked in the
transitive set).

**Root cause hypothesis:** the EN (enum-serializer) node emits its
output `stats_enums.h_serialized.cpp` correctly (the EN node itself is
L2/L3 clean per direct probe), but the **transitive `inputs` of CC
consumers** miss the peer-emitted `enum_serialization_runtime/*.h`
headers that the `enum_parser` tool's `PEERDIR` brings into every
PROGRAM/LIBRARY that uses `_ENUM_PARSER`.

**Sample:**

```
[CC] $(BUILD_ROOT)/devtools/ymake/include_parsers/xs_parser.cpp.o
  only-ref (5): stats_enums.h_serialized.cpp, stdlib_deps.h,
                dispatch_methods.h, enum_runtime.h, ordered_pairs.h
```

**Why it's likely L0 ↔ L2 coupled to Cluster D (§1):**
once the `enum_serialization_runtime` module emits on aarch64 (Cluster
D), its EmitsIncludes / peer-include propagation kicks in and these
headers thread into consuming CCs' inputs.

**Recommended PR (highest-yield):** enrich the `enum_parser` /
`enum_serialization_runtime` peer-include registry so consumers' CC
`inputs` carry these four `.h` + `stats_enums.h_serialized.cpp`. Likely
file: `/home/pg/monorepo/yatool/codegen_registry.go` (pattern matches
the recent PR-M3-F-7-D for protobuf — see commit `dcffbbf` for the
analogue: "PR-M3-F-7-D: enrich PB/EV EmitsIncludes with protobuf
runtime headers"). **Estimated L2 lift: +222 CC inputs = ≈ +2.55pp
on L2.**

### Cluster L3-A — `$PYTHON_INCLUDE` literal leak (143 CC nodes, all aarch64)

```
143× extra-in-ours: $PYTHON_INCLUDE   (UNEXPANDED variable!)
159× missing-from-ours: -I$(SOURCE_ROOT)/contrib/libs/python/Include
159× missing-from-ours: -DUSE_PYTHON3
151× missing-from-ours: -I$(BUILD_ROOT)/library/python/runtime_py3
```

We literally emit the unexpanded ymake variable `$PYTHON_INCLUDE` as a
cmd_arg (143 occurrences in `/home/pg/monorepo/yatool/.out/m3.json`).
Confirmed: `grep -c PYTHON_INCLUDE .out/m3.json → 143`.

Per upstream `build/conf/python.conf:1013-1033`, the macro
`_PYTHON3_ADDINCL` is:

```
macro _PYTHON3_ADDINCL() {
    SET(PYTHON3 yes); SET(PYTHON2 no); SET(_PYTHON_VER py3)
    when ($USE_ARCADIA_PYTHON == "yes") {
        CFLAGS+=-DUSE_PYTHON3
        ADDINCL+=GLOBAL $PY3_BASE_INCLUDE_DIR
        ...
    }
    otherwise {
        when ($USE_SYSTEM_PYTHON) { PEERDIR+=build/platform/python }
        otherwise { CFLAGS+=$PYTHON_INCLUDE }   # ← we're hitting THIS branch
    }
}
```

Even though `macros.go:338` claims `"USE_ARCADIA_PYTHON": true`, the
`_PYTHON3_ADDINCL` macro is **not being evaluated** by our walker —
neither branch of the `when/otherwise` is consulted. Instead, our
`gen.go:1108 case "USE_PYTHON3":` only appends peerdirs and skips the
ADDINCL / CFLAGS effect entirely; the literal `$PYTHON_INCLUDE` is
being emitted from somewhere else (likely a default CFLAGS template
that was meant to be substituted but isn't on aarch64).

**Sample:**

```
[CC] devtools/ymake/main.cpp.o (aarch64)  first diff at slot 9
  ours[9]: -I$(SOURCE_ROOT)/contrib/libs/linux-headers
  ref[9]:  -I$(SOURCE_ROOT)/contrib/libs/python/Include
```

The slot-9 swap propagates across ~all ymake-tree aarch64 CCs that
have `USE_PYTHON3` in their closure.

**Why aarch64-only:** likely the `$PY3_BASE_INCLUDE_DIR` expansion or
`USE_ARCADIA_PYTHON=yes` short-circuit applies on x86_64 (where the
peer-include propagation from `contrib/libs/python` already pulls the
right `-I`), but our aarch64 codegen path emits the otherwise-branch
literal instead.

**Recommended PR:** in `gen.go:1108 case "USE_PYTHON3":` (or the
`_PYTHON3_ADDINCL` analogue if separately handled), additionally
inject:

- `-DUSE_PYTHON3` into the module's CFLAGS,
- `-I$(SOURCE_ROOT)/contrib/libs/python/Include` (i.e. `$PY3_BASE_INCLUDE_DIR`'s expanded value) into the ADDINCLs,
- `-I$(BUILD_ROOT)/library/python/runtime_py3` for the runtime peer's gen tree.

And purge the `$PYTHON_INCLUDE` literal from wherever it's emitted on
aarch64 (likely a target-platform-conditional in `flags.go` or
`toolchain.go`).

**Estimated L3 lift: +143 CC nodes ≈ +1.64pp on L3.** Also recovers
the 159×`-DUSE_PYTHON3` + 159×`-I.../python/Include` + 151×runtime_py3
arg-missing counts toward L3 (these overlap heavily with the 143).

### Cluster L3-C — define-order swaps (~74 order-only nodes)

Order-only mismatch breakdown:

| Prefix | Count | Sample slot swap |
| --- | --- | --- |
| `contrib/tools/ragel5` | 22 | `tcmalloc + abseil` precede `ragel5/aapl` in ref; we put ragel5/aapl first |
| `devtools/ymake/lang` | 17 | (see Cluster L3-B — overlaps) |
| `contrib/restricted/libffi` | 15 | `libffi/include` swapped with `libffi` root |
| `contrib/libs/base64` | 10 | `-std=c11` comes LAST in ref, FIRST among the macro-prefix group in ours |
| `contrib/libs/asio` | 2 | `-DASIO_STANDALONE` vs `-DOPENSSL_RENAME_SYMBOLS=1` swap |

These are all **slot-ordering bugs** — same arg set, wrong order.
Each cluster has a distinct cause:

- **ragel5** — `tcmalloc` + `abseil-cpp` (transitive from a higher-level peer) appear AFTER the local peer (`ragel5/aapl`) in ours, but BEFORE in ref. This is a peer-visit-order mismatch.
- **libffi** — `libffi/include` should follow `libffi` root in ref, but our code puts include first. ADDINCL ordering inside the libffi module.
- **base64** — `-std=c11` is at the **end** of the cmd in ref (after `-fmacro-prefix-map=`), but at slot 105 (before macro-prefix-maps) in ours. This is the `_BASE_UNIT.CXXFLAGS` vs per-source override ordering.

**Recommended PR:** three small ordering-fix PRs, or one larger
"peer-include & flag-slot canonicalization" PR that audits the four
emit slots (CFlags, OwnCFlagsGlobal, PeerCFlagsGlobal, ADDINCL) per
`/home/pg/monorepo/yatool/cc.go:706-712` and reconciles them with
ref's order.

**Estimated L3 lift: ~74 nodes ≈ +0.85pp.**
