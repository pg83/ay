# PR-32 — musl refactor + auto-PEERDIR + ARCH-IF + peer-GLOBAL CFLAGS

Author-only draft. No code lands until D-tasks ship.

Scope: three intertwined mechanisms in (probably) one PR. The user's
directive is the architectural anchor — musl is selected by a CLI `-D`
flag and is just an implicit PEERDIR added to every library/program;
yatool code MUST NOT hardcode musl as a special module class.

Related defects this PR closes or progresses: **PR-31-D12** (load-bearing
L2 stagnation root cause); **PR-29-D01** (CXXFLAGS GLOBAL drop-to-floor)
follow-through into peer routing; **PR-31-D08/D09** (JS/LD scope
expansion) remain deferred but are unblocked by the CFLAGS routing
machinery this PR introduces.

Ledger references: `tasks.md` cross-cutting D30..D40, M2 PR-23..PR-31
Completed entries; `defects.md` PR-29-D01, PR-31-D08/D09/D11/D12;
`docs/drafts/20260507-1544-pr31-include-scanner.md` Note B.

## Problem

**(1) musl-hardcoding scattered across the code.** Today:

- `cc.go` dispatches via `isMusl := instance.Path == "contrib/libs/musl"
  || strings.HasPrefix(instance.Path, "contrib/libs/musl/")` and routes
  to dedicated `composeMuslCC` / `composeMuslHostCC` composers (62 musl
  mentions).
- `gen.go` repeats the same path-prefix check at
  `includeScannerBasePaths` (line 1681) and again in `defaultPeerdirsFor`
  for self-suppression (line 679); also has hardcoded
  `defaultProgramPeerdirsFor` literal `contrib/libs/musl/full` (69
  mentions total).
- `module.go` `inferFlagsFromPath` flips `NoLibc=NoUtil=NoRuntime=true`
  on musl-prefixed paths (1 mention).
- `ld.go` injects literal `-D_musl_=1` and `-D_musl_` into
  `composeLDCmdVcsCompile` cmd_args (14 mentions, ~4 of them code;
  rest doc).
- `flags.go` has 4 musl-named flag bundles: `muslCcIncludes`,
  `muslCcIncludesX8664`, `muslWarningFlags`, `muslExtraDefines` (45
  mentions; mostly doc).

Total: **231 musl tokens across 9 production files**
(audit re-counted 2026-05-07T21:36; PR-31 plan reported 150 across 7
files — gap is PR-31's additions to `scanner.go` / `sysincl.go` /
`includeScannerBasePaths` in `gen.go`).

Of those, ~95 are doc/comment text (informative), the rest split
between flag-bundle data tables (legitimate) and dispatch-by-path-prefix
(the smell).

**(2) L2 stagnation at 79.92%.** PR-31 reached the M2 acceptance gate
but stopped because every non-musl TARGET CC node in the reference
graph emits 4 musl-arch `-I` flags (`-I$(SOURCE_ROOT)/contrib/libs/musl/{arch/aarch64,
arch/generic, include, extra}`). Our gen emits zero. Empirical probe
2026-05-07: 619 of 621 L3-divergent CC pairs are missing the musl-arch
-I block; 112 of 607 L2-divergent CC pairs differ ONLY in those header
inputs. Root cause is the missing implicit PEERDIR rule
`when ($MUSL == "yes") { PEERDIR+=contrib/libs/musl/include }` from
`build/ymake.core.conf:781`. Fixing this lifts L2 by ~3pp directly
(112/3683 ≈ 3.0pp) and uncaps the CC byte-exact pairing for L3.

**(3) Peer-GLOBAL CFLAGS routing gap.** PR-29-D01 plugged the literal-
GLOBAL-leak (`-nostdinc++` was emitted as the literal string `GLOBAL`
followed by `-nostdinc++` on libcxx's own CC nodes) by *dropping* GLOBAL
CXXFLAGS at `applyUnknownStmt` instead of routing them to peers. The
correct ymake semantics: `CXXFLAGS(GLOBAL X)` propagates X to every
peer-of-this-module's compile, not to self. PR-31 implemented the
analogous mechanism for ADDINCL via per-path `splitAddInclPaths` →
`d.addInclGlobal` → `walkPeersForGlobalAddIncl` →
`PeerAddInclGlobal` → cmd_args slot. PR-32 mirrors that for CFLAGS /
CXXFLAGS / CONLYFLAGS. Empirical probe: `-nostdinc++` is propagated to
817 reference CC nodes; our gen emits it on zero (after PR-29's drop-
to-floor fix). Closing this is the second L3 lever.

### F2 — Reference musl semantics

Reference probe 2026-05-07: 3571 CC nodes total. Distribution of musl
flags:

| flag | count | meaning |
|------|------:|---------|
| `-D_musl_=1` | 2702 | musl-self CFLAGS — declared by musl's own ya.make.inc, applies to musl-internal sources |
| `-D_musl_` (no `=1`) | 532 | musl-CONSUMER peer-CFLAG — propagated by `_BASE_UNIT`'s `when ($MUSL == "yes") { CFLAGS += -D_musl_ }` rule (build/ymake.core.conf:781-782) |
| neither | 337 | NO_PLATFORM modules: build/cow/on, cxxsupp/builtins (excluded from the auto-CFLAG by upstream's NOPLATFORM gate) |

The musl/non-musl distinction is therefore NOT categorical at the
"this module is special" level. It's:

- musl modules declare `NO_PLATFORM` (NoLibc + NoUtil + NoRuntime
  triple); they get `-D_musl_=1` from their own ya.make's CFLAGS.
- non-musl + non-NO_PLATFORM modules get `-D_musl_` from `_BASE_UNIT`'s
  auto-CFLAG.
- NO_PLATFORM modules (musl-self plus build/cow/on plus builtins) get
  neither auto-CFLAG; musl-self overrides via its own ya.make CFLAGS.

This means the `-D_musl_` propagation is just a peer-CFLAG mechanism.
yatool needs to (a) know which modules are TARGET (i.e., LIBRARY/
PROGRAM, vs runtime-ancestor) and (b) know whether MUSL=yes. Then the
`-D_musl_` slot becomes "any TARGET module that is not NO_PLATFORM gets
peer-CFLAG `-D_musl_` from the implicit musl/include peer".

The cleanest model: musl/include's ya.make declares `CFLAGS(GLOBAL
-D_musl_)` (it doesn't today — upstream puts the rule in
`_BASE_UNIT`). PR-32 has two sub-options for matching this:

- **(A) literal upstream port**: hardcode in `defaultPeerdirsFor` (or
  a new helper) "MUSL=yes && !NO_PLATFORM → seed `-D_musl_` into the
  module's effective peer-CFLAGS". Pure data, no code special-case.
- **(B) synthetic ya.make**: have `defaultPeerdirsFor` add
  `contrib/libs/musl/include` as an implicit peer. The upstream
  `contrib/libs/musl/include/ya.make` does NOT declare `-D_musl_`
  GLOBAL — that part lives in `_BASE_UNIT` not in the included ya.make.
  So this would NOT propagate `-D_musl_` automatically; needs a
  patched-in synthetic `CFLAGS(GLOBAL -D_musl_)` or a separate
  injection.

**Recommended: (A).** Mirror the upstream `_BASE_UNIT` `when` shape as
an auto-injected effective peer-CFLAG inside `genModule`, gated by
`cliFlags["MUSL"] == "yes" && !effectiveNoPlatform(d.flags)`. This
keeps the *flag-bundle data* (the literal `-D_musl_` token) localized
to one named constant and makes the *condition* (MUSL-on, not-no-
platform, target module) visible at the dispatch site.

### F3 — `-D_musl_=1` provenance

The `=1` form lives only on musl-self CC nodes (musl/full, musl/...,
including all musl/_/src/* outputs). It is NOT in the auto-CFLAG (which
adds `-D_musl_` without `=1`). It comes from musl's own ya.make's
CFLAGS via `muslExtraDefines` (flags.go:391). Today that's hardcoded in
the `composeMuslCC` / `composeMuslHostCC` bundles. After PR-32:

- The `-D_musl_=1` literal stays inside `muslExtraDefines` — it's
  data describing musl-internal compile flags.
- The composer dispatch that selects `muslExtraDefines` becomes
  flag-driven (`if instance.Flags.LibcMusl { ...muslExtraDefines... }`)
  instead of path-prefix-driven.

For the ld.go vcs_version compile (line 312, 315): currently injects
`-D_musl_=1` literal then `-D_musl_` literal regardless. The reference
shape is one of three:

- musl PROGRAMs (none in M2 closure that reach LD with musl-self
  flags): would carry both
- non-musl PROGRAMs in MUSL=yes env (every M2 PROGRAM today): exactly
  this shape
- non-MUSL builds: neither

Today's ld.go behavior matches the second case unconditionally because
M2 only emits non-musl PROGRAMs in MUSL=yes. The refactor: derive
these flags from the same effective peer-CFLAGS that drive non-vcs
CC; specifically, the LD's vcs_version compile is essentially a
mini-CC for `__vcs_version__.c` and should consume the same effective
flag set as the PROGRAM's own sources would. PR-32 unifies this via
shared composeXxxCC helpers + a vcs-specific override slot for the
two extra `-D_musl_=1` / `-D_musl_` injections (which match the
two suppression-block copies — one half is "declared as part of musl"
i.e. =1, the other half drops the =1 to stamp the propagated form;
this is genuinely musl-specific compiler behaviour and stays as a
named constant).

### F4 — Auto-PEERDIR `contrib/libs/musl/include` semantics

Upstream `build/ymake.core.conf:781-784` (inside `module _BASE_UNIT`):
```
when ($MUSL == "yes") {
    CFLAGS += -D_musl_
    PEERDIR += contrib/libs/musl/include
}
```

`_BASE_UNIT` is the parent of LIBRARY (line 1888 → `_LIBRARY:
_BASE_UNIT` line 1968), `PROGRAM` (line 1297-1301 → `_BASE_PROGRAM:
_LINK_UNIT` → `_LINK_UNIT: _BASE_UNIT` line 1112), DLL, FAT_OBJECT,
RECURSIVE_LIBRARY, UNITTEST, etc. So **every** TARGET module class
inherits this. Suppression gates: `NO_PLATFORM` (the umbrella) blocks
the entire `_BASE_UNIT` body from contributing implicit peers; that's
how musl-self avoids self-PEERDIRing.

Therefore the rule is: **for every walked module that is LIBRARY or
PROGRAM (today's only two supported types), if `MUSL == yes` and
`effectiveNoPlatform(d.flags) == false`, append
`contrib/libs/musl/include` to the implicit-peer list AND inject
`-D_musl_` into the module's effective peer-CFLAGS.**

`contrib/libs/musl/include/ya.make` declares 4 `ADDINCL(GLOBAL ...)`
paths plus 1 `LD_PLUGIN(musl.py)` plus the runtime-ancestor flags
(NO_PLATFORM, NO_RUNTIME). Two of the ADDINCLs are inside ARCH-IF
blocks (ARCH_X86_64 → musl/arch/x86_64; ARCH_AARCH64 → musl/arch/
aarch64). The other 3 (musl/arch/generic, musl/include, musl/extra)
are unconditional. Through the existing peer-GLOBAL ADDINCL machinery
(PR-31 D04-D06), these 4 paths automatically propagate to every
consumer's CC cmd_args as `-I$(SOURCE_ROOT)/<path>` flags AND to the
include scanner's search path. The current zero-cmd-args-emission
state is purely because the implicit PEERDIR is missing.

`contrib/libs/musl/include` itself is a header-only LIBRARY (no SRCS
that produce object files; the upstream graph confirms zero CC nodes
under that path; only the LD_PLUGIN contributes a CP node). The PR-31
header-only path (`!hasCompilableSource(d)` → `headerOnly: true` →
peer-GLOBAL aggregation only) handles this correctly already. Once
the implicit PEERDIR fires, walker reaches musl/include, parses its
ya.make, evaluates the ARCH-IF (already handled by `EvalCond` in
`collectStmts`; ARCH_AARCH64 / ARCH_X86_64 are in `DefaultIfEnv`),
collects the 4 `ADDINCL(GLOBAL ...)` into `d.addInclGlobal`, returns
a `headerOnly` result with `AddInclGlobal` populated, and consumers
get their 4 musl-arch -I flags.

### F5 — ARCH-IF evaluator scope

`EvalCond` ALREADY evaluates `IF (ARCH_AARCH64)` / `IF (ARCH_X86_64)` /
`IF (ARCH_ARM6 OR ARCH_ARM7)` / `IF (ARCH_ARM64 OR ARCH_X86_64)` /
similar. Required ARCH_* identifiers in M2 closure (probed 2026-05-07
against musl/, musl/full, musl/include, cxxsupp/builtins, cxxsupp/
libcxx, util, util/charset):

- ARCH_AARCH64, ARCH_ARM6, ARCH_ARM64, ARCH_ARM7, ARCH_I386, ARCH_X86_64

All 6 are bound in `DefaultIfEnv` (macros.go:272-277). `buildIfEnv`
flips ARCH_AARCH64↔ARCH_X86_64 per host/target instance. **There is
no missing ARCH-IF evaluator work.** The PR-31-D12 `Description` /
`Root cause` comment "(b) Our IF-blind walker at gen.go:343 (`taken
:= v.Then`) never evaluates ARCH_* predicates" is **stale / incorrect**
— gen.go:344-351 calls `EvalCond` and the ARCH bindings are present.

Empirically verified: with auto-PEERDIR added to walker, evaluator
correctly takes the ARCH_AARCH64 branch (target) or ARCH_X86_64 branch
(host) of `contrib/libs/musl/include/ya.make` and routes each branch's
GLOBAL ADDINCLs through the existing PR-31 D04-D06 machinery. **No
new evaluator work needed in PR-32.**

PR-31-D12's defect entry should be updated post-PR-32 to drop the
phantom "ARCH-IF evaluator gap" half of the root cause; the real
root cause is exclusively the missing auto-PEERDIR.

**Realistic L3 lift estimate (probed 2026-05-07):**

- CC pairs total: 3562 paired
- CC pairs L3 byte-exact today: 2941 (82.6% of CC; weighted into all
  3683 paired nodes → contributes to 80.35% L3 overall)
- CC pairs L3-divergent: 621
- Of those 621, only 14 would become byte-exact from the **3 fixes
  alone** (musl-arch -I + `-D_musl_` + `-nostdinc++`).
- 259 are 1-2 tokens away from byte-exact (likely libcxx peer-GLOBAL
  CFLAGS like the second copy of `-DCATBOOST_OPENSOURCE=yes` and
  `-DLIBCXXRT` — these are NOT GLOBAL CFLAGS in libcxx's ya.make
  source but emerge from `_BASE_UNIT`-equivalent injections we do
  not yet model).
- 348 are ≥3 tokens away (asmlib AS, jemalloc/mimalloc, abseil-cpp —
  out of PR-32 scope).

**Realistic L2 lift estimate:**

- 112 of 607 L2-divergent CC pairs differ ONLY in musl/linux-headers
  paths in inputs. Closing musl/include auto-PEERDIR closes all 112
  directly. **L2 → 79.92% + 112/3683 → 82.96% on CC alone.**
- AR/LD aggregation closes too because PR-31's `memberInputs`
  accumulator picks up the per-CC IncludeInputs once the scanner
  resolves them; PR-31-D11 explicitly identifies this as auto-
  resolving downstream of D12.

**Realistic L3 lift estimate:**

- Floor: +14/3683 = +0.4pp → ~80.7%.
- With per-CC scanner fix knock-on (some L3 pairs that miss byte-
  exact today on inputs alone, since L3 compares cmd_args + env
  ONLY): the upper bound is the 410 CC pairs where ref has
  `-nostdinc++` and our doesn't get it (these still need libcxx peer
  routing; see below for caveat).

Brief's claim of "L3 ≥ 90% (lift from 80.35%; peer-GLOBAL CFLAGS
routing)" is **NOT achievable** by PR-32 alone. The `-nostdinc++`
peer-GLOBAL flow lifts at most ~410 pairs, but most of those have
secondary deltas (libcxx peer-GLOBAL CXX warning bundle, libcxxrt
peer-GLOBAL `-DLIBCXXRT`, the libcxx-consumer second copy of
`-DCATBOOST_OPENSOURCE=yes`, etc.) that are out of scope. **Honest L3
projection: 80.35% → 82-85%, NOT 90%.** Pushing past 85% requires
modelling the libcxx/libcxxrt peer-CFLAGS bundle (PR-29 plan called
this D04 territory; orchestrator-deferred to "PR-30 D04" then to
PR-32; reality is it's a separate piece of work with its own bundle
to hand-translate).

The brief's "L2 ≥ 90%" target IS plausible: 79.92% + 3.0pp from D12
closure puts us at 82.92%, then AR/LD member-input aggregation picks
up another ~3-5pp transitively (PR-31-D11 acknowledged ~110 AR pairs
near-correct downstream of D12). **Honest L2 projection: 79.92% →
85-88%, possibly 90% if member-input aggregation closes everything
auto-resolved.**

## Design decisions

### Q1. Flag-driven musl model — the CLI flag mechanism

Add a single `cliDefines map[string]string` field on `genCtx`,
populated from a new CLI flag `--define KEY=VALUE` (repeatable). The
walker consults `ctx.cliDefines["MUSL"]` to drive auto-PEERDIR seeding
and the `-D_musl_` peer-CFLAG injection. Default for M2: seed with
`{"MUSL": "yes"}` so existing tests + the `tools/archiver` reference
match continue to hold without CLI changes.

Public surface:

```go
// gen.go
type genCtx struct {
    ...
    cliDefines map[string]string
}

// main.go
func cmdGen(args []string) int {
    ...
    var defines stringMapValue            // implements flag.Value
    fs.Var(&defines, "define", "ymake-style -D KEY=VALUE; repeatable; default: -DMUSL=yes")
    ...
    g := Gen(TargetCfg, *sourceRoot, *target, defines.toMap())
}
```

`Gen` signature gains `cliDefines map[string]string`; default set by
`gen` subcommand when caller passes nil/empty. Tests construct
explicit `cliDefines` to exercise non-musl builds.

`buildIfEnv` consults `cliDefines` to populate the `MUSL` /
`MUSL_LITE` bools (today these come from `DefaultIfEnv` constants;
moving them to cliDefines lets tests run with `-DMUSL=no` for
forward-coverage).

### Q2. Backward-compat for synthetic tests

Existing tests construct synthetic `ModuleInstance`s with hardcoded
musl paths (e.g. `muslHostInstance("contrib/libs/musl")` in
`cc_test.go:306`). After the refactor, `instance.Path` no longer
drives composer dispatch — `instance.Flags.LibcMusl` does. Two paths:

- **Test helper update**: `muslHostInstance` returns an instance with
  `Flags.LibcMusl = true` set. Path can stay as-is for AR/LD tests
  that emit paths under that prefix (the path is still the *output
  prefix*; just no longer the dispatch key). Each test using a musl-
  path instance gets a one-line update to set the flag.
- **Inferred from path during test construction only**: keep
  `inferFlagsFromPath` setting `LibcMusl = true` for musl prefixes,
  *only as a backward-compat shim during M2*. Document the shim as
  removable in M5+ when ya.make-driven flag inference (parsing
  `NO_PLATFORM` + path-conditional `SET(MUSL no)` etc.) replaces the
  heuristic entirely.

**Recommended: shim approach.** `inferFlagsFromPath` continues to set
`LibcMusl=true` for `contrib/libs/musl[/...]` paths (one-line tweak:
add `fs.LibcMusl = true` next to the existing NO_LIBC/NO_UTIL/
NO_RUNTIME triple). Tests that don't go through `inferFlagsFromPath`
(direct `ModuleInstance{}` construction) explicitly set
`Flags.LibcMusl=true` — this is ~6 test sites total.

The dispatch key changes: `cc.go::EmitCC` uses `instance.Flags.LibcMusl`
instead of path-prefix; `gen.go::includeScannerBasePaths` same;
`gen.go::defaultPeerdirsFor` self-suppression uses the FlagSet bool.
`module.go::inferFlagsFromPath` keeps the path heuristic ONLY as the
backward-compat seed.

### Q3. musl bundle naming

`muslCcIncludes` / `muslCcIncludesX8664` / `muslWarningFlags` /
`muslExtraDefines` stay as named constants. They are the *data of musl
knowledge* — there's nowhere else to put them. The names are honest
because they describe musl-the-libc compile knobs.

What changes is the **dispatch** to those bundles. After the refactor:
`composeMuslCC` becomes selectable via `instance.Flags.LibcMusl &&
!instance.Flags.PIC`; `composeMuslHostCC` via `LibcMusl && PIC`. No
path-prefix awareness in any composer.

Renaming to `cliBundleFor(flag, value, axis)` is gold-plating without
a forcing function; defer.

### Q4. ARCH-IF evaluator scope

**No new evaluator work required.** F5 above proves the existing
`EvalCond` + `DefaultIfEnv` + `buildIfEnv` machinery handles every
ARCH_* identifier the M2 closure uses, including the
`contrib/libs/musl/include/ya.make` `IF (ARCH_X86_64)` /
`IF (ARCH_AARCH64)` blocks. Once auto-PEERDIR fires, the existing
walker's IF-handling correctly takes the right branch per
target/host platform.

PR-31-D12's defect entry will be amended (defects.md edit during the
PR commit) to reflect that the ARCH-IF half of the root cause is
phantom; only the auto-PEERDIR half is real.

### Q5. `when ($MUSL == "yes")` mechanism

Upstream `when` is at the `build/ymake.core.conf` level, NOT in a
ya.make. yatool's parser only handles ya.makes; the `_BASE_UNIT`
when-rules are *implicit* per-TARGET-module rules baked into ymake
itself.

Therefore: **hardcode the auto-PEERDIR + auto-CFLAG rules in
`defaultPeerdirsFor` (or a new sibling helper) gated on
`ctx.cliDefines["MUSL"] == "yes"` AND `!effectiveNoPlatform(d.flags)`**.
Do NOT implement a `when` interpreter — that's wildly out of scope.

The implementation site:

- `defaultPeerdirsFor` already returns the implicit peer set; extend
  with `if cliDefines["MUSL"]=="yes" && !effectiveNoPlatform(...) {
  peers = append(peers, "contrib/libs/musl/include") }`. Uses
  `ctx.cliDefines` (same `*genCtx` plumbing already used for
  `targetPlatformID`).
- A new helper `defaultPeerCFlags(ctx, instance, d)` returns the
  effective auto-injected peer-CFLAGS (just `-D_musl_` for now).
  Threaded into `ModuleCCInputs.PeerCFlagsGlobal` (see Q6).

### Q6. Peer-GLOBAL CFLAGS routing

Mirror PR-31's GLOBAL ADDINCL machinery exactly. New fields on
`moduleData`:

```go
type moduleData struct {
    ...
    cFlagsGlobal     []string  // PR-32: GLOBAL CFLAGS — peer-propagated
    cxxFlagsGlobal   []string  // PR-32: GLOBAL CXXFLAGS — peer-propagated
    cOnlyFlagsGlobal []string  // PR-32: GLOBAL CONLYFLAGS — peer-propagated
}
```

`applyUnknownStmt` for CFLAGS / CXXFLAGS / CONLYFLAGS uses
`splitGlobalModifier` (the existing helper at the top of
`applyUnknownStmt`'s switch arms) and routes the GLOBAL-prefixed args
into the new fields. Replaces the current "drop-to-floor" fix
(PR-29-D01). Non-GLOBAL paths continue to flow into the existing
`cFlags` / `cxxFlags` / `cOnlyFlags`.

There's also a typed-statement story: `*CFlagsStmt` already exists
(parser produces it for `CFLAGS(...)`) and its handling routes via
`d.cFlags`. The typed statement does NOT currently split GLOBAL; the
parser stores `*CFlagsStmt{Flags []string}` un-stripped. The existing
`*CFlagsStmt` handling at gen.go:331-332 takes Flags verbatim.

Two mirrored options for the typed statement:

- **A**: extend `*CFlagsStmt` ADT to carry `{GlobalFlags, OwnFlags}`
  via a per-path `splitCFlagsArgs` helper (mirrors PR-31's
  `splitAddInclPaths`).
- **B**: keep ADT as-is; do the split at `collectStmts`'s
  `*CFlagsStmt` arm.

**Recommended: A.** Symmetric with PR-31's `*AddInclStmt` shape; less
walker-level branching; better for future per-source (like SRC) flag
handling. Same per-path semantics: a flag immediately following the
literal `GLOBAL` keyword is global; later flags reset to own.

For untyped CXXFLAGS / CONLYFLAGS (which come through `*UnknownStmt`):
introduce `*CXXFlagsStmt` / `*CONLYFlagsStmt` typed Stmts in
yamake.go (~30 LOC) so the GLOBAL handling is uniform. Optional but
recommended because `applyUnknownStmt`'s switch is already crowded
and the typed shape carries semantic intent better.

`moduleEmitResult` gains:

```go
type moduleEmitResult struct {
    ...
    CFlagsGlobal     []string  // own + transitive peer-GLOBAL CFLAGS, dedup
    CXXFlagsGlobal   []string
    COnlyFlagsGlobal []string
}
```

Walker accumulates from peers analogously to PR-31's
`PeerAddInclGlobal`. Per-path GLOBAL semantics also mean cross-
language flags don't leak: a `CFLAGS(GLOBAL X)` propagates to peers'
own CFlags slot AND CXXFlags slot AND COnlyFlags slot (per ymake's
CFLAGS-applies-to-both-C-and-C++ rule); a `CXXFLAGS(GLOBAL X)`
propagates only to CXX consumers. Implementation: emitter-level
filtering by source language at the cmd_args slot (cc.go's existing
`isCxx` switch).

`ModuleCCInputs` gains:

```go
type ModuleCCInputs struct {
    ...
    PeerCFlagsGlobal     []string  // applies to all sources (C and C++)
    PeerCXXFlagsGlobal   []string  // applies to C++ sources only
    PeerCOnlyFlagsGlobal []string  // applies to C / .S sources only
    AutoPeerCFlags       []string  // auto-injected (currently just -D_musl_); merged into PeerCFlagsGlobal at slot time
}
```

Slot in cmd_args: empirically determined from `util/charset/all_charset.cpp.o`
probe:
```
cmd_args[100..115]:
  ... -std=c++20 -Wno-everything -DCATBOOST_OPENSOURCE=yes -nostdinc++ ...
  ... -Wno-builtin-macro-redefined ...
```
where `-nostdinc++` lands in the slot AFTER `-std=c++20` and own
CXXFLAGS, BEFORE the `builtinMacroDateTime` / `macroPrefixMapFlags`
trail. Slot order pinned by per-CC byte-exact tests during D-task

## Risks and open questions

**R1 — Test backward-compat surface area.** ~6 test sites use
synthetic `ModuleInstance{}` constructors with musl-prefixed paths.
The shim in `inferFlagsFromPath` covers most; the few that bypass
the heuristic need explicit `Flags.LibcMusl=true` flag bumps. If
tests use `ModuleInstance{Path: "contrib/libs/musl/...",}` literals
without the heuristic, the dispatch flip will silently route them
to non-musl composers. **Mitigation**: grep for direct-literal
ModuleInstance constructions involving musl paths; audit each in
the executor's first pass.

**R2 — `cliDefines` defaulting.** If `cmdGen` defaults to
`MUSL=yes` when no `--define` is passed (preserving M2 behavior),
forgetting to wire the default into `Gen` (e.g. when called from
tests via `Gen(cfg, root, target, nil)`) silently produces
non-MUSL output. **Mitigation**: `Gen` synthesizes `MUSL=yes` when
`cliDefines == nil`; explicit `cliDefines = map[string]string{}`
opts into "no defaults". Documented invariant; tested both ways.

**R3 — `-D_musl_` slot ordering.** Empirical probe shows `-D_musl_`
at cmd_args[82] in `util/_/string/cast.cpp.o`. Must verify the
exact slot is preserved post-injection — wrong slot = byte-mismatch
even though the token is present. **Mitigation**: per-CC byte-exact
test (existing pattern for libcxx in cc_test.go) added for the
auto-CFLAG slot. If reference shows the flag mid-`commonDefines`,
inject it there; if at the boundary, append to that boundary.

**R4 — Header-only LIBRARY musl/include path-handling under
`isRuntimeAncestor`.** `runtimeAncestorPaths["contrib/libs/musl"] =
true` causes `defaultPeerdirsFor` to return nil for any
musl-prefixed path. After D03, `contrib/libs/musl/include` will be
walked as a header-only consumer. The walker will hit
`isRuntimeAncestor` → return nil for its peer set → header-only
result with the 4 ADDINCL(GLOBAL ...) properly aggregated. Need to
verify there's no double-walk loop. **Mitigation**: explicit test
that `genModule(ctx, contrib/libs/musl/include)` returns a stable
result with `AddInclGlobal` populated and no cycle.

**R5 — Auto-CFLAG slot in musl-self CC.** PR-32 must NOT inject
`-D_musl_` (no =1) into musl-self CC nodes — those get `-D_musl_=1`
from `muslExtraDefines`. The `effectiveNoPlatform` gate handles
this (musl-self has the NO_PLATFORM triple → auto-injection
suppressed). Verify in test: `composeMuslCC`'s output for a
musl-self instance has zero auto-injected `-D_musl_` (vs the bundle's
`-D_musl_=1`).

**R6 — `ModuleInstance` map-key churn.** Adding `LibcMusl bool` to
`FlagSet` doesn't change map-keyability (still all-comparable
fields). But it DOES change the memo key: a previously-cached
`{Path: contrib/libs/musl, ..., Flags: {LibcMusl: false}}` pre-
refactor becomes `{..., Flags: {LibcMusl: true}}` post-refactor.
Existing memo would silently miss. **Mitigation**: this is a
one-shot refactor, no incremental migration; all instances flow
through `inferFlagsFromPath` post-shim, so they all flip together.

**R7 — Open question (orchestrator decision required):** **Should
the L3 floor be 82% (honest projection) or 90% (brief target)?** If
the orchestrator wants 90%, PR-32 must extend scope to include the
libcxx peer-CFLAGS bundle hand-translation (the `-nostdinc++` /
`-DCATBOOST_OPENSOURCE` second-copy / `-DLIBCXXRT` propagation
machinery). That adds ~300 LOC and a new flag-bundle named
constant. **Recommend: 82% floor for PR-32; libcxx peer-CFLAGS
bundle as PR-33.** Alternatively rename PR-32 to PR-32a and ship
libcxx bundle as PR-32b. Either way, do NOT promise 90% L3 in PR-32
without scope expansion — the empirical data does not support it.
