## TL;DR

- **53 leaf-divergences** account for **all 599 mismatched fingerprints**
  (53 leaves + 546 ancestor-cascade). Confirmed by ancestor-set union ==
  mismatch set, **599 / 599 = 100%** coverage.
- **41 PY-objcopy** leaves (yapyc3-chunk nodes) cover **44** mismatches.

  causes **92.8 %** of the 599-fingerprint cascade.

Common shape: the leaf node's `inputs[]` is byte-correct (L2=99.90 / L3

the codegen producer node(s)** whose `$(BUILD_ROOT)/...` outputs appear
in `inputs[]`. The producer nodes exist in our graph and on the right
platform; only the dep edge is dropped.

## Cascade scope

```
PY leaves: 41 -> covers  44 mismatches  (3 ancestors cascade)
CC leaves: 11 -> covers 556 mismatches  (cascade through AR/LD)
PR leaves:  1 -> covers   1 mismatch    (no cascade)
TOTAL:     53 -> covers 599 / 599 = 100%
```

The CC cascade dominates because each CC's wrong fingerprint flows
upward through `AR` (object archive) and `LD` (link) ancestors. The
PY-objcopy chunks have shallow upward reach (3 ancestors total) because
their downstream chain is only `AR(libpython3-Lib) → LD(python3 binary)`,
both of which are also affected by the broader CC cascade and thus
share fingerprints downstream.

## Root cause — where the wiring is dropped

`gen.go:344` `resolveCodegenDepRefs` walks `includeInputs` and looks up
`$(BUILD_ROOT)/...` paths in three maps:

- `ctx.enOutputs`          (kv.p=EN)
- `ctx.pbOutputs`          (kv.p=PB, per-platform)
- `ctx.evOutputs`          (kv.p=EV, per-platform)

It **does not** consult any registry for AR / BI / R5 / CF / JV / PR /
PY-yapyc3 producers. Those producers' output paths are registered in the
per-scanner `CodegenRegistry` (`m3_misc.go:876,925,966`, etc.) which
gives the scanner *header-closure-time* visibility — but the
`CodegenRegistry` carries no NodeRef and `resolveCodegenDepRefs` never
consults it.

Net effect: the scanner correctly *includes* the generated path in the
CC's `inputs[]` (so L2 / L3 stay near-perfect), but the *dep edge* that
turns that path into a producer→consumer link is silently dropped.

The PY-objcopy case is the same defect class but in a different code
path: `resource.go:1043-1048` `emitPySrcObjcopy` only adds
`rescompilerLDRef` and `rescompressorLDRef` to the chunk node's
`DepRefs`. The per-chunk `ch.pathInps` list carries the
`$(BUILD_ROOT)/<...>.yapyc3` paths but the yapyc3-producer NodeRefs are
never threaded.

The PR-residue case follows the same pattern in `m3_misc.go:810`
`EmitPR`: only the tool-LD `depRefs` are wired; the EN producer reached
through `inputs[]` is not.

## Single PR to close

**PR-L0-cascade-close — extend the codegen producer NodeRef registry**

Scope (one PR):

1. Add producer NodeRef maps to `genCtx`, mirroring `enOutputs` /
   `pbOutputs` / `evOutputs`:
   - `arOutputs`  map[`codegenOutputKey`]NodeRef — for `.pyc.inc` (AR-archiver)
   - `biOutputs`  map[`codegenOutputKey`]NodeRef — for `buildinfo_data.h`
   - `r5Outputs`  map[`codegenOutputKey`]NodeRef — for `.rl.tmp`
   - `cfOutputs`  map[`codegenOutputKey`]NodeRef — for generated `.cpp` / `.h` from CONFIGURE_FILE
   - `jvOutputs`  map[`codegenOutputKey`]NodeRef — for ANTLR4 `.cpp` / `.h`
   - `prOutputs`  map[`codegenOutputKey`]NodeRef — for RUN_PROGRAM `.cpp`
   - `pyOutputs`  map[`codegenOutputKey`]NodeRef — for `.yapyc3` per-source bytecode
2. Populate each map at the corresponding emitter site (AR/BI/R5/CF/JV/PR
   in `m3_misc.go`; `.pyc.inc` AR in `m3_misc.go:1446`; the per-source
   yapyc3 PY in `py.go`).
3. Extend `resolveCodegenDepRefs` (`gen.go:344`) to consult all seven
   maps in addition to EN/PB/EV.
4. For the PY-objcopy chunk path, after `chunkPySrcEntries`, walk
   `ch.pathInps` and append the matching pyOutputs NodeRef into the
   chunk node's `DepRefs` (analogue of `resolveCodegenDepRefs` but
   inlined in `resource.go:1042-1048`).
5. For the PR node (`m3_misc.go:810`), append the cross-EN dep refs
   resolved from its `inputs[]` to `depRefs`.
