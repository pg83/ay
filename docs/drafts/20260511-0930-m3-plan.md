# M3 plan — `devtools/ymake/bin` full byte-exact closure (sg2.json)

**Date:** 2026-05-11

**Acceptance criterion (verbatim from `GOALS.md` §Initial-target-subgraphs M3
and §Acceptance-target-primary):**

> **M3**: `devtools/ymake/bin` (the ymake binary itself — meta-bootstrap;
> 8750 nodes; reference at `/home/pg/monorepo/yatool_orig/sg2.json`).
> Same `--musl --target-platform=default-linux-aarch64 --sandboxing` flag
> set; produced by `srun2.sh`. Same L0..L4 = 100% bar; same gen ≤ 5 s
> hard gate (no relaxation despite ×2.35 node growth).

Plus: M1 (`build/cow/on`) and M2 (`tools/archiver`) byte-exact pins
preserved at all four levels; M2-L4 normalizer + comparator reused
unchanged on sg2.json.

**One-line approach:** Add 9 new emitter kinds (PY, EN, PB, PR, EV, JV,
CF, R5, BI), 13 new host PROGRAMs (the M3-extra LDs), broader
`foreign_deps` shape coverage (1 → 739 nodes), and verify (a) the
existing CC/AR/AS/LD/JS/R6/CP machinery scales to 8750 nodes without
unfamiliar cmd_args variants, (b) the M2-L4 normalizer projects the
larger fixture without algorithmic change, (c) gen wall stays under
5 s. All new surfaces close in one milestone (M3), not split. A
parallel perf track restarts only if the M3 baseline `gen` wall
exceeds 1 s; the 5 s gate is the absolute floor.

**"Out-of-graph dangling" entries** (12 distinct UIDs across 1297
references) are foreign_deps that reference a UID not present
anywhere in sg2.json's graph[]. These are equivalent to M2's
`OsvsEu9xnOqLNXi3INZ9CQ` (yasm) and `XO1d8CLk3qDKv0XQTlDKmQ` (host

that is itself outside the printed subgraph. Important: one of the
12 unresolved UIDs is `Ze_eMOLqyMsa6WlbbMhgvQ` — **the M2 archiver
LD UID**. ymake stamps that into the M3 graph as a dangling
reference even though the M3 graph does include a `tools/archiver`
LD (with a different uid `qkF8lsq4xdHFcID-Y1vUSA`). Cross-platform
duality: both the target-axis and host-axis AR nodes for the same
pyc carry `foreign_deps.tool` but reference *different* archiver
UIDs (one M2-shape, one M3-shape).

This duality is a UID derivation quirk of ymake that we do not need
to reproduce: per `GOALS.md` L4 normalization, the normalizer
re-UIDs the graph through OUR fingerprint cascade, which will
unify these references. For pre-normalization L0..L3, the
comparator already tolerates `foreign_deps` UID mismatches —
verify in PR-M3-A.

**Emitter design:**

1. **In-graph foreign_deps** (114 references / 11 distinct producer
   UIDs): the consumer emitter (`py.go`, `as.go`, `en.go`, etc.)
   acquires the producer's `NodeRef` via the host-walk machinery
   (PR-23's `instance.WithHost(cfg)` model) and populates
   `node.ForeignDepRefs["tool"] = []NodeRef{producerRef}`. Finalize
   resolves to UID strings. Each new emitter file gets a
   `hostToolRef NodeRef` parameter (mirrors M2's `ragel6LD NodeRef`
   per D31 / PR-23).
2. **Out-of-graph dangling foreign_deps** (1297 references / 12
   distinct UIDs): emit the literal UID string via
   `node.ForeignDeps["tool"] = []string{...}` (bypassing NodeRef
   resolution). This is the M2 D20 stub mechanism, scaled. 12
   stub UIDs declared as constants in a new `host_stubs.go` or
   in the per-kind emitter file (depending on which kind owns
   them). **All 12 stub UIDs must be enumerated explicitly** —
   no programmatic derivation; they come from ymake's own UID
   algorithm which we are not reproducing.

## 5. Performance budget

| Metric | M2 measured | M3 projected | M3 gate |
|---|---:|---:|---:|
| Total nodes | 3730 | 8750 (×2.35) | — |
| `gen` wall (warm) | 0.89-0.97 s | ~2.1-2.3 s (linear) | **≤ 5 s** |
| Memory peak | ~50 MB | ~120 MB (linear) | (no gate) |

Projected M3 gen wall at the M2 ms/node rate ≈ 2.2 s. Headroom
relative to the 5 s hard gate ≈ 2.8 s (~120% margin). Risks that
could erode the margin:

1. **PY emission (667 nodes)** — each PY node has a deeply transitive
   foreign_deps list and may exercise host-walk recursion
   more heavily than M2's R6/AS host-walks. Measure first; tune if
   gen > 3 s.
2. **EN emission (21 nodes) — large `inputs[]` sets** — 954-1500
   transitive headers per node. If the scanner re-walks rather than
   memoizing, throughput could fall. Memoization is already in
   PR-34b's memo-key minimization; verify it covers the EN case.
3. **CC scaling (3725 → 7621)** — strictly linear, no risk.
4. **DFS preorder graph reorder (PR-L4-C/06)** — O(N+E); 8750 nodes
   ≈ 8 ms additional cost; negligible.

**Perf agent dispatch rule (from MEMORY.md `perf_continuous_pipeline.md`,
2026-05-11):** spawn a perf-optimization subagent in parallel with
M3 closure work **only if** `./yatool gen --target devtools/ymake/bin`
wall exceeds 1 s on the M3 baseline. The 5 s acceptance gate
remains in force regardless. Realistic prediction: M3 trips the
1 s trigger immediately because it's already projected at ~2.2 s;
the perf pipeline restarts as a parallel track to the closure PRs
(PR-M3-perf-A onwards).

## 7. Risks / open questions

### 7.1 PROTO_LIBRARY multimodule semantics

The biggest structural unknown. `proto.conf:916` defines
`multimodule PROTO_LIBRARY` — a single ya.make declaration that
spawns BOTH a C++ library (PB nodes + linked AR) AND a Python
sub-module + Java sub-module + Go sub-module per flag selection.
For M3, we only need the C++ branch.

**Open question for user (if not resolvable from upstream code):**
Should our M3 parser model `multimodule` as a distinct concept (a
new yamake.go AST node `MultimoduleStmt`) or coerce it into a
LIBRARY-with-special-SRCS? The latter is simpler but bleeds
semantics; the former is principled but adds ~150 LOC. **Planner
default: coerce for M3; promote to MultimoduleStmt in M5 when
needed for PROTO_LIBRARY's Python/Java branches.**

### 7.2 PY3_LIBRARY language axis

PY3_LIBRARY inherits from `_LIBRARY` (proto.conf:738 actually
python.conf:738). It emits C++ AR for its glue code (`__res.cpp`,
`sitecustomize.cpp`) AND PY nodes for its `PY_SRCS`. The M2
`Language` axis (D35) has only `LangCPP`; whether we need a `LangPY`
or whether PY emission flows through LangCPP-with-PY-source-rules
needs an architectural call. **Planner default: flow through
LangCPP — `PY_SRCS` is a per-source rule (like `SRCS(*.cpp.in)`
triggers CF), not a separate Language.**

### 7.3 USE_PYTHON3 implicit PEERDIRs

`USE_PYTHON3()` at `devtools/ymake/bin/ya.make:9` implicitly adds
`contrib/tools/python3` and `contrib/tools/python3/lib2/py` to the
PEERDIR list (similar to ALLOCATOR adding `library/cpp/malloc/*`).
The implicit-peer machinery in `macros.go` needs a new branch for
USE_PYTHON3. Investigate during PR-M3-A; expected ~30 LOC.

### 7.4 The 12 dangling foreign_deps UIDs

Per §4.1, 12 distinct UIDs appear in `foreign_deps.tool` but are not
present in sg2.json's `graph[]`. These are stubs that ymake stamps
into the printed subgraph. We cannot derive them — they must be
listed empirically. Risk: if ymake re-runs (`srun2.sh` regeneration)
produces different UIDs, the stubs drift. **Mitigation:** check in
the 12 stubs as named constants in a new `host_stubs_m3.go`, with a
comment pointing at the regeneration command and the date.

### 7.5 PEERDIR cycle count

M2 tolerated ~7 PEERDIR cycles. M3's ymake closure (which transitively
PEERDIRs `library/cpp/eventlog`, `library/cpp/getopt`, etc., each
of which has multi-level PEERDIRs) likely has more cycles. Tolerate
and emit a stderr diagnostic per the M2 convention (see
`gen.go::genCtx` cycle handling). If the cycle count exceeds 25
(arbitrary threshold), investigate for systematic mis-declaration in
our parser; otherwise proceed.

### 7.6 The M2 archiver UID `Ze...` appearing in M3 foreign_deps

ymake emits the M2-shape archiver UID in 4 M3 AR nodes'
`foreign_deps.tool` (see §4.1). This is **not** the same UID as the
M3 archiver LD node (which has uid `qkF8lsq4xdHFcID-Y1vUSA`). Our
M3 emitter, going through the OUR fingerprint cascade, will produce
yet a third UID for the archiver and use that consistently for ALL
references. After the M2-L4 normalizer re-UIDs sg2.json through our
fingerprint, these references unify. **Pre-normalization:** the L0..L3
comparators tolerate foreign_deps UID mismatch (only the multiset is
compared at L2). **Post-normalization (L4):** the normalizer rewrites
both `Ze...` and `qkF8...` to OUR archiver UID. No emitter change
required; verify during PR-M3-G.

### 7.7 Regenerating sg2.json from upstream

Per the brief: sg2.json is large (356 MB) and regenerating from
upstream via `srun2.sh` is slow. The reference is checked-in (well,
present on disk at `/home/pg/monorepo/yatool_orig/sg2.json`). Do not
regenerate during M3 work. If the upstream tree changes
(`yatool_orig/` rev bump), regenerate as a one-off step gated on a
clear user prompt.

### 7.8 Per-CC flag bundle taxonomy explosion

PR-M3-F may face a long tail of small CC flag bundles. The 84
distinct fingerprints observed in REF (§3.1) bound the work above;
realistic deduplication likely brings that to ~30-50 distinct
bundles after sharing the "warning-free contrib" / "with-warnings
contrib" / etc. base classes. If bundle work approaches 800 LOC and
covers < 95% of CC nodes, **plan B**: shift the remaining bundles to
a PR-M3-Fbis follow-up, document the L3 deviation in `tasks.md`
Completed.

### 7.9 RECURSE_FOR_TESTS

ymake closure includes test subdirs via `RECURSE_FOR_TESTS(test)`
(e.g. `library/python/runtime_py3/ya.make:58-60`). M2 IGNORES this
macro. Verify M3 also ignores; otherwise the closure balloons by
~1000+ nodes that aren't in sg2.json's graph. (sg2.json was produced
without `--run-tests`, so RECURSE_FOR_TESTS should not appear.)

### 7.10 Cross-platform `module_tag=cpp_proto` AR nodes

37 AR nodes have `module_tag=cpp_proto` (vs 0 in M2). These are AR
archives that consume PB output (.pb.cc generated files). The CC
nodes for those .pb.cc files inherit the cpp_proto tag in their
target_properties. The AR aggregator must understand which sources
feed cpp_proto AR. Implementation hint: the AR's source list mixes
hand-written `.cpp` and generated `.pb.cc` — both compile to `.o`
identically, but the AR's `target_properties.module_tag` discriminates.
Wire through `module.go::ModuleType` extension.

## 8. Cross-cutting concerns

### 8.1 Naming

PRs prefixed `PR-M3-<letter>` (A, B, C, ...). Sub-tasks within a PR
labelled `PR-M3-<letter>/<NN>` (PR-M3-G/01, etc.) following M2's
PR-35a..z convention. Defects use `PR-M3-<letter>-D<NN>`
(PR-M3-B-D01, etc.). Resolved entries gain `Fix:` lines per
CLAUDE.md defect schema.

### 8.2 Worktree partition

When dispatching parallel executors across PR-M3-B..E, partition by
file:

| PR | Files owned |
|---|---|
| PR-M3-B | `py.go`, `py_test.go`, parser PY_SRCS/USE_PYTHON3 additions |
| PR-M3-C | `pb.go`, `ev.go`, `pb_test.go`, `ev_test.go`, parser PROTO_LIBRARY |
| PR-M3-D | `en.go`, `en_test.go`, parser GENERATE_ENUM_SERIALIZATION |
| PR-M3-E | `pr.go`, `r5.go`, `jv.go`, `cf.go`, `bi.go` + tests |

Each parallel executor uses `isolation: "worktree"` per CLAUDE.md
parallelism rules. Shared files (`gen.go`, `yamake.go`, `macros.go`)
are touched by PR-M3-A (serial) and possibly PR-M3-F (serial after
B..E). Within PR-M3-E, the 5 emitter files partition cleanly across
sub-worktrees if dispatched as parallel sub-executors.

### 8.3 L0..L3 invariance across PR-M3-x

Every PR-M3-x must keep M1 (`build/cow/on`) and M2 (`tools/archiver`)
byte-exact at all 4 levels. The orchestrator's review-loop discipline
runs the comparator against both targets after each PR's executor
returns:

```bash
./yatool gen --target build/cow/on --out /tmp/m1.json
./yatool compare --level=3 /tmp/m1.json /home/pg/monorepo/yatool_orig/g.json
./yatool gen --target tools/archiver --out /tmp/m2.json
./yatool compare --level=3 /tmp/m2.json /home/pg/monorepo/yatool_orig/sg.json
./yatool gen --target devtools/ymake/bin --out /tmp/m3.json
./yatool compare --level=3 /tmp/m3.json /home/pg/monorepo/yatool_orig/sg2.json
```

Any drop from 100% on M1/M2 is a blocker for the PR.

### 8.4 Dependency on M2-L4 closure

M3 depends on PR-L4-A (normalizer) for the L4 fixture pipeline.
PR-L4-A is already `[~] in progress` per `tasks.md:82`. M3 closure
PRs (PR-M3-A..F) can land *before* L4 is done — they only require
L0..L3. PR-M3-G (L4 closure on M3) requires PR-L4-A and PR-L4-B to
have landed. Sequencing:

```
M2-L4: PR-L4-A → PR-L4-B → PR-L4-C → PR-L4-D       (in flight)
M3:    PR-M3-A → {PR-M3-B || PR-M3-C || PR-M3-D || PR-M3-E} → PR-M3-F → PR-M3-G
              ↓                                                              ↑
              └─ PR-M3-perf-A → PR-M3-perf-B (conditional, parallel) ────────┘
```

PR-M3-G unblocks only after PR-L4-A and PR-L4-B are merged.

### 8.5 STYLE.md and throw.go discipline

All new files (py.go, pb.go, etc.) inherit M2's STYLE.md formatting
(blank-line discipline, no `if err != nil` chains, Throw/ThrowFmt for
internal errors). Reviewers check explicitly per D17/D18.

### 8.6 No path-prefix dispatch (D40 lock-in)

Any new flag-bundle or behavioral dispatch in `flags.go::cflagsForModule`
must be flag-driven (CLI `-D` or per-module declared), not path-prefix
matching. The `grep -E '(HasPrefix|HasSuffix|Contains).*"contrib/libs/...'`
production-code audit per GOALS.md §DoD item 7 stays clean.
