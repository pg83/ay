# T-5 — Promote sg7 out of XFAIL

## Goal

GOALS.md: `./dev/validate.py должен перестать XFAIL по существу` (validate.py must
stop XFAILing substantively). sg2–sg6 already gate byte-exact; sg7 is the last
case still marked XFAIL in `dev/validate.py`'s `CASES` table. T-2 and T-3 (this
ticket's deps) landed the deps-only root causes on trunk. Promote sg7 to a normal
gating case — but only because byte parity now holds, never by changing policy to
mask a residual difference.

## Mechanism / precondition verified (not a hypothesis)

This is a validation-policy promotion, not a graph-codegen change, so the
"upstream mechanism" here is the byte-exact equality of the residual normalized
graph. Verified on this host before touching policy:

- `dev/gen_sg7.sh` generated `sg7.our.json` (3.5 GB), exit 0, **empty stderr**
  (no unresolved-include warnings under `--keep-going`).
- `ay dev dump normalize | ay dev dump sort` on ours (faithful) and on the ref
  (`--ref-graph`) produced two JSONL files of identical size (3436828377 bytes,
  72284 nodes each).
- `cmp -s` of the two normalized files returns 0 → **byte-exact**.

So the promotion does not mask anything: sg7's normalized our/ref graphs compare
equal.

## Changes

1. `dev/validate.py` — `CASES`: flip sg7's `xfail` field `True → False`. It then
   takes the gating byte-compare path: prints `[sg7] OK` on parity, `[sg7] FAIL`
   + `status=1` on any divergence. No XFAIL line is emitted for sg7. (The `"auto"`
   self-promoting value is not used: a normal gating case is the explicit ask and
   it fails loudly on regression instead of silently dropping back to XFAIL.)

2. `dev/gen_sg7.sh` — drop `--keep-going` (and its now-stale comment). The flag's
   own comment says "Drop it once resolution is complete"; resolution is complete
   (empty stderr, byte parity). Keeping it would let a future unresolved include
   warn-and-continue into a divergent graph instead of aborting — i.e. it would
   mask a remaining difference, contrary to this ticket's intent. Without it sg7
   matches sg6 (no `--keep-going`). The byte-compare gate remains the primary
   guard regardless.

3. `dev/test_acceptance.py` — add a test that pins the promotion: load
   `dev/validate.py`'s `CASES` and assert sg7's xfail flag is `False` (gating).
   This fails on current trunk (sg7 is `True`) → test-first. The existing
   acceptance tests already cover "new FAIL rejected" and "all-SKIP → ACCEPT"
   generically; sg7 being gating routes a sg7 divergence through the same
   `new FAIL appeared` path, and a host without `/home/pg/monorepo/4` still emits
   `[sg7] SKIP` (CASES SKIP logic is unchanged).

## Expected effect on the gate

- `./dev/validate.py` on this host: `[sg7] OK`, exit 0, **no XFAIL line** for any
  evaluated case. Generation time still printed (`[sg7] gen time …s (no budget)`).
- Acceptance OLD(trunk: sg7 XFAIL) → NEW(branch: sg7 OK): OK count rises by 1,
  XFAIL count drops by 1, no new FAIL → ACCEPT.
- sg2–sg6 gating behavior unchanged (their CASES rows untouched; no shared code
  path altered).

## Verification

1. New test fails on current code, passes after the CASES flip.
2. `python3 dev/test_acceptance.py` all green.
3. Full gate `./dev/validate.py .out/digger-validate` PASSES: every present case
   `OK`, no XFAIL line, exit 0, sg5 within its gen-time budget.
