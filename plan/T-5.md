## Root-cause taxonomy

### A. AR ordering: pb.cc.o after h_serialized (2 AR nodes)
- **Nodes**: `libydb-core-tablet_flat.a`, `libydb-library-breakpad.global.a`
- **Upstream mechanism**: In upstream ymake, `*.pb.cc.o` files appear AFTER `h_serialized.cpp.o` in the AR object list. The current `reorderARMembers` puts `.pb.cc.o` (not `.ev.pb.cc.o`) in `regularSrcs` which appears before `hSerSrcs`.
- **Fix**: Add `pbCCSrcs` bucket for `.pb.cc.o` (excluding `.ev.pb.cc.o`) AFTER `hSerSrcs` in reorderARMembers output order.
- **File**: `gen.go:reorderARMembers`

### B. Global AR objcopy ordering: always first (1 global AR node)
- **Node**: `libydb-library-breakpad.global.a`
- **Upstream mechanism**: Upstream ALWAYS places objcopy resource objects before regular GLOBAL SRCS in global archives, regardless of their declaration order in ya.make. The `resourceBeforeGlobalSrcs` check uses declaration order which fails when SRCS(GLOBAL) appears before RESOURCE.
- **Fix**: Remove the `resourceBeforeGlobalSrcs` condition; always move objcopy objects to front of global archive (the `moveTailNodeRefsToFront` call should be unconditional when both objcopy refs and globalSrcs exist).
- **File**: `gen.go` (around line 1904)

### C. EN source resolution: missing d.srcDir (1 EN + cascade)
- **Node**: `interface.h_serialized.cpp` (purecalc_no_pg_wrapper)
- **Upstream mechanism**: GENERATE_ENUM_SERIALIZATION(interface.h) in a module with SRCDIR(yql/essentials/public/purecalc/common) must resolve `interface.h` relative to the SRCDIR. `emitEnumSrcs` passes `scanIn.SrcDir` (nil) to `resolveEnumHeaderInput` instead of `d.srcDir`.
- **Fix**: Pass `d.srcDir` (not `scanIn.SrcDir`) to `resolveEnumHeaderInput`. Also set `scanIn.SrcDir = d.srcDir`.
- **File**: `emit_en.go:emitEnumSrcs`

### D. Ragel extra inputs: over-inclusive closure (2 R6 nodes)
- **Nodes**: `httpfsm.rl6.cpp`, `parser.rl6.cpp` (ymq)
- **Upstream mechanism**: The R6 node's inputs should only include the ragel binary, the .rl6 source, ragel-native imported .rl6 files, and C/C++ includes from the .rl6's LOCAL include bucket. Our scanner follows includes from the .rl6's LOCAL bucket which includes headers from ragel-native-imported files' local bucket too (via some transitive path). Specifically, `util/datetime/parser.rl6` includes `<util/generic/ymath.h>` as a C include which bleeds into the httpfsm.rl6.cpp closure. Also, `ymq/http/parser.rl6` includes `params.h` â†’ `sqs.pb.h` (BUILD-generated), which should be excluded from RL6 inputs.
- **Fix**: Filter BUILD-root paths from `rl6Closure` before passing to `EmitR6` â€” keep only source-root ($(S)) files and the ragel binary. The build-generated .pb.h files should not be RL6 inputs.
- **Status**: Needs investigation - the exact chain that brings ymath.h in is unclear (may be from parsedIncludesLocal of files transitively reached)

### E. Flatc/CP inputs: missing wrapper and .fbs sources (2 CP nodes)
- **Nodes**: `stream_decoder.cpp`, `yt_codec_bc.cpp.16.bc`  
- **Upstream mechanism**: CP (WITH_CONTEXT) node for stream_decoder.cpp should include `cpp_flatc_wrapper.py` and the `.fbs` source files as inputs. These come from the `flatcCCExtraInputs` mechanism, but the CP code calls `keepOnlySourceVFS` which drops build-generated `.fbs.h` files before `flatcCCExtraInputs` can extract the corresponding source `.fbs` files and wrapper.
- **Fix**: In `emit_copy.go`, call `flatcCCExtraInputs` on the closure BEFORE `keepOnlySourceVFS`, to extract wrapper + .fbs sources from build-generated `.fbs.h` entries. Then add those source-root extras to the final closure (they survive `keepOnlySourceVFS`).
- **File**: `emit_copy.go`

### F. ONE_LEVEL ADDINCL ordering (5+ CC nodes)
- **Nodes**: `login.cpp.o`, `credentials.cpp.o`, `from_file.cpp.o`, `jwt_token_source.cpp.o`, `jwt.cpp.o`, etc.
- **Upstream mechanism**: `ADDINCL(ONE_LEVEL path)` flags should appear AFTER global peer ADDINCL flags, not before. Currently we add `AddInclOneLevel` to `d.addIncl` (OwnAddIncl) which comes before `PeerAddInclGlobal` in the CC command assembly.
- **Fix**: In `gen.go`, instead of appending `AddInclOneLevel` to `d.addIncl`, append them to a new field (or append directly to `PeerAddInclGlobal` after the global peer addincl are collected) so they appear AFTER the peer global addincl in the CC command.
- **File**: `gen.go`, `emit_cc.go`

### G. Proto-split ANTLR4: wrong codegen inputs (~80 CC nodes)
- **Nodes**: SQL v0/v1 lexer/parser, proto_split_antlr4 CC nodes, and all CC nodes that include the generated headers
- **Upstream mechanism**: `v1_proto_split_antlr4` module uses a 3-step chain: ANTLR4 â†’ RUN_PROGRAM(protoc) â†’ RUN_PYTHON3(multiproto). The final SRCS only include `pb.code*.cc`. CC nodes that include `pb.classes.h` or `pb.main.h` should reference the multiproto RUN_PYTHON3 output (`pb.code0.cc`) as the codegen dep, NOT the RUN_PROGRAM outputs (`pb.h`, `pb.cc`). Our codegen registry registers `pb.h` and `pb.cc` under the RUN_PROGRAM node, so CC includes of `pb.classes.h` (which presumably includes `pb.h`) resolve to the RUN_PROGRAM node and add `pb.cc` as a peer output.
- **Fix**: Complex. Need to understand how `pb.classes.h` is registered and why ref resolves it to `pb.code0.cc`. Possibly the multiproto RUN_PYTHON3 step registers `pb.classes.h` and `pb.main.h` in the codegen registry (associated with the multiproto node, whose first output is `pb.code0.cc`). If so, the fix is in how `emitRunPython` registers its OUT_NOAUTO outputs.
- **Requires deeper investigation**

### H. mkql_computation_node_codegen.h.txt (6 CC nodes)
- **Nodes**: kqp compute actor/state/runtime nodes
- **Upstream mechanism**: `mkql_computation_node_codegen.h` is produced by COPY_FILE(TEXT) from a `.h.txt` source. When a CC node in a DIFFERENT module includes this header, the source `.txt` file should appear as an input. The `withContextSourceExtras` function only covers COPY_FILE(TEXT) from the SAME module; cross-module TEXT copies need to be tracked differently.
- **Requires investigation**

### I. Bison .y inputs (2 CC nodes)
- **Nodes**: `re_parser.y.cpp.o`, `re_lexer.cpp.o`
- **Upstream mechanism**: `re_parser.y` bison file generates `re_parser.y.h`. When `re_lexer.cpp` is compiled, its include closure reaches `re_parser.y.h` (build-generated). The `re_parser.y` SOURCE file should appear as an input to `re_lexer.cpp.o`. Additionally, `re_parser.y.cpp.o` is missing flags `-Wno-deprecated-copy` and `-Wno-unused-but-set-variable`.
- **Requires investigation**
