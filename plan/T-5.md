# T-5 plan: parse-time include env must carry MODDIR and fold DEFAULT

## Symptom (reproduced)

`./dev/validate.py` baseline, sg7 pair
`$(B)/yabs/models_services/feature_store/build_configs/configs/caesar/features.gen.cpp`:

```
[field cmds differs]
  -ours +${FS_CODEGEN_MODULES_INCLUDED}
  -ref  +yabs.models_services.feature_store.feature_sets.models.ads_rsya_ml_selection_rank.caesar
  ... (full module list, in modules.lst source order)
```

Our RUN_PROGRAM command keeps the literal `${FS_CODEGEN_MODULES_INCLUDED}`;
the reference expands it to the concrete module list. Everything else in the
command (tool path `codegen_bin/fs_codegen`, `--bindir`, `--moddir`) is already
correct.

## Upstream mechanism

The caesar `ya.make` chain:

- `ya.make` -> `INCLUDE(build_config_name.inc)` sets `FS_CODEGEN_BUILD_CONFIG_NAME`,
  then `INCLUDE(${ARCADIA_ROOT}/.../codegen/build_config.inc)`.
- `build_config.inc` opens `LIBRARY()` then `INCLUDE(.../run_codegen.inc)`.
- `run_codegen.inc`:
  - `IF(MODDIR MATCHES .../configs) DEFAULT(FS_CODEGEN_CONFIG_PATH ${ARCADIA_ROOT}/${MODDIR}) ...`
  - `INCLUDE(${FS_CODEGEN_CONFIG_PATH}/modules.lst)`
  - `modules.lst` does `SET(FS_CODEGEN_MODULES_INCLUDED <list>)`
  - `SET(GENERATOR_ARGS ${FS_CODEGEN_MODULES_INCLUDED} --bindir ${BINDIR} --moddir ${MODDIR})`
  - `RUN_PROGRAM(.../codegen_bin ${GENERATOR_ARGS} ...)`

ymake resolves an `INCLUDE` path by `EvalExpr` against `Context->Vars()`
(makefile_reader.cpp:49-60). `Context->Vars()` always carries `MODDIR`
(the directory of the ya.make that opened the module) and the accumulated
`SET`/`DEFAULT` bindings. So `${FS_CODEGEN_CONFIG_PATH}` resolves to
`${ARCADIA_ROOT}/yabs/.../caesar`, the `modules.lst` include is read, its
`SET` enters the variable scope, and `GENERATOR_ARGS` expands the list.

## Gap in ay

Parse-time include resolution (`IncludeState.env`, threaded through inline
`INCLUDE` splicing in `yamake.go`) has two omissions vs `Context->Vars()`:

1. It is a bare `newEnvironment()` -- `MODDIR` is never seeded. So
   `${ARCADIA_ROOT}/${MODDIR}` cannot be expanded at parse time.
2. `parseMacroInto` folds only `SET` into the parse env; `DEFAULT` is not
   folded. So `FS_CODEGEN_CONFIG_PATH` stays unbound even with `MODDIR` present.

Result: `INCLUDE(${FS_CODEGEN_CONFIG_PATH}/modules.lst)` keeps an unresolved
`${...}` and falls through the silent missing-file skip (`expandOneInclude`).
`modules.lst` is never spliced, `FS_CODEGEN_MODULES_INCLUDED` is never set, and
the later `${FS_CODEGEN_MODULES_INCLUDED}` reference survives verbatim into the
command. (`peerdirs.lst` / `output.lst` already resolve because their path uses
only `SET`-defined vars.)

This is a generic make-include variable-propagation gap, not feature-store
specific: any included file that `SET`s a list later consumed by a command
argument, reached through a `${MODDIR}`/`DEFAULT`-derived include path, hits it.

## Change

`yamake.go` only:

1. `parseInternalWithStack`: seed the fresh `IncludeState.env` with
   `MODDIR = pathDir(name)` (the top-level file's directory). Do **not** seed
   `ARCADIA_ROOT`; the include-path resolver strips a literal `${ARCADIA_ROOT}/`
   prefix, so leaving it unexpanded is what routes the path to the source root.
2. `parseMacroInto`: fold `DEFAULT` into the parse env the same way `SET` is
   folded (set only when the name is not already bound, matching DEFAULT
   semantics).

No new data path, no feature-store special case. Parse stays a pure function of
(file, file-dir); `MODDIR` is the parsed file's own directory, so include
resolution remains context-independent and the per-file splice is unchanged
across modules.

## Test first

`yamake_test.go`: a memfs reproducing the nested pattern -- `ya.make` opens a
module and includes a `run.inc` that `DEFAULT`s a config path from
`${ARCADIA_ROOT}/${MODDIR}`, includes `${CONFIG_PATH}/modules.lst` (which
`SET`s a list), then uses the list in a `RUN_PROGRAM` argument. Drive
`parseFile` + `collectModule` and assert the collected `runPrograms[0].Args`
contain the expanded list entries, not the literal `${...}`. Must fail on
current code (literal survives), pass after the change.
