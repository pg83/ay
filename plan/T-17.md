# T-17 plan: flatc/flatc64 tool SBOM component name (REALPRJNAME fallback)

## Hypothesis (reproduced, not assumed)

The paired-but-divergent `$(B)/contrib/libs/flatbuffers64/flatc/flatc` binary
differs from the reference only in its `cmds` and `inputs` SBOM reference:

```
[field cmds differs]
  -ours +$(B)/contrib/libs/flatbuffers64/flatc/.CPP.component.sbom
  -ref  +$(B)/contrib/libs/flatbuffers64/flatc/flatc.CPP.component.sbom
[field inputs differs]  (same)
```

Our side emits the program's own SBOM component with an **empty REALPRJNAME**
(`.CPP.component.sbom`), while upstream names it by the program basename
(`flatc.CPP.component.sbom`). The empty-named node and the basename node are the
two extra differing outputs the ticket lists. The same defect hits every
from-source `PROGRAM()` declared with no name argument:

```
$(B)/contrib/libs/flatbuffers/flatc/.CPP.component.sbom      (ref: flatc.CPP...)
$(B)/contrib/libs/flatbuffers64/flatc/.CPP.component.sbom    (ref: flatc.CPP...)
$(B)/contrib/libs/luajit_21/compiler/.CPP.component.sbom     (ref: compiler.CPP...)
```

## Upstream mechanism

In ymake, every module has `REALPRJNAME`, which defaults to the **module
directory basename** when no explicit module-name argument is given. The program
binary, its `.ldcref`/`.map`, and its `<REALPRJNAME>.<LANG>.component.sbom`
(gen_sbom.py `--path`/output) all derive from this one resolved name
(ymake.core.conf:910-922, sbom.conf). For `PROGRAM()` with no argument under
`contrib/libs/flatbuffers64/flatc`, `REALPRJNAME = flatc`.

## Defect in our code

`programBinaryName` (gen.go:610) returns `""` for an unnamed `PROGRAM()`. Its own
comment says it "falls through to the module-dir basename default below", but the
basename fallback actually lives, duplicated, in two *consumers*:
`emitLD` (emit_ld.go:94-95) and `lDOutputPath` (emit_ld.go:218-219). The third
consumer — `emitSbomComponent` (gen.go:2403, resources.go:448) — has **no
fallback**, so it builds `<dir>/.<LANG>.component.sbom` from the empty name.

## Fix (centralize REALPRJNAME, net-negative)

Make `programBinaryName` the single source of truth: when there is no name
argument (or no moduleStmt), return `lastPathComponent(instance.Path.rel())`.
Then the duplicated fallbacks in `emitLD` and `lDOutputPath` become dead and are
removed. All five call sites already feed their binary name from
`programBinaryName`, so the binary/ldcref output paths are unchanged (still
basename); only the SBOM component name is corrected from `` to the basename.

Files touched:
- `gen.go`: `programBinaryName` no-arg / nil branches return the dir basename.
- `emit_ld.go`: drop the now-redundant basename fallbacks in `emitLD` and
  `lDOutputPath`.
- `gen_test.go`: update `TestProgramBinaryName_Py3ProgramBinArgWins` (it
  currently asserts the no-arg result is `""`); add a focused regression
  asserting an unnamed `PROGRAM()` resolves to its dir basename so its SBOM
  component is `<basename>.CPP.component.sbom`, not `.CPP.component.sbom`.

## Expected gate effect

- `$(B)/contrib/libs/flatbuffers64/flatc/flatc`, `.../flatc.CPP.component.sbom`,
  and `.../.CPP.component.sbom` stop appearing as paired self_uid / ref-only /
  our-only differences in sg7.diff.txt; same for the ordinary `flatbuffers/flatc`
  and `luajit_21/compiler` programs.
- sg7 `matched` increases (≈ binary + component nodes for the three programs),
  `our_only`/`ref_only` drop; the ordinary flatc binary does not regress.
- sg2..sg6 stay byte-exact; plutonium `.fbs64` outputs from T-6 stay paired
  (untouched).
- No generation-time regression (pure name resolution, no new walks).
