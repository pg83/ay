## Upstream mechanism

A `PROTO_LIBRARY` is a multimodule; its C++ side is the `_CPP_PROTO` submodule
(proto.conf), which carries `MODULE_TAG=CPP_PROTO`. RESOURCE() declared in the
PROTO_LIBRARY body belongs to that CPP_PROTO submodule. Therefore:

- The resfs objcopy packer folds the owning submodule's `MODULE_TAG` ("CPP_PROTO")
  into the objcopy output-name hash (`TObjCopyResourcePacker`), and the dumped
  `target_properties.module_tag` is the lowercased submodule tag (`cpp_proto`).
- The submodule's `.global.a` carries `<MODULE_TAG>_global` = `cpp_proto_global`.

This mirrors the existing PY3_PROTO path (`emitGeneratedPyProtoObjcopy`, which folds
`PY3_PROTO` into the hash and tags the objcopy `py3_proto` + the global archive
`py3_proto_global`). The CPP_PROTO resource path currently folds in NOTHING and
falls through to the generic `global` archive tag.

Hash arithmetic confirmed offline:
`md5("...")[:26]` over the sorted member list with `+ "CPP_PROTO"` appended yields
exactly `8f1adf26e50f9e4660bdaae178` (the ref hash); with nothing appended it yields
`0c4805f5d695bd430abe97e6c0` (our current hash).

## Current code path

PROTO_LIBRARY is a specialized library type (`isSpecializedLibraryType`), so the
CPP-proto submodule is emitted through the specialized branch in `gen.go`
(~lines 1038-1117), NOT the general library path:

- `emit_py_objcopy.go: emitResourceObjcopy` (called at gen.go:1080) computes the
  objcopy hash with `moduleTag := resourceLibTagForData(d)` (nil for PROTO_LIBRARY)
  and sets `resTargetProps.ModuleTag` only for py module kinds.
- `gen.go:1097-1112` chooses the global-archive `tag`; PROTO_LIBRARY hits the
  `default: tag = tagGlobal`.

The CPP-proto variant is identified exactly by the existing predicate
`cfModuleTag(d, instance) == tagCppProto`
(`d.moduleStmt.Name == tokProtoLibrary && instance.Language != LangPy`).
