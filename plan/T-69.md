# T-69 plan: feed RUN_PROGRAM-generated `.fbs` into flatc

## Hypothesis / divergence

A `RUN_PROGRAM(... Fbs STDOUT formula_parameters.fbs)` produces a build-root
`$(B)/.../formula_parameters.fbs`. Upstream re-feeds that auto-output as a module
source through `_SRC("fbs")` â†’ `FBS_CMD` â†’ `_CPP_FLATC_CMD`, exactly like a
checked-in `.fbs`. Our `emitRunProgramsForAR` only routes auto `STDOUT`/`OUT`
that are C-family or asm sources into a downstream compile; a `.fbs`/`.fbs64`
auto-output is ignored, so the flatc producer, its `.fbs.h`/`.fbs.cpp`/`.bfbs`
outputs, the `.fbs.cpp.o` compile, and the archive member never appear, and the
`.fbs` PR node is pruned (nothing consumes its output).

### Ground-truth (sg7 reference, `yabs/server/libs/protocol/proto_flat_buf`)

- PR node `P=PR`: `OUTPUTS=[formula_parameters.fbs]`, cmd `generator Fbs`,
  inputs = generator binary.
- FL node `P=FL`: `OUTPUTS=[.fbs.h, .fbs.cpp, .bfbs]`, cmd = flatc wrapper,
  `INPUTS=[flatc, $(B)/.../formula_parameters.fbs, $(S)/build/scripts/cpp_flatc_wrapper.py]`
  â†’ depends on the PR (the `.fbs` is the PR's build output).
- CC node `.fbs.cpp.o`: inputs lead with `.fbs.cpp`, `.fbs.h`, `cpp_flatc_wrapper.py`,
  flatbuffers runtime headers. It does **NOT** list `$(B)/.../formula_parameters.fbs`
  (a $(B) codegen intermediate reached via the producer dep edge, not a C include),
  nor any generator $(S) source.
- The sibling Cpp run's `formula_parameters.cpp` lists `formula_parameters.fbs.h`
  in `OUTPUT_INCLUDES`; its `.cpp.o` therefore must carry `.fbs.h` (currently
  missing on our side) â€” so the FL producer must be **registered before** the cc
  compiles.
- Target archive `liblibs-protocol-proto_flat_buf.a` members, in order:
  `[formula_parameters.cpp.o, formula_parameters.fbs.cpp.o]` â€” the direct
  cc-source STDOUT object first, the flatc-derived object second.

Contrast: a checked-in source `.fbs` (`catboost/.../model.fbs`) DOES list its
`$(S)/.../model.fbs` in `model.fbs.cpp.o` inputs. The source/build asymmetry is
the existing generic rule (a build intermediate is a dep edge, not an include),
matching `prEmitsIncludes`'s `v.isBuild()` skip.

## Upstream mechanism

`build/conf/fbs.conf` `_CPP_FLATC_CMD` (`-I $(B) -I $(S) -o X.fbs.h X.cpp X.fbs`)
is the same flatc rule for any `.fbs`, source or generated. ymake's auto-output
mechanism appends a `.fbs` STDOUT to `AUTO_INPUT`, then `_SRC("fbs")` routes it
through flatc; the resulting `.fbs.cpp` is itself a generated source compiled and
archived after the first-level objects.
