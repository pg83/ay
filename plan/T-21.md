## Upstream mechanism (ymake) — the real cause

`ads/caesar/libs/profiles/proto/with_transitive_headers/ya.make` is a `LIBRARY`
with a single `RUN_PROGRAM` (tool `ads/bsyeti/tools/transitive_proto … generate`)
that emits many `.pb.h` wrapper headers as `OUT`. `advm_banner.pb.h` is the
**first** `OUT`.

ymake models a multi-output command with `EDT_OutTogether` edges
(`compact_graph/dep_types.h`: "additional output -> main output"). The **main
output** is `FindMainElemOrDefault(GetOutput(), 0)` (`macro_processor.cpp:1041`):
the element flagged `.Main`, else **index 0** (the first declared output).
RUN_PROGRAM flags no `.Main`, so the main output is the first declared output
(STDOUT if present, else first `OUT`, else first `OUT_NOAUTO`).

When a node depends on a **non-main** output, the main output rides along as an
"AlsoBuilt" dep (`json_visitor.cpp` PrepareLeaving):

```cpp
const TNodeId outTogetherDependencyId = CurrData->OutTogetherDependency;
...
if (!prntDone && outTogetherDependencyId != TNodeId::Invalid && !CurrData->Fake) {
    prntDestSet.Add(outTogetherDependencyId);   // main output rides with any sibling
}
```

`prntDestSet` is the parent's `IncludedDeps` (or `NodeDeps` for a build-entry
parent); `IncludedDeps` propagate transitively up the include chain, so the main
output reaches every transitive consumer of any sibling output — exactly our
flat-closure model.

## Relationship to T-20 (landed on trunk)

T-20 (`fix(T-20): RUN_PROGRAM main-output sibling header rides the generated-cpp
consumer`) already records the main output (`mainOutputVFS`) and, via
`emit_codegen_cc.go withOutTogetherMain`, appends it to the SAME-run downstream
cpp/asm include window — fixing the `features.gen.h` sibling of `features.gen.cpp`.
That append looks only at the compiled source's OWN `OutTogetherMain`; it cannot
reach the cross-module `advm_banner.pb.h` case, where the consumer
(`features.gen.cpp.pic.o`) reaches the wrapper sibling through `features.gen.cpp`'s
include CLOSURE, and the wrapper is produced by a DIFFERENT module's RUN_PROGRAM.

T-20's root-only append is a special case of riding the main output through the
include closure. T-21 generalizes it to one mechanism: register the main output
as a ClosureLeaf of each non-main sibling (reusing T-20's `mainOutputVFS`), so the
scanner splices it into every window containing the sibling — root AND transitive.
`withOutTogetherMain` and the `OutTogetherMain` field are removed (subsumed). This
preserves the `features.gen.h` behavior (still rides, now via the leaf) without
re-deriving the main output or running a parallel mechanism.

## Our mechanism to reuse: ClosureLeaves

`CodegenRegistry.addClosureLeaf(node, leaf)` records a bare, non-expanded VFS the
scanner splices into any closure window containing `node` (`scanner.go` dfs
pass-2 / emitClosure leaf loops; `leafEver`/`windowSubsumed` keep resolution
context-free → caching intact). `emit_pr.go` already uses it to ride a PR's
`$(S)` generator sources with each output.

The OutTogether main output is exactly a "bare member that rides with a sibling":
register the command's **main output** as a ClosureLeaf of **every other**
distinct output of the same RUN_PROGRAM. No per-consumer re-resolution, no new
data path.

Bare leaf vs full closure: the main output's own include closure (advm_banner
wrapper → base `advm_banner.pb.h` + `basics/common.pb.h`) is independently
covered in the REF consumers, so a bare leaf is byte-exact and preserves the
resolution-caching invariant.

## Change

`emit_pr.go`, `emitRunProgram`: after all PR outputs are registered, if the run
declares ≥2 distinct outputs, take the first emitted output (same order emitPR
lists node `outputs`) as the main output and `addClosureLeaf(other, mainOut)`
for each other distinct output. The main output is NOT added to the PR
producer's own inputs (ticket: do not over-collect onto the PR producer).

regressed node disappearing while advm_banner stays fixed.
