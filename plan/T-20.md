# T-20 plan: model apphost cow well_known RUN_PROGRAM build-root OUT + ADDINCL

## Symptom

`$(B)/apphost/gp/modules/client/http/proto/http.pb.cc.o` (and the whole
`apphost_cow` consumer set) is missing, reference-only:

- cmds token `-I$(B)/apphost/gp/lib/proto/cow/generator/well_known`
- inputs `$(B)/apphost/gp/lib/proto/cow/generator/well_known/google/protobuf/*.cow.pb.h`
  plus their transitive `cow/lib.h`, `generic/*.h`, `.proto` closure.

The reproduced ay graph contained **zero** nodes mentioning
`apphost/gp/lib/proto/cow` (ref had 1580). The entire cow generation subtree
was absent — not merely an include flag.

## Upstream mechanism (root cause)

`apphost/gp/modules/client/http/proto/ya.make` does
`INCLUDE(${ARCADIA_ROOT}/apphost/gp/lib/proto/cow/gen.inc)`. That .inc gates the
whole apphost_cow plugin on:

```
IF(NOT DEFINED ARCH_TYPE_64) ENABLE(DISABLE_APPHOST_COW_PROTO_GEN) ENDIF()
IF (NOT DISABLE_APPHOST_COW_PROTO_GEN AND MODULE_TAG == "CPP_PROTO")
    CPP_PROTO_PLUGIN(apphost_cow apphost/gp/lib/proto/cow/generator .cow.pb.h
        DEPS apphost/gp/lib/proto/cow … apphost/gp/lib/proto/cow/generator/well_known …)
    ADDINCL(${ARCADIA_BUILD_ROOT}/apphost/gp/lib/proto/cow/generator/well_known)
ENDIF()
```

`ARCH_TYPE_64` is a target-architecture variable (`build/ymake_conf.py`:
`(self.is_64_bit, 'ARCH_TYPE_64')`), defined for every 64-bit target
(x86_64, aarch64). ay's `buildIfEnv` set `ARCH_X86_64` / `ARCH_AARCH64` but
never `ARCH_TYPE_64`, so `NOT DEFINED ARCH_TYPE_64` was always true →
`DISABLE_APPHOST_COW_PROTO_GEN` enabled → the CPP_PROTO_PLUGIN + ADDINCL block
skipped. No cow plugin, no well_known peer, no build-root include.

Once enabled, `well_known/ya.make` declares
`RUN_PROGRAM(... TOOL ${ARCADIA_BUILD_ROOT}/apphost/gp/lib/proto/cow/generator
OUT ${ARCADIA_BUILD_ROOT}/${MODDIR}/google/protobuf/*.cow.pb.h ...)`. The OUT
paths already resolve correctly through `copyFileOutputVFS` →
`moduleRootedVFS` (the `$(B)/` prefix maps to a build VFS verbatim). The TOOL,
however, is a build-root path `$(B)/apphost/gp/lib/proto/cow/generator`;
`resolveRunProgramAuxTools` fed that straight to `toolResult`, which tried to
open `$(B)/.../ya.make` as a source module and aborted. Upstream's
`${hide;tool:TOOL}` resolves the module named by the path (root prefix is just
an output reference) and the command spells the binary literally in $Args —
no tool-token substitution.

## Change

1. `env_consts.go` / `modules.go buildIfEnv`: define `ARCH_TYPE_64` for
   `ISAX8664` and `ISAAArch64` (and reserve `ARCH_TYPE_32` const). Mirrors
   ymake_conf.py is_64_bit.

2. `emit_pr.go resolveRunProgramAuxTools` + `emitPR`: a TOOL whose token is
   root-prefixed (`vfsHasPrefix`) resolves its module by stripping the prefix
   (`intern(token).rel()`), and is marked `rooted`. Rooted TOOLs contribute the
   dependency only — arg substitution is skipped (the literal
   `$(B)/dir/binary` already in the args must not be rewritten; the relative
   `contrib/.../cpp_styleguide` case still substitutes).

## Regression

`emit_pr_test.go`: a RUN_PROGRAM producer with
`OUT ${ARCADIA_BUILD_ROOT}/${MODDIR}/sub/any.cow.pb.h`,
`TOOL ${ARCADIA_BUILD_ROOT}/<tooldir>`, and
`ADDINCL(GLOBAL ${ARCADIA_BUILD_ROOT}/<moddir>)`; a consumer that PEERDIRs it
and #includes the generated header. Assert: producer node exists with the
build-root OUT; consumer CC command gets `-I$(B)/<moddir>`; generated header is
a build-root input of the consumer.
