## Upstream mechanism

`macro RUN_PROGRAM` (`build/ymake.core.conf:4781`) caesar invocation
(`run_codegen.inc`) declares, in order:

```
OUT
    ${BINDIR}/features.gen.h
    ${BINDIR}/features.gen.cpp
    ${BINDIR}/feature_sets_and_mutations.gen.cpp
    ${BINDIR}/input_proto_masks.pb
    ${BINDIR}/feature_to_precompile_info.pb
```

A multi-output command produces ONE build node, keyed on the **main output**.
`TMakeCommand::MainOutput = FindMainElemOrDefault(GetOutput(), 0)`
(`macro_processor.cpp:1041,71`) picks the first OUT (no element carries the
`Main` flag for RUN_PROGRAM), i.e. **`features.gen.h`**. Every other output of
the command (`features.gen.cpp`, â€¦) becomes an `EDT_OutTogether` sibling whose
`OutTogetherDependency` points back to the main output node.

When the json export computes the inputs of a node that consumes a non-main
output, `TJSONVisitor::PrepareLeaving` (`json_visitor.cpp:658-661`) adds the
consumed file's `OutTogetherDependency` (the main output) to the parent's dep
set:

```cpp
if (!prntDone && outTogetherDependencyId != TNodeId::Invalid && !CurrData->Fake) {
    prntDestSet.Add(outTogetherDependencyId);
}
```

So the compile of `features.gen.cpp` (a non-main OUT) carries `features.gen.h`
(the main OUT) as an input â€” a single file dep, NOT a re-walked include closure
(only the node id is added, not its `IncludedDeps`). `features.gen.h` is NOT in
`OUTPUT_INCLUDES`; it rides purely through the OutTogether main-output edge.

Verified: REF's single producer node carries all 5 outputs with `features.gen.h`
first; both generated `.cpp.pic.o` consumers list `features.gen.h`.

## Our model and the gap

`emitRunProgram` (emit_pr.go) emits ONE PR node with all outputs in OUT order
(features.gen.h first) â€” already matches REF's producer. The downstream compile
of each generated cc-source OUT is emitted by `emitCodegenDownstreamCC` /
`emitCodegenDownstreamAS` (emit_codegen_cc.go), whose inputs = the source's walked
include closure (T-16 made this find the OUTPUT_INCLUDES). Neither adds the
producer's main output. That is the missing `features.gen.h` input.

## Change (generic, registry-threaded)

Reproduce the OutTogether main-output edge as a property of each generated output:

1. **codegen_registry.go**: add `OutTogetherMain VFS` to `GeneratedFileInfo` â€”
   the producing command's main output, set on every NON-main output of a
   multi-output command (zero = this is the main output / single-output command).

2. **emit_pr.go** `emitRunProgram`: compute the run's main output VFS = first of
   `OUTFiles[0]`, else `OUTNoAutoFiles[0]`, else STDOUT (the GetOutput() order).
   In `registerPROutput`, after registering an output, set its
   `OutTogetherMain = mainOutputVFS` when the output != main.

3. **emit_codegen_cc.go** `emitCodegenDownstreamCC` / `emitCodegenDownstreamAS`:
   after walking the source closure, look up the compiled output's
   `GeneratedFileInfo`; if `OutTogetherMain != 0`, append it to the include-input
   window (dedup-safe: the source closure never contains the sibling main).

The main output rides as exactly one extra input, matching upstream's single
`prntDestSet.Add(outTogetherDependencyId)`. Scoped to the RUN_PROGRAM downstream
CC/AS path â€” proto/bison/etc. emitters are untouched, so sg2â€“sg6 byte-exact
producers do not move.
