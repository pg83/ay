# T-105 plan: YaFF/proto archive member-order residual

Representative: `$(B)/ads/peafowl/libs/profiles/proto/liblibs-profiles-proto.a`.
Our archive orders each `.pb.cc.pic.o` before its sibling `.yaff.cpp.pic.o`;
upstream orders `.yaff.cpp.pic.o` before `.pb.cc.pic.o` per proto.

## Upstream mechanism (verified against the reference graph)

The archive lists `$AUTO_INPUT` in the order ymake queues a command's outputs as
module sources (`module_builder.cpp` `QueueCommandOutputs` → `AddSource` over
`cmdInfo.GetOutput()`), i.e. the textual order of the `output` modifiers in the
protoc command = the order of `$CPP_PROTO_OUTS`. So the per-proto archive member
order = the order of the buildable (`.cc`/`.cpp`) entries in the proto command's
`--outputs` list.

`$CPP_PROTO_OUTS` accumulates in ya.make statement order, with the `${main}`
`.pb.h` floated to the front. The base `.pb.cc` (+ the `.deps.pb.h` for lite
headers) is (re)appended at the point `SET(PROTOC_TRANSITIVE_HEADERS "no")` runs
its conditional block (ymake `RecalcVars`); a plugin (`YAFF`/`GRPC`) appends its
outputs where its macro runs. Hence a plugin declared **before** that SET lands
ahead of `.pb.cc`; one declared after (or in a module with transitive headers,
where there is no SET) lands behind it.

Reference confirmation (3 cases, all consistent):
- `ads/peafowl/.../proto` (YAFF() then SET transitive=no): `--outputs` =
  `.pb.h, .yaff.h, .yaff.cpp, .pb.cc, .deps.pb.h`; archive = `.yaff.cpp.pic.o`
  then `.pb.cc.pic.o`.
- `ads/keywords/proto` (SET transitive=no then YAFF): `--outputs` =
  `.pb.h, .pb.cc, .deps.pb.h, .yaff.h, .yaff.cpp`; archive = `.pb.cc.pic.o` then
  `.yaff.cpp.pic.o`.
- `grut/libs/object_api/proto` (GRPC, default transitive headers): `--outputs` =
  `.pb.h, .pb.cc, .grpc.pb.cc, .grpc.pb.h`; archive = `.pb.cc.pic.o` then
  `.grpc.pb.cc.pic.o`.

## Existing machinery to reuse

`emitPB` (emit_pb.go) already builds the proto command `outputs` in exactly this
order via `pluginOutputsPrecedeCppGroup(plugin, liteHeaders)` (=
`liteHeaders && plugin.Spec.DeclaredBeforeLiteHeaders`). The PB node's
`--outputs` is therefore already correct (covered by
`TestEmitProtoSrcs_YaffOutputOrderFollowsLiteHeaderDeclarationOrder`).

The defect is only in `emitCPPProtoSrcs` (emit_proto.go): it builds
`codegenOutputs` as `.pb.cc`, then `.grpc.pb.cc`, then plugin extras in plugin
order — ignoring the precede/follow rule. The proto-codegen AR members share an
equal `reorderARMembers` sort key (generated, default prio, no Seq), so the
stable sort preserves this emission order → `.pb.cc.o` always lands before the
yaff `.cpp.o`.

## Change

Single source of truth for the per-proto output order: extract the ordering body
of `emitPB` into `assembleProtoCmdOutputs(...) []VFS` (full ordered output list).
`emitPB` uses it for the command; `emitProtoPB` calls it and filters buildables
(`isCCSourceExt`) into `ProtoPBEmission.orderedCC`. `emitCPPProtoSrcs` iterates
`orderedCC` instead of the hand-ordered `.pb.cc`/grpc/extras appends. Net code
reduction; the archive member order now follows the proto command output order
for every plugin/grpc/lite-header combination.

Files: emit_pb.go (extract helper), emit_proto.go (`ProtoPBEmission.orderedCC`,
emission loop), emit_proto_test.go (fail-first AR-order coverage).
