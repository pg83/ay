# T-86 plan: generated `.proto` CPP_PROTO compile-input `.pb.h` closure (`banner_flags`)

## Confirmed divergence (sg7, local probe)

`$(B)/irt/bannerland/proto/banner_flags/banner_flags.pb.cc.o` and `.pb.cc.pic.o`
pair-diff vs REF — both miss exactly 9 inputs (ref-only):

```
+$(B)/modadvert/bigmod/protos/core/{application,grut,object_types,options,
                                    regions,time,yt_path}.pb.h        (7)
+$(B)/modadvert/bigmod/protos/interface/markup_flags.pb.h            (1)
+$(S)/contrib/libs/protobuf/src/google/protobuf/descriptor.pb.h      (1, canonical only)
```

The `.pb.cc`/`.pb.h` PB producer node pair is already byte-exact (T-85). Only the
two C++ compile nodes under-carry the imported-proto `.pb.h` transitive closure.

## Upstream mechanism (verified)

Source (`irt/bannerland/proto/banner_flags/ya.make`):

```
RUN_PROGRAM(.../gen STDOUT_NOAUTO banner_flags.proto
            OUTPUT_INCLUDES modadvert/.../core/regions.proto
                            modadvert/.../interface/markup_flags.proto)
SRCS(banner_flags.proto)
```

The generated proto's ACTUAL imports (from gen.py template) are
`markup_flags.proto` + `options.proto` — NOT `regions.proto`. Yet the REF compile
closure includes `regions.pb.h`. That can only come from the OUTPUT_INCLUDES set
(`regions` + `markup_flags`), proving the induced `.pb.h` compile closure is
driven by the declared **OUTPUT_INCLUDES**, not the (configure-time-unreadable)
generated proto's real imports. OUTPUT_INCLUDES is upstream's
`${hide;output_include:OUTPUT_INCLUDES}` — induced deps attached to the generated
`.proto`: when protoc consumes it, the generated `.pb.cc` `#include`s the
imports' `.pb.h`, and the `.pb.cc.o` compile carries that header's transitive
closure.

For a **checked-in** proto this already works: `protoDirectPbHIncludes` parses
the `$(S)` proto's `import` statements → direct `.pb.h` includes registered on
`<proto>.pb.h`; the scanner then walks each direct import's own registered
`.pb.h` transitively (confirmed: `options.pb.cc.o` carries the full 6-header
closure, and descriptor resolves to the canonical `protobuf/descriptor.pb.h` via
the `google/protobuf/*.pb.h` → `$PROTOBUF_INCLUDE_PATH` remap in
`protoPbHIncludes`). For a **generated** proto the source does not exist at
configure time, so `protoDirectPbHIncludes(source(rel))` returns empty → the
`.pb.h` registers no direct imports → the compile closure is empty.

The fix supplies the generated proto's direct imports from its producer's
OUTPUT_INCLUDES `.proto` entries, mapped to `.pb.h` exactly as the checked-in
parse path maps them; the scanner's existing transitive walk then reconstructs
the full closure (incl. canonical descriptor) with no new mechanism. The
OUTPUT_INCLUDES `regions`+`markup_flags` transit to the full 8 modadvert `.pb.h`
+ canonical descriptor — exactly REF. Using only the DIRECT OUTPUT_INCLUDES (not
the full `ProducerSourceClosure`) avoids re-introducing the T-83
`protobuf_old/descriptor.pb.h` non-canonical false-add, because the transitive
walk reaches descriptor only through the registered `.pb.h`' canonical remap.

## Files to touch

- `codegen_registry.go`: add `ProtoImportRels []string` to `GeneratedFileInfo`
  (the generated `.proto`'s declared direct proto imports = its producer's
  OUTPUT_INCLUDES `.proto` entries) + `setProtoImportRels` setter, mirroring
  `setSourceInputs` / `setProducerSourceClosure`.
- `emit_pr.go` (`emitRunProgram`): when a registered PR output is a `.proto`,
  record the run's OUTPUT_INCLUDES `.proto` entries (normalized to rel) as that
  output's `ProtoImportRels`.
- `emit_proto.go` (`emitProtoPB`): when the proto is build-generated
  (`protoSrcOverride != 0`) and `info.ProtoImportRels` is non-empty, derive
  `directImports` from those rels (via a small `protoImportRelsToPbH` helper that
  reuses the `google/protobuf` → runtime / `protoOutputRel` mapping) instead of
  the empty source parse. Read `info.ProtoImportRels` in the existing
  `codegenRegForInstance(...).lookup(buildProto)` block.
- `emit_proto_test.go`: fail-first CPP_PROTO fixture + checked-in guard.

No new cross-module data path: `ProtoImportRels` rides the same per-output
`GeneratedFileInfo` the registry already threads (alongside `SourceInputs`,
`ProducerSourceClosure`), and the closure is rebuilt by the existing scanner walk
— resolution stays context-free.
