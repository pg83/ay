# T-102 plan: RUN_PROGRAM tool INDUCED_DEPS(h) producer closure for header outputs

## Representative divergence

`$(B)/ads/bsyeti/eagle/collect/generated/query_params.h` — paired sg7 node differs
only in `inputs`. The reference attaches the source/header/proto closure that the
generator tool's `INDUCED_DEPS(h ...)` induces; our producer node lists only the tool.

Source shape:

```make
# generated/ya.make
RUN_PROGRAM(ads/bsyeti/eagle/collect/generated/codegen query_params.h OUT query_params.h)

# generated/codegen/ya.make
PROGRAM()
INDUCED_DEPS(h
    ${ARCADIA_BUILD_ROOT}/ads/bsyeti/eagle/collect/proto/query_params.pb.h
    ${ARCADIA_ROOT}/ads/bsyeti/eagle/collect/errors.h
    ${ARCADIA_ROOT}/ads/bsyeti/libs/primitives/user_identifier.h
    ...
)
```

## Upstream mechanism

`INDUCED_DEPS(h …)` on a `_BASE_PROGRAM` (`.ALLOWED=INDUCED_DEPS`,
ymake.core.conf:1207) records the module's induced header deps. When the program is
used as a `${tool:Tool}` in `RUN_PROGRAM`, ymake attaches those induced deps to the
generated OUTPUT (`EVI_InducedDeps`). Upstream's flat-input model then lists the
output's full transitive induced-dep closure on the producing command node — the
SAME set a consumer that `#include`s the generated header picks up.

Our code already reproduces the CONSUMER side: `GeneratedFileInfo.GeneratorRefs`
records the tool refs, and `ScanCtx.resolveInducedDeps` (scanner.go) mixes the tool's
`InducedDeps.bucket(kind)` into the generated file's resolved children, with bucket
selection by output kind (header output → Header bucket, TU → Cpp bucket). The
PRODUCER node's own input closure (`prInputClosure`, emit_pr.go) did not fold this in
for a header-only RUN_PROGRAM with no IN / no OUTPUT_INCLUDES, so it returned nil.

## Change

`emit_pr.go: prInputClosure` — within the `fullSourceClosure` branch (no IN, no
compiled cc-source sibling), walk each header `OUT` file's own include closure
(`walkClosureTail`) and keep its `$(S)` source entries. The output is registered
with its `GeneratorRefs` before `prInputClosure` runs, so `resolveInducedDeps`
fires during the walk and surfaces exactly the tool's header induced-dep closure —
no new data path, no second parse, reusing the existing scanner machinery and the
output-kind bucket selection already in place.

Output-kind sensitivity is intact: the walk reads the header output's closure, whose
`resolveInducedDeps` selects the Header bucket; a cc-source OUT is never walked here,
so it cannot inherit a header-only bucket. `fullSourceClosure` gating keeps the
header-with-cc-sibling case (`features.gen.h`/`.cpp`, OUT gen.h gen.cpp) routing its
induced deps through the sibling to consumers, not onto the producer. OUT_NOAUTO
header outputs (yql `.pb.h`) are not OUTFiles and stay excluded from the producer
self-scan, avoiding the protobuf-closure over-emit.
