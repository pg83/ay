# T-41 plan: generated-output module_dir attribution for self-consuming RUN_PROGRAM producers

## Symptom (sg7, current trunk @583e52e)

`$(B)/ads/argus/proto/profile/generated/profile_traits.h` (a PR node) reports
`module_dir = ads/argus/libs/primitives` (a downstream consuming library); the
reference reports `module_dir = ads/argus/proto/profile/generated` (the module
that produces it). Same class for its `.cpp` sibling and for the bsyeti /
profile_printer `all_profiles.h` / `user_profile_traits.h` /
`user_timestamps_traits.h` / `printer_utils.{h,cpp}` generated families:

```
PR ads/argus/proto/profile/generated/profile_traits.h   ours=ads/argus/libs/primitives        ref=ads/argus/proto/profile/generated
PR ads/bsyeti/libs/profile/generated/all_profiles.h      ours=ads/bsyeti/libs/profile           ref=ads/bsyeti/libs/profile/generated
PR ads/bsyeti/libs/profile_printer/generated/printer_utils.h ours=ads/bsyeti/libs/profile_printer ref=ads/bsyeti/libs/profile_printer/generated
```

11 paired PR outputs (`ref == output-owner dir`, `ours == an external
consumer`). Each is target_properties-only drift (entangled with a `deps`
uid cascade on the same node).

## Upstream mechanism

`overrideGeneratedModuleDir` (attribute_generated.go) models upstream
`TJSONVisitor`'s `Node2Module[node] = FindModule(state)` on the FIRST DFS-leave
of a generated file node. The first leaver is the first module whose subtree
reaches the node in post-order. Two cases:

- **OUT_NOAUTO header-only producer** (LLVM `contrib/libs/llvm16/include`
  `*.h.inc`/`*.inc`, declared with `OUT_NOAUTO`): the producing module compiles
  nothing that includes the output. The producing module's subtree never
  descends into the output node; the FIRST module to traverse into it is an
  external consumer (`lib/IR/Attributes.cpp`). → `module_dir = consumer`.
  This is exactly what the existing consumer-claim override reproduces (sg5
  byte-exact).

- **Auto-`OUT` self-consuming producer** (`ads/argus/proto/profile/generated`
  is a `LIBRARY()` with `RUN_PROGRAM(... OUT profile_traits.h profile_traits.cpp)`;
  `profile_traits.cpp` is an auto `OUT` → compiled in the producing module):
  post-order processes peers before parents, so the producing module — a peer
  of every external consumer — compiles its own `profile_traits.cpp.o` and
  leaves `profile_traits.h` (the OutTogether main output it rides) BEFORE any
  consumer is visited. → `module_dir = producer`.

Our scanner only records a generated-output claim when a CONSUMER
include-resolves the output through `addBuild`/`lookupSplit` (scanner.go
generatedFirstClaim). The producing module's own compile of the sibling `.cpp`
pulls the header in as a non-expanded OutTogether closure leaf (T-20), NOT via
an include resolve, so it records no self-claim. The first external consumer
(`ads/argus/libs/primitives`) therefore wins the claim and the override
mis-attributes the header to it. features.gen.h does not exhibit the drift only
because no external module happens to include-resolve it.

The discriminator is upstream's auto/noauto distinction: a RUN_PROGRAM whose
module auto-compiles a cc-source (or asm) sibling self-consumes its outputs and
owns them; an OUT_NOAUTO/header-only RUN_PROGRAM does not.

## Change

`emit_pr.go` `emitRunProgram`: compute `selfConsumes` = the run has an auto
cc-source or asm-source output (same set `emitRunProgramsForAR` compiles via
`emitPRDownstreamCC`/`emitCodegenDownstreamAS`: `OUTFiles` + auto `STDOUT`,
never `OUT_NOAUTO`). When `selfConsumes`, record the producing module dir
(`instance.Path.rel()`) as the generated claim for every registered output, in
the producer's own scanner (`ctx.scannerFor(instance).generatedFirstClaim`),
inside `registerPROutput` — i.e. at registration, strictly before any consumer
can resolve the output (resolution requires the registry entry that
`registerPROutput` is creating). The recorded claim equals the producer's
emit-time `module_dir`, so `overrideGeneratedModuleDir` sees `claim == current`
and leaves the node attributed to the producer. OUT_NOAUTO/header-only runs
record nothing and keep the existing consumer-claim behavior.

New `IncludeScanner.markGeneratedProducerOwned(out VFS, dir string)` writes the
claim unconditionally (the producer is the guaranteed first writer).

No resolution-path, cache-key, command, input, or output change — the claim map
is a write-only side channel read only by the finalize attribution pass.

## Test-first

`emit_pr_test.go`:
1. Positive: `gen` LIBRARY `RUN_PROGRAM(... OUT gen.h gen.cpp)` (auto cpp
   compiled in `gen`), consumed by `cons` via `#include <gen/gen.h>`. Assert the
   `$(B)/gen/gen.h` PR node `TargetProperties.ModuleDir == "gen"`. Fails before
   the change (consumer `cons` claims it → `"cons"`).
2. Negative: `geninc` LIBRARY `RUN_PROGRAM(... OUT_NOAUTO geninc.inc)` consumed
   by `cons2`; assert the override still attributes `geninc.inc` to the consumer
   `"cons2"` (header-only producer keeps consumer-claim).

- sg2..sg6 byte-exact (no command/input/output movement; LLVM OUT_NOAUTO path
  unchanged).
- sg7: the 11 profile/printer PR outputs lose their `module_dir`

  (T-35) and cpp_import YaFF (T-37) families stay out of scope. Remaining

  `ours=owner ref=consumer` T-37 family, apphost cow.pb.h) reported separately.
