# T-4 — GLOBAL PROTO_NAMESPACE from transitive proto peers in the protoc `_PROTO__INCLUDE`

## Observed divergence (full sg7, normalize+diff)

`--by-token` reports `-I=$(S)/contrib/libs/googleapis-common-protos` as **ref-only
in 474 PB commands**. Matching outputs to the reference shows all 474 are the
**Python** protoc producers (`*__intpy3___pb2.py`); the C++ `.pb.h` producers
already carry the token.

Representative C++ node `ads/.../adv_machine/proto/plugin.pb.h`: cmds differ only
in **ordering** — ours emits `…googleapis, yt`, ref emits `…yt, googleapis`.

Representative pair `ads/caesar/libs/graph/proto/config` (PROTO_LIBRARY, no own
PROTO_NAMESPACE), `_PROTO__INCLUDE` peer band:

| position | ref | ours (C++) | ours (py) |
|---|---|---|---|
| 1 | protobuf/src | protobuf/src | protobuf/src |
| 2 | **yt** | googleapis | yt |
| 3 | **googleapis** | yt | *(missing)* |

So two facts: (a) the band is a **single ordered set** `[protobuf/src, yt,
googleapis]` — global (`protobuf/src`, `googleapis`) and bare (`yt`) namespaces
**interleave** by encounter order; (b) the py command omits the global members.

## Upstream mechanism (yatool/build/conf/proto.conf)

- `PROTO_NAMESPACE([GLOBAL] Ns)` → `PROTO_ADDINCL(GLOBAL $Ns)` — it **always**
  passes the literal `GLOBAL` to `PROTO_ADDINCL` (proto.conf:136), regardless of
  whether the caller wrote `GLOBAL`.
- `PROTO_ADDINCL(GLOBAL X)` → `_ORDER_ADDINCL(GLOBAL FOR proto $B/X SOURCE GLOBAL
  FOR proto $S/X)` plus `ADDINCL(GLOBAL $B/X)` (proto.conf:117-119). The
  `… FOR proto $S/X` arm is the `_PROTO__INCLUDE` member; the `GLOBAL` makes it
  propagate to every transitive proto consumer.
- Therefore **bare and GLOBAL PROTO_NAMESPACE contribute IDENTICALLY** to
  `_PROTO__INCLUDE` — one ordered set, both propagating. The `GLOBAL` keyword only
  governs the *separate* C++ `ADDINCL($B/X)` arm (already handled by
  `applyProtoNamespace` via `d.addInclGlobal`).
- Both the C++ (`_CPP_PROTO_CMDLINE_BASE`, ymake.core.conf:612 / proto.conf:457)
  and the Python (`PY_PROTO_CMDLINE`, proto.conf:516) commands expand the *same*
  `${pre=-I=:_PROTO__INCLUDE}`.

Our model split this set into two fields rendered global-then-tail
(`ProtoAddInclGlobal` then `ProtoNamespaceTail`). That (a) loses the interleave and
(b) was never wired into the py path. The split is an artifact, not upstream.

## Change — collapse the split into one ordered `_PROTO__INCLUDE` set

Replace the two carriers with a single ordered, first-encounter-deduped list:

- `ModuleEmitResult.ProtoAddInclGlobal` + `.ProtoNamespaceTail` → `.ProtoInclude`.
- `PeerGlobalContribs.protoAddIncl` + `.protoNamespaceTail` → `.protoInclude`.
- `ModuleCCInputs.PeerProtoAddInclGlobal` + `.ProtoNamespaceTail` → `.ProtoInclude`.

Build it in `gen.go` (general + specialized H paths) as
`dedupVFS(ownProtoInclude, <each peer's ProtoInclude in resolved order>)`, where
`ownProtoInclude = [source(ns) if d.protoNamespace != nil] ++ d.protoAddInclGlobal`
— **no `d.protoNamespaceGlobal` branch** (both go to the same set). `walkPeers`
aggregates a single first-encounter pass. (`d.protoNamespaceGlobal` stays only for
the C++ `$(B)` arm in `applyProtoNamespace`.)

Render the single list in each protoc command, deduped against tokens already
present (unchanged dedup logic, one loop instead of two):

- `composePBArgBlocks` (C++ PB) — emit_pb.go
- `newPyPBModuleEmission` (Python PB) — emit_py_proto.go  ← **gains the set**
- `evPeerProtoIncludes` / `emitEV` — emit_ev.go
- `descPeerClosure` / `descProtocIncludes` / `protoNamespaceContribs` — emit_proto_desc.go

`emitProtoPB` proto-import search paths use the single set (so bare namespaces like
`yt` now participate in import resolution — this also recovers the ref-only `yt`
`.proto` inputs the pair diff showed missing on plugin.pb.h).

## Why sg2–sg6 stay byte-exact

The segregated renderer emitted global-then-tail. sg2–sg6 are byte-exact today, so
in those graphs the reference order *is* global-then-tail (or no interleave) — i.e.
the true encounter order already places globals before bares there. The unified set
reproduces the true encounter order over the **same** peer traversal, so it equals
the segregated output wherever the segregated output was already correct, and only
changes the interleaved cases (sg7).

## Test-first

A focused test in `emit_proto_test.go`: a PROTO_LIBRARY whose transitive proto peer
chain reaches a `PROTO_NAMESPACE(GLOBAL …)` peer *after* a bare `PROTO_NAMESPACE(…)`
peer. Assert the C++ **and** Python protoc `-I` bands contain the GLOBAL namespace
exactly once, positioned after the bare one (encounter order), and not duplicated
when the module already provides it. Must fail on current code (py omits it; C++
mis-orders), pass after.
