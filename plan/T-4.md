# T-4 — Model FROM_SANDBOX fetched outputs consumed by RUN_PROGRAM inputs

## Hypothesis (reproduced by T-1 at 3f88253, re-verified here)
Under `--sandboxing`, the UID finalizer (`uid.go` `writeVFSSliceOS`, gated on
`v.isSource()`) content-hashes every *source* input of a node. The morph2blob
`RUN_PROGRAM(... IN trie suffixes paradigms ...)` in
`ads/clemmer/automorphology/uzb` has `trie` as an input. Because `FROM_SANDBOX`
is an acknowledged no-op (`gen.go` `acknowledgedMacros`), `trie` is NOT a
registered codegen output, so `runProgramInputVFS` falls through to a SOURCE VFS
`$(S)/ads/.../uzb/trie`. The finalizer then reads that nonexistent on-disk file
→ hard abort (`--keep-going` does not suppress the throw), 0-byte graph.

## Upstream mechanism
`build/ymake.core.conf` `macro FROM_SANDBOX(Id, OUT[], OUT_NOAUTO[], ...)` emits
a node that runs `build/scripts/fetch_from_sandbox.py`, producing the declared
OUT/OUT_NOAUTO files in the module build dir (`${hide;output:OUT}`,
`${hide;noauto;output:OUT_NOAUTO}`). In the reference graph this is a `p=SB`
node per FROM_SANDBOX with output `$(B)/<dir>/<name>`; the consuming morph2blob
`PR` node lists those outputs as inputs and the SB nodes as deps. A FROM_SANDBOX
OUT is a build-tree fetch output, never a source file.

## Fix (port of the FROM_SANDBOX slice of 3f88253; generic, not sg7-specific)
1. Parse `FROM_SANDBOX` into a typed `FromSandboxStmt` (Id + OUT/OUT_NOAUTO +
   OUTPUT_INCLUDES + FILE/PREFIX/EXECUTABLE), store on `ModuleData.fromSandboxes`,
   expand tokens in `collectStmts`; drop FROM_SANDBOX from `acknowledgedMacros`.
2. New `emit_from_sandbox.go`: emit one SB fetch node per statement (mirrors the
   upstream `fetch_from_sandbox.py` command, cwd=`$(B)/<dir>`, p=SB, pc=yellow,
   show_out, network=full, sandboxing) and register each OUT/OUT_NOAUTO via
   `registerBoundGeneratedParsedOutput(pkSB, ...)` bound to the SB node ref, so a
   consuming IN resolves to the `$(B)` path. OUTPUT_INCLUDES ride as the outputs'
   registered includes. Called per module in `genModule` before RUN_PROGRAM emit.
3. Add `pkSB` ("SB") to the ProcKind enum/string table.
4. `emitRunProgram`: resolve `resolveCodegenDepRefs` over `inVFSs ∪ inputClosure`,
   so a build-rooted IN that is a registered codegen output but carries no include
   parser (a FROM_SANDBOX OUT) still gets a dep edge to its SB producer. The build
   gate + registry probe + dedup make this a no-op for parsed/source INs.

## Test-first
`emit_from_sandbox_test.go::TestGen_FromSandboxOutputConsumedAsRunProgramInput`:
a module declares `FROM_SANDBOX(... OUT_NOAUTO trie)` and a `RUN_PROGRAM` consumes
`trie` via IN. Asserts: an SB node produces `$(B)/mod/trie`; the PR node lists
`$(B)/mod/trie` as input (and NOT `$(S)/mod/trie`); the positional arg resolves to
`$(B)/mod/trie`; the PR depends on the SB node's UID. Fails on current trunk
(FROM_SANDBOX is a no-op → source resolution).
