# T-103 plan: non-COW PY3 objcopy `py3_bin_lib` target-property attribution

## Upstream mechanism

`PY3_PROGRAM` is a multimodule (`build/conf/python.conf:347`) with two submodules:

```
module PY3_BIN: PY3_PROGRAM_BIN {
    .PEERDIRSELF=PY3_BIN_LIB
    .IGNORED=RESOURCE RESOURCE_FILES PEERDIR TASKLET_REG ...
    ENABLE(PROCESS_PY_MAIN_ONLY)   # process only PY_MAIN of PY_SRCS
    _PY_PROGRAM()
}
module PY3_BIN_LIB: PY3_LIBRARY {
    ENABLE(IGNORE_PY_MAIN)         # process everything except PY_MAIN
    NO_CLANG_TIDY()
}
```

The decisive fact: `RESOURCE` and `RESOURCE_FILES` are in the PROGRAM submodule's
`.IGNORED` list, so the resfs objcopy node a `RESOURCE()` produces is owned by the
LIBRARY twin `PY3_BIN_LIB`. Its `module_tag` is therefore `py3_bin_lib`. Only
`PY_MAIN` / `NO_CHECK_IMPORTS` (kv-only, program-side, `PROCESS_PY_MAIN_ONLY`) carry
`py3_bin`.

## The divergence

For the three representatives
(`ads/bsyeti/libs/events/codegen`, `ads/caesar/libs/events/codegen`,
`bigrt/lib/processing/graph/node_graph/events/codegen`) the upstream ya.make is a
`PY3_PROGRAM()` carrying both `RESOURCE(...)` and `PY_MAIN(...)`. The RESOURCE objcopy
node is the one whose `module_tag` diverges: ours stamps `py3_bin`, ref stamps
`py3_bin_lib`.

In `emit_py_objcopy.go:emitResourceObjcopy.flush()` the objcopy **output hash** is
already computed with `resourceLibTagForData(d)` (= `PY3_BIN_LIB` for a PY3_PROGRAM),
which is why the output path matches REF and the pair "prints only headers" except for
the field diff. But the **stamped** `TargetProperties.ModuleTag` switch (lines ~237-246)
sets `tagPy3Bin` for the `tokPy3Program` case, contradicting the hash tag. That single
inconsistency is the residual.

`emitKvOnlyObjcopyNode` already does the right thing: `kvOnlyLib` (py/namespace,
RESOURCE_FILES-style) → `py3_bin_lib`, `kvOnlyBin` (PY_MAIN, NO_CHECK_IMPORTS) →
`py3_bin`. `emitPySrcObjcopy` also already stamps `py3_bin_lib`. Only the
`emitResourceObjcopy.flush()` RESOURCE path is wrong.

## Change

`emit_py_objcopy.go`, `flush()` switch: collapse the `tokPy3Program` and
`programPairedLib` cases into one `py3_bin_lib` case (RESOURCE is library-side for both
the PROGRAM half — emitted then deduped against the twin — and the paired LIBRARY half).
This makes the stamped tag consistent with the hash tag (`resourceLibTagForData`).

No change to objcopy command args, resource paths, archive membership, generated
outputs, or the COW representative (a PROTO_LIBRARY cpp_proto path, untouched).
