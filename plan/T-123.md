# T-123 plan: cfgproto archive-member position parity (plain LIBRARY)

## Upstream mechanism

`.cfgproto` is a generated-source extension, identical in mechanism to `.ev`:

- `build/ymake.core.conf:3335` — `_SRC("cfgproto", SRC, …)` routes to
  `$_CPP_CFGPROTO_CMD($SRC)`.
- `build/conf/proto.conf:493-498` — `_CPP_CFGPROTO_CMD` expands `$CPP_EV_CMDLINE`
  (the EV protoc command), emitting `${output}` `.cfgproto.pb.{cc,h}`.
- The command outputs are queued back into the module source list
  (`module_builder.cpp` `QueueCommandOutputs()` → `AddSource("SRCS", output)`),
  so the generated `.cfgproto.pb.cc` compile runs in a later FIFO round and its
  object archives AFTER the module's direct (`.cpp`) compiles.

So a `.cfgproto`-generated object must sort as a generated member (after direct
compiles), exactly like `.ev`, `.proto`, etc.

## Defect (local discriminator gap)

`gen.go:isCodegenProducingSrc` (string space) already lists `.cfgproto`, but its
id-space twin `str_intern.go:isCodegenProducingSrcID` omits `srcExtCfgProto`
while it includes `srcExtEv`. Consequences in `gen.go`:

- Pass 1 (`gen.go:2141`, `if !isCodegenProducingSrcID(src) continue`): `.cfgproto`
  is NOT pre-emitted as codegen.
- Pass 2 first loop (`gen.go:2237`, `if isCodegenProducingSrcID(src) continue`):
  `.cfgproto` is treated as a DIRECT source and appended with `generated=false`.

A direct member sorts (via `SrcMeta.sortKey`) before any generated member. In
`SRCS(status_codes.cfgproto status_codes.cpp)` the cfgproto has the lower decl
seq, so `status_codes.cfgproto.pb.cc.o` archives before `status_codes.cpp.o` —
the observed divergence; upstream puts the direct `.cpp.o` first.

## Change

Add `srcExtCfgProto` to the `isCodegenProducingSrcID` switch
(`str_intern.go:248`). This makes id-space agree with string-space:

- Pass 1 pre-emits cfgproto (registering its `.pb.h` in the codegen registry
  before sibling header walks — same benefit `.ev` already has).
- Pass 2 second loop (`gen.go:2248`, `for _, ce := range codegenEmits`) appends
  it with `generated=true`, so it sorts after direct compiles via the existing
  `SrcMeta` mechanism. No new ordering concept introduced.

Mixed acceptance probe (`libmodules-balancer-algorithms.a`): cfgproto, carrying
its SRCS prio (4) + generated flag, sorts after EN serialized members (prio 2,
generated), which themselves sort after ordinary `.cpp.o` (direct). Order:
ordinary cpp → `h_serialized.cpp.o` → cfgproto — matches upstream.
