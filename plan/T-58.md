# T-58 plan: CPP_EVLOG models event2cpp as an ordinary C++ proto plugin on PB producers

## Upstream mechanism

`build/conf/proto.conf:605`:

```conf
macro CPP_EVLOG() {
    CPP_PROTO_PLUGIN0(event2cpp tools/event2cpp DEPS library/cpp/eventlog)
    ENABLE(_BUILD_PROTO_AS_EVLOG)
}
```

`CPP_PROTO_PLUGIN0(NAME TOOL DEPS ...)` (proto.conf:259) registers a C++ proto plugin
whose command tokens come from `_PROTO_PLUGIN_ARGS_BASE` (proto.conf:169):

```conf
.CMD=--plugin=protoc-gen-${Name}=${tool:"$Tool"} --${Name}_out=$ARCADIA_BUILD_ROOT/$PROTO_NAMESPACE ...
```

So for `event2cpp tools/event2cpp` with an empty `PROTO_NAMESPACE` the PB producer
command gains exactly two ref-only tokens observed in sg7:

- `--plugin=protoc-gen-event2cpp=$(B)/tools/event2cpp/event2cpp`
- `--event2cpp_out=$(B)/`

and the producer gains the tool input `$(B)/tools/event2cpp/event2cpp`.

`_BUILD_PROTO_AS_EVLOG` (proto.conf:733) only drives an `ASSERT` that CPP_EVLOG is
incompatible with `USE_VANILLA_PROTOC`; it has no effect on the PB command shape.
Therefore the entire observable CPP_EVLOG PB behavior is the `CPP_PROTO_PLUGIN0` half.

Our code already has a complete, tested C++ proto plugin path (`d.cppProtoPlugins` ->
`pe.extraPlugins`) that: resolves `ctx.tool(ToolPath)`, adds the binary to PB inputs,
emits the `--plugin=`/`--<name>_out=` command tokens (composePBArgBlocks), adds the
plugin foreign dep, and appends the plugin LD ref to the PB generated-output
`GeneratorRefs` (emit_proto.go) — which is precisely the T-47 induced-deps closure.

Today CPP_EVLOG instead sets a bespoke `d.buildProtoAsEvlog` flag that resolves
`tools/event2cpp` a second way (`pe.event2cppLDRef`) purely to carry the T-47
GeneratorRef, while NOT adding the command tokens or the tool input. That is the gap.

## Change (preferred shape from T-57 plan: remove the special case)

Route CPP_EVLOG through the ordinary plugin path, deleting the bespoke flag entirely.

- `modules.go`
  - Add helper `addCPPProtoPlugin(d *ModuleData, plugin CppProtoPlugin)`: appends to
    `d.cppProtoPlugins` and appends `plugin.Deps` to `d.peerdirs`.
  - `case tokCppProtoPlugin0, tokCppProtoPlugin, tokCppProtoPlugin2`: use the helper.
  - `case tokCppEvlog`: call the helper with
    `CppProtoPlugin{Name: "event2cpp", ToolPath: "tools/event2cpp", Deps: []string{"library/cpp/eventlog"}}`.
    Drop `d.buildProtoAsEvlog = true` and the open-coded eventlog peerdir.
  - Remove the `buildProtoAsEvlog` field.
- `emit_proto.go`
  - Remove `PbModuleEmission.event2cppLDRef`, the `if d.buildProtoAsEvlog { ... }` block
    in `newPBModuleEmission`, and the `if pe.event2cppLDRef != 0 { pbGenRefs = append... }`
    block in `emitProtoPB`. The `for _, p := range pe.extraPlugins` loop already appends
    the event2cpp LD ref to `pbGenRefs`, preserving T-47 with no duplicate ref.

No change to `emit_ev.go` or the `.ev` producer path in `emit_proto.go`: that path keeps
its distinct `--event2cpp_out=$(B)` (no slash) + `-I=$(S)/library/cpp/eventlog` shape.

## Test-first

Extend the existing CPP_EVLOG fixtures (which already stub `tools/event2cpp` with an
INDUCED_DEPS) with a regression that asserts, on the PB node producing
`$(B)/evlog/foo.pb.h`/`.pb.cc`:

- `Cmds[0].CmdArgs` contains `--plugin=protoc-gen-event2cpp=$(B)/tools/event2cpp/event2cpp`
- `Cmds[0].CmdArgs` contains `--event2cpp_out=$(B)/`
- both tokens appear after the source proto token
- `flatInputs()` contains `$(B)/tools/event2cpp/event2cpp`
- the event2cpp tool node appears exactly once in PB deps (no duplicate generator ref)
- T-47 induced dep `$(S)/runtime/eventlog_runtime.h` still reaches `foo.pb.cc.o`
- `library/cpp/eventlog` peer propagation unchanged (existing gen_test).

Before the fix the new command/input assertions fail (tokens absent); after, they pass.
