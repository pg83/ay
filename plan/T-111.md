# T-111: LD codegen resource-objcopy over-emission on PY3_PROGRAM link

## Symptom (verified on fresh sg7 artifacts)

Four codegen PROGRAM link (LD) nodes carry one `objcopy_*.o` member in both
`cmds` and `inputs` that the upstream reference lacks:

- `$(B)/ads/bsyeti/libs/events/codegen/codegen` — `objcopy_72dd50ba…o`
- `$(B)/ads/caesar/libs/events/codegen/codegen` — `objcopy_a2dd64f8…o`
- `$(B)/apphost/gp/lib/proto/cow/generator/generator` — `objcopy_120680d8…o`
- `$(B)/bigrt/lib/processing/graph/node_graph/events/codegen/codegen` — `objcopy_1d44c86f…o`

Pair-diff (ours-only, coupled cmds+inputs):

```
[field cmds differs]
  -ours +ads/bsyeti/libs/events/codegen/objcopy_72dd50ba72d82ee89c2707ebe2.o
[field inputs differs]
  -ours +$(B)/ads/bsyeti/libs/events/codegen/objcopy_72dd50ba72d82ee89c2707ebe2.o
```

All four are `PY3_PROGRAM()` with `RESOURCE`/`RESOURCE_FILES`. Raw-record probe
on `ads/bsyeti/libs/events/codegen/codegen`:

- REF link members under that dir: `libpy3libs-events-codegen.global.a` +
  `objcopy_c1580ddb…o` (the PY_MAIN objcopy).
- OURS additionally: `objcopy_72dd50ba…o` (the RESOURCE/ya.make-capture objcopy).
- `objcopy_72dd50ba…o` is ALSO a member of `libpy3libs-events-codegen.global.a`
  on BOTH sides (module_tag `py3_bin_lib_global`).

So our side double-links the RESOURCE objcopy: once as a direct LD member and
once (correctly) as a member of the PY3_BIN_LIB twin's global archive. REF links
it only via the global archive.

## Upstream mechanism

`build/conf/python.conf:348-352`:

```
module PY3_BIN: PY3_PROGRAM_BIN {
    .PEERDIRSELF=PY3_BIN_LIB
    .IGNORED=RESOURCE RESOURCE_FILES PEERDIR ...
    ENABLE(PROCESS_PY_MAIN_ONLY)
}
```

The PROGRAM half (PY3_BIN) `.IGNORED`s the `RESOURCE`/`RESOURCE_FILES` (and thus
`onresource`/`onresource_files`, including PY_CONSTRUCTOR's `onresource`) macros.
The resfs objcopy is owned solely by the PY3_BIN_LIB (LIBRARY) twin, which packs
it into its `<MODULE_TAG>_global` archive; the PROGRAM links it through
`.PEERDIRSELF=PY3_BIN_LIB`, never as a direct member. PY_MAIN / NO_CHECK_IMPORTS
(`PROCESS_PY_MAIN_ONLY`) remain PROGRAM-side and are genuine direct LD members.

Our `emitPySrcObjcopy` already mirrors `PROCESS_PY_MAIN_ONLY` for PY_SRCS
(`emit_py_objcopy.go:656`), which is why PY_SRCS-only PY3_PROGRAMs do not
diverge. The RESOURCE/RESOURCE_FILES path in `emitResourceObjcopy` had no
equivalent guard, so the PROGRAM side emitted the resource objcopy and gen.go
threaded it into `ldObjcopyRefs` (`gen.go:2527`).

## Change

`emit_py_objcopy.go::emitResourceObjcopy`: gate the two `emitEntries(...)` calls
(`d.resources`, `d.pyPyiResources`) behind `!(d.moduleStmt.Name == tokPy3Program
&& !d.programPairedLib)` — the same predicate `emitPySrcObjcopy` uses. The
PROGRAM side then returns only PROGRAM-owned objcopies (PY_MAIN /
NO_CHECK_IMPORTS / ya.conf.json), so `gen.go` no longer adds the resource objcopy
to `ldObjcopyRefs`. The objcopy node is still emitted by the LIBRARY twin and
packed into `py3_bin_lib_global.a`; the program links it via the peer global.

No change to AR/LD source closures, dump-diff cancellation, or the T-103 PY3
objcopy `module_tag` attribution.

## Files

- `emit_py_objcopy.go` — the guard (only change to production code).
- `emit_py_objcopy_test.go` — fail-first test
  `TestGen_Py3ProgramLDDoesNotDirectlyLinkResourceObjcopy`: a PY3_PROGRAM with
  RESOURCE + PY_MAIN; asserts the LD does NOT carry the RESOURCE objcopy
  (cmds/inputs/deps), DOES carry the PY_MAIN objcopy (control), and the
  `py3_bin_lib_global` archive still packs the RESOURCE objcopy. Control
  `TestEmitProgramResource_CppProgramLinksObjcopy` (plain C++ PROGRAM links its
  RESOURCE objcopy directly) stays green — the guard does not touch non-PY3
  PROGRAMs.

## Expected gate effect

- sg2–sg6 stay byte-exact; sg5 under budget.
- The four representative LD pairs lose their coupled cmds+inputs objcopy diffs.
- sg7 exact parity improves (≥4 LD nodes converge; ≥8 fewer field diffs) and does
  not regress.
- bsyeti AR/LD T-109 representatives stay header-only under `--pair`.
