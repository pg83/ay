## Upstream mechanism

### Issue 1: Missing Wno flags on re_parser.y.cpp.o

Upstream ymake defines `_LANG_CFLAGS_BISON = -Wno-unused-but-set-variable -Wno-deprecated-copy` (ymake.core.conf:3593). 
These flags are applied to the CC compilation of any bison-generated `.cpp` (for `.y` or `.ypp` sources) via `_LANG_CFLAGS_VALUE_NEW` (ymake.core.conf:3603-3604).

The flags go in `PerSrcCFlags` position in the CC command (same position as `-Wno-implicit-fallthrough` for ragel-generated files).

### Issue 2: Missing .y source input on re_lexer.cpp.o

`re_lexer.cpp` includes `re_parser.y.h` (bison-generated header). Upstream propagates the source `.y` file as an input to any CC node that includes the generated bison header.

In our system, `bisonCppHeaderParsed(srcVFS)` already receives `srcVFS` (the `.y` file) but ignores it — it only includes `bisonPreprocessPyVFS` and `bisonCppSkeletonInputs`. This means any CC node including `re_parser.y.h` gets only the skeleton/preprocess files in its closure but NOT `re_parser.y`.

The fix: add `srcVFS` to the parsed includes of the bison header in `bisonCppHeaderParsed`. Any CC node whose closure reaches `re_parser.y.h` will then also get `re_parser.y` as a closure entry.

This mirrors how `bisonGeneratedCPPParsed` works — it includes `{headerVFS, srcVFS, localIncludes}`, making the generated `.cpp.o` node include `re_parser.y` as an input. The header should do the same.
