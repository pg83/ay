# T-8 plan: proto generated C++ object path-shape for SRCDIR/external-namespace cases

## Symptom (sg7 output-only buckets)

For `market/proto/content/ir/common` (a `PROTO_LIBRARY` with
`SRCDIR(market/proto/content/ir)` + `SRCS(BusinessCleanWebStatus.proto …)`),
the compiled proto object diverges only in PATH SHAPE:

- ours:  `$(B)/market/proto/content/ir/common/_/market/proto/content/ir/BusinessCleanWebStatus.pb.cc.o`
- ref:   `$(B)/market/proto/content/ir/common/__/BusinessCleanWebStatus.pb.cc.o`

The generated `.pb.cc` FILE path matches on both sides
(`$(B)/market/proto/content/ir/BusinessCleanWebStatus.pb.cc`); only the
compiled-object output name differs. So this is purely an object-path selection
defect in `composeCCPaths`, not a reachability or content gap.

## Upstream mechanism

ymake names a compiled object by rebasing the source's path under the module's
BINDIR. When the source lies OUTSIDE the module dir (SRCDIR points at a parent /
sibling directory), each `..` ascent in the module-relative path becomes a `__`
segment. This is the SAME rule ymake uses for an ordinary `$(S)` SRCDIR C++
source — our `composeSrcDirOutputRel` already reproduces it and the
`srcVFS.isSource() && !canonMatches()` branch already uses it.

The generated `.pb.cc` is a `$(B)` (build) VFS, so it never reached that branch.
Its logical location (`market/proto/content/ir/BusinessCleanWebStatus.pb.cc`) is
one directory ABOVE the module `market/proto/content/ir/common`, exactly the
SRCDIR-ascent case (the `.proto` was found via SRCDIR one level up, so the
codegen output inherits that ascent). Upstream rebases it to `__/…`.

Our build-path branch instead fell through to
`normalizeDotDotSegments(srcRel)`, where `srcRel` is the FULL module-rooted path
(the `TrimPrefix(pbCC, moduleDir+"/")` was a no-op because the path is outside
the module), producing the wrong `_/<full-path>` shape.

## Change

`emit_cc.go` `composeCCPaths`: before the existing `srcRel`-based switch, add a
branch — when `srcVFS` is a build VFS whose rel is NOT under the module dir,
rebase it under the module BINDIR via `composeSrcDirOutputRel(moduleDir,
srcVFS.rel())` (the `..`→`__` mechanism), mirroring the `$(S)` SRCDIR source
branch. In-module build sources (proto subdir, bison `_/`-prefixed
intermediates) keep `srcVFS.rel()` under the module dir, so they are untouched
and the existing switch handles them exactly as before.

No new data flow, no context-dependence; reuses the existing
`composeSrcDirOutputRel` upstream-faithful helper.
