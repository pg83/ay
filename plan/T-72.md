# T-72 plan: OUTPUT_INCLUDES source closure on a generated-C++ `RUN_PROGRAM` producer

## Hypothesis / divergence

`yabs/server/libs/protocol/proto_flat_buf` has a sibling
`RUN_PROGRAM(... Cpp OUTPUT_INCLUDES <3 generated headers> STDOUT formula_parameters.cpp)`.
The reference PR producer of `$(B)/.../formula_parameters.cpp` carries a 1602-input
closure: the **transitive `$(S)` source closure of its 3 `OUTPUT_INCLUDES`
headers** (`.fbs.h` → flatbuffers; `search_formula_params.pb.h` and
`formula_parameters.pb.h` → their `.proto` import sources + protobuf/grpc/abseil/
libcxx headers). It keeps a single dep (the generator `LD`) and **no** `$(B)`
codegen `.pb.h`/`.fbs.h` intermediates as inputs.

Our PR producer carries only the generator binary (1 input). The downstream
compile `formula_parameters.cpp.o` is already byte-exact (T-69); only the producer
drifts.

## Upstream mechanism (ya/ymake)

`RUN_PROGRAM`'s `${hide;output_include:OUTPUT_INCLUDES}` attaches induced includes
to the run's **MAIN output** (`FindMainElemOrDefault(GetOutput(), 0)` —
macro_processor.cpp:1041; output order `OUT, OUT_NOAUTO, STDOUT`). Whether that
induced-include closure surfaces on the PRODUCER's own `inputs` is determined,
empirically across 9 real sg7 cases, by the **type of the main output**:

| run | main output | producer closure |
|-----|-------------|------------------|
| formula_parameters (STDOUT .cpp) | `.cpp` (cc) | YES (1602) |
| by_name / sys_const_transfer (STDOUT .cpp) | `.cpp` (cc) | YES |
| registrar (OUT_NOAUTO .cpp) | `.cpp` (cc) | YES (already via headerOnlyNoIN) |
| features.gen (OUT .h .cpp) | `.h` (header) | NO |
| profile_traits / printer_utils / user_profile_traits / adfox (OUT .h .cpp) | `.h` | NO |

Rule: **when the run's MAIN output is a C/C++ translation unit, the
OUTPUT_INCLUDES source closure rides the producer; when the main output is a
header with a compiled cc-source sibling, it rides the header to its consumers
instead.** A header-only / OUT_NOAUTO-only no-IN run (no cc-source output at all)
already rides the producer in our code (the `headerOnlyNoIN` path), and registrar
(OUT_NOAUTO .cpp) is covered by it because OUT_NOAUTO is not an auto cc-source.

## Current code (emit_pr.go `prInputClosure`)

`hasAutoCCSourceOut` is true for any auto cc-source OUT/STDOUT; `headerOnlyNoIN =
noIN && !hasAutoCCSourceOut`. For `formula_parameters` (no IN, cc STDOUT)
`hasAutoCCSourceOut` is true ⇒ `headerOnlyNoIN` false ⇒ the function early-returns
`nil`. So the OUTPUT_INCLUDES closure is never walked for the producer.

## Change (minimal, reuse the existing OUTPUT_INCLUDES walk)

In `prInputClosure`:
1. Compute the run's MAIN output rel (OUT[0] → OUT_NOAUTO[0] → STDOUT) and
   `mainIsCC := isCCSourceExt(mainOut)`.
2. Replace `headerOnlyNoIN` with a `fullSourceClosure` flag:
   `fullSourceClosure := len(IN)==0 && (!hasAutoCCSourceOut || mainIsCC)`.
   - header-only / OUT_NOAUTO (registrar): `!hasAutoCCSourceOut` ⇒ true (unchanged).
   - cc-main (formula_parameters): `mainIsCC` ⇒ true (new).
   - header-main with cc sibling (features.gen): both false ⇒ early-return nil.
3. Early-return gate becomes `len(IN)==0 && !fullSourceClosure`.
4. Use `fullSourceClosure` in place of `headerOnlyNoIN` in the OUTPUT_INCLUDES
   block: `keep` = `v.isSource()`, the source-tree-header branch, and the
   WKT-`.pb.h`-sibling synthesis guard (all unchanged semantics for IN runs).
5. Guard the generated cc-source `walkOne` blocks (the producer scanning its own
   `.cpp` OUT/STDOUT) on `len(IN) > 0`: for no-IN cc-main runs the producer must
   NOT scan its own generated `.cpp` (that would drag the `$(B)` codegen headers
   the consumer carries); the OUTPUT_INCLUDES `keep=isSource` walk supplies the
   producer's closure. For headerOnlyNoIN these blocks were already no-ops, and
   IN runs are unchanged.

`keep=isSource` keeps only `$(S)` files, so no `$(B)` intermediate is added and
`resolveCodegenDepRefs`' build-gate adds no extra producer dep — the single
generator-`LD` dep is preserved.
