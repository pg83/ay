# T-37 — cpp_import yaff producer module_dir attribution (MULTIMODULE / host-vs-target claim conflict)

## Symptom
After T-26, sg7 has 3 `*.yaff.h` RUN_PROGRAM producers under
`yabs/server/cs/libs/cpp_import/yaff` whose `target_properties.module_dir`
diverges from the reference (only `target_properties` differs; inputs/cmds are
byte-exact):

| output | ours | reference |
|---|---|---|
| plutonium_aggregated_socdem_profile_local.yaff.h | yabs/server/libs/handlers/meta | yabs/server/cs/libs/plutonium_traits |
| plutonium_ctv_content_tags_local.yaff.h | yabs/server/libs/server | yabs/server/cs/libs/plutonium_traits |
| plutonium_telegram_brief_filter_local.yaff.h | yabs/server/libs/server | yabs/server/cs/libs/plutonium_traits |

All other 815 `*.yaff.h` producers already match. The 3 affected headers are
each directly `#include`d by several modules (plutonium_traits, server,
handlers/meta, rtb, aggregated_ctv_socdem_profile).

## Upstream mechanism
`devtools/ymake/json_subst.cpp:265` sets `module_dir = Node2Module[node].GetDir()`.
`Node2Module` is populated in `json_visitor.cpp:638` with `try_emplace` (first
write wins) during a SINGLE post-order DFS of the export graph: the value is the
nearest enclosing module on the visitor stack at the node's first leave
(`FindModule(state)`). There is one unified graph/DFS — no host/target split.

## Our mechanism and the defect
`scanner.go` records `generatedFirstClaim[output] = OwnerModuleDir` on the first
include-scan that resolves a generated header (mirroring Node2Module). We run TWO
include scanners — `targetScanner` and `hostScanner` — and
`mergeGeneratedFirstClaims(targetScanner, hostScanner)` (gen.go) merges them
first-wins with **target first**, i.e. "target wins on conflict", under the
documented assumption that "the host scanner only sees CC compiles for tool
builds, which are an orthogonal claim space."

That assumption is empirically false. Instrumenting the run shows EXACTLY 3
host/target claim conflicts across all of sg7 — precisely these 3 headers. In
every conflict the host scanner claims `plutonium_traits` (the reference-correct
owner) and the target scanner claims a downstream peer (server / handlers/meta);
"target wins" then discards the correct host claim.

`plutonium_traits` directly `PEERDIR`s the producer module
`yabs/server/cs/libs/cpp_import/yaff` and is reached inside the producer-side
(host) closure where the co-located trait library first consumes the generated
header — which is what upstream's single Node2Module DFS attributes it to. The
target scanner only re-encounters the same header through downstream peers later.

## Fix
On a host/target claim conflict, prefer the HOST scanner's claim. Implemented by
merging with the host scanner first (first-wins), and correcting the doc comment.
This changes ONLY keys where both scanners claim AND disagree — exactly the 3
conflicts in sg7; every other generated file is claimed by at most one scanner
(or both agree) and is unaffected.

Also collapse the 4 duplicated inline claim-recording blocks in scanner.go into a
single `recordFirstClaim` helper (net simplification).

## Files
- `gen.go` — `mergeGeneratedFirstClaims` call order + doc.
- `scanner.go` — `recordFirstClaim` helper replacing 4 inline blocks.
- `gen_test.go` — focused test pinning host-wins-on-conflict and
  single-claimant-preserved.
