# T-44 — Fix FROM_SANDBOX script-input over-collection

## Upstream mechanism

The `FROM_SANDBOX` macro `.CMD` (yatool `build/ymake.core.conf:4894`) lists its
script inputs as three *explicit* directives:

```
$YMAKE_PYTHON3 ${input:"build/scripts/fetch_from_sandbox.py"}
              ${hide;input:"build/scripts/process_command_files.py"}
              ... -- $OUT $OUT_NOAUTO
              ${hide;input:"build/scripts/fetch_from.py"} ...
```

ymake's `${input:"..."}` adds exactly the named file as a command input. It does
**not** scan that Python file's `import` graph and add the transitive closure —
which is why the macro author names `process_command_files.py` and
`fetch_from.py` explicitly (they would be redundant if ymake expanded imports).
The closures seen on LD/AR nodes likewise come from those macros explicitly
listing every helper (`${hide;input:process_command_files.py}` etc.), not from
import scanning.

So the upstream FROM_SANDBOX SB node carries exactly three script inputs:
`fetch_from_sandbox.py`, `process_command_files.py`, `fetch_from.py`.

## Our divergence

`emit_from_sandbox.go:83` sets the SB node inputs to
`ctx.scripts[buildScriptsFetchFromSandboxPy]` — the *import closure* table entry,
which is `{fetch_from_sandbox, process_command_files, fetch_from, retry, error}`
(fetch_from_sandbox imports process_command_files + fetch_from; fetch_from
imports retry; both inline-`import error`). That appends `error.py` and
`retry.py`, which upstream never lists.

Confirmed against sg7:
- ours: error.py, fetch_from.py, fetch_from_sandbox.py, process_command_files.py, retry.py
- ref:  fetch_from.py, fetch_from_sandbox.py, process_command_files.py

This self-content divergence makes every reachable SB node's self_uid differ and
cascades into the dependent AR/link nodes (incl. the MKL `libintel-mkl-*.a`
archives made reachable by T-39).

## Change

- vfs_consts.go: add `buildScriptsFetchFromPy` and
  `buildScriptsProcessCommandFilesPy` constants (the two explicit helpers).
- emit_from_sandbox.go: replace the closure-expanded
  `ctx.scripts[buildScriptsFetchFromSandboxPy]` input with the explicit three
  scripts the macro names. `ctx` is then unused by the input set — keep it for
  the codegen-registry calls.

This is generic to FROM_SANDBOX emission; no MKL / feature_store / resource-path
special-casing.

## Files touched

- `emit_from_sandbox.go`
- `vfs_consts.go`
- `emit_from_sandbox_test.go` (regression)

## Test

`TestGen_FromSandboxScriptInputsExplicitThree`: build a memFS with
fetch_from_sandbox.py (imports process_command_files, fetch_from), fetch_from.py
(imports retry), retry.py, error.py present under build/scripts; emit a
FROM_SANDBOX; assert the SB node inputs contain the three named scripts exactly
once each and exclude error.py and retry.py. Fails on current closure-expanded
code (it lists 5), passes after the fix.

## Expected gate effect

- sg7: SB nodes (and their AR/link dependents, incl. MKL archives) lose the
  error.py/retry.py over-inclusion → self_uid matches ref → matched up / only
  down.
- sg2–sg6: byte-exact gating must not regress (verify whether they carry SB
  nodes; if so they move toward the three-script form, else unchanged).
