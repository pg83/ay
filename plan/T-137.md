# T-137 plan: yabs plugin copied-source PY_SRCS path/closure parity

## Representatives (sg7)

Module `yabs/server/cs/pylibs/servant_app` (PY23_LIBRARY) stages two sources
from another module via COPY_FILE, then lists them in PY_SRCS:

```
COPY_FILE(yabs/server/cs/package/plugin/keys.py     keys.py)
COPY_FILE(yabs/server/cs/package/plugin/settings.py settings.py)
PY_SRCS(__init__.py argv.py iface.py run.py secrets.py keys.py settings.py settings_args.py)
```

Current normalized sg7 diffs (ours missing the ref-only rows):

```
keys.py.yapyc3   (py3cc bytecode):  +$(S)/build/scripts/fs_tools.py
                                     +$(S)/build/scripts/process_command_files.py
                                     +$(S)/yabs/server/cs/package/plugin/keys.py
settings.py.yapyc3 (bytecode):      same three (the settings.py copy closure)
objcopy_8b… (pysrc .py chunk):      +$(S)/…/package/plugin/keys.py
                                     +$(S)/…/package/plugin/settings.py
objcopy_4a6f… (settings.py.yapyc3): ours $(B)/…/servant_app/settings.py
                                     ref  $(S)/…/package/plugin/settings.py
```

## Upstream mechanism

`COPY_FILE(File Dest)` is a NOAUTO copy (`build/scripts/fs_tools.py copy`).
`PY_SRCS(keys.py)` then references the staged `$(B)/…/servant_app/keys.py`.
Upstream's flat-input model lists, on every node that consumes a build output,
that output's producer's transitive **$(S)** input closure. The copy node's
$(S) inputs are the original source plus the copy tooling
(`fs_tools.py` → `process_command_files.py`). Hence:

- The py3cc bytecode node compiling the staged copy carries
  `{original $(S) source, fs_tools.py, process_command_files.py}`
  in addition to the staged `$(B)` copy it compiles.
  (Verified: ref `keys.py.yapyc3` node, sg7.json.)
- The resource-objcopy node that embeds the staged `.py` / its `.yapyc3`
  carries the same copy producer source closure. Normalization
  (`filterObjcopyInputs`, ref-side only) drops the `$(S)/build/*.py` scripts but
  keeps the original `$(S)` source — so only the original source must surface in
  OUR objcopy inputs; adding the scripts to ours would create ours-only rows
  (our objcopy inputs are NOT normalized).

This is the same `ProducerSourceClosure` machinery already used for RUN_PROGRAM
/ swig generated PY_SRCS (emit_py_codegen.go reads it). COPY_FILE simply never
recorded it.

## Files to touch

1. **emit_copy.go** — when registering a COPY_FILE dst (`srcVFS != dstVFS`),
   record `ProducerSourceClosure = {srcVFS} ++ scripts[copyFsToolsVFS]`
   (= original source + fs_tools.py + process_command_files.py). This is the CP
   node's own $(S) input set, already computed there. emit_py_codegen.go picks
   it up unchanged → fixes both `*.py.yapyc3` bytecode nodes.

2. **resource.go `buildPySrcEntriesFor`** — capture the `GeneratedFileInfo`
   (not just `!= nil`). For a copy-staged entry (`info.SourcePath != 0`):
   - the `.py` entry's payload input stays the staged `$(B)` copy (cmd parity),
     but the entry additionally carries the original `$(S)` source
     (`info.SourcePath`) as a node input;
   - the `.yapyc3` entry carries the original `$(S)` source instead of the
     staged `$(B)` copy.
   Generalize `PySrcEntry.extraSrcInput *VFS` → `extraInputs []VFS`, attached to
   both entries via `chunkPySrcEntries`. Non-copy generated and ordinary source
   entries keep their current single-VFS behavior.

The objcopy reads only `info.SourcePath` (one path), NOT the scripts, because
ours-side objcopy inputs are unfiltered.

## Rework: restrict copy-closure propagation to source-root copies

Reviewer found the first pass leaked into the deferred feature_store bucket
(plan T-132 bucket C). `COPY_FILE(${ARCADIA_BUILD_ROOT}/.../gen_consts/generated_consts.py
generated_consts.py)` stages a `$(B)` generated output, not a `$(S)` source.
The unconditional `ProducerSourceClosure = [srcVFS, scripts...]` therefore put a
build-root input (`$(B)/.../gen_consts/generated_consts.py`) onto the parent
PY_SRCS bytecode node, and the resource `copyStaged` branch named that same
build-root path on the objcopy — work T-137 explicitly scopes out.

A packaging-stage copy whose original lives in `$(S)` is the only case with a
real `$(S)` closure to lift onto the staged copy's consumers. A copy of a `$(B)`
generated output has none here — that producer's source closure is modeled by
the generated-resource buckets. Guard both sites on the original being a source:

- `emit_copy.go`: set `ProducerSourceClosure` only when `srcVFS.isSource()`.
- `resource.go`: `copyStaged` also requires `genInfo.SourcePath.isSource()`,
  so a build-root staged copy keeps the pre-existing staged-`$(B)`-copy edge.

Regression test `TestGen_CopyFileStagedBuildRootSrcCarriesNoOriginalClosure`
stages a `${ARCADIA_BUILD_ROOT}` source and asserts neither the bytecode node
nor the objcopy names the original `$(B)` producer path. sg7 matched=67536
(yabs reps still clean), feature_store `-ours +$(B)/.../gen_consts/...` leak gone.
