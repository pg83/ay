# T-3: Fix two LLVM16 BC node divergences in sg5

## Upstream mechanism

### Issue 1: `${__COMMA__}` in OP node

`build/plugins/llvm_bc.py` passes `'\'-passes="{}"\''.format('${__COMMA__}'.join(passes))`
to `unit.onllvm_opt(...)`. ymake expands `${__COMMA__}` → `,` (defined in
`build/ymake.core.conf` as `__COMMA__=${comma:""}`) and strips the outer
single-quote shell-quoting syntax before writing the graph JSON.

Our `emit_llvm_bc.go:98` emits the string BEFORE expansion, leaving both
`${__COMMA__}` unexpanded AND the outer single quotes. Fix: emit `,` directly
and drop the outer single quotes.

### Issue 2: BC compile node missing full flag set and wrong source

`LLVM_COMPILE_CXX` in `build/ymake.core.conf:4961`:
```
.CMD=$YMAKE_PYTHON3 ${input:"build/scripts/clang_wrapper.py"} $WINDOWS
  ${CLANG_BC_ROOT}/bin/clang++
  ${pre=-I:_C__INCLUDE}
  $BC_CXXFLAGS        ← = $CXXFLAGS (all module CXX flags)
  $C_FLAGS_PLATFORM   ← --target=x86_64-linux-gnu -B/usr/bin (comes AFTER CXXFLAGS)
  -Wno-unknown-warning-option $LLVM_OPTS ${NO_LTO_CFLAGS} ${NO_PGO_CFLAGS}
  -emit-llvm -c ${input:Input} -o ${noauto;output:Output} $Opts
```

Comparing BC vs CC for the same module (sg5 `yt_codec_bc.cpp`):
- BC starts with `python3 clang_wrapper.py no clangBC++` (wrapper prefix)
- BC includes: include paths, hostDefines, OwnCFlags, noLibcBlock×2, catboost×1,
  moduleScopeCFlags, -std=c++20, cxxStdWarnings, OwnExtras, OwnGlobalBucket,
  composePostCatboostBucket
- BC puts `--target=... -B/usr/bin` AFTER all CXXFLAGS (unlike CC which puts them early)
- BC OMITS: bundle.CFlags (x86TargetCFlags), warningBundle, debugPrefixMapFlags,
  xclangDebugCompilationDir, the extra catboostOpenSourceDefine (2nd one),
  builtinMacroDateTime, macroPrefixMapFlags, PerSrcCFlags
- BC source: build-root copy for generated sources (already handled by `llvmBcInputVFS`)

Our current code only emits: `-emit-llvm -c src -o out` (missing everything else).

### Issue 3 (rework 2): BC node `inputs` missing include closure

Reviewer found via `./ay dump diff --pair '$(B)/yt_codec_bc.cpp.16.bc'` that the BC node's
`inputs` field still diverges: our side has only `[inputVFS]` while upstream has the full
transitive include closure (all headers the source includes).

Upstream mechanism: normal CC/codegen compiles call `walkClosure` to get the transitive
include set, `dropTransitiveGeneratedProto` to trim intermediate `.proto` files, and
`resolveCodegenDepRefs` to find extra producer refs from build-root closure hits.
`emit_codegen_cc.go:8-46` shows the canonical pattern (`emitCodegenDownstreamCC`).

Our `emitLLVMBC` hand-rolled `Inputs: []VFS{inputVFS}` with only a single input and no
closure, bypassing the normal mechanism entirely.

Fix: apply the same `walkClosure` + `dropTransitiveGeneratedProto` + `resolveCodegenDepRefs`
pattern in the BC source loop, setting `Inputs = [inputVFS] + closure` and
`DepRefs = producer + extra_from_closure`.

- `emit_llvm_bc_test.go`: add failing tests first
- `gen.go`: pass `moduleInputs` to `emitLLVMBC`

- OP node: `-passes="default<O2>,globalopt,globaldce,internalize"` (no ${__COMMA__}, no quotes)
- BC node cmds: full flag set with includes, defines, CXX flags, --target/-B after flags
- BC node inputs: [inputVFS] + full transitive include closure
- Both `./ay dump diff --pair` checks pass for both ticket nodes

`sg5 matched` should not decrease, gating cases OK.
