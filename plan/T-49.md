`$(B)/yabs/server/libs/aggregated_ctv_socdem_profile/enums/socdem_type.h_serialized.cpp`
(EN) and `…/_/enums/socdem_type.h_serialized.cpp.o` (CC) are **byte-identical**

yabs/server/libs/aggregated_ctv_socdem_profile`. socdem is a plain

### Remaining attribution class: solomon submodule (reported separately)

Exactly three serialized-enum EN nodes drift on `module_dir` only (CC compile
nodes already match):

| output (`_serialized.cpp`) | ours (EN) | ref (EN) | CC ours/ref |
|---|---|---|---|
| `yabs/server/libs/server/degradation_stats.h` | `…/server` | `…/server/solomon` | `…/server` both |
| `yabs/server/libs/server/hit_cost_param.h` | `…/server` | `…/server/solomon` | `…/server` both |
| `yabs/server/libs/server/solomon/types.h` | `…/server` | `…/server/solomon` | `…/server` both |

All three are `GENERATE_ENUM_SERIALIZATION_WITH_HEADER(...)` declared in the
**server** LIBRARY (`yabs/server/libs/server/ya.make:1550-1554`). The other
server WITH_HEADER enums (`ctr_prediction`, `dynamic_top_user_features`,
`secrets_enums`) do NOT drift — they stay `…/server` in both.

Discriminator: the generated `_serialized.h` of exactly these three is
explicitly `#include`d by `yabs/server/libs/server/solomon/services.h` — a
header physically located in the **solomon** submodule's directory. ymake's
`Node2Module` (json_visitor.cpp:638, set on first DFS-leave via
`FindModule` = nearest module on the visitor stack) leaves the EN node under
the solomon submodule before the declaring parent (server) completes, so the
generation node is attributed to solomon. The compile (`.cpp.o`) is a SRC of
server and stays with server.

### Why this is NOT reproducible at emit time / via the existing first-claim path

- Parents of the EN node in BOTH graphs are identical (server / handlers /
  phantom / daemons CC nodes + the LD root). The solomon LIBRARY is **not** a
  dep-graph parent of the EN node in either graph — its attribution comes purely
  from ymake's DFS module-stack visitation through solomon's directory-owned
  header (`services.h`), a path that survives as no dep edge.
- Our scanner-based first-claim (`recordFirstClaim`, OwnerModuleDir = the
  *compiling* module) records **server** for these outputs: server compiles
  `solomon/services.cpp` and `degradation_stats.cpp`, both of which reach the
  `_serialized.h` under a server scan context. solomon compiles only
  `config_service.cpp` (→ `config_service.h` → `<inttypes.h>`), which has no
  path to these headers — so solomon never enters our first-claim map for them.
- Empirical confirmation: extending `overrideGeneratedModuleDir` to `pkEN`
  leaves all three UNCHANGED (`…/server`) and REGRESSES 30+ correct
  `yabs/server/libs/enums/*` nodes (declaring module → first consumer). The
  consumer-claim override is the wrong mechanism for EN.

Reproducing the solomon attribution requires modelling ymake's module-owns-its-
directory-headers visitation plus DFS post-order (submodule-before-parent)
claim ordering — machinery our architecture does not have and which cannot be
derived from emit-time or current first-claim data. That is a separate,
larger change; this ticket reports it as a remaining attribution class.
