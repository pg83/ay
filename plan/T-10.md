# T-10: LLVM16 objcopy PY node missing BC closure inputs

## Upstream mechanism identified

In upstream ymake, the `RUN_PYTHON3` macro for the objcopy step uses `${input:__PATHS}` which
references the optimized BC file. When ymake resolves this input reference, it propagates the
producer node's inputs (the OP node's full source closure) into the consumer (the objcopy PY node).

In our code, this propagation is handled by the `prOutputInputs` mechanism:
- `emit_pr.go` and `emit_py_run.go` populate `d.prOutputInputs[output]` with the producer node's inputs
- `emitResourceObjcopy` calls `prResourceExtraInputs(d, e.Path)` to fetch these inputs and add them as `extraInputs` to the PY node

## Root cause

`emitLLVMBC` correctly sets `d.prOutputProducer[optOutName] = opRef` (so the OP node is linked
as the dependency), but it never sets `d.prOutputInputs[optOutName]`. As a result,
`prResourceExtraInputs` returns nil, and the PY objcopy node only gets 4 inputs instead of 1628.

## Files to touch

- `emit_llvm_bc.go`: add `d.prOutputInputs[optOutName] = optInputs` after building the OP node
- `emit_llvm_bc_test.go`: add failing repro test that asserts the PY node includes BC closure scripts

## Expected effect

After the fix:
- PY node inputs for the LLVM16 objcopy will include `clang_wrapper.py`, `llvm_opt_wrapper.py`,
  and all the BC compilation closure headers
- sg5 matched count will increase by 1 (the objcopy_eb223d69... node stops diverging)
- The remaining 15 sg5 divergences are unrelated to LLVM BC (pire regex, ydb modules, etc.)
- sg2/sg2_x86_64/sg3/sg4 remain byte-exact

## Validation

1. `go test ./...` — existing tests pass, new test passes after fix
2. `./dev/validate.py .out/digger-validate` — sg5 matched increases, gating cases stay OK
