# T-10 plan: ARCHIVE(NAME ...) generated build-dir include propagation

## Upstream mechanism

`build/ymake.core.conf` line 4164:

    macro ARCHIVE(NAME="", DONTCOMPRESS?"-p":"", Files...) {
        .CMD=$ARCH_TOOL -q -x $DONTCOMPRESS ${suf=\:;input:Files} -o ${addincl;noauto;output:NAME} ...
    }

The `-o` argument uses the `${addincl;noauto;output:NAME}` modifier chain:

- `output:NAME` — emit the build output `$(B)/<moddir>/<NAME>` (already modeled by
  `emit_archives.go`, which registers the output in the CodegenRegistry).
- `addincl` — side effect: add the output's *directory* (`$(B)/<moddir>` when NAME
  has no subdir) as an include directory reaching the declaring module and its
  PEERDIR consumers.
- `noauto` — per conf line 76, only suppresses auto-wiring the output as an input
  to other module commands; it does NOT affect the `addincl` side effect.

This is the identical modifier already handled for `CONFIGURE_FILE`
(`${addincl;output:Dst}`, conf line 4261) via `addGeneratedHeaderInclude`, which is
validated byte-exact in sg2–sg6. So the upstream-correct fix is to route ARCHIVE's
NAME output through the same generated-include path.

## Observed gap

Reference `$(B)/ads/bsyeti/libs/features/formula.cpp.o` carries
`-I$(B)/kernel/lemmer/context/default_decimator`; ours does not. Producer:
`kernel/lemmer/context/default_decimator/ya.make` declares
`ARCHIVE(NAME data.inc default_decimator.lst)`. Our `applyArchiveStmt` records the
entry and emits the output, but never adds the build-dir include, so it never
reaches `ModuleEmitResult.AddInclGlobal` / `AddInclUserGlobal` and never propagates
to the transitive consumer.

## Change

Upstream `addincl` resolves to `Module.IncDirs.Add(Parent(output), Global)` which
pushes the dir into the module's local + user-global + global buckets, at the
ARCHIVE statement's processing position. Reproduced as `applyArchiveAddIncl`
(`modules.go`), run during module finalization **after** `applyPython3AddIncl`
(our python include is itself a finalization append, so the build-root dir must
follow it to match upstream declaration order). For each `d.archives` entry it
appends `$(B)/<dir-of-NAME>` to `d.addIncl`, `d.addInclGlobal`,
`d.addInclUserGlobal`. The dir computation is factored into `generatedIncludeDir`,
shared with `addGeneratedHeaderInclude` / `addGeneratedHeaderIncludeCF`.

This **subsumes and deletes** the former `library/python/runtime_py3`-specific
local-include hardcode in `applyPython3AddIncl` (runtime_py3 declares two
`ARCHIVE(NAME *.pyc.inc ...)`, so the generic mechanism now supplies its
build-root include). The runtime_py3 propagation-position splice in
`gen.go` (insert after abseil for the consumer-propagated `effectiveAddInclGlobal`)
is retained: it places the propagated dir at the upstream peer-traversal position,
which the own-module bucket order does not reproduce. `emit_archives.go` producer
registration is unchanged.
