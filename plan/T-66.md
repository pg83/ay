# T-66 plan: model `YMAPS_SPROTO` command `.PEERDIR=maps/libs/sproto`

## Upstream mechanism

`build/internal/conf/project_specific/maps/sproto.conf`:

```
macro _YMAPS_GENERATE_SPROTO_HEADER(File) {
    .CMD=${cwd;rootdir;input:File} ${tool:"maps/libs/sproto/sprotoc"} ...
    .PEERDIR=maps/libs/sproto
}
macro YMAPS_SPROTO(FILES...) {
    foreach (FILE : $FILES) { _YMAPS_SPROTO_DISPATCH(${lastext:FILE} $FILE) }
}
macro _YMAPS_SPROTO_DISPATCH("proto", FILE) { .CMD=$_YMAPS_GENERATE_SPROTO_HEADER($FILE) }
```

`_YMAPS_GENERATE_SPROTO_HEADER` runs once per `.proto` named by `YMAPS_SPROTO`
and declares a command-level `.PEERDIR=maps/libs/sproto`. That peer is added to
the *module that runs the macro* — the maps proto library — not only to the
`sprotoc` host tool. The peer propagates the target-side (non-PIC)
`$(B)/maps/libs/sproto/libmaps-libs-sproto.a` through ordinary peer-archive
closure, so a final non-PIC program/daemon link reaches it.

Today our `case tokYmapsSproto` records only the source `.proto` args
(`d.ymapsSprotoSrcs`) for `emit_ymaps_sproto.go`. The PIC archive already exists
in our graph because the host `sprotoc` PROGRAM declares `PEERDIR(maps/libs/sproto)`
and is built on the host platform (PIC=yes). The missing piece is the non-PIC
target archive reachable through the proto module's own peer closure.

## Change

`modules.go`, `case tokYmapsSproto`: after validating each arg is `.proto`,
append `maps/libs/sproto` to `d.peerdirs` when the statement has at least one
arg. Dedup is handled by the existing `peerSeen` peer-assembly pass, so repeated
files / repeated `YMAPS_SPROTO(...)` statements are harmless. This rides the
existing `d.peerdirs -> genModule -> PeerArchiveClosure` path; no link-level
patch, no `yabs-server` allowlist, no change to the `.sproto.h` producer nodes.

`str_consts.go`: add `strMapsLibsSproto = internStr("maps/libs/sproto")`.

`emit_ymaps_sproto.go` / `emitYmapsSprotoHeaders`: unchanged — those producer
nodes are already byte-exact.

## Tests

1. New fixture: a `PROTO_LIBRARY()` with `EXPORT_YMAPS_PROTO()` + `YMAPS_SPROTO(...)`,
   a normal `maps/libs/sproto` library (`sproto.cpp`), a host `sprotoc` PROGRAM
   peering `maps/libs/sproto`, and a `PROGRAM()` consumer that `PEERDIR`s the
   proto library. Assert:
   - both `$(B)/maps/libs/sproto/sproto.cpp.o` (non-PIC) and
     `$(B)/maps/libs/sproto/sproto.cpp.pic.o` (PIC) objects exist;
   - two `libmaps-libs-sproto.a` AR nodes exist, one with the `.o` member and
     one with the `.pic.o` member;
   - the consumer program link lists `maps/libs/sproto/libmaps-libs-sproto.a`
     before the proto library's own archive.
2. Negative guard: a proto module with `EXPORT_YMAPS_PROTO()` + ordinary
   `.proto` SRCS but NO `YMAPS_SPROTO(...)` does not gain `maps/libs/sproto`.

## Expected gate effect

- `$(B)/maps/libs/sproto/sproto.cpp.o` becomes present on the left, pairs
  byte-exact with ref.
- `$(B)/maps/libs/sproto/libmaps-libs-sproto.a` gains the non-PIC variant; pair
  diff stays empty.
- `yabs-server` pair loses only the `maps/libs/sproto/libmaps-libs-sproto.a`
  cmds/inputs line; libm / `.gztproto` / uppercase `.C` residuals unchanged.
- sg2–sg6 byte-exact, sg5 within time budget.
