# T-91 plan: enum-serialization / generated-registrar proto over-closure

## Symptom (reproduced from trunk baseline)

`./dev/validate.py` sg7, representative pairs:

```
ay dev dump diff --pair '$(B)/modadvert/bigmod/protos/config.pb.h_serialized.cpp'
ay dev dump diff --pair '$(B)/ads/caesar/libs/events/generated/registrar.cpp'
```

Both differ from ref ONLY in `inputs`: ours carries extra ours-only
`$(S)/taxi/schemas/schemas/proto/modadvert/bigmod/protos/...` `.proto` sources
(circuit, external_data_v2_items, grut, object_key, object_types, options,
markup_flags, moderation_units, verdict). The same over-emission rides the
proto compile (`config.pb.cc.pic.o`) and the PB producer (`config.pb.cc`) — by
`--by-token` it leaks into 800+ nodes (object_types.proto alone into 187).

## Root cause (originates at the PB producer, not EN/PR)

`config.proto` has `import "modadvert/bigmod/protos/core/circuit.proto"` — a file
that exists BOTH at the source root (`$(S)/modadvert/bigmod/protos/core/circuit.proto`,
the real PROTO_LIBRARY) AND under a peer `PROTO_NAMESPACE(taxi/schemas/schemas/proto)`
GLOBAL proto addincl, whose subtree mirrors the modadvert tree
(`$(S)/taxi/schemas/schemas/proto/modadvert/bigmod/protos/core/circuit.proto`).

`emitProtoPB` walks the proto's transitive imports (`walkClosureTail(config.proto,
protoWalkInputs)`) and weaves each source-rooted import into `config.pb.h`'s parsed
includes (emit_proto.go:381-394), from where it rides to every `.pb.h` consumer
(the EN serialized.cpp, the PR registrar, the .pb.cc compile).

Our include scanner resolved that import via ADDINCL (the taxi namespace) BEFORE
the arcadia source root, so it bound the spurious mirror copy.

## Upstream mechanism

`devtools/ymake/module_resolver.cpp` `ResolveSingleInclude` (lines 238-245,
322-352): for a Local include in a lang in `LANGS_REQUIRE_BUILD_AND_SRC_ROOTS`
(`c asm cython proto flatc swig ydl nlg`, build/conf/settings.conf), upstream
resolves `MakeResolvePlan(srcDir, BldDir, SrcDir)` FIRST and returns on success,
consulting the module IncDirs (ADDINCL) only on a miss. So the arcadia roots
(build + source) outrank ADDINCL. protoc itself confirms it: the generated
`-I` list is `-I=./ -I=$(S)/ -I=$(B) -I=$(S) ... <namespaces>` — source root first.

Our `resolveContextSearchTier` (scanner.go) inverted this: ADDINCL (own/peer,
incl. the ranked index) was tried before `BaseSearchPaths`, so a target present
under any ADDINCL never reached the arcadia roots.

## Fix

`scanner.go resolveContextSearchTier`: before the ADDINCL resolution, try the
arcadia roots (`bld` then `v`, matching upstream's BldDir/SrcDir local-plan
order). On a hit, return immediately. The includer-dir arm of the local plan is
already handled by the quoted `resolveSourceUnder` branch in `resolveSearchPath`;
the linux-headers fallback in `BaseSearchPaths` stays AFTER ADDINCL (it models a
peerdir'd contrib addincl, not an arcadia root). The change is inside the cached
tier, so cache hits pay nothing extra.

This is generic (every required-roots lang, every context), not proto-specific —
it just happens to surface only where a source file is mirrored under an addincl.
