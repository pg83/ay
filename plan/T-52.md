# T-52 plan: serialized-enum (EN) module_dir drift through nested-submodule directory-owned headers

## Upstream mechanism

ymake sets `Node2Module` for a generated node on its **first DFS leave**
(`devtools/ymake/json_visitor.cpp:638`): `FindModule(state)`
(`compact_graph/query.h:13`) returns the *nearest module on the visitor stack*.
A module owns every file physically located in its directory. When a nested
submodule's directory-owned header includes a generated `_serialized.h`, the
generated node is reached **through that submodule** and, because a peerdir
submodule completes (post-order) before its enclosing parent module, the
submodule wins `Node2Module`.

### Concrete sg7 representatives (`yabs/server/libs/server`)

`yabs/server/libs/server/solomon` is a nested LIBRARY (own `ya.make`), a peerdir
of the enclosing `server` LIBRARY. The header `solomon/services.h` (owned by the
solomon submodule, by directory) `#include`s three server-declared generated
serialized headers:

| `_serialized.cpp` (EN) declared in | output dir | ref module_dir |
|---|---|---|
| `degradation_stats.h` (server) | server | server/solomon |
| `hit_cost_param.h` (server) | server | server/solomon |
| `solomon/types.h` (server) | server/solomon | server/solomon |

All drift `server → server/solomon`. The CC `.cpp.o` compile nodes stay `server`
(server compiles the `solomon/*.cpp` SRCS) — only the EN producer drifts.

### Why the other server `WITH_HEADER` enums do NOT drift

- `secrets_enums.h` ← `program/components/secrets/secrets.h` — that subdir has
  **no** `ya.make` (owned by server) → no nested submodule → stays server.
- `dynamic_top_user_features.h` ← `l1/feature_computer/*.cpp` — same, no
  submodule → stays server.
- `ctr_prediction.h` — included only by server-dir files → stays server.

### Why enums-family serialized headers (also included by `solomon/services.h`)
keep their owner

`rtb_decline_reasons.h` etc. are declared in `yabs/server/libs/enums` and their
outputs live in the **enums** dir. solomon is **not** nested under enums, so the
post-order pre-emption does not apply — the declaring `enums` module owns the
directory and keeps the node. This is the discriminator that the blanket
consumer-claim (T-49) got wrong (it regressed 30+ enums nodes).

## Generic rule to reproduce

For an EN producer declared in module `D` whose generated `_serialized.h` is
`#include`d, during scanning, by an includer file `F`:

  claim = nearestModuleDir(dir(F))

The node drifts to `claim` iff `claim` is a **real module dir strictly nested
under `D`** (`HasPrefix(claim, D+"/")`). Among multiple qualifying includers the
deepest nested module wins (post-order: the most-nested submodule leaves first).
This reproduces FindModule's directory-ownership + submodule-before-parent
post-order without a dep-graph edge (solomon is NOT a producer dep of the EN
node) and without touching the PR/CF/CP consumer first-claim path.

## Files touched

1. `scanner.go` — new `generatedENIncluderDirs map[VFS][]string`: during
   `resolveSearchPath`, when an include resolves to a registered EN output,
   record `pathDir(includerAbs.rel())`. Intrinsic to the includer file, so
   resolution caching stays context-free (the recorded value does not change
   the resolved path).
2. `gen.go` — merge the per-scanner maps (union) onto the emitter, mirroring
   `mergeGeneratedFirstClaims`.
3. `emitter.go` — new field on `BufferedEmitter`.
4. `attribute_generated.go` — finalize pass for `pkEN` nodes: build the module-dir
   set (every distinct `module_dir` is a real module dir — verified), resolve
   `nearestModuleDir(includerDir)`, and override to the deepest module strictly
   nested under the declaring dir.
