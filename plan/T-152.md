# T-152 plan: same-module forward proto import generated-header registration

## Residual

`$(B)/yt/yt/library/query/proto/query_service.pb.cc.pic.o` is missing exactly one
ref-only input: `$(B)/yt/yt/library/query/proto/functions_cache.pb.h`.

`yt/yt/library/query/proto/ya.make` is a plain `LIBRARY()` (not `PROTO_LIBRARY`)
with `SRCS(query.proto query_service.proto functions_cache.proto)`.
`query_service.proto` imports `yt/library/query/proto/functions_cache.proto`,
a same-module sibling declared LATER in SRCS.

## Upstream mechanism

protoc resolves `query_service.proto`'s import of `functions_cache.proto` against
its `-I` roots and emits `#include "yt/library/query/proto/functions_cache.pb.h"`
into `query_service.pb.h`. ymake's include scanner resolves that generated header
to the same-module PB output regardless of SRCS declaration order — the generated
`.pb.h` edge survives a forward import. The downstream `query_service.pb.cc`
compile therefore lists `functions_cache.pb.h` as an input.

## Local defect

For a specialized `PROTO_LIBRARY`, `emitCPPProtoSrcs` is two-phase: it emits ALL
proto PB nodes (registering every `.pb.h` in the codegen registry) in its first
loop, then walks each generated `.pb.cc` closure in a later loop. A forward
same-module import resolves because the imported `.pb.h` was already registered.

For a plain `LIBRARY`/`PROGRAM`, proto sources flow through the general module
path in `gen.go` (the Pass-1 codegen loop at ~2140 → `emitOneSource` →
`emitLibraryProtoSource`). `emitLibraryProtoSource` emits the PB node AND
immediately walks the `.pb.cc` closure in a single call, per source, in SRCS
order. When `query_service.proto` (declared before `functions_cache.proto`) is
processed, `functions_cache.pb.h` is not registered yet, so the walk misses it.

The general path already has this exact two-phase pre-pass for `.fbs`
(`emitFlatcProducer`) and bison `.y` (`emitBisonProducer`) — register every
producer first, then compile. Proto lacks the equivalent pre-pass.

## Change

Mirror the `.fbs`/bison split for proto, matching `emitFlatcProducer` /
`emitLibraryFlatcSource`:

1. `emit_proto.go`: split `emitLibraryProtoSource` into
   - `emitProtoProducer(ctx, instance, d, srcRel, in)` — emits the PB node and
     registers its `.pb.h`/`.pb.cc` outputs (the existing `emitProtoPB` call).
   - `emitLibraryProtoSource(ctx, instance, d, srcRel, in)` — looks up the
     already-registered producer ref from the codegen registry and compiles the
     generated `.pb.cc` (+`.grpc.pb.cc` under GRPC). Pure compile, no PB emit.

2. `gen.go`: add a proto producer pre-pass in the general module path, next to
   the existing `.fbs` and `.y` pre-passes (~line 2119-2138): for each
   `srcExtClassOf(src) == srcExtProto` in `d.srcs`, call `emitProtoProducer`.

3. `emit_gztproto.go`: `emitLibraryGztProtoCompile` produces a `$(B)` `.proto`
   not present in `d.srcs`, so the pre-pass cannot cover it. Have it call
   `emitProtoProducer` then `emitLibraryProtoSource` (producer-then-compile, as
   today, just via the split functions).

No new data passed between modules; the registry/`GeneratedFileInfo` seam already
used by `.fbs`/bison carries the producer ref. No reordering of PB commands or
archive members — the PB nodes are emitted in the same SRCS order, only earlier.
The `emitCPPProtoSrcs` (`PROTO_LIBRARY`) path is untouched.
