# T-42 plan: rebase builtin protobuf DESC_PROTO `.desc` outputs under the builtin module output root

## Upstream mechanism

`build/conf/proto.conf` `_PROTO_DESC_RAWPROTO_CMD` declares the two producer
outputs with different ymake output modifiers:

- `.desc`     via `${output;suf=.desc:File}`            — rebases an SRCDIR-resolved
  `File` under the current module output dir, preserving the relative ascent as
  `__` path segments (ymake SRCDIR output semantics).
- `.rawproto` via `${norel;output;suf=.<hash>.rawproto:File}` — keeps the output
  at the resolved physical source path (`norel` = no module rebasing).

Our `emitDescProtoSubmodule` currently uses the resolved physical source path
(`protoSourceRelPath` → `contrib/libs/protobuf/src/...`) for BOTH outputs, so the
`.desc` lands at the physical source root instead of the builtin module root.

The exact `${output:File}` SRCDIR rebasing already exists in `emit_cc.go` as
`composeSrcDirOutputRel(instancePath, target)` (used by `composeCCPaths`), which
maps `..` ascent to `__`. Reuse it; do not invent a second dot-dot encoder.

## Files to touch

- `emit_proto_desc.go`: add `descProtoOutputRel(instancePath, srcRel, resolvedRel)`
  helper; change only `descOut` construction in `emitDescProtoSubmodule` to use it.
  `rawprotoOut` (incl. hash), `--proto-file`, imports, merge, peer closure unchanged.
- `emit_proto_desc_test.go`: add a helper unit test (3 exact cases) and a graph
  regression for a `protos_from_protobuf`-style SRCDIR builtin.

## Expected gate effect

- sg2..sg6 stay byte-exact (no SRCDIR-backed DESC_PROTO in gating cases moves).
- sg7: the 12 builtin `.proto.desc` outputs (protos_from_protobuf any/descriptor/...,
  protos_from_protoc plugin) move from structural-only to same-path; the two builtin
  `.self.protodesc` merge nodes no longer show the `.desc` path split. sg7 stays XFAIL
  (remaining paired-node input/protoc-include residuals are out of scope).
