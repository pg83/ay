## Upstream mechanism

`CPP_EVLOG()` = `CPP_PROTO_PLUGIN0(event2cpp tools/event2cpp DEPS
library/cpp/eventlog)` (build/conf/proto.conf:605). The plugin DEPS feed
`CPP_PROTOBUF_PEERS` (proto.conf:265), and the C++ `module LIBRARY` base injects
them via `when ($HAS_CPP_PROTOBUF_PEERS == "yes") { PEERDIR+=$CPP_PROTOBUF_PEERS
}` (ymake.core.conf:2002) — emitted **ahead of** the body's declared `PEERDIR`
in the module's single peer order. ymake builds the link/archive closure
(`UniqPeers`, module_restorer.cpp `TTransitivePeersCollector::Collect`) from that
one peer order via first-occurrence-dedup post-order DFS: for each direct peer,
merge the peer's `UniqPeers` then append the peer. So the plugin-runtime peer's
whole archive subtree lands before the declared peers' subtrees.

This is the same induced-peer-first rule T-14 identified and applied to the
GLOBAL ADDINCL (`-I`) order, where it leaves the link/archive order untouched.
Upstream uses **one** peer order for both: the plugin-runtime peer leads the
ADDINCL order *and* the archive/link order. T-14 was scoped to ADDINCL; this
ticket extends the identical front-loading to the link-order passes.

`contrib/libs/protobuf` is deliberately NOT a plugin DEP (it is not in
`d.protoCmdPeers`; `GRPC()` likewise appends `contrib/libs/grpc` as a plain
declared peerdir, not a plugin DEP). So front-loading `protoCmdPeers` moves only
the genuine plugin-runtime peers (e.g. `library/cpp/eventlog`), matching
upstream; the protobuf/grpc runtimes keep their declared positions (the regress
T-14 saw came from hoisting protobuf, which this change does not touch).

## Change

`gen.go` `walkPeersForGlobalAddIncl` (the PROTO_LIBRARY / specialized-library
closure builder; `d.protoCmdPeers` is set only for `tokProtoLibrary`):

- Replace the bespoke three-loop front-ordering used only by the ADDINCL pass
  with a single `orderedResolved` slice (prefix defaults/cppSelf/googleapis →
  front `protoCmdPeers` declared peers → remaining declared peers), and iterate
  **every** peer-closure aggregation over it: addIncl, archive, sbom, global,
  wholeArchive, wholeArchiveCmd, dynamic, ldPlugin, resourceGlobals,
  protoInclude, flags. This makes the propagated `PeerArchiveClosurePaths` (and
  the other link-order closures) reflect the same induced-peer-first order as the
  includes — net code shrink (one ordering, not two).

No change to the general PROGRAM/LIBRARY path (gen.go:1335 `archiveOrder`): a
PROGRAM is never a `PROTO_LIBRARY`, so it has no `protoCmdPeers`; the corrected
closure flows into the program link through the normal per-peer merge.
