# T-99 plan: COW well-known generated-header consumer attribution + cpp_proto tag

## Divergence (reproduced on current trunk, sg7)

The single `PR` (RUN_PROGRAM) node produced by
`apphost/gp/lib/proto/cow/generator/well_known` — outputs the seven
`google/protobuf/*.cow.pb.h` headers — pairs structurally with ref
(cmds/inputs/outputs/deps byte-identical) but diverges on `target_properties`:

```
ours: {module_dir: apphost/gp/lib/proto/cow/generator/well_known}
ref:  {module_dir: apphost/lib/proto_answers, module_tag: cpp_proto}
```

`ay dev dump diff --pair '$(B)/apphost/gp/lib/proto/cow/generator/well_known/google/protobuf/any.cow.pb.h'`
shows ONLY the `target_properties` field differs.

## Upstream mechanism

ymake `Node2Module` (json_visitor.cpp:638) attributes every generated FILE node
to the module that FIRST leaves it in DFS post-order (`FindModule(state)` on the
visitor stack, `try_emplace` = first-write-wins). The node inherits that
module's directory AND tag.

The well_known headers are pulled into a consuming CPP_PROTO module through the
`CPP_PROTO_PLUGIN(apphost_cow …)` window declared in
`apphost/gp/lib/proto/cow/gen.inc` (included by `apphost/lib/proto_answers`).
The plugin's `DEPS` add `…/well_known` as a PEERDIR whose GLOBAL build-root
ADDINCL exposes the cow headers; the consuming module's generated `*.pb.h` /
`*.pb.cc` reach `google/protobuf/any.cow.pb.h` as a **rooted induced-dep**
(`${ARCADIA_BUILD_ROOT}/…/any.cow.pb.h`). The first consuming CPP_PROTO module
to leave the node in DFS post-order (here `apphost/lib/proto_answers`, the
cpp_proto submodule) owns it → `module_dir=apphost/lib/proto_answers`,
`module_tag=cpp_proto`.

## Why our attribution misses it

`overrideGeneratedModuleDir` (attribute_generated.go) re-attributes a PR/CF/CP
producer to the first consumer recorded in `generatedFirstClaim`. That map is
written by `recordFirstClaim` from the addincl / registry-split resolve paths.
But the cow headers reach consumers as a **fully-rooted** include directive,
which `IncludeScanner.resolve` binds directly (scanner.go:763,
`d.target.vfs() != 0`) and returns WITHOUT calling `recordFirstClaim`. So no
claim is recorded and the node keeps its well_known producer attribution. The
module_tag is never propagated at all — `overrideGeneratedModuleDir` only ever
sets `module_dir`.

Verified by instrumentation: `recordFirstClaim` is never called for
`any.cow.pb.h`; the rooted-bind path resolves it, first by
`apphost/lib/proto_answers/debug_info.pb.cc` (owner dir
`apphost/lib/proto_answers`, a CPP_PROTO module → `perModuleCCTag = cpp_proto`).

## Change

1. **Carry the claiming module's tag with the claim.** `generatedFirstClaim`
   value `string` → `GenOwner{Dir string; Tag STR}`. `recordFirstClaim` gains a
   tag arg; `markGeneratedProducerOwned` records `Tag:0`. Merge + emitter field
   types updated.
2. **Record a first-claim on the rooted-bind path** (scanner.go:763) when the
   bound target is a registered codegen output and the scan has an owner — the
   induced-dep edge upstream's Node2Module also attributes. Gated on
   `v.isBuild() && OwnerModuleDir != ""`, single `codegen.lookupSTR(d.target)`.
3. **Thread the consumer module tag** into `ScanContext.OwnerModuleTag` (not a
   hashed field — pure claim side-channel, resolution stays context-free), set
   once at the module CC scan context (gen.go:2092 = `perModuleCCTag`).
4. **Apply the tag** in `overrideGeneratedModuleDir`: when a claim re-attributes
   a node's `module_dir`, also set `module_tag` from the claim (when non-zero).
