# T-136 plan: propagate generated-source closure through `COPY_FILE`

## Upstream mechanism

ymake's input model is flat and transitive: a node's `inputs` list every
`$(S)` source leaf reachable through its producer chain, while `$(B)`
intermediates stay behind the producer-node edge. For the representative

```
# child gen_consts/ya.make
RUN_PROGRAM(const_generator ... OUT_NOAUTO generated_consts.py CWD ${BINDIR})
# parent generated/ya.make
COPY_FILE(${ARCADIA_BUILD_ROOT}/.../gen_consts/generated_consts.py generated_consts.py)
PY_SRCS(generated_consts.py)
```

upstream therefore lists the RUN_PROGRAM producer's full transitive `$(S)`
closure on:

1. the `CP` action that copies the generated file (`$(B)/.../generated_consts.py`),
   alongside the copy tool scripts `fs_tools.py` / `process_command_files.py`
   it already carries; and
2. the `py3cc` bytecode action (`generated_consts.py.yapyc3`) that compiles the
   *copied* file — its flat closure folds the CP's own `$(S)` scripts PLUS the
   `$(B)` source's producer closure (the RUN_PROGRAM 1554 source/proto/header
   leaves).

ay records the RUN_PROGRAM producer's closure as
`GeneratedFileInfo.ProducerSourceClosure` (`emit_pr.go:358`), and the bytecode
emitter already reads it (`emit_py_codegen.go:55`). The missing seam: a
`COPY_FILE` over a registered generated output does not carry that closure onto
the CP node, nor store it on the copied dst's `GeneratedFileInfo`, so the
parent's `PY_SRCS` consumer sees nothing.

## Files touched

- `emit_copy.go`: in `emitCopyFiles`, when `reg.lookup(srcVFS)` resolves to a
  generated output carrying `ProducerSourceClosure`:
  - store `producerClosure ∪ ctx.scripts[copyFsToolsVFS]` on the destination
    `GeneratedFileInfo.ProducerSourceClosure` (so the `PY_SRCS` bytecode node
    folds the producer closure + the CP's own copy-tool scripts), and
  - add `producerClosure` to the CP node's extra inputs.
  Ordinary source-to-build copies and `COPY_FILE(TEXT)` (whose `srcVFS` is a
  `$(S)` file, never a registered generated output) resolve `srcInfo == nil`,
  so they keep current behavior.

No change to `codegen_registry.go` (reuses `ProducerSourceClosure`),
`emit_pr.go`, or `emit_py_codegen.go` — the existing consumer seam suffices.

## Ordering

Peers resolve via `genModule` (gen.go ~1423) before the parent's
`emitCopyFiles` (gen.go 2159) and `emitPySrcs` (gen.go 2710) run, and the
codegen registry is per-platform (shared across modules). So the child
RUN_PROGRAM's `ProducerSourceClosure` is already registered when the parent's
COPY looks it up, and the parent's `PY_SRCS` reads the dst closure after
`emitCopyFiles` set it.

## Test (fail-first)

`emit_copy_test.go`: a child PY3_LIBRARY RUN_PROGRAM produces
`generated_consts.py` (OUT_NOAUTO) with `IN_NOPARSE gen.h`; gen.h includes
`other/other.h`. A parent PY3_LIBRARY PEERDIRs the child, COPY_FILEs the
build-root output into `generated_consts.py`, and `PY_SRCS`es it. Assert:

- CP node `$(B)/mod/generated_consts.py` carries `$(S)/mod/gen/gen.h` and
  `$(S)/other/other.h`.
- bytecode node `$(B)/mod/generated_consts.py.yapyc3` carries those PLUS
  `$(S)/build/scripts/fs_tools.py` and `$(S)/build/scripts/process_command_files.py`.
- control: a plain `COPY_FILE(src.txt dst.txt)` of a `$(S)` source registers no
  `ProducerSourceClosure` on its dst (current behavior preserved).

## Expected gate effect

- sg7: CP `inputs` drift −1, PY `inputs` drift −1; `matched` does not decrease.
- sg2–sg6 stay byte-exact (the pattern is specific to copying a registered
  generated output with a producer closure; the upstream-correct closure is
  what we now emit).
- No new `cmds` / `target_properties` drift; T-127/T-129 copy ownership
  independent (this is input-closure only).
