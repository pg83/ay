# T-82 plan: formula.cpp data-IN producer over-emission (Ticket A of T-81)

## Upstream mechanism

Upstream's `RUN_PROGRAM` registers each `IN` as a scan-on-include input
(`${hide;input:IN}`); the scanner walks an IN's parsed-include closure only when
the file's extension maps to a real include parser. The producer node's include
graph is **rooted at IN**. A data IN such as `formula.in` (YAML-like, no
registered parser) contributes no include edges, so upstream lists only the tool
and the direct `$(S)/.../formula.in` on the producer. The generated `.cpp`/`.h`
OUTs are `${output;...}`; their include closure surfaces on the downstream
COMPILE node (which C-scans the generated source independently), never back onto
the producer command.

Our `prInputClosure` self-scans the run's generated cc-source OUT/STDOUT whenever
`len(stmt.INFiles) > 0`. For `formula.cpp` that self-walk drags the
`OUTPUT_INCLUDES` source closure (`util/generic/string.h` + libcxx, 755 extra
headers) onto the producer. But `formula` ALSO generates `formula.h` as an `OUT`,
and the generated `formula.cpp` `#include "formula.h"`. Upstream routes the
`OUTPUT_INCLUDES` (`array`, `util/generic/string.h`) through that generated header
sibling: they ride `formula.h` to the cc-source's consumers (and the cc-source
itself includes the header), so the producer command does not self-carry them.
This is the IN-rooted analog of T-72's accepted no-IN header-sibling guard:
when a generated header sibling carries the includes, the producer stays clean.

## Counterexample that fixes the discriminator (geocoding_data.cc)

A prior attempt gated the self-walk on `hasParsedIN || toolInducesCC` (tool
`INDUCED_DEPS(cpp)`). That dropped `formula` correctly but REGRESSED
`$(B)/contrib/libs/libphonenumber/cpp/src/phonenumbers/geocoding/geocoding_data.cc`
(sg7 matched 66353→66352): a data-IN (`IN resources/geocoding/**/*.txt`),
no-`INDUCED_DEPS` tool (`generate_geocoding_data`, a plain `PROGRAM`), whose
REF producer DOES carry the OUTPUT_INCLUDES closure (234 inputs: `cstdint`→libcxx
+ the source header `geocoding_data.h`). The distinguishing fact vs `formula`:
geocoding's `OUT` is a single `geocoding_data.cc` with **no generated header
output** — `geocoding_data.h` is a checked-in source. With no generated header to
route the OUTPUT_INCLUDES through, the closure surfaces on the producer's own
self-scan. So the real discriminator is **whether the run generates a header
output**, not whether the IN is parsed or the tool induces C++.

`bcrypt._bcrypt.cffi.py3.c` (sg6) confirms: data `IN build_bcrypt.py`, `OUT` a
single `.c` (no header), REF producer keeps the cffi/Python closure — kept here
by `!generatesHeader`, with no need to inspect the tool's INDUCED_DEPS.

## Change

`emit_pr.go`, `prInputClosure`:

1. Compute `hasParsedIN` once: true iff some `stmt.INFiles` entry satisfies
   `includeDirectiveParsers.hasRegisteredParser(rel)`.
2. Compute `generatesHeader`: true iff any `stmt.OUTFiles` entry or the
   `StdoutFile` is a header (`isHeaderSource`).
3. Gate the generated cc-source OUT/STDOUT `walkOne` blocks on
   `selfScanGeneratedCC := len(stmt.INFiles) > 0 && (hasParsedIN || !generatesHeader)`
   instead of `len(stmt.INFiles) > 0`. This is the original `len(IN)>0` gate minus
   exactly the formula class (data IN, no parsed IN, generates a header sibling).
4. Leave the T-72 `fullSourceClosure` / `mainIsCCSource` / header-main guard,
   `prMainOutputRel`, the OUTPUT_INCLUDES walk, tool deps, and the IN walk
   unchanged. The earlier `toolInducesCC` / `res.InducedDeps` plumbing is removed
   (net-negative): `!generatesHeader` subsumes the bcrypt case.

- formula: data IN, no parsed IN, generates `formula.h` → `selfScanGeneratedCC=false`
  → producer inputs 757→2 (tool + IN).
- geocoding_data.cc: data IN, no header OUT → `!generatesHeader=true` → self-walk
  preserved (234 inputs).
- bcrypt cffi `.c`: data IN, no header OUT → preserved.
- control_board / `tmpl.cpp.in`: parsed IN → `hasParsedIN=true` → preserved.
