# T-73 — Generated PY_SRCS packaged as objcopy resfs, not rescompiler `_raw.auxcpp`

## Classification of the representatives (against normalized sg7 ref)

- `yabs/models_services/feature_store/mappers/argus/py/objcopy_*.o` — **archive-member
  drift / wrong node kind.** REF packages the build-generated `__init__.py` (PY_SRCS +
  RUN_PROGRAM OUT_NOAUTO) as a single `objcopy_<hash>.o` resfs node embedding
  `$(B)/.../__init__.py` and `$(B)/.../__init__.py.yapyc3` (with deps on both producers).
  OURS instead emits (a) a rescompiler `<hash>_raw.auxcpp` PR node + its `.o` CC node, and
  (b) a *spurious* `py/namespace/...` kv-only objcopy node. So the global archive member set
  diverges: we contribute an auxcpp.o + a namespace.o where ref contributes one resfs objcopy.o.
  This is the bulk of the OUR-only `_raw.auxcpp` set (all 24 feature_store mappers + a few
  yabs/server/cs pylibs + irt/bannerland — every one a generated PY_SRCS).

- `yabs/models_services/feature_store/lib/py3/objcopy_*.o` — **separate root cause (reported
  separately).** lib/py3 uses `ALL_PY_SRCS()` over checked-in `.py` (no generated py here);
  its 60 ref-only objcopy nodes package `.json` configs pulled in by
  `ALL_RESOURCE_FILES(json PREFIX … ${ARCADIA_ROOT}/<dir>/*)` from `INCLUDE(.../ya.inc)`. The
  `/*`-suffixed DIR glob is not handled by `allResourceDir`/`globDirSuffix`, so we emit none of
  them. That is an ALL_RESOURCE_FILES DIR-glob defect, orthogonal to the generated-py routing
  this ticket fixes; it gets its own follow-up.

## Upstream mechanism (build/plugins/pybuild.py `onpy_srcs`)

For py3, every `PY_SRCS` entry — generated or checked-in — flows through
`unit.onresource_files(res)`. The objcopy-vs-rescompiler routing turns on the
**PY_SRCS token FORM**, the same signal T-70 already established for the py3cc
module name: a *bare* token (a checked-in source OR a RUN_PROGRAM-generated
`$(B)` file referenced by its plain name) is packaged into an `objcopy_<hash>.o`
resfs node; a *full* `${ARCADIA_BUILD_ROOT}/<full>.py` token — the form swig
injects via `PY_SRCS(unit, swg_py + '=' + mod)` — is embedded through the
rescompiler `<hash>_raw.auxcpp` path (compiled as C++, carrying its include
closure). Empirically (sg3 contrib/python/python-libarchive/py3): the three
checked-in pys → one objcopy node, the swig `_libarchive.py` → `_raw.auxcpp`,
same module. (The PY proto `_raw.auxcpp` flow, `emit_py_proto.go` tag PY3_PROTO,
is yet another, separate path.)

So a RUN_PROGRAM-generated `__init__.py` (bare token, `pySrcsFullName=false`)
becomes a normal `objcopy_<hash>.o` resfs node, differing from a checked-in py
only by binding the `.py` from `$(B)` and depending on the producer. A swig py
(`pySrcsFullName=true`) keeps the `_raw.auxcpp` path.

The `py/namespace` resource is gated on `is_extended_source_search_enabled(path)` →
`is_arc_src(path)` (pybuild.py:388). A `$(B)` generated file is NOT an arc source, so it is
excluded from `py_namespaces`. A module whose only PY_SRCS is generated therefore emits no
namespace resource. (`mod_list_md5` still folds every mod name, but the namespace node is
emitted only when `py_namespaces` is non-empty.)

## Files to touch

1. `resource.go` `buildPySrcEntriesFor` — replace the blanket "skip if in codegen registry"
   `continue` with a token-form gate. A *full-name* generated token (swig) is still skipped (the
   `_raw.auxcpp` path owns it). A *bare-token* generated srcRel is now resourced here: bind the
   `.py` input to `build(modulePath+"/"+srcRel)` ($(B)) instead of `source(resolvedRel)`, and set
   the yapyc3 entry's `extraSrcInput` to the same build path (dedups against the py input rather
   than fabricating a `$(S)` source). hash/key/kvHash strings are unchanged (rootrel-independent
   of $S vs $B), so the objcopy name matches ref. Producer deps come for free via
   `resolveCodegenDepRefs(ch.inps)` in `emitPySrcObjcopy`.

2. `emit_py_objcopy.go` `emitPyNamespaceForGroup` — exclude generated (registry-resolved)
   sources from the namespace decision, mirroring `is_arc_src` (neither swig nor RUN_PROGRAM
   generated py is an arc source). md5 still folds the full mod list; emit only when ≥1 arc
   source remains, so a generated-only module emits no namespace node.

3. `emit_py_aux.go` `emitGeneratedPyAuxChunks` — restrict to *full-name* (`pySrcsFullName`)
   generated entries, so it keeps owning swig py while ceding the bare-token RUN_PROGRAM case to
   the objcopy path. (`emitRawAuxResourceChunks` etc. unchanged; PY proto unaffected.)

## Expected gate effect

The OUR-only `_raw.auxcpp` set (~24 feature_store + a few others) and the spurious
feature_store namespace objcopy nodes collapse to ref-matching resfs objcopy nodes; the
feature_store objcopy structural bucket shrinks substantially / pairs for the argus
representative. sg2–sg6 stay byte-exact (no generated PY_SRCS there); sg5 budget unaffected
(net-fewer nodes on the hot path). Remaining lib/py3 `.json` ALL_RESOURCE_FILES drift reported
separately.
