## Upstream mechanism

`target_properties.module_dir` and the late-resolved `${BINDIR}` in `cwd` both come
from the command node's owning module = `Node2Module[node]`
(devtools/ymake/json_visitor.cpp:638, devtools/ymake/export_json.cpp:101,
devtools/ymake/json_subst.cpp:265, devtools/ymake/mkcmd.cpp:42). `Node2Module` is
set once, on the FIRST DFS post-order leaving of the node, to the nearest enclosing
module on the visitor stack.

`generated_consts.py` is an `OUT_NOAUTO` output: the producing `gen_consts` module
does NOT auto-add it as a source and does not otherwise reference it, so it never
appears inside `gen_consts`'s own DFS subtree. The only module that references the
node is the PARENT `generated`, through its `COPY_FILE(${ARCADIA_BUILD_ROOT}/.../gen_consts/generated_consts.py ā€¦)`
build-path `EDT_BuildFrom`. The parent is therefore the first (and only) module to
leave the node ā†’ `Node2Module = parent`. Because `${BINDIR}` (mkcmd.cpp:42, set from
the owning module's dir) is resolved at command-materialization time against the
owning module, `cwd` becomes the PARENT bindir while the OUT path ā€” fixed at graph
construction when `gen_consts` created the node ā€” stays under `gen_consts`.

This is the same first-consumer-wins ownership ay already models for generated
headers via `generatedFirstClaim` (attribute_generated.go / scanner.go). The only
gap: a cross-module `COPY_FILE` source consumption does not record a claim, and the
already-baked `${BINDIR}` cwd is not re-resolved when module_dir is overridden.

## Change

Files: `emit_copy.go`, `attribute_generated.go`, plus a focused test in `emit_pr_test.go`.

1. `emit_copy.go` (`emitCopyFiles`): when a `COPY_FILE` source resolves to a
   cross-module registered generated output (`srcVFS.isBuild()`, `srcVFS != dstVFS`,
   producer found), record a first-claim attributing that output to THIS (consumer)
   module via `recordFirstClaim(srcVFS, instance.Path.rel(), moduleCCTag(d.moduleStmt.Name))`.
   First-wins + the existing `claim.Dir == current` self/same-module guard make this a
   no-op for same-module and already-claimed outputs. `moduleCCTag(PY3_LIBRARY)`==0, so
   no spurious `module_tag` (ref node carries none; upstream emits a tag only for
   multimodule owners, json_subst.cpp:280).

2. `attribute_generated.go` (`overrideGeneratedModuleDir`): when a `pkPR` node's
   module_dir is overridden from `current` (producer) to `claim.Dir` (owner), also
   re-resolve a `${BINDIR}`-derived cwd: if `cmd.Cwd` has prefix `$(B)/<current>`,
   rewrite that prefix to `$(B)/<claim.Dir>`. This reproduces upstream's late
   `${BINDIR}` resolution against the owning module without changing the OUT path.
   Restricted to `pkPR`; an explicit non-`${BINDIR}` cwd does not match the prefix and
   is left untouched.
