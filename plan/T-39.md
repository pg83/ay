# T-39 plan: FROM_SANDBOX auto OUT `.a`/`.o` become module link/archive members (MKL reachability)

## Upstream mechanism

`FROM_SANDBOX(... OUT file)` declares `file` as an **auto** module output
(`${hide;output:OUT}` — without `noauto`), whereas `OUT_NOAUTO` uses
`${hide;noauto;output:OUT_NOAUTO}`. In ymake an auto output is folded into the
module's `AUTO_INPUT`. For a `LIBRARY`, the module archive command is
`LINK_LIB = … $_LD_ARCHIVER $TARGET $_LD_TAIL_LINK_LIB`, and
`_LD_TAIL_LINK_LIB = $AUTO_INPUT …` (build/conf/linkers/ld.conf:372,384). So the
module's own `.a` is produced by `link_lib.py llvm-ar … -- -- $TARGET $AUTO_INPUT`,
archiving every auto output — including a fetched `.a`/`.o` — as a member.

`contrib/libs/intel/mkl/{core,lp64,threads}` are `LIBRARY()` modules whose only
content is `FROM_SANDBOX(<id> OUT libmkl_*.a)` (no compiled sources). Upstream
therefore emits, for `core`:

```
link_lib.py llvm-ar LLVM_AR gnu $(B) None -- -- \
  $(B)/contrib/libs/intel/mkl/core/libintel-mkl-core.a \   # $TARGET
  $(B)/contrib/libs/intel/mkl/core/libmkl_core.a            # $AUTO_INPUT (fetched)
```

The dependent program (`yabs/.../fs_codegen`) reaches `contrib/libs/intel/mkl`
through its peer closure; the umbrella peers core/threads/lp64, and each leaf's
module archive `libintel-mkl-{core,threads,lp64}.a` flows up the existing
`PeerArchiveClosure` and lands in the program's LD as both a `rootrel` command
token and a `$(B)` input.

## Defect in our code

`emit_from_sandbox.go` emits the SB fetch node and registers the OUT files as
codegen producers, but never feeds the auto `.a`/`.o` outputs into the module's
archive member set. In `gen.go` the module archive is emitted only
`if len(ccRefs) > 0`; a FROM_SANDBOX-only `LIBRARY` has no `ccRefs`, so **no**
`libintel-mkl-core.a` is produced, its `ARRef` stays nil, and the peer closure
delivers nothing — the three MKL archives are ref-only in the fs_codegen link.

## Files to touch

- `emit_from_sandbox.go`: `emitFromSandbox`/`emitFromSandboxes` collect and return
  the auto OUT (`stmt.OUTFiles`, never `OUT_NOAUTO`) outputs whose extension is
  `.a` or `.o`, paired with the SB node ref — these are the module's AUTO_INPUT
  link members.
- `gen.go` (main module path): capture the returned `(memberRefs, memberPaths)`
  from the `emitFromSandboxes` call and append them to `ccRefs`/`ccOutputs`
  **after** the clang-toolchain-SBOM gate (which must stay keyed on real compiled
  C-family TUs, not fetched archives) and before the program-LD / AR emission, so
  a `LIBRARY` archives them and a `PROGRAM` links them, matching `$AUTO_INPUT`.

## Test (test-first)

`emit_from_sandbox_test.go`: a `LIBRARY()` with `FROM_SANDBOX(id OUT libfoo.a)`
and no sources, peered by a `PROGRAM()`. Assert:
1. the library emits AR `$(B)/mod/libmod.a` whose cmd + inputs carry the fetched
   member `$(B)/mod/libfoo.a`;
2. the program LD carries `$(B)/mod/libmod.a` (input) and `mod/libmod.a` (cmd);
3. control — an `OUT_NOAUTO libbar.a` member is **not** archived, and a sibling
   non-FROM_SANDBOX library is unchanged.

## Expected effect on the gate

fs_codegen pair loses the three `contrib/libs/intel/mkl/{core,lp64,threads}/
libintel-mkl-*.a` ref-only cmd/input tokens; the three `libintel-mkl-*.a` AR
nodes (plus their fetched `libmkl_*.a` SB outputs) become paired. sg2..sg6 stay
byte-exact; sg7 matched increases / ref_only shrinks. No global default-link
change; only modules that actually `FROM_SANDBOX OUT *.a/*.o` are affected.

## Scope guard

py-proto archive naming (T-32), models_meta `ALL_RESOURCE_FILES` (T-31), and
pycxx/protobuf SBOM contour are out of scope and untouched.
