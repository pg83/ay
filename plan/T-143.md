# T-143 plan: bare PROTO_NAMESPACE C++ build-root ADDINCL must be GLOBAL

## Symptom (reproduced)

`$(B)/yt/yt/client/formats/config.cpp.o` (and the bigrt `shard_number.cpp.pic.o`
twin): command token multiset matches reference, but the peer build-root include
band is mis-ordered. Reproduced against the sg7 reference:

- ours: `… -I$(B)/yt/yt/client/api/rpc_proxy … -I$(B)/yt/yt/build -I$(B)/yt -I$(S)/contrib/libs/protobuf/src …`
- ref:  `… -I$(B)/yt -I$(B)/yt/yt/client/api/rpc_proxy … -I$(B)/yt/yt/build -I$(S)/contrib/libs/protobuf/src …`

Every token is identical; only `-I$(B)/yt` is placed last in ours and first in ref.

## Upstream mechanism

`yt/yt/client/ya.make` is a plain `LIBRARY()` with bare `PROTO_NAMESPACE(yt)`.

In upstream `build/conf/proto.conf`:

```
macro PROTO_NAMESPACE(GLOBAL?"GLOBAL":"", WITH_GEN?"WITH_GEN":"", Namespace) {
    SET(PROTO_NAMESPACE $Namespace)
    PROTO_ADDINCL(GLOBAL $WITH_GEN $Namespace)   # <-- literal GLOBAL
}
macro PROTO_ADDINCL(GLOBAL?"GLOBAL":"", Path, WITH_GEN?"BUILD":"") {
    _ORDER_ADDINCL(... $GLOBAL FOR proto $(B)/$Path SOURCE $GLOBAL FOR proto $(S)/$Path)
    ADDINCL($GLOBAL ${ARCADIA_BUILD_ROOT}/$Path)
}
```

`PROTO_NAMESPACE` passes the **literal** token `GLOBAL` to `PROTO_ADDINCL`, so
`$GLOBAL` inside `PROTO_ADDINCL` is always `"GLOBAL"`. The C++ build-root
`ADDINCL(GLOBAL $(B)/$ns)` is therefore **always GLOBAL**, regardless of whether
the `GLOBAL` keyword was given to `PROTO_NAMESPACE`. The `GLOBAL`/`WITH_GEN`
keywords on `PROTO_NAMESPACE` do not gate this arm (the body never reads
`$GLOBAL`). `addincls.cpp` `TLanguageIncDirs::Add(dir, Global)` then puts the dir
into `LocalUserGlobal`, `UserGlobal`, and `Global`; on `PropagateTo` the
consumer receives it via `UserGlobalPropagated` (rendered first, ahead of the
peers' `GlobalPropagated`).

Because every `PROTO_NAMESPACE(yt)` module (including the plain LIBRARY
`yt/yt/client`, the first PEERDIR of `formats`) contributes `$(B)/yt` as a
UserGlobal dir, upstream renders `-I$(B)/yt` early — right where `yt/yt/client`
propagates, before the rpc_proxy build-root subdir include carried in its
`GlobalPropagated`.

## Our defect

`applyProtoNamespace` (modules.go) currently gates the C++ build-root arm:

```go
if d.protoNamespaceGlobal || moduleStmt.Name == tokProtoLibrary {
    d.addInclGlobal = append(...)
    d.addInclUserGlobal = append(...)
}
```

For bare `PROTO_NAMESPACE(yt)` in a plain `LIBRARY`, `$(B)/yt` is added only to
`d.addIncl` (local), never to the global/user-global closure. It therefore does
not ride `yt/yt/client`'s propagation; `$(B)/yt` reaches `formats` only via a
deeper `PROTO_LIBRARY(yt)` that is visited late, landing the include last.
The `proto` SOURCE arm (`ProtoInclude`/`protoAddInclGlobal`) was already
correctly treated as always-global (gen.go comment at the `ownProtoInclude`
assembly); only the C++ `$(B)/<ns>` arm kept the wrong gate.

## Fix

Remove the gate in `applyProtoNamespace`: the C++ build-root `ADDINCL` is always
GLOBAL, matching `PROTO_ADDINCL`'s literal `GLOBAL`. Always append `protoBuildRoot`
to `d.addInclGlobal` and `d.addInclUserGlobal` (it already always lands in
`d.addIncl`). This makes the `global` parameter and the `protoNamespaceGlobal`
field dead; delete them and the GLOBAL-keyword detection loop at the
`tokProtoNamespace` case (net-negative diff).

This is a pure ordering correction at the token level: `$(B)/<ns>` already
reaches every affected consumer via some `PROTO_NAMESPACE(<ns>)` module — the
change only moves its first-encounter earlier (to the bare-PROTO_NAMESPACE
peer), so dedup keeps the earlier position and the multiset is unchanged.

## Files

- `modules.go`: `applyProtoNamespace` — drop the gate; remove `global` param,
  `protoNamespaceGlobal` field, and the `tokProtoNamespace` GLOBAL-detection loop;
  fix the `tokExportYmapsProto` call site.
- `emit_proto_addincl_test.go` (or a new `emit_cc_proto_ns_order_test.go`): a
  fail-first test asserting `-I$(B)/mid` precedes a peer-propagated
  `-I$(B)/mid/sub` build-root include in a consumer of a bare-`PROTO_NAMESPACE`
  plain LIBRARY.

## Expected gate effect

- sg2–sg6 stay byte-exact (if the gate could change them they would not currently
  be byte-exact, since upstream already always-globals this arm).
- sg7: `CC cmds` bucket shrinks by the YT/proto-namespace ordering rows; no CC
  `inputs` drift, no AR/LD `inputs` drift, `matched` does not decrease.
- Both representative pairs lose their `cmds` diff.
- sg5 generation stays under budget (no new work; a gate removed).
