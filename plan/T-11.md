# T-11 — Expand SET_APPEND-defined variables before resource input resolution

## Observed divergence (reproduced)

Direct generation of `yabs/plutonium/libs/dictionaries_resource` (source root
`/home/pg/monorepo/4`) aborts with:

    open yabs/plutonium/libs/dictionaries_resource/${PLUTONIUM_IN_YABS_DICTIONARIES}: no such file or directory

The module sets its file list with `SET_APPEND(PLUTONIUM_IN_YABS_DICTIONARIES …)`
and then references it as `${PLUTONIUM_IN_YABS_DICTIONARIES}` in both
`FROM_SANDBOX(… OUT_NOAUTO ${PLUTONIUM_IN_YABS_DICTIONARIES})` and
`RESOURCE_FILES(${PLUTONIUM_IN_YABS_DICTIONARIES})`. The literal `${…}` survives
into resource/sandbox input resolution and is treated as a path under the module
dir, which cannot exist. `dictionaries_resource_for_workers` is identical.

## Upstream mechanism

Upstream `SET_APPEND(VAR x)` is a builtin that sets `VAR` to `"$VAR x"` (see
`devtools/ymake/macro_processor.cpp:247-248` — "SET_APPEND(VAR x) => 2:VAR=$VAR
x", and `vars.h` `TVars::SetAppend`, doc `builtin.cpp` "Appends appendvalue to
varname's value using space as a separator"). The variable becomes an ordinary
module variable visible to every subsequent macro's argument expansion. So
`${VAR}` resolves to the space-joined accumulated list before the resource and
sandbox-output inputs are resolved.

In ay today:
- `SET` is parsed into a typed `SetStmt` and bound into `env`
  (`modules.go` case `*SetStmt` → `env.setFromString`). `${VAR}` then expands via
  `expandBracedVars`/`env.lookup`.
- `SET_APPEND` is **not** a typed statement; it falls through to `UnknownStmt` →
  `applyUnknownStmt` → `case tokSetAppend`, which only routes a few hard-coded
  flag names (SFLAGS, _PROTOC_FLAGS, RPATH_GLOBAL) into typed flag slices and
  **never records the variable into `env`**. The reference therefore stays
  literal — the variable is silently dropped, not kept as a diagnostic.
- `RESOURCE_FILES` (`case *ResourceFilesStmt`) calls
  `expandResourceFiles(strStrings(v.Args))` directly, i.e. it does **not** run
  its args through `expandStmtTokensSTR(env)` the way the other typed macro
  cases do — so even a bound `${VAR}` would not expand there.

## Fix (generic, two seams in existing machinery)

1. `applyUnknownStmt` `case tokSetAppend` (`modules.go`): after the existing
   flag dispatch, bind the variable generically into `env` (and `d.setVars`,
   matching the `SET` case) with append semantics — `prev ? prev + " " + vals :
   vals`, reproducing upstream `VAR = "$VAR x"`. Args reach `applyUnknownStmt`
   already expanded, so nested refs are resolved exactly as upstream's eval pass.
   Only actually-defined variables get bound; a genuinely unknown `${VAR}` still
   stays literal (diagnostic preserved).

2. `case *ResourceFilesStmt` (`modules.go`): expand `v.Args` through
   `expandStmtTokensSTR(v.Args, env)` before `expandResourceFiles`, mirroring the
   other typed cases. A `${VAR}` arg then splits into the individual file tokens;
   `PREFIX`/`DEST`/`STRIP`/literal args carry no `$` and pass through unchanged.

`FROM_SANDBOX` already expands `OUTNoAutoFiles` via `expandStmtTokensSTR(env)`, so
seam (1) alone fixes its path; seam (2) fixes `RESOURCE_FILES`.

## Files touched

- `modules.go` — `tokSetAppend` generic var binding; `ResourceFilesStmt` arg
  expansion through env.
- `modules_test.go` — regression `TestCollectModule_SetAppendExpandsResourceAndSandboxInputs`:
  `SET_APPEND(VAR …)` then `FROM_SANDBOX(OUT_NOAUTO ${VAR})` + `RESOURCE_FILES(${VAR})`;
  asserts both the sandbox OUT_NOAUTO list and the RESOURCE_FILES paths receive
  the expanded files, not the literal `${VAR}`. Fails before, passes after.

## Expected effect on the gate

- `go test ./...` green; new regression fails before / passes after.
- `./dev/validate.py` keeps sg2/sg2_x86_64/sg3/sg4 OK, XFAIL count, sg5 matched
  count, and the per-case generation-time budget. sg2–sg6 stay byte-exact: a
  referenced-but-dropped SET_APPEND var would already diverge those (upstream
  expands it), so binding it changes nothing there.
- Direct `yabs/plutonium/libs/dictionaries_resource` (and `_for_workers`) no
  longer emits the literal `${PLUTONIUM_IN_YABS_DICTIONARIES}` missing-file path.
  Any remaining sg7 generation blocker is reported (own macro + example) for a
  separate ticket; graph divergence classification is out of scope here.
