# T-46 plan: Cython-induced pyx Use closure on the generated `.c`/`.cpp` compile

## Symptom (reproduced on trunk after T-27, sg7)

The generated cython C++ compile
`$(B)/contrib/python/lxml/py3/_/lxml/objectify.pyx.c.pic.o` is missing **46
ref-only inputs** (zero `+ours` over-collection):

```
+ $(B)/contrib/python/lxml/py3/lxml/etree.c                 (producer main output)
+ $(S)/contrib/python/lxml/py3/lxml/etree.pyx               (producer pyx)
+ …/apihelpers.pxi … xsltext.pxi                            (producer .pxi set)
+ …/cvarargs.pxd, includes/{c14n,config,…,xslt}.pxd         (producer .pxd closure)
+ contrib/tools/cython/Cython/Includes/cpython/{mem,object,ref,type}.pxd
+ contrib/tools/cython/Cython/Includes/libc/stdlib.pxd
```

The deps of the node already match the ref exactly (the producer `etree` CY node
is already a dep via the `etree_api.h` include resolution). Only the flat input
list diverges. The named producer nodes `etree.h`, `etree_api.h`, `objectify.pyx.c`
remain byte-exact from T-27 — this is purely the *compile-consumer* input closure.

## Verified upstream mechanism

`devtools/ymake/include_processors/cython_processor.cpp`
`TCythonIncludeProcessor` rule:

```
Rule.Actions: EVI_InducedDeps "pyx"   -> Use
              EVI_InducedDeps "cpp"   -> Pass
              EVI_InducedDeps "h+cpp" -> Pass
Rule.PassInducedIncludesThroughFiles = true;
```

The `etree` CY node carries an induced **"pyx"** property = its resolved
cimport/include/pxd closure (`etree.pyx` + its `.pxi`/`.pxd`). With
`PassInducedIncludesThroughFiles`, that induced set rides *through files that
include any of the node's outputs* onto the consumer. The generated
`objectify.pyx.c` `#include`s the sibling `etree_api.h` (a `${output}` of the
`etree` CY node whose **main** output is `etree.c`). So the C++ compile of
`objectify.pyx.c`:

1. Uses the `etree` node's "pyx" induced closure (`etree.pyx` + `.pxi`/`.pxd`).
2. lists the producer's main output `etree.c` (the `etree_api.h` OutTogether
   main: `etree_api.h`/`etree.h` are `${output}` siblings of `etree.c`).

This is the SAME "pyx" Use set T-27 already routes onto the *consuming CY node*
(`objectify.pyx.c` producer, via `cythonInducedPyxClosure` reading
`GeneratedFileInfo.CythonInducedPyx`). T-27 deliberately kept it off the C++
compile; the ref shows the compile DOES carry it — that is this ticket.

Empirical confirmation (sg7 ref): `etree.c` is an input of exactly two nodes —
`etree.c.pic.o` (its own compile) and `objectify.pyx.c.pic.o` (the sibling-api
consumer). It is NOT a broad leaf on every `etree_api.h`/`etree.h` consumer, and
`etree_api.h`'s only closure consumer here is the generated cython compile. So
the fix must be targeted to *generated cython compile nodes*, not a global
ClosureLeaf on the header (which would also attach the pyx source closure to
unrelated hand-written C++ compiles that include the header).

## Our gap

`emitCythonCpp` (emit_cython_cpp.go) augments only the consuming **CY node**'s
tool inputs with the induced "pyx" closure (`cythonInducedPyxClosure`, line 272).
The generated `.c`'s **C++ compile** inputs (`ccIn.IncludeInputs =
walkClosure(generatedVFS)`, line 352) are NOT augmented, so the compile misses
the api-header producer's pyx closure and main output.

The pyx files must ride as **non-expanded** inputs: re-resolving `etree.pyx` /
the `.pxd`s through the cython parser in the C++ walk would pull the producer's
`cdef extern from` C headers (libxml/libxslt) — which the ref does NOT carry.
`CythonInducedPyx` is already the *cython-language-only* transitive closure
(`cythonPyxLangClosure` drops C headers), so appending it verbatim is correct.

## Change (test-first, surgical, generic to generated cython compiles)

1. `codegen_registry.go`: record the producing CY node's **main output** on the
   generated header alongside `CythonInducedPyx`. Add `CythonMainOut VFS` to
   `GeneratedFileInfo`; extend `setCythonPyxInduced(node, pyx, mainOut)` to set
   both, and add accessor `cythonPyxInducedInfo(node) (pyx []VFS, mainOut VFS)`.
2. `emit_cython_cpp.go`:
   - phase 1: pass `generatedVFS` (the CY node main output, e.g. `etree.c`) to
     `setCythonPyxInduced` for each header output.
   - phase 2: after `ccIn.IncludeInputs = walkClosure(generatedVFS)` and the
     `resolveCodegenDepRefs` over it (so deps stay byte-identical), augment the
     compile inputs: `ccIn.IncludeInputs = cythonCompileInducedInputs(ctx,
     instance, ccIn.IncludeInputs)`. The helper scans the compile's resolved
     includes for `$(B)` registered cython headers carrying `CythonInducedPyx`
     and unions in that pyx set + the recorded main output (dedup, order-stable).
3. `emit_cython_cpp_test.go`: regression `TestGen_CythonGeneratedCompileCarriesInducedPyx`
   — module with `CYTHON_C app/cons.pyx` (consumer, `#include`s `prod_api.h` via
   `pub.pxd`'s `cdef extern from`) + `CYTHON_C_API_H app/prod.pyx` (producer,
   closure = `prod.pyx`+`helper.pxd`+`h.pxi`, main `prod.c`) + `SRCS(helper.cpp)`.
   Assert the compile `$(B)/pkg/_/app/cons.pyx.c.o` gains `prod.pyx`, `helper.pxd`,
   `h.pxi` (producer pyx closure) and `$(B)/pkg/app/prod.c` (producer main); the
   un-cimported `unrelated.pxd` stays absent; the hand-written `helper.cpp.o`
   carries none of the producer pyx closure (no broad attachment). Fails before
   (compile has only cons.pyx/pub.pxd/prod_api.h), passes after.
