# T-46 plan: yt/yt_proto PY3_PROTO include duplicate + PY_NAMESPACE resource key

## Scope (from T-39B)

Two narrow PY3_PROTO residuals for `PROTO_LIBRARY` modules that declare
`PROTO_NAMESPACE(yt)` together with `PY_NAMESPACE(...)`:

1. **Command gap.** Paired `__intpy3___pb2.py`, `.pyi` and `_pb2_grpc.py`
   producers are missing the extra own `-I=$(S)/yt` that upstream renders in the
   `_PROTO__INCLUDE` region.
2. **Resource gap.** Aux resource keys collapse nested source subdirectories
   when `PY_NAMESPACE` is set (`yt_proto/yt/client/<base>_pb2.py`), while the
   reference preserves the module-local proto subdirectory
   (`yt_proto/yt/client/chunk_client/proto/data_statistics_pb2.py`).

The `libpy3yt_proto-*` archive churn is in scope only as a downstream
consequence of (2). No generic archive ordering / AR / link-closure work.

## Upstream mechanism

`PROTO_NAMESPACE(yt)` (proto.conf:134) always calls `PROTO_ADDINCL(GLOBAL …)`
with a *literal* `GLOBAL` regardless of the macro's own `GLOBAL?` keyword. The
SOURCE arm `GLOBAL FOR proto $(S)/yt` and the build arm
`ADDINCL(GLOBAL $(B)/yt)` therefore propagate through the proto-addincl /
global-addincl closure into every consumer's `_PROTO__INCLUDE`.

`_PY_PROTO_CMD_BASE` (proto.conf:515) renders
`-I=./$ns -I=$(S)/$ns ${pre=-I=:_PROTO__INCLUDE} -I=$(B) -I=$PROTOBUF_INCLUDE_PATH`.
`_PROTO__INCLUDE` renders the module's own namespace once **plus one more copy
whenever a peer re-contributes `$(B)/<ns>`** (deduped to a single extra). This
is exactly the rule the C++ side already implements:

    emit_proto.go:452  cfg.duplicateOutputRootInclude =
        containsVFS(peerContribs.addIncl, build(cfg.cppOutRoot))
    emit_pb.go:356-361 render `-I=$(S)/<root>` once, then a second time iff
        duplicateOutputRootInclude.

The asymmetry between the C++ and Python sides of the *same* module is real and
verified in the sg7 reference:

- `yt/core` C++ `.pb.cc`: one `-I=$(S)/yt` in `_PROTO__INCLUDE` (no proto peer).
- `yt/core` Python `_pb2.py`: **two** `-I=$(S)/yt` in `_PROTO__INCLUDE`.

The extra Python copy comes from the **CPP_PROTO self-sibling** that every
optimized PY3_PROTO PROTO_LIBRARY peers (`walkPeersForGlobalAddIncl`,
gen.go:3049). That sibling carries `ADDINCL(GLOBAL $(B)/yt)`, so the Python
module's `peerContribs.addIncl` contains `$(B)/yt` even when no *real* proto
peer does. `protos_from_protoc` (`NO_OPTIMIZE_PY_PROTOS`) has no self-sibling,
so `containsVFS` is false and it correctly gets no duplicate — matching the
reference (`contrib/libs/protoc/src` py command has one copy).

So the correct, general Python condition is **identical** to the C++ one,
evaluated against the Python module's own `peerContribs.addIncl`:

    duplicateOutputRootInclude = containsVFS(peerContribs.addIncl, build(protoRoot))

This subsumes the current `d.grpc && d.protoNamespaceGlobal` special case
(verified: `contrib/proto/grpc/grpc/health/v1` and
`contrib/libs/googleapis-common-protos` are GLOBAL grpc PROTO_LIBRARYs with a
CPP_PROTO sibling, so `containsVFS` is true for them too) — that branch is
deleted.

### Resource key

`protoPythonResourceKey` collapses nested sources to `filepath.Base(base)` when
`PY_NAMESPACE` is set. The sg7 reference preserves the module-local proto path
under the namespace, e.g. SRC `api/rpc_proxy/proto/api_service.proto` in a
`PY_NAMESPACE(yt_proto.yt.client)` module →
`resfs/file/py/yt_proto/yt/client/api/rpc_proxy/proto/api_service_pb2.py`.
Root-level sources (no `/`) are unaffected, so the fix is to drop the
`filepath.Base` collapse and use the full module-local `base`. This mirrors the
already-correct no-namespace and `PY_NAMESPACE(.)` branches.

   i.e. three total including the output-root arg; order asserted.
3. `TestGen_PyProtoLibrary_GrpcRootSourceSharesDuplicateInclude` — a GRPC
   root-level `yt/orm/api` shape: `_pb2_grpc.py` is emitted and its producer

(T-13 consumer, protoRoot == "") must stay green: the consumer renders exactly
one `-I=$(S)/yt` from the tail and the duplicate logic does not apply.
