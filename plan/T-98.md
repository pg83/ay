# T-98 plan: explicit-dot source path canonicalization for object/archive names

## Representative

`ysite/yandex/pure/ya.make` declares `SRCS(./generated/default_pure.cpp)`.

- ours: `$(B)/ysite/yandex/pure/_/./generated/default_pure.cpp.o` (+ `.pic.o`)
- ref:  `$(B)/ysite/yandex/pure/_/generated/default_pure.cpp.o` (+ `.pic.o`)

The downstream archive member path carries the same extra `./`. Compile command,
inputs, and include closure already match — the only divergence is the literal
`./` from the SRCS token leaking into the localized object name.

## Upstream mechanism

ymake canonicalizes a source path (NPath / TFileConf path normalization removes
`.` segments) before `TCommandInfo::InitDirs` localizes a subdir source's object
under `_/<dir>`. The leading `./` of an explicit-dot SRCS token is therefore
stripped before the object name is formed; the object dir is the canonical
module-relative directory, not the raw token.

In our emitter, `sources.go:resolveSourceVFS` already path-cleans the *resolved*
`srcVFS` (its `.rel()` is canonical), but `composeCCPaths` (emit_cc.go) builds the
localized object name from the *raw* `srcRel` token. For `SRCS(./generated/x.cpp)`
the source resolves canonically inside the module (`canonMatches()` is true), so
control falls through to the generic naming switch, which feeds the raw
`./generated/x.cpp` into `normalizeDotDotSegments`, producing `_/./generated/x.cpp`.

## Fix

Canonicalize `srcRel` with the existing `cleanRel` helper immediately before the
generic naming switch in `composeCCPaths`. `cleanRel` strips `.`/`//` segments and
leaves a leading `..` intact (so the `../` → `__` mapping in
`normalizeDotDotSegments` is unchanged) and is the identity on already-clean tokens
(so ordinary `generated/foo.cpp`, root-relative, and the `_/`-prefixed rl6 build
leaf are untouched). The SRCDIR / build-rooted branches run *before* this point and
keep seeing the raw token, so SRCDIR and `${BINDIR}` cases are preserved.
