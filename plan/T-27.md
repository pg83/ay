# T-27 plan: residual yabs/lua_templates LJ_21_ARCHIVE sg7 mismatch

## Goal

Eliminate the three residual sg7 self_uid mismatches T-26 identified for the
`yabs/lua_templates` `LJ_21_ARCHIVE` surface:

- `$(B)/yabs/lua_templates/LuaScripts.inc` â€” ref has one extra input
  `$(S)/yabs/lua_templates/tests/ft/yabs_md5.lua`.
- `$(B)/yabs/lua_templates/templates.cpp.o` â€” ref command has `-I$(B)/yabs/lua_templates`.
- `$(B)/yabs/lua_templates/templates_sources.cpp.o` â€” ref command has
  `-I$(B)/yabs/lua_templates` and 356 `$(S)/.../*.lua` inputs.

## Upstream mechanism (verified)

`yatool/build/ymake.core.conf`:

```
macro ARCHIVE(NAME="", DONTCOMPRESS?"-p":"", Files...) {
    .CMD=$ARCH_TOOL -q -x $DONTCOMPRESS ${suf=\:;input:Files} -o ${addincl;noauto;output:NAME} ...
}
macro ARCHIVE_BY_KEYS(NAME="", KEYS="", DONTCOMPRESS?"-p":"", Files...) {
    .CMD=$ARCH_TOOL -q -x $DONTCOMPRESS ${input:Files} -k $KEYS -o ${addincl;noauto;output:NAME} ...
}
```

`LJ_21_ARCHIVE` (`build/plugins/lj_archive.py`) compiles each `.lua` to a `.raw`
and emits two `ARCHIVE_BY_KEYS` calls â€” `LuaScripts.inc` over the raws and
`LuaSources.inc` over the source luas, both `-k <colon-joined module-relative
lua names>`.

Two upstream effects that we currently miss, both shared with plain `ARCHIVE`:

1. `${addincl;noauto;output:NAME}` puts the archive output's build dir into the
   module's include buckets at **Global** scope (already modeled for plain
   `ARCHIVE` by `applyArchiveAddIncl` in `modules.go`). For `yabs/lua_templates`
   that is `$(B)/yabs/lua_templates`, giving its own C++ compiles
   `-I$(B)/yabs/lua_templates`.
2. The archived source members ride into a C++ unit that `#include`s the
   generated header as that unit's input closure (already modeled for generated
   `.raw` members via `GeneratedFileInfo.SourceInputs` â†’ archive `ClosureLeaves`).
   `LuaSources.inc`'s members are direct source luas, which have no producer and
   so register no leaves today.

## Divergence from T-26 (flagged)

T-26's "`UseSourceInputsForGeneratedMembers`: use SourceInputs for the archive
node Inputs instead of the generated output path" is **incorrect** for the
`LuaScripts.inc` input symptom and is NOT implemented here. Measured facts:

- The raw upstream `LuaScripts.inc` node has 713 inputs: 356 `$(B)/.../*.raw` +
  356 `$(S)/.../*.lua` + the archiver. Our node has 356 raw + archiver = 357.
- The normalizer applies AR/LD input pruning **only to the reference graph**
  (`--ref-graph`); our side is faithful. `filterARLDInputs` keeps an input iff
  the command names its basename. The archiver command names the `.raw` members
  and the keys, never the source luas â€” so all 356 source luas are over-emission
  the AR filter is meant to discount.
- The filter drops 355 of them but keeps `yabs_md5.lua`, because
  `nodeCmdBasenames` extracts a file basename from the `-k` value (one
  colon-joined argument `â€¦:tests/ft/yabs_md5.lua`); `baseName` of that whole
  token is `yabs_md5.lua`, which collides with the over-emitted input.

So the residual `yabs_md5.lua` is a **reference-side AR-filter artifact**: the
`-k $KEYS` value is an archive key list, not input file paths, and must not
contribute "command-named input" basenames. Emitting source luas on our AR node
(as T-26 suggested) would make our faithful side carry all 356 luas (713 total)
and *widen* the gap to 355 tokens, not close it. Verified empirically: skipping
the `-k` value in `nodeCmdBasenames` drops the phantom lua and the ref node
normalizes to 357, matching ours exactly.

## Changes

1. `dump.go` `nodeCmdBasenames`: skip the argument that follows `-k` (the
   archiver KEYS value). Generic across `ARCHIVE_BY_KEYS`; `-k` appears only on
   archiver nodes (none in sg2..sg6).
2. `modules.go` `applyArchiveAddIncl`: when `d.lj21 != nil`, add `build(modulePath)`
   to the local + global + user-global include buckets â€” same Global-scope
   `addincl;output` effect already applied to `d.archives`, but driven from
   `d.lj21` because the LJ archive entries are synthesized later (`emitLuaJit21`).
3. `modules.go` `ArchiveEntry`: add `PropagateSourceMembers bool`.
4. `emit_archives.go` `emitArchive`: when `a.PropagateSourceMembers` and a member
   resolves to a direct source (no producer in the registry), add that source VFS
   to the archive `ClosureLeaves` (member order, deduped). Generated members keep
   the existing `SourceInputs` folding.
5. `emit_lj.go`: set `PropagateSourceMembers: true` on both LJ archive entries.

`LuaScripts.inc` node content (cmd args, inputs, deps) is unchanged on our side;
its parity is achieved by the normalizer fix. `LuaSources.inc` node content is
unchanged; only its registered closure leaves grow.
