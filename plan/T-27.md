## Verified upstream mechanism

`devtools/ymake/include_processors/cython_processor.cpp`
`TCythonIncludeProcessor::ProcessIncludes`, `CimportFrom` case (lines 95-159):

For `from X cimport name1, name2`:
1. `searchPath` = X with leading-dot ŌåÆ `../` and `.`ŌåÆ`/` (relative-import rule).
2. skip if first path part == `cython`.
3. resolve `searchPath/__init__.pxd` ŌåÆ keep if resolved (package init, always tried).
4. if X non-empty: resolve `searchPath.pxd` ŌåÆ keep if resolved, then STOP (X is a
   module, names are names-within).
5. else (X is a package): for each name resolve `searchPath/name/__init__.pxd`
   then `searchPath/name.pxd` ŌåÆ keep resolved (submodule cimport).

`CimportSimple` (`cimport a.b`): resolve `a/b.pxd` then `a/b/__init__.pxd`.

Every candidate is funnelled through `module.ResolveLocalIncludes`, which
**silently drops unresolved** entries ŌĆö cython cimport resolution is best-effort,
not an error. The kept set rides the CY node as "pyx"-language includes
(`AddIncludesToNode`, rule `pyx ŌåÆ Use`), so it lands on the producer (both the
Header pyx-only closure and the non-Header full closure). Steps 4-vs-5 are
mutually exclusive in practice (a path is a file XOR a directory), so emitting
all candidates and letting resolution filter is byte-equivalent to the
conditional.

## Our gap

## Re-implementation (arbiter CONTINUE after reviewer REWORK)

The first cut closed etree.h / etree_api.h but left `objectify.pyx.c` reporting
the full ref-only etree pyx class, and over-collected the per-name submodule
candidates. Root causes found by dumping the lxml producers on trunk:

1. **`addincl;output` for the generated api header.** `_BUILDWITH_CYTHON_*_H /
   _API_H` (ymake.core.conf) emit the `.h` / `_api.h` with
   `${hide;addincl;tobindir;output ŌĆ”}` ŌĆö the header's build dir enters the
   module include dirs at Global scope (macro_processor.cpp AddToIncl). Without
   it `includes/etreepublic.pxd`'s `cdef extern from "etree_api.h"` never
   resolved, so the consumer never reached the producer. New
   `applyCythonHeaderAddIncl` (modules.go) reproduces it, like
   `applyArchiveAddIncl`.

2. **Register all header outputs before any source closure walk.** PY_SRCS lists
   the consumer (`CYTHON_C objectify.pyx`) before the producer
   (`CYTHON_C_API_H etree.pyx`). `emitCythonCpp` now runs a phase-1 pass that
   reserves each CY node and registers its header outputs (+ the producer's
   pyx-induced set), then a phase-2 pass that walks closures and emits nodes ŌĆö
   mirroring upstream building the dep graph before include resolution.

3. **Cython-induced "pyx" Use passthrough (typed, not a window splice).**
   Upstream's TCythonIncludeProcessor attaches the resolved cimport/include
   closure to the node as EVI_InducedDeps "pyx" (action Use). A CYTHON consumer
   that cdef-externs the generated header Uses that set; a C++ consumer (the
   generated `.c`'s compile) does not. Modeled by recording the producer's pyx
   closure on the header (`GeneratedFileInfo.CythonInducedPyx`) and unioning it
   into a consuming CY node's tool inputs (`cythonInducedPyxClosure`) ŌĆö NOT a
   ClosureLeaves splice, which would also reach the C++ compile and (re-resolving
   the producer's `.pxd` `cdef extern from`) pull in the producer's libxml/libxslt
   C headers.

4. **needCheckLists=false (reviewer ask).** `addCythonCimportFrom` emits the
   module `searchPath.pxd` candidate as `includeCythonModule` and the per-name
   submodule candidates as `includeCythonName`; `forEachResolvedChild` skips the
   name probes once the module .pxd resolves (upstream sets needCheckLists=false).
   Only observable across distinct addincl roots (a path is a file XOR a
   directory in one root), so the corpus is unaffected ŌĆö covered by a focused
   two-root regression.

Out of class: `objectify.pyx.c.pic.o` (the generated `.c`'s C++ compile) still
under-collects etree's flat pyx set + `etree.c`. That is the C++-side induced
passthrough (PassInducedIncludesThroughFiles to the compile, + the api header's
OutTogether main) ŌĆö a distinct mechanism, no over-collection introduced.

## Addendum: first-resolved fallback suppression (reviewer rework, 2026-06-20)

### Upstream mechanism (re-read cython_processor.cpp 75-145)

Both candidate pairs upstream are *first-resolved-wins ordered chains*, not
unconditional collect-both:

- **CimportSimple** (`cimport a.b`): resolve `a/b.pxd`; **only if it does not
  resolve** try `a/b/__init__.pxd`; push exactly one path.
- **CimportFrom per-name**: resolve `X/name/__init__.pxd`; **only if it does not
  resolve** try `X/name.pxd`; push exactly one path (or skip the name).

Our parser emits both members of each pair, and `forEachResolvedChild` collected
both ŌĆö over-collecting when the two members resolve through *different* cython
ADDINCL roots (a path is a file XOR a directory within one root, so this is the
only way both exist). Same edge class as the already-tested module-vs-name case.

### Change

- `scanner.go`: add kind `includeCythonFallback` (resolves like a cython probe,
  dropped no-warn). `forEachResolvedChild` tracks `prevProbeMissed`: a fallback
  candidate is probed **only when the immediately preceding probe was attempted
  and did not resolve**. A probe suppressed by `suppressNames` counts as
  not-missed, so a module-suppressed name's `.pxd` fallback is also dropped.
  `suppressNames` is now cleared by any non-name/non-fallback directive (the
  statement opener `includeCythonOptional`), so a name's fallback no longer
  clears the name-list suppression mid-statement.
- `parsers.go`: emit each pair's second member as `includeCythonFallback`
  (`addCythonPxdCandidates` `path/__init__.pxd`; `addCythonCimportFrom` per-name
  `base+".pxd"`).

### Regressions

- `TestGen_CythonCimportSimpleFirstResolvedFallback`: `cimport sub.leaf` with
  `sub/leaf.pxd` via root ra and `sub/leaf/__init__.pxd` via root rb ŌĆö only
  `sub/leaf.pxd` rides the CY node.
- `TestGen_CythonCimportFromNameFirstResolvedFallback`: `from sub cimport leaf`
  (no `sub.pxd`) with `sub/leaf/__init__.pxd` via ra and `sub/leaf.pxd` via rb ŌĆö
  only `sub/leaf/__init__.pxd` rides.

Both fail on collect-both, pass after the fallback rule. Gate: lxml targets stay
exact; sg2..sg6 byte-exact; sg7 matched not regressed.
