# Plan: T-2 – Split-codegen shard input wiring

## Upstream mechanism identified

`v1_proto_split_antlr4` and `v0_proto_split` use a `RUN_PYTHON3(multiproto.py ...)` step that splits
a monolithic protobuf `pb.cc`/`pb.h` pair into N shard `.cc` files (`pb.code0.cc`…`pb.code9.cc`,
`pb.data.cc`) and two helper headers (`pb.main.h`, `pb.classes.h`).

The upstream build graph contracts:

- **Shard CC nodes** (`pb.codeN.cc.o`): their include-input closure lists the *source-level* leaf
  inputs of the split step (the generator script, the antlr/configure chain, the protobuf runtime
  headers from induced deps).  They do **not** reference the build-generated `pb.h`, `pb.cc`, or the
  intermediate `.proto`/`.g` files.

- **`pb.main.h`**: its registered parsed-includes list the **shard CC files** (`pb.code0.cc`…
  `pb.data.cc`), making every consumer of `pb.main.h` carry the shard paths in their input closure.
  That is how downstream CC nodes end up with `pb.code0.cc` as an input.

## Current divergence (from `bytoken.txt`)

| Token | Ours-only | Ref-only |
|-------|-----------|----------|
| `$(B)/SQLv1Antlr4Parser.pb.cc` | 60 | — |
| `$(B)/SQLv1Antlr4Parser.pb.h`  | 60 | — |
| `$(B)/SQLv1Antlr4Parser.proto` | 61 | — |
| `$(B)/SQLv1Antlr4.g`           | 62 | — |
| `$(B)/Java.stg`                | 62 | — |
| `$(B)/SQLv1Antlr4Parser.pb.code0.cc` | — | 59 |

Root cause: `pyEmitsIncludes` (in `emit_py_run.go`) registers all `IN` files of the RUN_PYTHON3
statement as include-directives of EVERY generated output, regardless of type.  For the split-codegen
pattern the `IN` files are `pb.h` and `pb.cc` (both build-generated). The scanner then walks their
registered chains into build-generated intermediates, producing the extra inputs above and missing the
shard-CC inputs in consumer nodes.

## Files to change

- `emit_py_run.go` — modify `emitRunPython` (new helper `isSplitCodegenStmt`) and `pyEmitsIncludes`
  (new split-codegen branch for CC-shard outputs and header outputs)

No changes to `emit_pb_primitive.go`, `emit_proto.go`, or `emit_py_proto.go` needed — the bug is
purely in the generic `RUN_PYTHON3` emission path.

## Fix design

### Detection

A RUN_PYTHON3 is "split-codegen" if its `OUTNoAutoFiles` contains **both** CC source files
(`isCCSourceExt`) **and** header files (`isHeaderSource`).  This matches `multiproto.py` and any
structurally identical step.

### `splitSources` computation

Before registering any output, compute the source-level transitive closure of the IN files:

```
for each IN file that is build-generated (IsBuild):
    walk its registered parsedIncludes via walkClosure
    filter result to source-only VFS entries
splitSources = [scriptVFS] + source-only VFS from all IN files (deduped)
```

`walkClosure` at this point is safe because the build-generated IN files (`pb.h`, `pb.cc`) were
already registered by the preceding `emitRunProgramsForAR` call.

### Registration for CC shard outputs (`pb.codeN.cc`, `pb.data.cc`)

Replace the default `[scriptVFS, $(B)/pb.h, $(B)/pb.cc]` with `splitSources`.  The scanner then
finds only source-level files when walking the shard's registered includes.

### Registration for header outputs (`pb.main.h`, `pb.classes.h`)

Register with `[shard CC OUT_NOAUTO paths…] + splitSources`.
The shard CC paths make every consumer of `pb.main.h` carry the shards in their input closure.

### Other OUT_NOAUTO (non-CC, non-header)

Unchanged — use existing `pyEmitsIncludes` logic.

## Expected effect on gate

After the fix:
- `pb.cc`, `pb.h`, `proto`, `SQLv1Antlr4.g`, `Java.stg` disappear from CC node inputs (—60 each).
- `pb.code0.cc` appears in consumer CC inputs (+59).
- sg5 token diff shrinks substantially toward zero.
- No other modules affected (the split-codegen condition is narrow).
- No perf change (no new walks; the existing `pyInputClosure` walk path remains the same,
  just reading different registered includes).
