# T-134 plan: constrain cfgproto cpp-only `wire_format.h` induced-dep over-inclusion

## Defect (reproduced hypothesis)

Ordinary downstream C++ consumers of a generated `.cfgproto.pb.h` carry an extra
input `$(S)/contrib/libs/protobuf/src/google/protobuf/wire_format.h`. Reference
keeps `wire_format.h` only on the generated translation units
(`.pb.cc.o` / `.ev.pb.cc.o` / `.yaff.cpp.o`). Representative pair:
`$(B)/balancer/kernel/balancing/backends.cpp.o` — its only `inputs` diff is this
one ours-only header (commands / outputs / target_properties all match).

## Upstream mechanism

`.cfgproto` is built with the EV protoc wrapper (`CPP_EV_OUTS`), which marks NO
`main` output. So a unit that `#include`s the generated `.cfgproto.pb.h` must
also reach its sibling `.cfgproto.pb.cc` via `EDT_OutTogether` (json_visitor
`PrepareLeaving`) — a bare output-sibling edge, NOT a C++ include. Upstream does
not re-resolve the sibling `.pb.cc`'s own includes / induced deps per consuming
module; the sibling rides as a non-expanded leaf.

Our emitter (`emit_cfgproto.go`) modeled that OutTogether edge as a *parsed
quoted include* of `cfgPbCC` on `cfgH`. A parsed include is **traversed** during
a consumer's closure walk, so the scanner descends into `cfgPbCC` and runs
`resolveInducedDeps(cfgPbCC)`. `cfgPbCC`'s `GeneratorRefs` include protoc, and
protoc's cpp-only `INDUCED_DEPS(cpp … wire_format.h)` bucket
(`contrib/tools/protoc/ya.make.induced_deps`) is then mixed into the consumer.
`wire_format.h` is the unique cpp-only induced header that no runtime header
`#include`s, so it has no other path into the consumer — hence it is the only
ours-only leak (the other cpp-bucket headers reach consumers via `message.h`'s
real include closure, which is independently walked).

The EV path (`evWitnessExtras` → `evPbCC`) registers its sibling `.pb.cc` with
`GeneratorRefs = [event2cppLDRef]` only (no protoc), so it induces no
`wire_format.h` and is not part of this defect — left untouched (surgical).

## Fix

In `emit_cfgproto.go`, model the OutTogether sibling `.pb.cc` with the existing
`ClosureLeaves` seam instead of a traversed parsed include: drop the
`cfgHParsed = append(…, cfgPbCC)` line and add
`reg.addClosureLeaf(cfgH, cfgPbCC)`. The sibling still rides into every consumer
that includes `cfgH` (bare, transitively) — EDT_OutTogether parity — but its own
`#includes` / induced deps are no longer re-resolved per consumer, so the
cpp-only `wire_format.h` stops leaking. The self `.cfgproto.pb.cc.o` compile
still walks `cfgPbCC` directly and keeps `wire_format.h` via its GeneratorRefs.

No change to `resolveInducedDeps` (must stay context-independent), no path
allowlists, net removal of a code path.

## Test (fail-first)

`emit_cfgproto_test.go`: a `.cfgproto` library whose generated `.pb.h` is
`#include`d by an ordinary C++ source in a peer module; protoc tool declares
`INDUCED_DEPS(cpp …/wire_format.h)`. Assert:
- generated `.cfgproto.pb.cc.o` KEEPS `wire_format.h` (required runtime input),
- the ordinary consumer object does NOT list `wire_format.h`,
- the consumer still carries the sibling `.cfgproto.pb.cc` and `.pb.h`
  (EDT_OutTogether parity preserved).

Fails before the fix (consumer carries `wire_format.h`); passes after.
