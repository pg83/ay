# T-13 — RESOURCE_FILES root-relative source resolution for the resfs/src kv input

## Symptom (repro)

Direct sg7 on `yabs/server/libs/banner_flags` aborts while content-hashing
`yabs/server/libs/banner_flags/modadvert/dyn_disclaimers/disclaimers_config.pb.txt`.
That path does not exist: the file lives at the arcadia root,
`modadvert/dyn_disclaimers/disclaimers_config.pb.txt`. The consumer module
`yabs/server/libs/banner_flags/ya.make` declares
`RESOURCE_FILES(modadvert/dyn_disclaimers/disclaimers_config.pb.txt)`.

## Upstream mechanism

`build/plugins/res.py:RESOURCE_FILES` expands each `path` into two RESOURCE
pair members:

- the payload `path key` (key = `resfs/file/<prefix><path>`)
- a `- resfs/src/<key>=${rootrel;context=TEXT;input=TEXT:"<path>"}` member.

Doc (res.py): *resfs/src/{key} stores a source root (or build root) relative
path of the source of the value of the {key} resource.* The `input=TEXT:"path"`
makes ymake resolve `path` as an input; `rootrel` then yields its root-relative
path. For an ordinary source file that exists at the arcadia root, ymake binds
the input to `$(S)/<path>` and rootrel = `<path>` — never module-prefixed. This
is the SAME input resolution the payload member uses.

## Our gap

`emit_py_objcopy.go`:

- payload member (L290) resolves through `resolveResourceInput(..., copyFileInputVFS(...))`.
  `copyFileInputVFS`→`sourceInputVFS` checks the fs: a root-relative path that
  exists at `srcRootVFS` resolves to `$(S)/<path>`. Correct already.
- resfs/src kv member (L285) used the fallback `source(instance.Path.rel()+"/"+inner)`
  — naive module-dir concatenation, no fs check. That fabricates the phantom
  `$(S)/yabs/server/libs/banner_flags/modadvert/...` input, which sg7 then tries
  to content-hash and aborts.
- the kv command string was expanded by `expandRootrel(kv, unitPath)` = the same
  naive `unitPath + "/" + inner` concat, so the emitted resfs/src value was the
  phantom path too.

The generated-output case (RUN_PROGRAM STDOUT/OUT, FROM_SANDBOX, BUNDLE) is
unaffected: `resolveResourceInput` still hits the codegen registry first and
binds to `$(B)` with the producer dep; for those the rootrel equals
`module/inner`, which the resolved `$(B)` VFS `.rel()` already yields.

## Fix (generic, no special-casing)

Resolve the resfs/src kv input the same way as the payload input:

1. kv fallback uses `copyFileInputVFS(ctx.fs, instance.Path.rel(), inner)` instead
   of `source(instance.Path.rel()+"/"+inner)`.
2. Derive the emitted resfs/src command value from the *resolved* input VFS
   `.rel()` (its rootrel), not from a naive concat. Replace `expandRootrel`
   (concat) with `rootrelExpand(kv, resolvedRel)` that substitutes the resolved
   root-relative path into the `${rootrel…}` marker. For a generated $(B) output
   `resolvedRel` = `module/inner` (unchanged); for a root source it = `<path>`.

The objcopy hash is unchanged: it is computed over the RAW `cur.paths` /
`cur.kvs` (with the `${rootrel…}` marker), not over the expanded values.

## Files

- `emit_py_objcopy.go`: kv fallback + carry expanded kv command (`cur.kvsCmd`).
- `resource.go`: replace `expandRootrel` with `rootrelExpand(kv, resolved)`.
- `emit_py_objcopy_test.go`: regression — nested consumer module RESOURCE_FILES
  a source at the arcadia root outside the module; assert objcopy Inputs contain
  `$(S)/<root path>`, never the module-prefixed phantom, resfs/file key
  unchanged, and the resfs/src command value uses the root path.

## Gate

`go test ./...` + `./dev/validate.py` green; direct sg7 no longer reports the
disclaimers_config.pb.txt phantom; any later sg7 blocker is its own mechanism.
