# T-145 plan: RESOURCE objcopy carries RUN_PROGRAM producer's build-root OUTPUT_INCLUDES closure

## Symptom (reproduced, sg7)

`$(B)/yabs/models_services/feature_store/build_configs/configs/caesar/objcopy_39091f2a7032395522afdc2649.o`

- ref inputs: 518 — 1 source (`build/scripts/objcopy.py`) + 517 build:
  511 `.pb.h`, `features.gen.h`, `ads/argus/proto/profile/profile.yaff.h`,
  the two `.pb` resource payloads, and the two rescompiler/rescompressor tools.
- our inputs: 5 — objcopy.py + 2 tools + 2 `.pb` payloads. Missing the **513**
  build-root generated headers (511 `.pb.h` + `features.gen.h` + `profile.yaff.h`).
- ref deps: 507 (the producers of those headers). our deps: 3.
- `cmds` and `outputs` are byte-identical; ONLY `inputs`/`deps` diverge.

The module's `RUN_PROGRAM` (run_codegen.inc) is:

```
RUN_PROGRAM(... <no IN>
  OUTPUT_INCLUDES <source headers> + ${FS_CODEGEN_IMPORT_FILES_INCLUDED} (many generated .pb.h)
  OUT ${BINDIR}/features.gen.h features.gen.cpp feature_sets_and_mutations.gen.cpp
      input_proto_masks.pb feature_to_precompile_info.pb)
RESOURCE(${BINDIR}/input_proto_masks.pb input_proto_masks.pb
         ${BINDIR}/feature_to_precompile_info.pb feature_to_precompile_info.pb)
```

## Upstream mechanism

1. `macro_processor.cpp` (`addOutputIncludes`, ~L1250) attaches the
   `OUTPUT_INCLUDES` induced-deps (`EVI_InducedDeps`) parsed-includes property to
   **every** output of the command, not just the main output — the loop runs
   `for (auto [outNodeRef, _] : outs) addOutputIncludes(outNodeRef)`. So
   `input_proto_masks.pb` (a non-main OutTogether sibling) genuinely carries the
   full `OUTPUT_INCLUDES` set.
2. The RESOURCE objcopy command (`plugins/resource_handler/objcopy.h`) is
   `--inputs ${input:__PATHS}` — the embedded `.pb` is an `${input:…}`, so the
   include processor walks its parsed-includes (the `OUTPUT_INCLUDES`) and their
   transitive closure onto the objcopy node.
3. Our `dump normalize` already drops the **$(S)** half of upstream's objcopy
   input over-emit (`filterObjcopyInputs` / `objcopyOverEmitExts`, refGraph-only),
   keeping every **$(B)** entry. So the normalized ref objcopy lists exactly the
   build-root generated-header closure; the source `OUTPUT_INCLUDES` (util/…,
   library/cpp/resource/resource.h, the `.proto` leaves) are pruned. Our
   generator must therefore emit **only the $(B)** subset.

This is the same closure the sibling CC compile already carries correctly:
`features.gen.cpp.pic.o` matches ref byte-for-byte (514 $(B), incl. the 511
`.pb.h`). The objcopy must reproduce the $(B) subset of the same window.

## Change

### Producer (emit_pr.go, `prEmitsIncludes`)

Currently returns nil for a non-carrying output (`.pb`), so `input_proto_masks.pb`
has no parsed-includes window. Make it register the `OUTPUT_INCLUDES` on **every**
output (faithful to `addOutputIncludes` over all outs). A carrying output
(cc/header/inc) additionally models its `$(S)` IN sources and re-exported proto
`.pb.h` as `#include`s, as today; a data output carries `OUTPUT_INCLUDES` only.
Blast radius: only RUN_PROGRAM non-carrying outputs gain a window, and nothing
walks those windows except the new objcopy consumer below (CC/PB never `#include`
a `.pb`; opaque-IN consumers use `SourceInputs`, not the window).

### Consumer (emit_py_objcopy.go, `emitResourceObjcopy`)

Thread the module's `ModuleCCInputs` (the same `ScanCfg` the CC compile uses, so
the generated `.pb.h` resolve identically) into `emitResourceObjcopy`. When a
RESOURCE path resolves to a registered generated build output, `walkClosureTail`
its window and append the **$(B)** entries to the objcopy node inputs (NOT to the
`--inputs` command payload / hash — `cmds` and the `objcopy_<hash>` name stay
byte-stable). Resolve the producer dep-refs of all data inputs (payload + closure
+ kv) via `resolveCodegenDepRefs`, replacing the hand-rolled `pathDeps` loop
(net-negative).

T-137 guard preserved: a build-root COPY_FILE-staged source is not a RUN_PROGRAM
`OUTPUT_INCLUDES` producer, so its window has no `$(B)` generated-header closure;
nothing new is added to PY_SRCS objcopy inputs.
