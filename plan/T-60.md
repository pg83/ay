# T-60 — inline-proto protobuf/abseil GLOBAL ADDINCL propagation

## Divergence (reproduced)

Representative regular compiles `library/cpp/html/html5/parser.cpp.o` and
`library/cpp/html/spec/...` (and ALL ordinary C++ sources of `library/cpp/html/face`)
miss three `-I` include roots that the reference compile carries at the tail of the
`-I` band:

```
-I$(S)/contrib/libs/protobuf/src
-I$(S)/contrib/restricted/abseil-cpp-tstring
-I$(S)/contrib/restricted/abseil-cpp
```

Confirmed by dumping the paired nodes from `sg7.{our,ref}.norm.jsonl`:
`face/out.cpp.o`, `face/propface.cpp.o`, `html5/parser.cpp.o` — ref has the band,
ours does not; `spec/tagstrings.cpp.o` (no proto, no protobuf provider) has it on
neither side.

## Upstream mechanism

`contrib/libs/protobuf/ya.make` declares `ADDINCL(GLOBAL contrib/libs/protobuf/src)`
and `PEERDIR(contrib/restricted/abseil-cpp-tstring)`; abseil-cpp-tstring peers
abseil-cpp and both carry GLOBAL ADDINCL on themselves. So the protobuf GLOBAL band
is `{protobuf/src, abseil-cpp-tstring, abseil-cpp}`.

The protobuf peer is attached generically: `conf/proto.conf:461`

```
macro _CPP_PROTO_CMD(File) {
    .CMD=...
    .PEERDIR=contrib/libs/protobuf
}
```

`_CPP_PROTO_CMD` runs for every `.proto` compiled to C++ — whether the module is a
`PROTO_LIBRARY` or a plain `LIBRARY()` with an inline `.proto` in `SRCS` (the
`library/cpp/html/face` shape: `propface.proto` + ordinary `.cpp`). The macro's
`.PEERDIR` makes the module peer `contrib/libs/protobuf`, whose GLOBAL ADDINCL then
lands on every compile in the module and propagates transitively to consumers
(`html5` peers `face`).

`library/cpp/html/face` declares no protobuf/abseil peer directly; it obtains the
band purely through `_CPP_PROTO_CMD`'s induced peer.

## Our defect

`modules.go` added the base `contrib/libs/protobuf` peer only when
`d.moduleStmt.Name == tokProtoLibrary`. An inline-proto plain `LIBRARY()` has
`hasProto == true` but is not a `PROTO_LIBRARY`, so it never peered protobuf and
never inherited the GLOBAL band — neither for its own ordinary/generated sources nor
for consumers. (The grpc/inline path reached protobuf only transitively via
`contrib/libs/grpc`; a non-grpc inline proto like `face` had no route.)

## Change

`modules.go` (`collectModule`): broaden the protobuf-peer arm to fire for any
C++ module that compiles a `.proto` (`hasProto && !hasEv`), not only
`PROTO_LIBRARY`. A `PROTO_LIBRARY` keeps the `PY3_PROTO` suppression (python-only
proto runs no C++ compile) and the `optimizePyProtos` default; a plain inline-proto
LIBRARY always compiles the proto to C++, so it peers protobuf unconditionally. No
new data path, no allowlist — one condition broadened to match upstream's
per-source `.PEERDIR`.

## Files

- `modules.go` — broaden the protobuf-peer condition.
- `emit_proto_addincl_test.go` — fail-first coverage.

## Tests

`TestGen_InlineProtoLibrary_ProtobufGlobalAddInclReachesOrdinaryAndConsumer`:
inline-proto `LIBRARY()` with `SRCS(svc.proto use.cpp)`, a consumer peering it, and
an unrelated plain library. Asserts the protobuf/abseil band lands on the ordinary
source (`use.cpp.o`), the generated `svc.pb.cc.o`, and the consumer's `c.cpp.o`, and
is ABSENT from the unrelated module (negative guard).

## Expected gate effect

`library/cpp/html` parser/face regular compiles lose the protobuf/abseil command +
input drift; T-56 gperf outputs need no custom include patch (they ride the same
module band). sg2–sg6 byte-exact, sg5 within budget; sg7 matched up, ref_only down.
Verify with greps for `library/cpp/html`, `protobuf/src`, `abseil` in `sg7.diff.txt`.
