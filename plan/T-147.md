# T-147 plan: PY_SRCS Cython variant-bucket ordering for AR members

## Upstream mechanism (verified, not assumed)

`yatool/build/plugins/pybuild.py:onpy_srcs` parses `PY_SRCS(...)` into FIVE
separate cython lists, selected by directive (lines 301-348):

```python
pyxs_c = []; pyxs_c_h = []; pyxs_c_api_h = []; pyxs_cpp = []; pyxs_cpp_h = []
pyxs = pyxs_cpp                       # default bucket is C++
... 'CYTHON_C' -> pyxs = pyxs_c
    'CYTHON_C_H' -> pyxs = pyxs_c_h
    'CYTHON_C_API_H' -> pyxs = pyxs_c_api_h
    'CYTHON_CPP' -> pyxs = pyxs_cpp
    'CYTHON_CPP_H' -> pyxs = pyxs_cpp_h
    'CYTHONIZE_PY' -> cythonize_py = True   # only flips a flag
```

A `.pyx` source, and a `.py` source while `cythonize_py` is set, are appended to
the *currently selected* `pyxs` list (lines 417-423). After the full arg parse,
the cython emission loop iterates the five lists IN FIXED ORDER and, within each,
in textual append order (lines 480-512):

```python
for pyxs, cython, ... in [
    (pyxs_c, ...), (pyxs_c_h, ...), (pyxs_c_api_h, ...),
    (pyxs_cpp, ...), (pyxs_cpp_h, ...),
]:
    for path, mod in pyxs:
        cython(cython_args)          # emits the generated .c/.cpp compile
        py_register(unit, mod, py3)  # emits the .reg3.cpp registration
```

So BOTH the generated cython compile members (regular archive) and the implicit
`.reg3.cpp` registration members (global archive) are ordered by
`(variant-bucket, within-bucket textual seq)`, NOT by raw PY_SRCS textual order.
`CYTHONIZE_PY` `.py` entries inherit whatever bucket the last directive selected
(default = `pyxs_cpp`). Ordinary `SRCS(...)` members are non-generated (ymake
FIFO round 0) and already precede the generated cython members (round 1) via the
existing `SrcMeta.Generated` round bit, matching gevent's `callbacks.c` first.

Explicit `PY_REGISTER(...)` is a separate macro (`onpy_register`) ordered by
statement position — left untouched.

## Local defect

`modules.go` (`case tokPySrcs`) appends `d.cythonCpp` and the implicit
`d.pyRegister`/`d.pyRegisterExplicit` entries in raw textual order. `gen.go`
gives every cython compile and every reg3.cpp compile the same AR sort key
(`SrcMeta{Prio:2, Generated:true}`), so the stable `reorderARMembers` preserves
that textual emission order instead of upstream's variant-bucket order.

## Change

In `modules.go`, at the END of the `tokPySrcs` case (after cython directives are
applied), stable-reorder the cython statements added by THIS `PY_SRCS` call
(`d.cythonCpp[cythonStmtStart:]`) by bucket, and apply the same permutation to
their paired implicit `pyRegister` entries (tracked via a per-call index slice so
interleaved swig/other registrations stay anchored at their textual positions).
The reorder is stable, so within-bucket textual order is preserved.

Files touched:
- `modules.go`: add `cythonVariantBucket`, capture per-cython register index,
  reorder at end of `tokPySrcs`.
- `emit_cython_cpp_test.go`: fail-first gevent-like fixture (regular + global
  reg3 member order).
- `emit_py_codegen_test.go`: fail-first lxml-like fixture (regular + global
  reg3 member order).

No new data path between modules, no scanner/closure change, no new field on
generated-file info. Emission order (= `d.cythonCpp` order) flows through the
existing `reorderARMembers` stable sort.
