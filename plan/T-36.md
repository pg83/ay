## Deliverable: regression guard for the tool-dep-identity mechanism

Upstream behavior: a RUN_PROGRAM-produced header that runs a built PROGRAM tool
records the tool as a dependency carrying that tool's node identity (its LD
`uid`, which is a Merkle hash over the tool's own source/header link closure), so
the producer re-fires when the tool's binary identity changes. In `ay` this is
`emitRunProgram` → `ctx.toolResult(toolPath)` → `LDRef`, placed on the producer
as `ForeignDepRefs` (emit_pr.go:710-741). A plain source-file argument to the
tool invocation stays a plain `$(S)` input (`runProgramInputVFS`), never promoted
to a built-tool dep.

Test (`emit_pr_tool_identity_test.go`,
`TestGen_RunProgramBuiltToolDepIdentityTracksClosure`):

- a `PROGRAM(genhdr)` tool whose `main.cpp` `#include`s `tool.h` (binary
  identity depends on its own header closure);
- a `LIBRARY` with `RUN_PROGRAM(tools/genhdr template.h.in gen.h IN template.h.in
  OUT gen.h)`;
- assert the `$(B)/gen/gen.h` producer's foreign (tool) deps contain the
  `$(B)/tools/genhdr/genhdr` LD `uid`, and the source argument `template.h.in`
  is a plain producer input, NOT a foreign dep;
- regenerate with a changed `tool.h`; assert the tool LD `uid` changes and the
  producer's tool dep follows it to the new identity, while the source argument
  stays the identical `$(S)/gen/template.h.in` input (source-only arg unchanged).
