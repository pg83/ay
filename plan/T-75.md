# T-75 plan: py-proto include-order command residual

## Upstream mechanism

Upstream renders the protoc include set identically for the C++ and Python proto
commands. Both `_CPP_PROTO_CMDLINE_BASE` (ymake.core.conf:612) and
`_PY_PROTO_CMD_BASE` (proto.conf:516) expand to:

```
$PROTOC -I=./$PROTO_NAMESPACE -I=$ARCADIA_ROOT/$PROTO_NAMESPACE \
        ${pre=-I=:_PROTO__INCLUDE} -I=$ARCADIA_BUILD_ROOT -I=$PROTOBUF_INCLUDE_PATH \
        --{cpp,python}_out=...
```

There is **no standalone `-I=$PROTOBUF_INCLUDE_PATH` before the
`_PROTO__INCLUDE` band**. `protobuf/src` reaches the command in exactly two ways:

1. As a member of `_PROTO__INCLUDE`, contributed by `contrib/libs/protobuf`'s
   `ADDINCL(GLOBAL FOR proto contrib/libs/protobuf/src)`
   (yatool/contrib/libs/protobuf/ya.make:36-39). Every C++-compiling proto module
   transitively peers `contrib/libs/protobuf`, so this band carries `protobuf/src`
   in its **encounter order** relative to the other namespace addincls
   (e.g. a transitive `PROTO_NAMESPACE(yt)` → `-I=$(S)/yt`).
2. As the trailing `-I=$PROTOBUF_INCLUDE_PATH` after `-I=$ARCADIA_BUILD_ROOT`.

Because `_PROTO__INCLUDE` is an ordered, first-encounter-deduped set, the relative
order of `-I=$(S)/yt` and `-I=$(S)/contrib/libs/protobuf/src` inside the band is
**per-module** (whichever peer is encountered first). Raw sg7 reference evidence
(`/home/pg/monorepo/4/sg7.json`) shows both orders across modules:

- `ads/bsyeti/libs/events/proto/adfox` (transitive yt): `… -I=$(S)/yt
  -I=$(S)/contrib/libs/protobuf/src -I=$(S)/contrib/libs/protoc/src …`
  (yt **before** protobuf-src).
- another module: `… -I=$(S)/contrib/libs/protobuf/src -I=$(S)/yt
  -I=$(S)/contrib/libs/protoc/src …` (protobuf-src **before** yt).

Our C++ `composePBArgBlocks` (emit_pb.go) already reproduces this faithfully — the
cpp sibling `$(B)/ads/bsyeti/libs/events/proto/adfox.pb.cc/.pb.h` diffs are empty
against the reference, proving `protoInclude` (our `_PROTO__INCLUDE`) is ordered
correctly and already contains `protobuf/src`.

## The defect

`newPyPBModuleEmission` (emit_py_proto.go:206) appends an **extra** standalone
`argISContribLibsProtobufSrc` (`-I=$(S)/contrib/libs/protobuf/src`) *before* the
`protoInclude` band loop. The band loop then dedupes the band's own `protobuf/src`
(slices.Contains), so `protobuf/src` is pinned to a fixed position **before** every
other band namespace — destroying the per-module encounter order. For modules where
the reference encounters a peer namespace first (e.g. adfox's `yt`), our py command
emits `protobuf-src, yt` where the reference has `yt, protobuf-src`. This is the
sg7 PB `cmds` residual: argument multiset matches, order differs.

The C++ side has no such standalone line; line 206 is a py-only divergence with no
upstream counterpart.

## Fix

Delete emit_py_proto.go:206 (`mid = append(mid, argISContribLibsProtobufSrc.str())`).
The `protoInclude` band then renders `protobuf/src` at its true encounter position,
exactly like the cpp side, and the trailing `-I=$(B) -I=$(S)/contrib/libs/protobuf/src`
pair (line 239-243) is preserved unchanged. Net: one line removed.

## Files touched

  now-stale neighbouring comment context.
- `emit_proto_test.go`:
  - `TestGen_PyProtoLibrary_TransitivePROTONamespaceReachesPyProtoCmd`: invert the
    order assertion to the upstream-correct `yt < protobuf-src < protoc-src <
    python_out`, fix the misleading comment, and make the synthetic
    `contrib/libs/protobuf` faithful by giving it
    `ADDINCL(GLOBAL FOR proto contrib/libs/protobuf/src)` so `protobuf/src` enters
    `_PROTO__INCLUDE` (the band) as it does in the real graph. This test fails on
    current code (line 206 places protobuf-src before yt) and passes after the fix.
  - `assertYtNamespaceDuplicated` consumers
    (`TestGen_PyProtoLibrary_OwnPROTONamespaceDuplicatesNamespaceInclude`,
    `TestGen_PyProtoLibrary_GrpcRootSourceSharesDuplicateInclude`) and
    `TestGen_ProtoLibrary_TransitiveGlobalNamespaceInterleavesInBothCmds`: add the
    same faithful `FOR proto` addincl so `protobuf/src` rides `_PROTO__INCLUDE`;
    without it, removing line 206 would (correctly) drop protobuf-src from the band
    in the unfaithful synthetic FS and break the `protobuf-src after the duplicated
    namespace` assertion.
