Representative `$(B)/ads/peafowl/libs/profiles/proto/alice_music_search_profile.yaff.h`

## Upstream mechanism

`CPP_PROTO_OUTS` (ymake.core.conf `_BASE_UNIT`, build/conf/proto.conf) is a list
accumulated by `SET_APPEND`/`+=` in statement order. The wrapper command is
`--outputs $CPP_PROTO_OUTS -- $PROTOC вА¶`; the node's outputs and the `--outputs`
arg list are exactly this accumulation, with the `${main;вА¶suf=.pb.hвА¶}` output
floated to the front.

Three contributors:
- base `CPP_PROTO_OUTS+=${вА¶suf=.pb.ccвА¶} ${main;вА¶suf=.pb.hвА¶}` (the cpp_out group).
- `when ($PROTOC_TRANSITIVE_HEADERS == "no")`: `CPP_PROTO_OUTS+=${main;вА¶suf=.deps.pb.hвА¶}`.
- `YAFF()` вЖТ `CPP_PROTO_PLUGIN2(yaff вА¶ .yaff.h .yaff.cpp вА¶)` вЖТ
  `_ADD_CPP_PROTO_OUT(.yaff.h)` + `_ADD_CPP_PROTO_OUT(.yaff.cpp)`
  (build/internal/conf/project_specific/yabs.conf, build/conf/proto.conf).

Empirically (sg7 ref, distinct `--outputs` suffix orderings):
- 361 `.pb.h .pb.cc .deps.pb.h .yaff.h .yaff.cpp`  вАФ SET(PROTOC_TRANSITIVE_HEADERS "no") **before** YAFF() (e.g. yabs/server/proto/log).
- 248 `.pb.h .pb.cc .yaff.h .yaff.cpp`             вАФ no lite headers (e.g. grut/libs/proto/enums).
- 11  `.pb.h .yaff.h .yaff.cpp .pb.cc .deps.pb.h`  вАФ YAFF() **before** SET(PROTOC_TRANSITIVE_HEADERS "no") (peafowl, yabs/yacs/libs/proto/yaff, adv_machine).

So the yaff plugin output group precedes the cpp_out group ({.pb.cc} + {.deps.pb.h})
iff the plugin was declared **before** lite headers (PROTOC_TRANSITIVE_HEADERS="no")
were enabled. When lite headers are off, or were already enabled before the
plugin, the plugin group follows the cpp_out group (our current, already-correct
behavior for the 361 + 248 majority).

## Change

1. `modules.go`: `CppProtoPlugin` gains `DeclaredBeforeLiteHeaders bool`,
   captured at parse time for YAFF / YAFF_SCHEMA as
   `protoTransitiveHeadersEnabled(d)` (true вЗТ deps not yet turned on at this
   statement). event2cpp / grpc unaffected.
2. `emit_proto.go` / `emit_pb.go`: thread the flag onto
   `ResolvedCPPProtoPlugin`; in `emitPB`, a plugin whose group must precede the
   cpp_out group (`liteHeaders && DeclaredBeforeLiteHeaders`) has its suffix
   outputs emitted right after `pb.h` and before `pb.cc`/`deps.pb.h`; all other
   plugins keep emitting after the cpp_out + grpc group as today.

The first output stays `pb.h` (the main output), so node identity/pairing is
unchanged. No new cross-module data path; the discriminator is local module
parse state.
