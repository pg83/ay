## Upstream mechanism

ymake `Node2Module` (json_visitor.cpp:638) attributes a generated node to the
module that FIRST leaves it in DFS post-order, and that node inherits the
owning module's directory AND tag.

### sys_const.h — self-owned producer pre-empts an external OUTPUT_INCLUDES claim

`yabs/server/libs/constant/generated/ya.make` is a `LIBRARY()` whose
`RUN_PROGRAM(generate_sys_const … OUT sys_const.h sys_const.cpp)` produces
sys_const.h AND compiles the sibling sys_const.cpp — a self-consuming producer.
The parent `yabs/server/libs/constant/ya.make` names
`yabs/server/libs/constant/generated/sys_const.h` in a *different* RUN_PROGRAM's
`OUTPUT_INCLUDES`. The parent is a consumer; the `generated` submodule is a
PEERDIR dependency, so post-order leaves the producer node inside `generated`
first → `generated` owns it.

Our bug: `emit_pr.go:679 recordNodeClaim` records the parent's OUTPUT_INCLUDES
claim keyed by the producer node ref, and `attribute_generated.go`
unconditionally lets `generatedNodeClaim` win over the producer-self-claim that
`markGeneratedProducerOwned` stored in `generatedFirstClaim`. The node-level
claim must NOT pre-empt a self-owned producer.

Discriminator: a self-owned producer has `generatedFirstClaim[out] == current`
(its own module is the first claimer — recorded at registration, first-write
wins) for its outputs. When that holds, ignore the external `generatedNodeClaim`.
The `with_transitive_headers` wrapper (the case the node-claim exists for) is
NOT self-consuming, so its outputs' first-claim is a far peer ≠ current; the
node-claim still applies there.

### orc-format SBOM — the CPP_PROTO submodule owns the component

A `PROTO_LIBRARY()` with `LICENSE()` and no `EXCLUDE_TAGS(CPP_PROTO)` builds a
CPP_PROTO submodule, the only proto submodule that keeps `_NEED_SBOM_INFO=yes`
(`sbomQualifies` already returns false otherwise). That submodule carries
`MODULE_TAG=CPP_PROTO`, so its `_GEN_SBOM_COMPONENT` output is tagged cpp_proto.
`emitSbomComponent` currently sets `module_tag` only for the py multimodule
variants; add the PROTO_LIBRARY → cpp_proto case.

### any.cow.pb.h — NOT fixed in this ticket

`apphost/gp/lib/proto/cow/generator/well_known` is a `LIBRARY()` whose
`RUN_PROGRAM` OUTs `*.cow.pb.h` (header-only, NOT self-consuming). ref
attributes the header to the consuming `apphost/lib/proto_answers` CPP_PROTO
submodule (module_dir + cpp_proto tag). Our graph records no first-claim for it
because the consumer reaches it through the cow-plugin output-include / closure
window, not a scanned `#include`, so the PR override never fires. Correcting
this needs a claim recorded along the plugin-output consumption path AND
tag propagation from the claiming module — a larger change than the two
attribution fixes above and out of this ticket's safe scope. Documented for the
lead; this ticket leaves the node unchanged (no regression).
