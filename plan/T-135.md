# T-135 plan: serialized-enum sibling + runtime input propagation

## Upstream mechanism

`GENERATE_ENUM_SERIALIZATION_WITH_HEADER(File)` runs `enum_parser` over `File`,
emitting `File_serialized.{cpp,h}` whose includes are resolved exactly like any
other generated header. When a module declares several such macros, one
declaration's enum header can `#include` another declaration's generated
`*_serialized.h` through an ordinary header chain. Upstream resolves those
generated headers transitively, so the EN producer and every ordinary CC
consumer that reaches the chain carry the sibling's `_serialized.{cpp,h}` plus
the enum-runtime closure (the enum_parser `INDUCED_DEPS`).

Representative (balancer/kernel/cookie):
- `set_cookie.h` (declared first) includes `cookie_errors.h`, which includes
  `<.../set_cookie_errors.h_serialized.h>` (declared second).
- ref EN `set_cookie.h_serialized.cpp` and ordinary CC `set_cookie.cpp.o` carry
  `set_cookie_errors.h_serialized.{cpp,h}` and the enum-runtime headers.

## Defect

`emitEnumSrcs` registered each declaration's serialized outputs and walked its
closure in a single per-statement loop. The scanner caches a file's resolved
children by file id (shared module-wide). Processing the FIRST declaration walked
the including header (`cookie_errors.h`) before the SECOND declaration's outputs
were registered, so the sibling `<*_serialized.h>` resolved to nothing and that
empty resolution was cached — poisoning both the EN producer's inputs and the
later ordinary CC compile of `set_cookie.cpp`.

## Fix

Split `emitEnumSrcs` into two passes (mirrors the .fbs / bison / proto
register-all-then-compile pre-passes in `gen.go`):
1. Pass 1: for every declaration resolve the header, reserve the EN ref, and
   register all `_serialized.{cpp,h}` outputs + parsed includes.
2. Pass 2: walk each header's closure (now seeing every sibling registered) and
   emit the EN node and downstream CC.

This keeps include resolution context-free (no per-context resolution); it only
reorders registration before any closure walk within the module. The
enum-runtime headers ride in automatically through the sibling outputs' windows
(enum_parser `INDUCED_DEPS` via `GeneratorRefs`). No ownership / module_dir /
archive-order change.

## Files

- `emit_en.go`: two-pass restructure of `emitEnumSrcs`.
- `emit_en_test.go`: fail-first `TestGen_EnumSerializationSiblingSerializedOutputsPropagate`.

## Expected gate effect

- EN `inputs` drift for this bucket: 1 -> 0.
- CC serialized-enum sibling `inputs` rows disappear (cookie representatives
  print only headers).
- sg7 `matched` does not decrease; sg2..sg6 byte-exact; sg5 under budget.
- T-122 archive-order probes unchanged; AR/LD input filtering intact.
