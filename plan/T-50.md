# T-50 plan: enum_serialization_runtime archive position is statement-position peer closure, not a PY3_PROGRAM relocation

## Hypothesis (reproduced)

For the representative linked program
`$(B)/yabs/.../codegen_bin/fs_codegen` the regular `-Wl,--start-group` archive
list places `tools/enum_parser/enum_serialization_runtime/...a` at index 50,
while the reference places it at index 39 — immediately after
`ads/bsyeti/quota_accounts/...a` (38) and before
`ads/bsyeti/libs/primitives/counter_proto/...a` (40). Whole-archive list and the
rest of the regular list are already byte-identical; the only difference is this
single shifted element (diff window 39..50, one element moved).

## Upstream mechanism

`GENERATE_ENUM_SERIALIZATION*` expands inline to
`PEERDIR(tools/enum_parser/enum_serialization_runtime)` at its textual position
(ymake.core.conf). `counter_proto`'s effective peer order is therefore
`[ads/bsyeti/quota_accounts, tools/enum_parser/enum_serialization_runtime]`.
ymake's `module_restorer.cpp` builds `UniqPeers` by merging each peer's
`UniqPeers` before adding the peer itself, so any downstream link reaching
`counter_proto` lists `quota_accounts` closure, then
`enum_serialization_runtime`, then `counter_proto`'s own archive. The position of
`enum_serialization_runtime` is determined entirely by the peer closure order; it
is not a fixed slot.

Our code already reproduces this: `modules.go` appends the enum runtime peer to
`d.peerdirs` at the `GenerateEnumSerializationStmt` position, and
`walkPeersForGlobalAddIncl` puts it correctly inside `counter_proto`'s
`PeerArchiveClosurePaths` (verified by instrumentation: quota_accounts[14],
enum_serialization_runtime[15], counter_proto.a[16]). The pre-relocation
`peerLinkCmdPaths` for fs_codegen already lists enum runtime right after
quota_accounts and before counter_proto.

## Defect

`gen.go` (program LD path, `tokPy3Program && allocatorName == strJ`) applies a
hardcoded relocation: `movePathsBefore(... enum_serialization_runtime ...,
library/cpp/json/common)` (and the archive-ref twin `moveArchivePathsBefore`).
This was added by T27 to force enum runtime immediately before `json/common` for
`devtools/ya/bin` (ya-bin). After the later statement-position-peer fix in
`modules.go`, the natural closure order is already correct for ya-bin AND for
fs_codegen — but the relocation now *moves enum runtime out of its correct slot*
in fs_codegen (39 → 50, before json/common), creating the divergence. It is a
path-specific reordering with no upstream counterpart; the generic mechanism
already in place subsumes it.

## Change

- Remove the enum-runtime `moveArchivePathsBefore` / `movePathsBefore` relocation
  block from the `tokPy3Program && strJ` branch in `gen.go`. Keep the malloc
  (`build/cow/on` anchor) relocation, which is a separate, still-needed ordering.
- Remove the now-unused `moveArchivePathsBefore` and `movePathsBefore` helpers
  (no other callers) — net code shrink.

## Test-first regression

Add a focused test: a `PY3_PROGRAM` peering a `PROTO_LIBRARY` that has an
ordinary `PEERDIR` before `GENERATE_ENUM_SERIALIZATION`. Assert the program LD
regular start-group archive sequence lists the ordinary peer, then
`enum_serialization_runtime`, then the proto library's own archive, each exactly
once. This must fail with the relocation present (enum runtime shifted to before
json/common) and pass after removal.

## Expected gate effect

- sg7: fs_codegen pair diff goes empty (enum-runtime/counter_proto window closes);
  matched does not regress.
- sg2..sg6 stay byte-exact (ya-bin's enum runtime position is unchanged because
  the natural closure already places it immediately before json/common there).
