# T-139 plan: enum-serialization AR order by generated-`.pb.h` provenance

## Upstream mechanism

ymake processes a module's statements in `(StatementPriority, name)` order
(`module_loader.cpp:38`); lower priority first. `GENERATE_ENUM_SERIALIZATION`
is a default-priority (2) macro; `SRCS` is priority 4. Every non-temp command
output is re-queued onto the module source list at its declaring statement's
processing point (`module_builder.cpp` `QueueCommandOutputs()` →
`AddSource("SRCS", output)`), and the queued generated compiles run in the next
FIFO round in queue order. So among first-level generated members, a member
queued during a prio-2 statement archives before one queued during a prio-4
statement.

Two consequences for an enum serialization `File_serialized.cpp`:

- If its input header `File` is NOT produced by this module (a checked-in
  header, or a build-rooted `.pb.h` produced by a *peer* module), the EN compile
  is a first-level generated compile (round 1), queued at the EN statement's
  prio-2 band. It therefore archives **before** this module's proto `.pb.cc.o`
  objects (round 1, queued in the prio-4 SRCS band).
- If its input header `File` IS a `.pb.h` produced by one of this module's own
  proto/ev SRCS, the EN cannot run until the proto command produced the header.
  Its compile is therefore deferred one further FIFO round (round 2 /
  `SecondLevel`), and archives **after** every round-1 member, i.e. after the
  `.pb.cc.o` that produces the header.

This is the same generation-round model T-122 introduced (`SrcMeta.SecondLevel`
for a flatc `.fbs.cpp` produced from a RUN_PROGRAM-emitted `.fbs`). An EN whose
input header is itself an in-module generated source is the proto analogue.

The provenance signal is exactly membership in this module's own proto/ev source
set: `OffersBatch.pb.h` ← `OffersBatch.proto ∈ SRCS` (same module → round 2);
`ExportMessage.pb.h` ← `ExportMessage.proto ∉ SRCS` (built by peer
`market/idx/datacamp/proto/external` → round 1). The global codegen registry
cannot make this distinction (it holds peer outputs too), so the classification
keys on the module's own proto/ev SRCS.

## Files touched

- `emit_en.go`: classify each EN statement's header against this module's own
  proto/ev SRCS (`<base>.proto`→`<base>.pb.h`, `<base>.ev`→`<base>.ev.pb.h`).
  Carry the result in a new `EnumSrcsResult.SecondLevel []bool` parallel to
  `Seqs`.
- `gen.go` (LIBRARY path, ~2310): consume `enCCRes.SecondLevel[i]` in the
  `SrcMeta` for each EN member (instead of always first-level).
- `emit_proto.go` (`emitCPPProtoSrcs`): give every archive member an
  `arDeclMeta` `SrcMeta` key and `reorderARMembers` the final member list:
  proto/ev codegen = round 1, prio `stmtPrioSrcs`, seq = declIdx (preserves the
  T-131 SRCS interleave and gzt-at-tail); RUN_ANTLR `.cpp` = round 1, prio
  `stmtPrioDefault`; EN = round 1 prio `stmtPrioDefault` (external/source) or
  round 2 (same-module generated). This merges the EN members into the same
  ordering model instead of blind tail-append.

## Tests (fail-first)

- `emit_en_test.go`: `LIBRARY(); SRCS(data.proto); GENERATE_ENUM_SERIALIZATION(data.pb.h)`
  — assert `data.pb.cc.o` precedes `data.pb.h_serialized.cpp.o`. Fails now
  (EN keyed round 1, ties/precedes the proto pb.cc).
- `emit_proto_test.go`: `PROTO_LIBRARY()` with a peer that produces
  `external.pb.h`, `SRCS(own.proto tail.proto)`, EN over the external build-root
  header and over the same-module `own.pb.h`. Assert order
  `external.pb.h_serialized.cpp.o < own.pb.cc.o < tail.pb.cc.o <
  own.pb.h_serialized.cpp.o`. Fails now (both EN appended at tail).
