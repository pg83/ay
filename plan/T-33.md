## Symptom (reproduced)

```

```

## Upstream mechanism (verified, not assumed)

1. The YaFF protoc plugin (`library/cpp/yaff/tools/protoc_plugin/proto_plugin.cpp`,
   `Generate`) writes the `.yaff.cpp` with a single line:
   `#include "<stem>.yaff.h"` â€” by BASENAME (`StripPath(headerPath)`).
   So the `.yaff.cpp` translation unit's only direct include is the sibling
   header. The header itself is generated by the same protoc invocation.

2. `wire_format.h` is NOT reachable by include-scan from that closure: no header
   in the closure `#include`s it (only comments mention it). It arrives via the
   protoc TOOL's induced deps. `contrib/tools/protoc/ya.make.induced_deps`:

   ```
   INDUCED_DEPS(cpp   ... google/protobuf/wire_format.h ...)   # .cc/.cpp only
   INDUCED_DEPS(h+cpp ... message.h, arena.h, ... )            # both .h and .cc
   ```

   `wire_format.h` is in the `cpp`-only bucket, so it is induced onto generated
   `.cc`/`.cpp` protoc outputs but NOT onto generated headers. This is exactly
   why the regular `.pb.cc.o` carries `wire_format.h` while `.pb.h` does not, and
   why the `.yaff.cpp.o` should carry it too.

3. The sibling `.yaff.h` is absent from the upstream `.yaff.cpp.o` inputs (raw
   ref node confirmed). Upstream resolves the basename self-include for its
   transitive closure (all the `.pb.h`, yaff runtime, libcxx ride in) but does
   not record the sibling header itself as a compile input. This is the same
   shape the codebase already encodes for `.ev.pb.cc` (drop the self header from
   the walked closure).

## Current code (the defect)

`emit_proto.go`:

- `yaffCC` is registered with `GeneratorRefs == nil` (line ~441), denying it
  protoc's `INDUCED_DEPS(cpp â€¦)` â†’ no `wire_format.h`. The comment claiming the
  plugin "declares no INDUCED_DEPS" overlooks that the relevant induced deps come
  from the protoc TOOL (which produces every `.yaff.*` output), exactly as the
  `.pb.cc`/`.pb.h` registrations already use `pbGenRefs`.
- `yaffCCParsed = ["<stem>.yaff.h"]` (quoted). The `.yaff.cpp.o` compile walks
  this and records the sibling `.yaff.h` as an input.

`.yaff.cpp` is compiled in the `codegenOutputs` loop (it is appended as
`extraSourceCC`), so the same post-walk filtering hook used for `.ev.pb.cc`
applies.

## Fix

1. Register `yaffCC` with `pbGenRefs` (instead of `nil`) so protoc's
   `INDUCED_DEPS(cpp â€¦)` resolve onto the `.yaff.cpp` output during the include
   walk â†’ `wire_format.h` enters the `.yaff.cpp.o` closure via the existing
   `scanner.resolveInducedDeps` path. (`yaffH` registration stays `nil` â€” header
   bucket would only re-add already-present `h+cpp` headers; leaving it untouched

   `.yaff.cpp` translation unit's `IncludeInputs` â€” mirroring the existing
   `.ev.pb.cc` self-header removal. The closure (`.pb.h`, yaff runtime, libcxx)
   was already walked through the header and survives; only the header node is
   removed. `wire_format.h` rides in from step 1, not a hand-woven append.
