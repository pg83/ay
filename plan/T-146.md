# T-146 plan: FROM_SANDBOX OUT_NOAUTO main-output spurious input on RESOURCE_FILES objcopy

## Upstream mechanism

A ymake build command with several outputs has exactly one *main* output —
`FindMainElemOrDefault(GetOutput(), 0)` (macro_processor.cpp:1041): the first
output unless one is explicitly flagged `Main`. The main-output node carries the
build command; every additional output is linked to it with `EDT_OutTogether`
(macro_processor.cpp:1194-1195), and `MainOutputId[out] = mainOutId` records the
mapping for all of them.

When another command consumes an *additional* (non-main) output, ymake records
the dependency against the **main-output node** (because the build command lives
there) and stores it in `NodeDeps` together with the node's real file inputs.
`TJSONVisitorNew` then cannot distinguish that main-output node-dep from a real
input, so it lists the main output's name in the consumer's `inputs` section even
though the file is absent from the command line — see the explicit comment at
`devtools/ymake/json_visitor.cpp:999-1023` ("the main output file name will be
listed in the 'inputs' section of JSON node, even though this file is not present
in the command arguments and is not really used").

For `FROM_SANDBOX(12019934890 OUT_NOAUTO ${VAR})` the fetch is one SB command
with all dict files as outputs; the main output is the first listed file,
`plutonium_dicts/2.dict`. `RESOURCE_FILES(${VAR})` packs the dicts into objcopy
chunks (≈14 files/chunk by `maxCmdLen`). Every chunk depends on the single SB
node, so ymake lists `$(B)/<mod>/plutonium_dicts/2.dict` in every chunk's inputs.
The chunk that already embeds 2.dict (first chunk) has it via its payload; the
other three chunks per module list it *only* through this main-output edge — the
6 sg7 residual rows (2 modules × 3 non-first chunks).

Our objcopy emission resolves each embedded dict to its `$(B)` output and adds
the SB producer dep, but never adds the producer's main output as the spurious
input. Hence ours omits `2.dict` on the non-first chunks.

## Files to touch

1. `codegen_registry.go` — add `GeneratedFileInfo.ProducerMainOut VFS` (the
   producing node's main/first output) plus a `setProducerMainOut` setter,
   mirroring `setSourceInputs`.
2. `emit_from_sandbox.go` — record `outVFSs[0]` (the SB node's main output) on
   every registered FROM_SANDBOX output.
3. `emit_py_objcopy.go` — `resolveResourceInput` also returns the producer's main
   output; `emitResourceObjcopy.flush` adds, per chunk, each distinct producer
   main output not already present among the chunk's inputs (deduped). This does
   NOT feed `objcopyHash` (which hashes only paths/keys/kvs), so resource names,
   chunking, commands, output paths and hashes are unchanged.

The rule is generic: for every producer a chunk depends on, ensure the
producer's recorded main output is among its inputs. Single-output producers
have main == referenced output → deduped, no change. Only emitters that record a
main output (FROM_SANDBOX here) participate, so feature_store / sprav buckets
(other tickets) are untouched.

## Expected gate effect

- The 6 `yabs/plutonium/libs/dictionaries_resource{,_for_workers}` objcopy roots
  lose their `inputs` diff (ref-only `+$(B)/.../plutonium_dicts/2.dict`).
- PY `inputs` paired divergence shrinks by those rows; sg7 matched does not
  decrease (T-141 baseline matched=67618).
- No objcopy hash / command / output-name change; no AR/LD input drift; sg2–sg6
  byte-exact; sg5 under budget.
