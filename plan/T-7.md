# T-7 Plan: Fix ONE_LEVEL ADDINCL include ordering (rework)

## Upstream mechanism (ymake)

In `addincls.cpp`, `TModuleIncDirs::TLanguageIncDirs`:
- `Add(dir, EIncDirScope::User)` (ONE_LEVEL) → puts dir in `LocalUserGlobal` AND `UserGlobal`
- `Add(dir, EIncDirScope::Global)` (GLOBAL) → puts dir in `LocalUserGlobal`, `UserGlobal`, AND `GlobalPropagated`

`PropagateTo(other)` when module M peerdirs P:
```cpp
propagate_dirs(P.UserGlobal, M.UserGlobalPropagated);       // P's own ONE_LEVEL+GLOBAL in declaration order → M
propagate_dirs(P.GlobalPropagated, M.UserGlobalPropagated); // P's transitive GLOBAL → M
```

`Get()` returns: `LocalUserGlobal` (own) + `UserGlobalPropagated` (peer-propagated).

**Key invariant**: `UserGlobal` preserves **declaration order** — for `ADDINCL(GLOBAL a ONE_LEVEL b)`,
`UserGlobal = [a, b]`, not `[b, a]`. The prior implementation always emitted all ONE_LEVEL paths
before all GLOBAL paths (wrong for mixed declarations).

## Prior rework verdicts

Reviewer finding (R1): the first implementation used separate `AddInclOneLevel` and `AddInclGlobal`
bags and emitted all ONE_LEVEL before all GLOBAL for each user peer. For `ADDINCL(GLOBAL a ONE_LEVEL b)`,
this produced `b, a` instead of upstream `a, b`.

Arbiter ruling (R1): preserve each direct peer's mixed GLOBAL/ONE_LEVEL in declaration order before
transitive globals. Added `UserGlobalPaths` field to track GLOBAL+ONE_LEVEL in declaration order.

Reviewer finding (R2): `addInclUserGlobal` was only populated from explicit `ADDINCL(GLOBAL/ONE_LEVEL)`
statements. Implicit own-global include sources — generated-header build dirs (`SRCS(*.h.in)`),
Python includes, build-info dirs, `PROTO_NAMESPACE` — still went only into `addInclGlobal`, after
the emission of `AddInclUserGlobal`. A peer with `SRCS(gen.h.in)` then `ADDINCL(ONE_LEVEL mydir)`
would emit ONE_LEVEL before the generated-header build dir, violating upstream declaration order.

Arbiter ruling (R2): widen `AddInclUserGlobal` to cover every own-global include source.

## Fix (R2 pass)

Every site that appends to `d.addInclGlobal` for own (non-transitive) includes must also append
to `d.addInclUserGlobal`, so the emission order matches upstream's UserGlobal accumulation order:

### modules.go: implicit own-global sites

1. `addGeneratedHeaderInclude` — `SRCS(*.h.in)` → generated-header build dir
2. `addGeneratedHeaderIncludeCF` — CF-variant generated header → same build dir logic
3. `applyPython3AddIncl` — Python include injection → `$(S)/contrib/libs/python/Include`
4. `applyBuildInfoAddIncl` — build-info dir → `$(B)/<modulePath>`
5. `applyUnknownStmt` case `PROTO_NAMESPACE` — proto namespace build root (when GLOBAL or PROTO_LIBRARY)

`filterInvalidAddIncl` already handles all new entries correctly:
- Generated-header build dirs (Build paths) are never filtered by `filterExistingSourceDirs` (only
  source paths are checked), so they'll always be in `validGlobal` after filter → survive rebuild.
- Python/BuildInfo are added after `filterInvalidAddIncl` runs (in `collectModule`) and are appended
  to `addInclUserGlobal` post-filter; no rebuild collision.
- PROTO_NAMESPACE (inline in collectStmts) also appends to `addInclGlobal`, so survives as `validGlobal`.
