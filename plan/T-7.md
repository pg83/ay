# T-7 — BUNDLE-produced file consumed as a RESOURCE input

## Upstream mechanism (verified against /home/pg/monorepo/4)

`BUNDLE(<Dir> [SUFFIX <s>] [NAME <n>])` is a Python plugin
(`build/plugins/bundle.py`): it splits each `<Dir> [SUFFIX s] [NAME n]` group
and calls `unit.on_bundle_target([target, name, suffix])`, where `name`
defaults to `basename(target)+suffix`. `on_bundle_target` runs
`_BUNDLE_TARGET` (`build/ymake.core.conf:2883`):

    .CMD=$MOVE_FILE ${result:Target}$Suffix ${noauto;output:Destination} \
         ${hide;kv:"p BN"} ${hide;kv:"pc light-cyan"} …

with `MOVE_FILE = $FS_TOOLS rename` and
`FS_TOOLS = $YMAKE_PYTHON3 ${input:fs_tools.py} ${hide;input:process_command_files.py}`.

So a BUNDLE declaration emits one **BN** node per group:
- output `$(B)/<module>/<name>` (a `noauto` build output),
- cmd `python3 build/scripts/fs_tools.py rename ${result:Target} $(B)/<module>/<name>`,
- inputs: `${result:Target}` (the bundled module's primary output) + fs_tools.py
  + process_command_files.py,
- a dep on the node producing `${result:Target}`,
- `kv p=BN pc=light-cyan`, `target_properties.module_dir=<module>`.

A later `RESOURCE(<name> <key>)` in the SAME module embeds `<name>`. Upstream
resolves `<name>` to the BN build output `$(B)/<module>/<name>` (not
`$(S)/<module>/<name>`) and takes the BN node as the resource producer.

Reference (sg7, `grut/libs/misc/protobuf`):
`BUNDLE(grut/libs/proto/objects/descriptions NAME proto.desc)` +
`RESOURCE(proto.desc …)` → BN node renaming
`$(B)/grut/libs/proto/objects/descriptions/proto-objects-descriptions.protodesc`
(the bundled PROTO_DESCRIPTIONS module's primary output) →
`$(B)/grut/libs/misc/protobuf/proto.desc`; the resource objcopy lists the
`$(B)` proto.desc and depends on the BN node.

## Divergence (reproduced)

`BUNDLE` is currently an acknowledged no-op (`acknowledgedMacros`), so no BN
node is emitted and `$(B)/<mod>/<name>` is never registered. The resource
objcopy's codegen-registry probe
(`emit_py_objcopy.go` `emitEntries`) misses, falls back to
`copyFileInputVFS` → `source(<mod>/proto.desc)`, and sandboxing content-hashing
faults: `open grut/libs/misc/protobuf/proto.desc: no such file`.

`./ay make … --sandboxing --keep-going --source-root /home/pg/monorepo/4
yabs/server/daemons/bs_static` → stderr contains exactly that line.

## Scope boundary (non-authoritative file hint)

The sg7 bundled target `grut/libs/proto/objects/descriptions` is a
`PROTO_DESCRIPTIONS()` module — NOT a modeled module type (it has no
`*ModuleStmt` opener; its primary output name `proto-objects-descriptions.protodesc`
derives from REALPRJNAME conventions that require modeling the
PROTO_DESCRIPTIONS module). For such an unmodeled bundled target, `emitBundles`
still emits the BN node and registers the `$(B)/<mod>/<name>` output (which
removes the proto.desc-as-source fault and wires the consumer to the producer
dep), but cannot attach the faithful rename SOURCE / producer dep. That piece —
modeling the PROTO_DESCRIPTIONS bundled target so `${result:Target}` resolves —
is reported via `replan` as the next hard blocker, NOT broadened into this
ticket.

> File hint, non-authoritative: bundle producer = `build/ymake.core.conf:2883`
> `_BUNDLE_TARGET`; plugin = `build/plugins/bundle.py`; consumer probe already
> in `emit_py_objcopy.go`. Confirm against the live tree.
