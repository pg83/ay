# T-30 plan: model EXPORT_YMAPS_PROTO's GLOBAL PROTO_ADDINCL(maps/doc/proto) into the C++ proto include closure

## Upstream mechanism

`build/internal/conf/project_specific/maps/sproto.conf`:

```
macro EXPORT_YMAPS_PROTO() {
    PROTO_NAMESPACE(maps/doc/proto)
    ...
}
```

`build/conf/proto.conf`:

```
macro PROTO_NAMESPACE(GLOBAL?"GLOBAL":"", WITH_GEN?"WITH_GEN":"", Namespace) {
    SET(PROTO_NAMESPACE $Namespace)
    PROTO_ADDINCL(GLOBAL $WITH_GEN $Namespace)   # literal GLOBAL — always
}
macro PROTO_ADDINCL(GLOBAL?"GLOBAL":"", Path, WITH_GEN?"BUILD":"") {
    _ORDER_ADDINCL($WITH_GEN $GLOBAL FOR proto $(B)/$Path SOURCE $GLOBAL FOR proto $(S)/$Path)
    ADDINCL($GLOBAL $(B)/$Path)
}
```

`EXPORT_YMAPS_PROTO()` therefore always emits a **GLOBAL FOR proto** addincl whose
SOURCE arm is `$(S)/maps/doc/proto`. That GLOBAL proto addincl rides the
CPP_PROTO peer closure (`_PROTO__INCLUDE`) and renders as `-I=$(S)/maps/doc/proto`
via `${pre=-I=:_PROTO__INCLUDE}` in `_CPP_PROTO_CMDLINE_BASE` in every transitive
C++ PROTO_LIBRARY consumer's protoc command (and, post T-16, the shared EV path).

The maps/doc/proto C++ proto modules (e.g. `maps/doc/proto/yandex/maps/proto/*`,
each `PROTO_LIBRARY()` + `EXPORT_YMAPS_PROTO()`) are reached transitively by
`extsearch/goods/plutometa/proto` through its real PEERDIR/import closure, so its
`host_info.pb.cc` / `events.ev.pb.cc` protoc commands carry `-I=$(S)/maps/doc/proto`
in the reference — last in the peer include block (after yt/googleapis/taxi/market).

## Our defect

`EXPORT_YMAPS_PROTO` sits in `acknowledgedMacros` (gen.go) as a no-op, so the maps
proto modules contribute nothing. T-9 propagates PROTO_NAMESPACE GLOBAL; T-16
reuses it for EV. Neither sees `maps/doc/proto` because the macro that produces it
is silently dropped. Result: the sole remaining reference-only include token on
both `host_info.pb.cc` (PB) and `events.ev.pb.cc` (EV) is `-I=$(S)/maps/doc/proto`.

## Change (narrow, scoped per ticket — include only, no output-path remodel)

Model the include-propagation half of `EXPORT_YMAPS_PROTO` only: contribute the
GLOBAL proto SOURCE-arm addincl `maps/doc/proto` into the existing
`d.protoAddInclGlobal` channel — the exact carrier our `GLOBAL FOR proto X`
parsing already uses (yamake.go `splitAddInclPaths`, modules.go:1140). This is the
same chain T-9/T-16 already render; ordering falls out of the existing peer walk.

Deliberately NOT modeled here (out of scope, would be "broad attempts to zero the
whole sg7 proto diff"): the `SET(PROTO_NAMESPACE)` half (which moves the maps
modules' own outputs to `$(B)/maps/doc/proto`) and the `ADDINCL(GLOBAL $(B)/...)`
C++ build-root addincl (would be C++ `-I` leakage, which the ticket forbids).
Appending only to `protoAddInclGlobal` adds nothing to any C++ addincl set.
