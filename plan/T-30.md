## Upstream mechanism (build/conf/proto.conf, build/internal/conf/sbom.conf)

- `module _DESC_PROTO: _BARE_UNIT { ... DISABLE(_NEED_SBOM_INFO) ... }`
  (proto.conf:869) — the DESC_PROTO submodule turns SBOM info OFF and is a
  Library, so it neither generates nor materializes any `.component.sbom`.
- `module PROTO_DESCRIPTIONS: _BARE_UNIT { .FINAL_TARGET=yes ... }`
  (proto.conf:976) does **not** disable `_NEED_SBOM_INFO`. On linux/x86_64 with
  the internal contour `_NEED_SBOM_INFO=yes` (sbom.conf:4).
- Every `_BARE_UNIT` carries `PEERDIR+=$YMAKE_PYTHON3_PEERDIR`
  (ymake.core.conf:574, python.conf). `build/platform/python/ymake_python3` is a
  `RESOURCES_LIBRARY` with `TOOLCHAIN(python3)`; under SBOM it emits the global
  `toolchain.component.sbom` (`_GEN_SBOM_TOOLCHAIN_COMPONENT`, sbom.conf).
- A FINAL_TARGET materializes its peer-closure `SRCS_GLOBAL` as node inputs+deps.
  `_PROTO_DESC_MERGE_PEERS_CMD` renders only the `.protodesc` subset
  (`${ext=.protodesc:SRCS_GLOBAL}`), but the universal python toolchain SBOM
  global also rides in as an input+dep — it just is not referenced in the
  command text.

This is the same "universal python toolchain peer contribution" that
`pythonToolchainSbomComponent` already models for bare PREBUILT_PROGRAM
(resources.go) and the LD closure (gen.go). DESC_PROTO submodules stay untouched
(they DISABLE the feature), so PR/PB/CC/YAFF nodes are unaffected.

## Change

`emit_proto_desc.go`, `emitProtoDescriptions` only: after assembling the
`.self.protodesc` peer inputs/deps, resolve `pythonToolchainSbomComponent(ctx,
instance.Platform)`. When non-nil (sbomActive: sbomEnabled && linux/x86_64),
append its path to `inputs` and its ref to `deps`. Gate is sbomActive — the
helper returns nil otherwise, so non-SBOM contours (sg2–sg5) and AArch64 are
unaffected. No allowlist; the relationship is producer(PROTO_DESCRIPTIONS) ↔
universal python toolchain.
