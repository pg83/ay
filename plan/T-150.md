# T-150 plan: generated C/C++ source carries its same-RUN_PROGRAM header sibling's include closure

## Divergence

sg7 CC `inputs` residual on:

- `$(B)/yabs/server/libs/constant/generated/sys_const.cpp.o`
- `$(B)/yabs/server/libs/constant/generated/sys_const.cpp.pic.o`

Reference lists the generated `const_options.pb.h` proto closure (1292 ref-only
inputs); ours does not.

## Upstream mechanism

`yabs/server/libs/constant/generated/ya.make`:

```
RUN_PROGRAM(generate_sys_const sys_const.proto $(B)/.../sys_const.h $(B)/.../sys_const.cpp
    IN  sys_const.proto
    OUT sys_const.h sys_const.cpp)
```

The generator (`generate_sys_const/main.cpp`) writes `#include "sys_const.h"` at
the top of the generated `sys_const.cpp`. Upstream scans the generated source at
build time, resolves that quoted include to the same-producer generated header
`sys_const.h`, and from there reaches `sys_const.h`'s own `#include
"…/const_options.pb.h"` and the full generated proto closure. So the generated
`.cpp`'s compile inputs include the header sibling's induced proto-header closure
purely because the `.cpp` physically includes the header.

We never scan generated bodies; we model parsed includes. We already model the
header sibling (`sys_const.h`) carrying its proto imports' `.pb.h` via
`protoImportPbH` / `carryProtoImportPbH` (T-83). The gap: the generated `.cpp`
output's modeled parsed-include window does NOT include the header sibling, so
`walkClosure` over the `.cpp` never transits into the header's window and misses
`const_options.pb.h` and its closure.

The current `addClosureLeaf(out, mainOutputVFS)` rides the main output (the
header, here) onto the `.cpp` consumer as a NON-expanded bare leaf — enough to
list `sys_const.h` itself, but not to expand its window.

## Change

In `emitRunProgram` (`emit_pr.go`):

1. Collect the run's generated header outputs (`isHeaderSource`) among the auto
   outputs (OUT, auto STDOUT) into a set of quoted include directives keyed by
   each header's `.rel()` (the same rel-keyed form `emit_proto.go` uses; a
   `$(B)` includer resolves it through the codegen registry's rel key).
2. When registering a cc-source output (`isCCSourceExt`), append those
   header-sibling include directives to its parsed-include window. `walkClosure`
   then expands the header window (which already carries the proto-import `.pb.h`
   closure) into the generated `.cpp`'s compile inputs.
3. Avoid the double edge: when the cc-source output now includes the main output
   (a header) as a parsed include, skip the non-expanded
   `addClosureLeaf(ccOut, mainOutputVFS)` for that output — the header rides
   (expanded) via the parsed include instead. Keep the closure leaf for every
   other non-main output and for the case where the main output is not a header.

Restriction (per ticket): only cc-source outputs gain the header-sibling edge,
and only when the same producer also emits a header. Non-cc/asm outputs and runs
without a header sibling are untouched.
