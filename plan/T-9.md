# T-9 plan: correct PIC vs non-PIC archive member contour in the diff comparison

## Symptom and the upstream-vs-ours reproduction

Ticket T-9 (from T-1 recon cause #3) reports that the representative archive
pair

    $(B)/library/cpp/yaff/experiments/codecs/libyaff-experiments-codecs.a

shows `bitset.cpp.o` (ours) vs `bitset.cpp.pic.o` (ref) in `cmds`/`inputs`, i.e.
"upstream archives PIC objects but our graph archives non-PIC objects".

### What the graphs actually contain

Generating sg7 from current trunk and inspecting the raw nodes (not the diff
summary) shows **both** graphs emit two archive nodes at that output path:

- a non-PIC variant with members `*.cpp.o`
- a PIC variant with members `*.cpp.pic.o`

Our PIC variant's member set (`bitset.cpp.pic.o`, … 8 members) is byte-identical
to ref's PIC variant; our non-PIC variant matches ref's non-PIC variant. The
archive **member selection emitter is already correct** — member PIC-ness is
derived generically from `Platform.PIC` (emit_cc.go:112: `.pic.o` iff
`instance.Platform.PIC`), and the host platform (`flags["PIC"]=="yes"`,
make.go:142) vs target (`"no"`) gives the two variants. The yaff codecs PIC
variant exists because it is reached on the host platform (the YAFF protoc
plugin / proto_reflect host tool closure).

### Where the `.cpp.o` vs `.cpp.pic.o` report actually comes from

`ay dev dump diff` keys same-output nodes for pairing via
`dumpDiffNodeMatchKey` = `kv.p` + `platform` + host-bit (dump_diff.go:744). It
does **not** distinguish the PIC contour. When two nodes (PIC + non-PIC) share
one output path, `findMatchingNodePair` (used by `--pair`) and
`findDumpDiffFieldMatch` (used by `--by-field`/`--by-token`) can pair our
**non-PIC** node against ref's **PIC** node and report their member multiset
difference — even when the two graphs are byte-identical.

Minimal reproduction (identical variant sets on both sides):

    L.jsonl / R.jsonl each: one AR node with member foo.cpp.o + one with foo.cpp.pic.o
    ay dev dump diff --left L --right R --pair '$(B)/x/lib.a'
    => spurious "[field cmds differs] -ours +foo.cpp.o / -ref +foo.cpp.pic.o"

This is exactly the recon's cause-#3 signal. The defect is in the comparison's
member contour, not in archive emission.

## The fix (generic, mechanism-faithful)

Make the diff node match key carry the PIC contour so PIC variants pair with PIC
variants and non-PIC with non-PIC. Add a single discriminator derived from the
node content — a node is a "PIC variant" when any of its outputs / inputs /
cmd-arg object tokens ends in `.pic.o`. Append it to `dumpDiffNodeMatchKey`;
both `dumpDiffExactOutputKey` and `dumpDiffAxisOutputKey` inherit it, and
`findMatchingNodePair` (which calls `dumpDiffNodeMatchKey` directly) inherits it
too. No path allowlist, no per-module special case — it is purely the object
suffix contour, identical to how the emitter derives `.pic.o`.

Effect: identical PIC/non-PIC variant sets at one output report no diff; a
genuine PIC-vs-non-PIC member divergence (one side missing a variant) still
shows up against the correct same-contour counterpart.

## Files touched

- `dump_diff.go`: add `dumpDiffNodePicVariant(n)`; append it to
  `dumpDiffNodeMatchKey`.
- `dump_diff_test.go` (new): failing-first test reproducing the spurious
  same-output PIC/non-PIC mispairing; passes after the key change.
- `emit_ar_test.go`: focused regression test proving that on a PIC platform the
  gen'd archive members are the `.pic.o` variants (and `.o` on a non-PIC
  platform) — the contour the ticket asks to pin.

## Expected effect on the gate

- The gate's `matched` count is a byte-line comparison of sorted normalized
  JSONL (validate.py `normalized_node_parity_counts`); it does **not** use the
  diff tool's pairing, so this change does not move sg7 `matched` and cannot
  regress sg2..sg6 byte-exactness.
- `go test ./...` gains the new tests.
- The representative archive pair no longer reports `.cpp.o` vs `.cpp.pic.o`
  for the right reason (PIC pairs with PIC), robust to node ordering.
