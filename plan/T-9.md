# T-9 — RUN_PROGRAM OUT headers consumed by GENERATE_ENUM_SERIALIZATION

## Upstream mechanism

A module's generated build-tree outputs are known to the whole module's
resolution before any consumer resolves its inputs. `GENERATE_ENUM_SERIALIZATION(H)`
where `H` is a `RUN_PROGRAM` `OUT` (e.g. `ads/bsyeti/libs/features/formula.h`,
produced by `RUN_PROGRAM(... --header formula.h ... OUT formula.cpp formula.h)`)
must resolve `H` to the `$(B)` producer output and take a dependency on the
producing node — never to a `$(S)` source that does not exist on disk.

In `ay` the codegen registry (`CodegenRegistry`) is exactly that mechanism: a
producer reserves its NodeRef and registers each build output (path + parsed
includes + producer ref) so a consumer's `resolveEnumHeaderInput` lookup
(`codegenRegForInstance(...).lookup(build(rel))`) resolves the `$(B)` output and
`resolveCodegenDepRefs` over the EN include closure pins the producer dep.

`resolveEnumHeaderInput` ALREADY performs this lookup. The only defect is emit
ordering in the main library path: `emitEnumSrcs` ran BEFORE
`emitRunProgramsForAR` / `emitRunPythonForAR` / `emitMiscNodes`, so the registry
had no `formula.h` entry yet and EN fell back to the nonexistent `$(S)` source.
The header-only / global path (gen.go ~938-991) already orders producers before
`emitEnumSrcs`; the main path diverged.

## Divergence (reproduced)

`emit_en_test.go::TestGen_EnumSerializationConsumesRunProgramGeneratedHeader`:
a `RUN_PROGRAM` `OUT gen.h` with no `$(S)` counterpart, consumed by
`GENERATE_ENUM_SERIALIZATION(gen.h)` in the same module. Before the fix the EN
node input is `$(S)/mod/gen.h` (nonexistent) with no producer dep; after the fix
it is `$(B)/mod/gen.h` plus a dep on the RUN_PROGRAM producer.

## Files touched

- `gen.go` — in the main library emit path, move `emitMiscNodes`,
  `emitRunProgramsForAR`, `emitRunPythonForAR` ahead of `emitEnumSrcs`, matching
  the header-only path. AR member append order is unchanged (it is fixed at the
  `genCC` append site, independent of emit order; final order is the stable
  `reorderARMembers` sort by (prio, line)).
- `emit_en_test.go` — regression test (above).
- `gen_test.go` — `mustNodeByAnyOutput` helper (producer lists gen.cpp as
  Outputs[0]).

## Expected effect on the gate

- `go test ./...` green; new regression fails before, passes after.
- `./dev/validate.py` keeps sg2/sg2_x86_64/sg3/sg4 OK counts, XFAIL count, sg5
  matched count, and the per-case generation-time budget; sg2–sg6 stay
  byte-exact (the reorder changes only NodeRef allocation order, normalized away
  to content hashes; no module without the EN-consumes-generated-header pattern
  changes content). Direct sg7 no longer aborts on the `formula.h`
  generated-header EN input. Subsequent independent generated-input blockers are
  out of scope (their own tickets).
