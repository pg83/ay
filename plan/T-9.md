## Upstream mechanism

`header.ya.make.inc` (included by `ydb/library/yql/dq/comp_nodes/llvm16/` via `ya.make.inc`) defines:
```
COPY_FILE(TEXT
    yql/essentials/minikql/computation/mkql_computation_node_codegen.h.txt
    ${BINDIR}/yql/essentials/minikql/computation/mkql_computation_node_codegen.h
    OUTPUT_INCLUDES ...
)
```

This produces:
- dst: `$(B)/ydb/library/yql/dq/comp_nodes/llvm16/yql/essentials/minikql/computation/mkql_computation_node_codegen.h`
- src: `$(S)/yql/essentials/minikql/computation/mkql_computation_node_codegen.h.txt`

kqp modules (`ydb/core/kqp/compute_actor/`, etc.) PEERDIR â†’ `ydb/library/yql/dq/comp_nodes/llvm16/` which contributes GLOBAL ADDINCL at `$(B)/ydb/library/yql/dq/comp_nodes/llvm16/`, so `#include <yql/essentials/minikql/computation/mkql_computation_node_codegen.h>` resolves to the COPY dst.

In upstream ymake, when a CC node's include closure reaches a COPY_FILE(TEXT) dst, the original `.txt` source is also added as a leaf input. This is the generic TEXT-copy input-tracking mechanism.

## Root cause in our code

`withContextSourceExtras` (in `emit_copy.go`) adds `.txt` sources for TEXT copies, but only from the **current module's** `d.copyFiles`. It builds `dstToSrc` only from `d.copyFiles` entries where `entry.Text == true`.

For kqp modules, `d.copyFiles` is empty â€” the COPY_FILE(TEXT) lives in `ydb/library/yql/dq/comp_nodes/llvm16/`. So `withContextSourceExtras` returns nil and the source is never added.

The fix: use the `CodegenRegistry` (which is platform-wide and contains ALL modules' CP registrations) to also pick up TEXT copies from other modules.

## Files to touch

1. `codegen_registry.go` â€” add `IsText bool` to `GeneratedFileInfo`
2. `emit_copy.go`:
   - Set `IsText: true` when registering TEXT copies in `emitCopyFiles`
   - Extend `withContextSourceExtras` to scan the closure for cross-module TEXT copies via registry lookup (remove the `len(d.copyFiles) == 0` early return that prevents cross-module resolution)
