# T-34 plan: C++ GRPC producer outputs for inline-proto LIBRARY modules with GRPC()

## Upstream mechanism

`build/conf/proto.conf`:

```
macro GRPC() {
    ENABLE(_GRPC_ENABLED)
    CPP_PROTO_PLUGIN2(grpc_cpp contrib/tools/protoc/plugins/grpc_cpp $_GRPC_SUF_CC $_GRPC_SUF_H DEPS contrib/libs/grpc ...)
}
```

`GRPC()` turns on the `grpc_cpp` C++ protoc plugin for the module. protoc then
emits `<base>.grpc.pb.cc` + `<base>.grpc.pb.h` in addition to `.pb.{cc,h}`,
is passed `--grpc_cpp_out=$(B)/` and
`--plugin=protoc-gen-grpc_cpp=$(B)/contrib/tools/protoc/plugins/grpc_cpp/grpc_cpp`,
depends on the grpc_cpp plugin tool as a command input, and the generated
`.grpc.pb.cc` is compiled into the module archive.

This applies to any C++ proto producer in the module, regardless of whether the
module is a `PROTO_LIBRARY()` or a plain `LIBRARY()` that lists a `.proto` in
`SRCS()` (the `ads/dssm/inference` shape: `LIBRARY()` + `SRCDIR(...)` +
`SRCS(... inference.proto ...)` + `GRPC()` inside `IF (NOT OS_WINDOWS)`).

## Root cause (verified, not assumed)

- Collection is correct: `modules.go` `case tokGrpc:` sets `d.grpc = true` for
  any module kind. A debug print confirmed `d.grpc == true` for the `LIBRARY()`
  inline-proto reproduction.
- Emission is the defect. Inline `.proto` in a `LIBRARY()` is routed through
  `emitLibraryProtoSource` (emit_proto.go), NOT `emitCPPProtoSrcs`. That
  function hardcodes `ProtoPBConfig{}` for both `newPBModuleEmission` and
  `emitProtoPB`, so `cfg.grpc` is always false: no grpc outputs, no
  `--grpc_cpp_out`, no grpc_cpp plugin input, and the `.grpc.pb.cc` is never
  compiled. (`emitCPPProtoSrcs`, the `PROTO_LIBRARY` path, already wires
  `cfg.grpc = d.grpc` and compiles the grpc.pb.cc — confirming the seam.)

A debug print in `emitCPPProtoSrcs` did NOT fire for the reproduction, proving
the inline-proto-in-LIBRARY case uses `emitLibraryProtoSource` exclusively.

## Files to touch

- `emit_proto.go` — `emitLibraryProtoSource`: build `ProtoPBConfig{grpc: d.grpc}`
  once, pass it to both `newPBModuleEmission` and `emitProtoPB`; when `d.grpc`,
  also compile the generated `.grpc.pb.cc` (same closure/dep wiring as the
  PROTO_LIBRARY path) and return it as an extra archived object.
- `emit_sources.go` — add `Extra []SourceEmit` to `SourceEmit` so a single
  source can yield more than one compiled object.
- `gen.go` — `appendCC`/`genCC` drain `emit.Extra` with the same `SrcMeta`.
- `emit_proto_test.go` — new test: a `LIBRARY()` with inline `.proto` + `GRPC()`
  emits `.grpc.pb.{cc,h}` producer outputs, includes the grpc_cpp plugin input
  and `--grpc_cpp_out`, and produces the `.grpc.pb.cc.o` compile node; an
  otherwise identical module WITHOUT `GRPC()` gains none of these.

Scope guard: do NOT change `cppOutRoot`/`moduleTag` for this path — the current
`ProtoPBConfig{}` already matches the reference pb.h producer for the non-grpc
case (the T-28 diff is grpc-only: cmds+inputs+outputs, no target_properties).
Only the grpc flag is added.
