# T-59 — generic `NO_OPTIMIZE()` compile-flag modeling

## Upstream mechanism (reproduce, do not invent)

`build/ymake.core.conf:4336`:

```
macro NO_OPTIMIZE() {
    ENABLE(NO_OPTIMIZE)
}
```

`build/conf/compilers/gnu_compiler.conf`:

```
OPTIMIZE=$_OPTIMIZE_RELEASE            # release x86 → -O3, debug → empty
...
when ($NO_OPTIMIZE == "yes") {
    OPTIMIZE = -O0
}
```

`OPTIMIZE` is a single variable spliced into `CFLAGS` between the `-pipe -m64`
prefix and the debug-info / `-fno-common…` group (`CFLAGS+=$_C_FLAGS
$DEBUG_INFO_FLAGS $_C_FOPTIONS`). In a release build its value is `-O3`; the
`when ($NO_OPTIMIZE == "yes")` clause reassigns it to `-O0`. This is a generic,
module-level compile option — every module that declares `NO_OPTIMIZE()` gets
`OPTIMIZE = -O0`, independent of which tool/library it is.

### Reference (sg7.json, `contrib/tools/gperf/getopt.c.pic.o`)

The reference compile carries the host/release C-flag vector
`[-pipe, -m64, -O0, -fno-common, -ffunction-sections, …]` — i.e. hostCFlags with
the optimize token `-O3` reassigned to `-O0`. `contrib/tools/gperf/ya.make`
declares `NO_OPTIMIZE()`; after T-56 made the gperf tool closure reachable, ay
emits the same nodes but with `-O3` (the default OPTIMIZE), producing paired
cmd drift on `getopt.c.pic.o`, `getopt1.c.pic.o`, `all_srcs.cpp.pic.o`, etc.

In our model the OPTIMIZE token with a value exists only in `hostCFlags`
(release x86, `flags.go:60` — `argO3` after `-m64`). The debug/target vectors
carry no optimize token (OPTIMIZE empty upstream), so there is nothing to
reassign there and no reference exercises a debug `NO_OPTIMIZE` build. The
faithful, minimal reproduction is therefore: when a module declares
`NO_OPTIMIZE()`, replace the optimize token `-O3` with `-O0` in its compile
C-flag vector. This is exactly `when ($NO_OPTIMIZE=="yes"){OPTIMIZE=-O0}` for the
case where OPTIMIZE has a value, and touches only vectors that contain `-O3`
(release x86 host), so the byte-exact debug-target gating cases sg2–sg6 are
untouched.

## Files touched

1. `arg_consts.go` — add `argO0 = internArg("-O0")`.
2. `modules.go` — `ModuleData.noOptimize bool`; dispatch `case tokNoOptimize:
   d.noOptimize = true` (the `NO_OPTIMIZE` token already exists but was ignored).
3. `emit_cc.go` — `ModuleCCInputs.NoOptimize bool`; in
   `composeCCModuleArgBlocks`, when `in.NoOptimize`, rebuild `bundle.CFlags`
   with the optimize token suppressed (`-O3` → `-O0`).
4. `gen.go` — set `NoOptimize: d.noOptimize` on the module's `ModuleCCInputs`.
5. `emit_cc_no_optimize_test.go` (new) — fail-first coverage + negative guard.

## Test (fail-first)

`TestGen_NoOptimizeSuppressesOptimization` — release x86 host + aarch64 target,
`contrib/tools/gperf` PROGRAM declaring `NO_OPTIMIZE()` referenced via a
`.gperf` source so it builds on the release host. Assert the host compile
`$(B)/contrib/tools/gperf/main.cpp.pic.o` carries `-O0` and not `-O3`.
Negative guard `TestGen_DefaultOptimizationIntact`: same module WITHOUT
`NO_OPTIMIZE()` keeps `-O3` and has no `-O0`.

Fails on current trunk: without the dispatch + suppression the host compile
carries the default `-O3`.
