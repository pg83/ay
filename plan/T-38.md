## Upstream mechanism

1. **pycxx ŌĆö MODULE_LANG of PY23_NATIVE_LIBRARY's modeled submodule is CPP.**
   `build/conf/python.conf:1237` `multimodule PY23_NATIVE_LIBRARY` defines the
   modeled half as `module PY3: LIBRARY { ... SET(MODULE_LANG CPP) ... }`
   (line 1251), prefix `libpy3c`, OBJ_SUF `.py3`. The SBOM component output
   suffix and `--lang` come from `${...suf=.${MODULE_LANG}.component.sbom...}`
   and `${pre=--lang :MODULE_LANG}` (`build/internal/conf/sbom.conf:43`).
   So a PY23_NATIVE_LIBRARY emits `<name>.CPP.component.sbom`, not `.PY3`.
   Its MODULE_TAG is `PY3_NATIVE` (PYTHON3_ADDINCL, python.conf:995) ŌĆö already
   correct in our emitter.

2. **py-only PROTO_LIBRARY emits no component.**
   `protos_from_protobuf` / `protos_from_protoc` are `PROTO_LIBRARY()` with
   `EXCLUDE_TAGS(CPP_PROTO GO_PROTO)` and `LICENSE(BSD-3-Clause)`. The
   CPP_PROTO submodule (the only proto submodule that carries
   `_NEED_SBOM_INFO=yes` and would emit a `.CPP.component.sbom`) is excluded.
   The remaining py-proto submodule `_PY3_PROTO` does `DISABLE(_NEED_SBOM_INFO)`
   (`build/conf/proto.conf:806`; `_PY_PROTO` line 783), so it emits no
   component either. Net: a PROTO_LIBRARY with CPP_PROTO excluded emits no SBOM
   component. Our model represents exactly this case (`defaults.go:133`
   switches such a module to the py-only/C++-peer contour) and currently emits
   a spurious `.CPP.component.sbom`.
