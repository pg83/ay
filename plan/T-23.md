# T-23 plan: link resource objcopy members for plain C++ PROGRAM modules

## Upstream mechanism

A reachable C++ `PROGRAM()` that declares `RESOURCE`/`RESOURCE_FILES` embeds
those payloads into its binary exactly like any other module: `_SRC_RESOURCE`
(`build/conf/project_specific/resources.conf`) packs each resource batch into an
`objcopy_<hash>.o` (TObjCopyResourcePacker) that becomes a member of the link.
Upstream applies this to the link side of every module, not only Python ones.
For `ads/argus/proto/profile/generated/codegen`:

```
LD  $(B)/ads/argus/proto/profile/generated/codegen/codegen
 -> PY $(B)/.../codegen/objcopy_f1b4b3539b79a9a8a902252c63.o   (resource objcopy)
 -> BN $(B)/.../codegen/proto.desc                             (BUNDLE rename)
 -> PD $(B)/ads/argus/proto/profile/descriptions/proto-profile-descriptions.protodesc
```

The objcopy embeds `RESOURCE(proto.desc ads/argus/proto/profile/descriptions/proto.desc)`:
file arg `proto.desc` resolves to the module-local `$(B)/.../codegen/proto.desc`
produced by `BUNDLE(ads/argus/proto/profile/descriptions NAME proto.desc)`, with
the second arg as the embed key (base64 in `--keys`).

## Our current behavior (the defect)

`gen.go` PROGRAM LD branch (`isProgramModuleType`, ~2361) gates resource objcopy
emission on `resourceLibTagForData(d) != nil`. That helper returns non-nil only
for Python-flavored module types (PY3_LIBRARY/PY3_PROGRAM/…). For a plain
`PROGRAM()` it returns `nil`, so `emitResourceObjcopy` is never called on the LD
path even when `d.resources` is non-empty. Consequently the LD node has no
objcopy member, and the `BN proto.desc` / `PD …protodesc` producer chain is
pruned as unreachable from the normalized closure.

The producers already exist: `emitBundles` (gen.go:1312) registers the BN, and
`resolveResourceInput` binds the resource path to the registered BN build output
through the codegen registry. Only the link-side objcopy edge is missing.

## Change (files touched)

1. `gen.go` ~2361: widen the PROGRAM LD objcopy gate from
   `resourceLibTagForData(d) != nil` to also fire when the module has real
   resource entries:

   ```go
   if resourceLibTagForData(d) != nil || len(d.resources) > 0 {
   ```

   `RESOURCE_FILES` is already expanded into `d.resources`, so this covers both
   forms. `emitResourceObjcopy` already returns `nil` when there are no
   resources/pysrc/kv, so non-resource PROGRAMs are unaffected. The objcopy hash
   uses `resourceLibTagForData(d)` (nil for plain C++ PROGRAM), matching the
   reference `objcopy_f1b4b3539b79a9a8a902252c63.o` name — no synthetic tag.

No change to `resolveBundleSource`, no PROTO_DESCRIPTIONS root traversal, no
Python resource handling change.

## Test (test-first)

`gen_test.go` (or a new focused test): a reachable plain `PROGRAM()` with
`SRCS(main.cpp)`, `BUNDLE(dep NAME x.bundle)`, `RESOURCE(x.bundle dep/key)`,
where `dep` is a `LIBRARY()`. Assert:
- BN `$(B)/prog/x.bundle` exists and depends on `$(B)/dep/libdep.a`;
- objcopy node `$(B)/prog/objcopy_*.o` exists, input is `$(B)/prog/x.bundle`
  (not `$(S)/prog/x.bundle`);
- the PROGRAM LD node depends on the objcopy object.

Must fail before the gate widening (LD lacks objcopy; objcopy node absent).

## Gate effect

- `go test ./...` green.
- `./dev/validate.py` keeps sg2/sg2_x86_64/sg3/sg4 byte-exact OK, sg5 matched
  not decreasing, XFAIL not growing.
- sg7 focused probes show the ads/argus codegen objcopy, `proto.desc` BN, and
  `proto-profile-descriptions.protodesc` PD present in our normalized closure.

## Parity risk

Any reachable plain C++ `PROGRAM()` in sg2–sg6 with `RESOURCE`/`RESOURCE_FILES`
gains objcopy members; byte-exact gates catch any divergence from upstream.
`emitResourceObjcopy` returns nil for modules without resources, so no spurious
emissions.
