# T-23 plan: lxml Header-variant Cython producer input-closure drift

## Symptom (reproduced)

Current trunk (post-T-18), sg7 baseline:
`matched=62248 our_only=9719 ref_only=10036`.

The lxml `etree.c` / `etree.h` / `etree_api.h` outputs are **paired** (present on
both sides) but carry different normalized content. The single differing field is
`inputs`:

```
ay dev dump diff --pair '$(B)/contrib/python/lxml/py3/lxml/etree.c'
[field inputs differs]
  -ours +$(S)/contrib/libs/cxxsupp/libcxx/include/__algorithm/...   (×~1200)
```

Ours carries ~1272 inputs; ref carries **88**. Our node over-collects the whole
libcxx / Python / numpy / libxml closure; ref carries only:

- `contrib/tools/cython/cython.py`
- the 36 `contrib/tools/cython/Cython/Utility/*` embedded files (bare)
- `etree.pyx`
- the pyx's cimport/include/`.pxd`/`.pxi` closure (lxml `.pxi`/`.pxd` +
  cython `Includes/{cpython,libc}/*.pxd`)

No `Python.h`, no libcxx, no numpy, **no libxml** (the `cdef extern from` targets).

## Verified upstream mechanism

Classifying every CY node in sg7's ref graph by source extension and whether it
carries libcxx: 58 of 60 carry the full induced "cpp" closure; the **only two
that do not** are:

- `contrib/python/lxml/py3/lxml/etree.pyx`  → `CYTHON_C_API_H`  (88 inputs)
- `contrib/python/gevent/py3/gevent/libev/corecext.pyx` → `CYTHON_C_H` (74 inputs)

These are exactly the `_H` / `_API_H` macro variants
(`_BUILDWITH_CYTHON_C_H`, `_BUILDWITH_CYTHON_C_API_H`, `_BUILDWITH_CYTHON_CPP_H`).
`CYTHON_C` / `CYTHON_CPP` (e.g. lxml `objectify.pyx`, gevent `resolver/cares.pyx`)
and `CYTHONIZE_PY` `.py` (gevent) all carry the full closure.

The `_H`/`_API_H` macros emit their generated `.h`/`_api.h` as an `addincl`
output (`${hide;addincl;tobindir;output;...h;noext:Src}`). In ymake the cython
include-processor routes the induced **"cpp"** deps (cdef-extern headers,
`CYTHON_OUTPUT_INCLUDES` Python headers, embedded-file include closures —
`AddParsedIncls("cpp", …)`, rule `cpp → Pass`) onto that header output for
consumers, **not** onto the producer node. The "pyx"-language includes
(cimport/include/`.pxd`/`.pxi`, `AddIncludesToNode`, rule `pyx → Use`) always
ride the producer.

Net rule: a **Header / ApiHeader** CY node carries only
`{cython.py, bare embedded utility files, src, pyx-language cimport/include/pxd
closure}` — never the induced cpp closure (cdef-extern C/C++ headers, the
`CYTHON_OUTPUT_INCLUDES` python headers, or the embedded-file include closures).

The non-Header path is unchanged.

## Change

`emit_cython_cpp.go`:

- Add `cythonPyxLangClosure(scanner, src, cfg)`: a filtered DFS that reuses the
  scanner's cached per-file child resolution but recurses only into
  cython-language files (`.pyx/.pxd/.pxi/.py`), dropping `cdef extern from`
  C/C++ header targets. It builds no closure-window cache entry, so the
  context-free window cache is untouched (perf-neutral; fires only for the 2
  Header nodes).
- Add `cythonHeaderToolInputs(...)`: `keepOnlySourceVFS(dedup(
  [cython.py, src, bare embedded files], cythonPyxLangClosure(src)))`.
- In `emitCythonCpp`, when `stmt.Header`, override the CY node `Inputs` with
  `cythonHeaderToolInputs`. `emitsIncludes` (consumer-side parsed includes of
  the generated output) and the command stay as-is.
