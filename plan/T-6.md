# T-6 plan: flatbuffers64 / `.fbs64` producer family

## Upstream mechanism

`build/conf/fbs.conf` defines two C++ flatc command macros:

- `_CPP_FLATC_CMD` (regular `.fbs`), dispatched by `_SRC("fbs")` →
  kind `FL`, tool `FLATC = contrib/libs/flatbuffers/flatc`,
  `.PEERDIR=contrib/libs/flatbuffers`.
- `_CPP_FLATC64_CMD` (`.fbs64`), dispatched by `_SRC("fbs64")` (ymake.core.conf
  line 3535) → kind `FL64`, tool `FLATC64 = contrib/libs/flatbuffers64/flatc`,
  `.PEERDIR=contrib/libs/flatbuffers64`.

The two commands differ only in a few constant spans:

```
FL  : ... --schema -b ${ITER_FLAG} --gen-object-api --filename-suffix .fbs   ${FLATC_FLAGS} ${pre=-I :_FLATC__INCLUDE}  -o <h> <cpp> <src> <iter?> <bfbs;noext>
FL64: ... --schema -b                            --filename-suffix .fbs64 ${FLATC_FLAGS} -I ${ARCADIA_ROOT} -I ${ARCADIA_BUILD_ROOT} -o <h> <cpp> <src> <bfbs64;noext>
```

Concretely FL64 vs FL:
1. tool: `FLATC64` (`contrib/libs/flatbuffers64/flatc`) instead of `FLATC`.
2. drops `--gen-object-api` (and the yandex-maps-iter flag/header, which we
   already do not model).
3. `--filename-suffix .fbs64` instead of `.fbs` (so a generated header that
   imports a sibling references `<sibling>.fbs64.h`).
4. include args are the fixed pair `-I ${ARCADIA_ROOT} -I ${ARCADIA_BUILD_ROOT}`
   (= `-I $(S) -I $(B)`), opposite order to FL's `-I $(B) -I $(S)`.
5. `.bfbs64` output suffix instead of `.bfbs`.
6. kv `p FL64`, `.PEERDIR=contrib/libs/flatbuffers64`.

`contrib/libs/flatbuffers64/flatc/ya.make` is an ordinary PROGRAM with
`INDUCED_DEPS(h+cpp contrib/libs/flatbuffers64/include/flatbuffers/flatbuffers.h)`.
Our tool registry (`ctx.tool` → `genModule(newToolInstance)`) builds any PROGRAM
by path, and the scanner mixes a producer's `GeneratorRefs` INDUCED_DEPS into the
generated file's closure generically, so the 64-bit runtime header rides to
consumers via the FLATC64 tool ref with no special casing.

`.fbs64` sources are listed directly in module SRCS (e.g.
`search/plutonium/impl/chunkler/idl/ya.make`); they are not part of the
`FBS_LIBRARY` multimodule (which is `.fbs`-only).

## Existing machinery to reuse

`emit_flatc.go` already models the FL path end to end: `emitFL` (node),
`emitFlatcProducer` (pre-pass producer + codegen-registry registration of
.h/.cpp/.bfbs, closure leaves, generated-header includes), and
`emitLibraryFlatcSource` (compile the generated .cpp). The `.fbs64` variant is
the same shape with the constant spans above swapped. The generated-header
include resolver (`flatcDirectGeneratedHeaderIncludes`) appends `.h` to the
resolved import path and is already generic across `.fbs`/`.fbs64`.

## Change

Parameterize the FL emitter with a small `flatcVariant` value (tool arg, const
flags, IO-lead args, proc kind, src/bfbs suffix, runtime header VFS); keep two
package-level variants `flatcVariantFL` / `flatcVariantFL64`. The consumer side
(`emitLibraryFlatcSource`) is suffix-agnostic and is reused unchanged.

Files:

- `arg_consts.go`: `argContribLibsFlatbuffers64Flatc`, `argFbs64` (`.fbs64`).
- `vfs_consts.go`: `flatc64RuntimeVFS`
  (`contrib/libs/flatbuffers64/include/flatbuffers/flatbuffers.h`).
- `str_consts.go`: `strContribLibsFlatbuffers64`.
- `node_attrs.go`: `pkFL64` enum + `"FL64"` label.
- `emit_flatc.go`: introduce `flatcVariant`, thread it through `emitFL` and
  `emitFlatcProducer`; add the two variant values. The bfbs path uses
  `variant.srcExt`/`variant.bfbsExt`.
- `emit_sources.go`: dispatch `.fbs64` → `emitLibraryFlatcSource` (shared).
- `str_intern.go`: `srcExtFbs64` class; `classifySrcExt`,
  `isCodegenProducingSrcID` classify `.fbs64` distinctly.
- `gen.go`: pre-pass emits `.fbs64` producers with `flatcVariantFL64`;
  `isCodegenProducingSrc` adds `.fbs64`; auto-peer adds
  `contrib/libs/flatbuffers64` when `d.hasFbs64`.
- `modules.go`: `hasFbs64` flag set in the src triage.
- `dev/gen_parsers.py`: add `.fbs64` to `FlatbuffersIncludeDirectiveParser`;
  regenerate `parsers_generated.go`.
