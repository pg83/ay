## Divergence analysis

**Ours** cmd_args members: `minidumps.cpp.o`, `objcopy_c3cc8e61a1f1a4fb5113609cb8.o`
**Reference** cmd_args members: `objcopy_c3cc8e61a1f1a4fb5113609cb8.o`, `minidumps.cpp.o`

The reference always puts the RESOURCE objcopy output FIRST, before SRCS(GLOBAL) objects.

## Upstream mechanism

The breakpad `ya.make` declares:
```
SRCS(GLOBAL minidumps.cpp)   ← GLOBAL src declared first
RESOURCE(about.txt ...)      ← RESOURCE (→ objcopy output) declared second
```

Despite GLOBAL SRCS coming first in declaration order, upstream's global archive
always places the objcopy (RESOURCE → `OUT_GLOBAL`) before the SRCS(GLOBAL) objects.
The `resourceBeforeGlobalSrcs(d)` predicate in gen.go checked declaration order and
returned false here (since SRCS(GLOBAL) comes before RESOURCE), so the reordering
`moveTailNodeRefsToFront` was never called. The objcopy ended up last.

The correct generic mechanism: upstream always orders objcopy objects before
SRCS(GLOBAL) objects in the global archive, regardless of declaration order.

## Fix

Remove the `resourceBeforeGlobalSrcs(d)` condition — make `moveTailNodeRefsToFront`
unconditional when `globalSrcMemberCount > 0 && len(objcopyRes.Refs) > 0`.

Clean up the now-unused infrastructure:
- `resourceBeforeGlobalSrcs` function in gen.go
- `firstResourceEvent` and `firstGlobalSrcsEvent` fields in moduleData (modules.go)
- All code that set these fields

## Why `len(d.resources) > 0` as the discriminant

The upstream behavior differs by how the objcopy was produced:
- RESOURCE → `d.resources` → always placed first regardless of declaration order
- PY_SRCS → `d.pySrcs` → follows declaration order (after SRCS(GLOBAL))

Evidence from reference graph:
- cctz tzdata (LIBRARY, RESOURCE before SRCS(GLOBAL)): objcopy first ✓ (already worked)
- breakpad (LIBRARY, SRCS(GLOBAL) before RESOURCE): objcopy first ✓ (was broken)
- py3 runtime (PY3_LIBRARY, SRCS(GLOBAL) before PY_SRCS): compiled .o first ✓ (already correct)
- MarkupSafe (PY3_LIBRARY, no SRCS(GLOBAL)): reorderARMembers handles reg3 last ✓

Using `len(d.resources) > 0` as the condition catches breakpad while correctly
leaving py3 runtime unchanged.
