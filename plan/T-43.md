# T-43 plan: non-whitelisted YaFF `.yaff.cpp` rides its proto's main `.pb.h`

## Symptom (reproduced on trunk after T-33)

`./dev/validate.py` sg7, pair
`$(B)/ads/bsyeti/libs/experiments/proto/config.yaff.cpp.o`:

```
[field inputs differs]
  -ref  +$(B)/ads/bsyeti/libs/experiments/proto/config.pb.h
  -ref  +$(S)/ads/bsyeti/libs/experiments/proto/config.proto
  -ref  +$(S)/build/scripts/cpp_proto_wrapper.py
```

`config.proto` is **outside** the module's `YAFF(FILES source.proto tree_info.proto)`
allowlist. The reference `.yaff.cpp.o` carries the proto's own `.pb.h` plus the
producer-source bundle (`.proto`, `cpp_proto_wrapper.py`); ours carries neither.
Our closure is otherwise complete (1292 inputs; ref 1295 — exactly the 3 above).

The whitelisted siblings (`source.yaff.cpp.o`, `tree_info.yaff.cpp.o`) are already
byte-exact: their non-empty `*.yaff.h` `#include`s `*.pb.h`, so the `.pb.h` (and its
`.proto` closure-leaf + `cpp_proto_wrapper.py` window include) ride via include scan.

## Upstream mechanism (verified, not assumed)

1. The YaFF plugin (`library/cpp/yaff/tools/protoc_plugin/proto_plugin.cpp`,
   `Generate`) ALWAYS writes `<proto>.yaff.cpp` as `#include "<proto>.yaff.h"`, and
   writes `<proto>.yaff.h` empty when `NeedToProcessFile` is false (proto not in the
   FILES whitelist). So an include scan of a non-whitelisted `.yaff.cpp` reaches only
   the empty `.yaff.h` and nothing else — no `.pb.h`.

2. The protoc command emits `config.pb.h` (main), `config.pb.cc`, `config.yaff.h`,
   `config.yaff.cpp` from one invocation. `CPP_PROTO_OUTS` floats `.pb.h` to the
   front with the `${main;…:.pb.h}` flag (`ymake.core.conf` `_BASE_UNIT`), making
   `.pb.h` the command's MAIN output; every other output is an `EDT_OutTogether`
   sibling pointing at it.

3. `TJSONVisitor::PrepareLeaving` (`devtools/ymake/json_visitor.cpp:658-661`): a node
   depending on a non-main sibling output rides that sibling's
   `OutTogetherDependency` — the MAIN output — into its inputs, transitively expanded
   (`config.pb.h` node → its own include closure: `cpp_proto_wrapper.py`, the
   `config.proto` generated-from edge, the protobuf runtime). This is exactly the
   mechanism `emit_pr.go` already models for RUN_PROGRAM main outputs
   (`reg.addClosureLeaf(out, mainOutputVFS)`), but the PB main output `.pb.h` carries
   a non-trivial include window (wrapper, .proto leaf), so it must ride EXPANDED, not
   as a bare leaf.

So the `.yaff.cpp.o` compile rides `config.pb.h` (expanded) via OutTogether,
independent of the FILES whitelist. In the whitelisted case this coincides with the
include-scan path (`.yaff.h` content includes `.pb.h`) and dedupes to no change.

## Current code (the defect)

`emit_proto.go` registers `yaffCC` with parsed includes `[<stem>.yaff.h]` only. For a
non-whitelisted proto the sibling `.yaff.h` is empty, so the walk yields no `.pb.h`
and the OutTogether main-output ride is absent. The protobuf runtime still arrives via
T-33's `INDUCED_DEPS(cpp …)` (yaffCC GeneratorRefs), but `.pb.h` / `.proto` /
`cpp_proto_wrapper.py` do not.

## Fix

`emit_proto.go`, in the YaFF registration loop: add the proto's own `pbH` (the protoc
command's MAIN output) to `yaffCCParsed`, modeling the `EDT_OutTogether` main-output
ride. Generic — applied to every YaFF `.yaff.cpp` (whitelisted dedupes against the
`.yaff.h`-content path, non-whitelisted gains the only `.pb.h` path). `pbH` expands to
its cached window (`cpp_proto_wrapper.py` include + `config.proto` closure-leaf +
protobuf runtime, all deduped), so the `.yaff.cpp.o` gains exactly the 3 ref inputs.
T-33's sibling self-header (`.yaff.h`) filter is unchanged — it drops only the empty
`.yaff.h`; `pbH` survives.
