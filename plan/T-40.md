# T-40 plan: GRPC inline-proto LIBRARY include-band order (plugin-runtime peer leads ADDINCL)

## Symptom (reproduced)

After T-34 the inline-proto `LIBRARY()` `ads/dssm/inference/lib/core` emits the
`.grpc.pb.cc` compile, but its compile output
`$(B)/ads/dssm/inference/lib/core/__/__/inference.grpc.pb.cc.o` diverges from
upstream by command include-path order only (the cmd_args multiset matches).
The sibling `inference.pb.cc.o` shows the identical drift — this is a
module-wide ADDINCL band order issue, not specific to the grpc.pb.cc compile.

Observed `-I` band (after the libc_compat defaults):

- ours: `protobuf/src, abseil-cpp-tstring, abseil-cpp, libiconv, libidn,
  yaml-cpp, sentencepiece, grpc/include, c-ares, openssl, re2, intel/mkl`
- ref:  `grpc/include, abseil-cpp, c-ares, openssl, re2, abseil-cpp-tstring,
  protobuf/src, libiconv, libidn, yaml-cpp, sentencepiece, intel/mkl`

Ref hoists the whole `contrib/libs/grpc` GLOBAL ADDINCL closure
(grpc/include → abseil-cpp, c-ares, openssl, re2, abseil-tstring, protobuf) to
the front of the band; ours leaves it in declared-PEERDIR position (the GRPC()
peer is appended late), so its closure-unique includes (grpc/include, c-ares,
openssl, re2) land late while protobuf/abseil — also reachable through an
earlier declared peer — appear early.

## Upstream mechanism

`build/conf/proto.conf` `GRPC()` is `CPP_PROTO_PLUGIN2(grpc_cpp … DEPS
contrib/libs/grpc)`. The proto plugin DEP `contrib/libs/grpc` is a
plugin-runtime peer injected via `CPP_PROTOBUF_PEERS` ahead of the body PEERDIR
(`ymake.core.conf:2002`, `PEERDIR+=`). ymake derives ONE peer order for both the
GLOBAL ADDINCL (`-I`) closure and the link/archive closure, so the plugin-runtime
peer leads both. This is the exact mechanism already modelled by `protoCmdPeers`
+ `walkPeersForGlobalAddIncl`'s front-hoist (T-14/T-42) for `PROTO_LIBRARY`.

## Root cause

`modules.go:985` populates `d.protoCmdPeers` ONLY for
`d.moduleStmt.Name == tokProtoLibrary`. The `ads/dssm/inference` shape is a plain
`LIBRARY()` that compiles an inline `.proto` + `GRPC()`. For it `d.protoCmdPeers`
stays empty, so `contrib/libs/grpc` is never in `frontSet` and never hoisted —
its closure lands in declared (late) order.

The proto plugin DEPS lead the peer order whenever the module compiles a `.proto`
to C++, regardless of whether the module is a `PROTO_LIBRARY` or a `LIBRARY` with
an inline `.proto` — `protoCmdPeers` is already keyed off `d.grpc` /
`d.cppProtoPlugins`, both of which are populated for the inline case.

## Change

`modules.go`: broaden the gate that populates `d.protoCmdPeers` to also fire when
the module has an inline `.proto` source (`hasProto`), not only for
`tokProtoLibrary`. For inline proto without grpc/plugins `protoCmdPeers()` returns
empty → no reordering (zero effect); only grpc / cpp-proto-plugin inline modules
gain the front-hoist, matching upstream.

## Test (test-first)

`emit_proto_grpc_library_test.go`:

- GRPC fixture: inline-proto `LIBRARY()` + `PEERDIR(declared/peer)` (declared
  before `GRPC()`), where `contrib/libs/grpc` carries `ADDINCL(GLOBAL grpc/inc)`.
  Assert the `.grpc.pb.cc.o` compile has `-I$(S)/contrib/libs/grpc/inc` BEFORE
  `-I$(S)/declared/peer/inc`. Fails before the fix (grpc not hoisted), passes
  after.
- Non-GRPC control: inline-proto `LIBRARY()` with `PEERDIR(a)` then `PEERDIR(b)`,
  each with a GLOBAL ADDINCL. Assert `a/inc` precedes `b/inc` (declared order
  preserved) — the fix must not reorder non-grpc proto compiles.
