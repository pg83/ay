# T-148 plan: split explicit-resource vs PY_SRCS objcopy placement in global archives

## Hypothesis / divergence

For a Python global archive that has BOTH explicit `RESOURCE_FILES` objcopy
members AND `PY_SRCS` `.py`/`.pyi` resource objcopy members AND `SRCS(GLOBAL)`
C++ sources, the reference (.global.a) member order is:

```
explicit RESOURCE_FILES objcopy, GLOBAL src band, PY_SRCS .py objcopy,
PY_SRCS .pyi objcopy, PY_REGISTER .reg3.cpp
```

`ay` currently produces:

```
explicit RESOURCE_FILES objcopy, PY_SRCS .py objcopy, PY_SRCS .pyi objcopy,
GLOBAL src band, PY_REGISTER .reg3.cpp
```

i.e. ay over-moves the PY_SRCS/.pyi objcopy members ahead of the GLOBAL band.

Representatives (T-142 survey, AR cmds x2):
- `$(B)/contrib/python/tree-sitter/libpy3contrib-python-tree-sitter.global.a`
- `$(B)/contrib/python/pycryptodome/py3/libpy3python-pycryptodome-py3.global.a`

Member multisets already match; only the global-archive member sequence differs.

## Upstream mechanism

`yatool/build/plugins/res.py:RESOURCE_FILES` expands explicit files into an
`onresource(...)` call at the statement's position; ymake's source FIFO sees that
explicit resfs objcopy output before the `SRCS(GLOBAL)` C++ band concludes.
`yatool/build/plugins/pybuild.py:PY_SRCS` expands `.py`/`.pyi` resources through
later `onresource_files` calls (a step after GLOBAL handling), so those objcopy
outputs are ordered AFTER the global C++ symbols and before the generated
`PY_REGISTER` `.reg3.cpp` compile (which is a later FIFO round).

So the explicit-resource objcopy band belongs before GLOBAL; the PY_SRCS/.pyi
objcopy band belongs after.

## Current ay defect

`emitResourceObjcopy` (emit_py_objcopy.go) returns one combined
`ObjcopyEmitResult` whose `Refs` are appended in this order:
`[PY_MAIN?, NO_CHECK_IMPORTS?, YA_CONF_JSON?, explicit RESOURCE_FILES,
PY_SRCS .py, PY_SRCS .pyi]`. The PY_SRCS `.py`/`.pyi` members are the trailing
span.

`gen.go` appends that whole result after the GLOBAL/`reg3.cpp` band, then moves
the WHOLE combined objcopy tail in front of the global sources when
`globalSrcMemberCount > 0 && len(d.resources) > 0`. That over-moves the PY_SRCS
trailing span. `reorderARMembers` then stable-sorts by generation round, which
preserves this (defective) relative order for the round-0 members.

## Fix

Move only the LEADING (non-PY_SRCS) objcopy band before the global sources;
leave the PY_SRCS `.py`/`.pyi` trailing band after the global sources. The
trailing band is a contiguous suffix of the combined result, so a single count
identifies it.

Files touched:

1. `emit_py_objcopy.go`
   - Add `PySrcTrailCount int` to `ObjcopyEmitResult` = number of objcopy
     members emitted by `emitPySrcObjcopy` plus the `.pyi`
     `emitEntries(d.pyPyiResources)` (the trailing PY_SRCS band). Set it in both
     emission paths (the no-resources early return and the resources path).

2. `gen.go`
   - Replace the `moveTailNodeRefsToFront`/`moveTailVFSToFront` whole-tail move
     with a move of only the leading band
     `leadCount = len(Refs) - PySrcTrailCount` starting at the objcopy base
     index. Keep the `len(d.resources) > 0` guard (the move is for explicit
     resources; without them nothing moves and PY_MAIN/etc. stay after globals,
     as today).
   - Add generic `moveSubrangeToFront[T any](in, start, count)`; delete the two
     type-specialized `moveTail*ToFront` helpers (now unused). Net code shrink.

T-145/T-146 generated-resource input behavior is untouched: this change only
reorders already-emitted member refs; objcopy keys, batching, hashes,
inputs, and command args are unchanged.
