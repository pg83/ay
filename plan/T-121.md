# T-121 plan: GZT-generated build `.proto` closure propagation for PB consumers

## Upstream mechanism

A `.gztproto` is converted by `dict/gazetteer/converter` into a build-rooted
`$(B)/<moddir>/<base>.proto`. That generated `.proto` is an ordinary protoc
input: a later PB node whose source `.proto` imports it must see, exactly as
protoc would, the generated proto's own parsed imports — the rewritten
`.gztproto`→`.proto` imports, the ordinary `.proto` imports, and the converter's
`INDUCED_DEPS(proto …)` base.proto — and, as a generated-from source edge, the
raw `.gztproto` producer-source leaves behind each generated `.proto` in the
chain. This is upstream's flat input closure: the reference
`$(B)/search/idl/blender/blender_setup.pb.h` lists the generated nested
`xml_parse_request.proto`, `http.proto`, `base.proto`, and the two
`.gztproto` leaves.

Our emitter registers the generated proto's parse only under its SOURCE-rooted
VFS (`injectSourceParse`), which serves the node compiling that generated proto
itself (`protoSrcOverride`). A *different* source PB node resolves the import to
the BUILD-rooted `$(B)/…/<base>.proto`; `parsedIncludes` consults only
`buildParsed` for build paths (parser_manager.go:210-217), so the nested imports
are invisible and the `.gztproto` leaves never ride. That is the gap.

## Files to touch

- `emit_gztproto.go` — register the generated proto's build parse and splice its
  `.gztproto` source leaves onto the generated `$(B)` proto.
- `emit_proto.go` — add producer deps for build-rooted generated transitive
  proto imports on a source PB node (Merkle-graph parity, not a `--pair`
  content field).
- `emit_gztproto_test.go` — fail-first consumer test.

## Implementation

In `emitLibraryGztProtoSource`:

1. Compute `generatedParse := gztGeneratedProtoParse(ctx, gztSource, inducedProtos)`
   once; keep `injectSourceParse(source(gen), generatedParse)` for the
   self-compile path.
2. Add `ctx.parsers.registerBuildParsedIncludes(genProto, generatedParse.bucket(parsedIncludesLocal))`
   so a consumer that resolves `$(B)/…/<base>.proto` as a transitive import
   walks its nested proto imports (rewritten `.gztproto`→`.proto`, ordinary
   `.proto`, induced base.proto). Context-free, keyed by VFS, set once at emit.
3. After registering the generated proto in the codegen registry, splice the
   `sourceInputs` (self `.gztproto` + transitively-imported `.gztproto` only)
   as closure leaves of `genProto`: `reg.addClosureLeaf(genProto, s)`. They ride
   as non-expanded generated-from leaves in the consumer's closure window.

In `emitProtoPB`, after `transitiveImports`/`extraProtoDeps` are computed, append
producer refs for build-rooted generated transitive imports:
`extraProtoDeps = append(extraProtoDeps, resolveCodegenDepRefs(ctx, instance, transitiveImports, extraProtoDeps...)...)`.
The raw upstream `blender_setup.pb.h` has 4 deps where ours has 2 (the two GZ
producers for the transitively-imported generated protos); deps are not a
`dumpContentFields` member, so this does not alter the `--pair` content but
keeps the Merkle graph aligned.

Do NOT broaden `SourceInputs` to ordinary `.proto` imports; do NOT add a
`dump.go` PB filter; do NOT touch AR/LD/PY normalization; do NOT change protoc
args or output paths.

(PROTO_LIBRARY, `consumer.proto` imports `gzt/model/model.proto`,
PEERDIR gzt/model). `model.gztproto` imports `gzt/peer/peer.gztproto` and
`gzt/data/data.proto`. Assert `$(B)/gzt/consumer/consumer.pb.h` PB node inputs
include `$(B)/gzt/model/model.proto`, `$(B)/gzt/peer/peer.proto`,
`$(S)/gzt/model/model.gztproto`, `$(S)/gzt/peer/peer.gztproto`,
`$(S)/gzt/data/data.proto`, `$(S)/kernel/gazetteer/proto/base.proto`, and do NOT
invent `$(B)/gzt/peer/peer.gztproto.pb.h`. Fails pre-fix (consumer cannot parse
the build-rooted generated proto's nested imports nor splice its `.gztproto`
leaves).
