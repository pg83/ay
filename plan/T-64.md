# T-64 plan: generic `.gztproto` source generation

## Upstream mechanism

`ymake.core.conf:3324` defines `_SRC("gztproto")`:

```
.CMD=${tool:"dict/gazetteer/converter"} -I$PROTOBUF_INCLUDE_PATH ${pre=-I:_PROTO__INCLUDE} \
     -I$ARCADIA_ROOT ${SRCFLAGS} ${input:SRC} ${norel;output;suf=.proto;nopath;noext:SRC} \
     ${hide;kv:"p GZ"} ${hide;kv:"pc yellow"}
.PEERDIR=kernel/gazetteer/proto
```

So a `.gztproto` source is compiled in **two stages**:

1. **GZ stage (`p=GZ`, `pc=yellow`)** — `dict/gazetteer/converter/gztconverter`
   reads `<base>.gztproto` and writes `$(B)/<moddir>/<base>.proto`. Include
   roots: `-I$(S)/contrib/libs/protobuf/src` (`PROTOBUF_INCLUDE_PATH`), then the
   full `_PROTO__INCLUDE` chain (`-I$(B) -I$(S)` base + each proto namespace),
   then `-I$(S)` (`ARCADIA_ROOT`). The converter rewrites `.gztproto` imports to
   `.proto` and injects `import kernel/gazetteer/proto/base.proto`
   (`dict/gazetteer/converter` declares `INDUCED_DEPS(proto …/base.proto)` +
   `INDUCED_DEPS(h+cpp …/base.pb.h)`).
2. **PB stage** — the generated `$(B)/<base>.proto` runs through the **ordinary**
   protoc path (`emitProtoPB` / `emitPB`, `protoSrcOverride`), producing
   `<base>.pb.{h,cc}`, compiling `<base>.pb.cc.o`, and archiving it into
   `libinit-<name>-proto.a`. The PB node already supports a build-generated
   `.proto` via the codegen-registry `protoSrcOverride` lookup (jsonpath
   RUN_ANTLR precedent), pinning the GZ producer as a dep and riding the GZ
   producer's `$(S)` `.gztproto` sources as `producerSourceInputs`.

Reference GZ + PB + compile nodes dumped from `/home/pg/monorepo/4/sg7.json`
(`search/begemot/rules/init/xml_auth/proto`, `xml_parse_request/proto`) confirm
this shape.

The generated `.proto` is **not on disk** at configure time, but its import
structure equals the `.gztproto`'s imports with `.gztproto`→`.proto` rewriting,
plus the injected `base.proto`. The `.pb.h` it yields therefore `#include`s the
imports' `.pb.h` siblings — so a `.gztproto` import must resolve through the
generated `.pb.h` name (`X.gztproto` → `X.pb.h`), **not** the `.cfgproto.pb.h`
rule.

## Generated-proto parse injection (reuse, not a parallel path)

`emitProtoPB` reads the proto's parse from its **source** path
(`sourceParsedBuckets`, transitive-import walk seed). For the generated proto
that file is absent. Rather than fork `emitProtoPB`, inject the gzt-derived
`ParsedIncludeSet` into the shared parse cache under the generated proto's
**source** VFS (a context-free pre-computed parse — same value for every asker,
so the resolution-cache invariant holds). The set is:

- `parsedIncludesLocal` = `[base.proto] ++ (gztproto imports, .gztproto→.proto)`
- `parsedIncludesHeader`/`Cpp` = each import's `.pb.h` sibling.

The PB node's transitive-import walk then resolves `xml_parse_request.proto` to
the peer's generated `$(B)` output (codegen registry) and `base.proto`/
`extensions.proto`/… to `$(S)`; `descriptor.proto` fans to its 3 contrib
variants exactly as for an ordinary proto. The `.pb.h` `#include` set is derived
(and sorted) by the existing `protoDirectPbHIncludes`.
