# T-151 plan: Cython generated header visible to ordinary SRCS scanning

## Symptom (reproduced)

sg7 pair probe `$(B)/contrib/python/gevent/py3/_/gevent/libev/callbacks.c.pic.o`
differs from upstream only in `inputs`: 788 ref-only inputs, 0 ours-only.
`callbacks.c` is a handwritten `SRCS` C file that `#include "corecext.h"`, where
`corecext.h` is the `CYTHON_C_H` generated header of the same `PY3_LIBRARY`.

The 788 missing = `corecext.c` (Cython main output) + `corecext.h` (generated
header) + the Cython induced "cpp"/"pyx" closure that passes through the header:
libcxx (603), numpy (83), Cython Includes pxds + Utility files + cython.py +
generated_c_headers.h (74), python Include headers (24), and the source
`corecext.pyx` + `libev.pxd` (2). It does NOT include `corecext.c`'s own
`cdef extern` C closure (libev/ev.c, linux-headers, structseq ŌĆö 75 inputs that
`corecext.c.pic.o` carries but the header does not pass through).

## Upstream mechanism

`_BUILDWITH_CYTHON_C_H` (ymake.core.conf) emits the generated `.c` plus
`${hide;addincl;tobindir;output;...h}` ŌĆö the header is an `addincl;output` of the
CY node, carrying `$CYTHON_OUTPUT_INCLUDES`. `TCythonIncludeProcessor`
(cython_processor.cpp) sets `PassInducedIncludesThroughFiles = true`: the induced
"cpp"/"h+cpp"/"pyx" deps attached to the generated header pass through to any file
that `#include`s it. So a handwritten C source including the generated header
receives the generated header, the Cython main output, the pyx/pxd source, and
the Cython induced C/C++ header closure ŌĆö but not the source's own `cdef extern`
C closure (that is resolved at the generated `.c`'s compile, not passed through
the header).

## Root cause in our graph

`emitCythonCpp` runs in `gen.go` AFTER the ordinary `SRCS` are emitted
(non-codegen loop at gen.go:2236, emitCythonCpp at gen.go:2296). Phase 1 already
pre-registers the generated header outputs (for sibling cythonŌåÆcython consumers),
but it registers `corecext.h` with `parsed=nil` and no closure leaves, and it runs
too late: when `callbacks.c` is scanned the header is not yet visible, and even
when visible it carries none of the induced closure for a C consumer.

## Fix

1. Split `emitCythonCpp` into `planCythonCpp` (phase 1: reserve CY refs, register
   header outputs WITH their induced closure, compute per-statement plan data) and
   `emitCythonCppPlanned` (phase 2: emit CY nodes + generated compiles). Call
   `planCythonCpp` in gen.go BEFORE the ordinary SRCS loop; call
   `emitCythonCppPlanned` at the current emitCythonCpp position (AR ordering and
   node emission unchanged).

2. Register each generated header output with:
   - parsed includes = the Cython induced "cpp" closure (CYTHON_OUTPUT_INCLUDES +
     embedded utility files + their transitive closures) as quoted include
     directives ŌĆö walked and cached like the generated `.c`'s own window, so the
     libcxx/numpy/python closures ride as normal cached windows (no leafEver
     pollution of those hot common headers). The Cython source itself is NOT a
     parsed directive (walking it would re-pull the `cdef extern` C closure).
   - closure leaves = the source's pyx-language closure (the `.pyx`/`.pxd` files,
     which `cdef extern` C headers and so must ride bare, not be walked) + the
     CY node's main generated output (`corecext.c`). These few module-specific
     files ride bare into any consumer that includes the header.

3. Refactor `cythonGeneratedOutputInputs` so the cpp-induced singles/closures are
   computed once (`cythonCppInducedSets`) in phase 1 and reused by phase 2 ŌĆö no
   double closure walk (perf neutral).

The generated `.c`'s own compile does not `#include` the public header, so its
inputs are unchanged. CY node command/outputs, archive member ordering, and the
existing cythonŌåÆcython induced-pyx paths are untouched.
