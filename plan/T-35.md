# T-35 plan: model EXPORT_YMAPS_PROTO's SET(PROTO_NAMESPACE maps/doc/proto) output root

## Upstream mechanism (verified in T-34)

`build/internal/conf/project_specific/maps/sproto.conf`:

```conf
macro EXPORT_YMAPS_PROTO() { PROTO_NAMESPACE(maps/doc/proto) }
```

`yatool/build/conf/proto.conf`:

```conf
macro PROTO_NAMESPACE(GLOBAL?..., WITH_GEN?..., Namespace) {
    SET(PROTO_NAMESPACE $Namespace)
    PROTO_ADDINCL(GLOBAL $WITH_GEN $Namespace)
}
```

So `EXPORT_YMAPS_PROTO` is exactly `PROTO_NAMESPACE(maps/doc/proto)` with no GLOBAL.
The three observable effects of `PROTO_NAMESPACE(maps/doc/proto)`:

1. `SET(PROTO_NAMESPACE maps/doc/proto)` — sets the proto output root, driving
   `protoCPPOutRoot(d)` so the module's own protoc commands use `-I=./maps/doc/proto`,
   `-I=$(S)/maps/doc/proto`, `--cpp_out=:$(B)/maps/doc/proto`,
   `--cpp_styleguide_out=:$(B)/maps/doc/proto`, and the import search path roots
   at `$(S)/maps/doc/proto`. **This is the half still missing (T-35).**
2. `PROTO_ADDINCL`'s SOURCE arm `GLOBAL FOR proto $(S)/maps/doc/proto` — transitive
   protoc include (T-30, already in `d.protoAddInclGlobal`).
3. `PROTO_ADDINCL`'s `ADDINCL(GLOBAL $(B)/maps/doc/proto)` — transitive C++ build-root
   include (T-32, already in `d.addInclGlobal`/`d.addInclUserGlobal`).

Our `tokExportYmapsProto` currently applies (2) and (3) but leaves `d.protoNamespace`
nil, so `protoCPPOutRoot(d)` returns "" and the maps modules' own PB producers emit
root-relative roots (`-I=./`, `-I=$(S)/`, `--cpp_out=:$(B)/`).

## Local code already honors d.protoNamespace

- `protoCPPOutRoot(d)` (emit_pb.go) returns the cleaned namespace, "" when nil.
- `composePBArgBlocks` (emit_pb.go) roots `-I=./`, `-I=$(S)/`, `--cpp_out=:$(B)/`,
  `--cpp_styleguide_out=:$(B)/` at `cppOutRoot`.
- `emitProtoPB` prepends `source(cfg.cppOutRoot)` to the proto import search path,
  so `import "yandex/maps/proto/common2/attribution.proto"` resolves through
  `$(S)/maps/doc/proto`.
- gen.go computes `ownProtoTail = source(ns)` (non-GLOBAL) and dedups C++ addincl
  (`dedupVFS(d.addIncl, d.addInclGlobal)` at gen.go:1893), so a build-root include
  added to both `addIncl` and `addInclGlobal` renders exactly once.

The generic command construction is already correct; the only missing thing is that
`EXPORT_YMAPS_PROTO` must express the same namespace state as `PROTO_NAMESPACE`.

## Change

1. Factor the `tokProtoNamespace` body into a shared helper
   `applyProtoNamespace(d, namespace STR, global bool)` so the two macros cannot drift:
   - `d.protoNamespace = strPtr(namespace)`;
   - `d.protoNamespaceGlobal = true` only when `global`;
   - append `build(clean(namespace))` to `d.addIncl`;
   - for GLOBAL or PROTO_LIBRARY, append it to `d.addInclGlobal`/`d.addInclUserGlobal`.
2. `tokProtoNamespace` computes `global` from its args and calls the helper.
3. `tokExportYmapsProto` keeps the T-30 source arm
   (`d.protoAddInclGlobal = append(..., mapsDocProto)`) and then calls
   `applyProtoNamespace(d, mapsDocProtoNS, false)`. The helper subsumes the T-32
   `addInclGlobal`/`addInclUserGlobal` appends (build root = `mapsDocProtoBuild`),
   so those explicit lines are removed (no duplicate tokens).
4. Add interned STR constant `mapsDocProtoNS = internStr("maps/doc/proto")` for the
   raw namespace string (the existing `mapsDocProto`/`mapsDocProtoBuild` are VFS).

Files: `modules.go` (helper + two cases), `vfs_consts.go` (one STR const).

## Latent defect found while implementing

Setting `d.protoNamespace` on the maps modules first exercised the multi-segment
non-GLOBAL arm of `protoCPPModulePath` (emit_pb.go), which returned
`source(filepath.Dir(namespace))` (= `maps/doc`). That shifted the cpp compile
instance path, so the generated object rebased to
`$(B)/maps/doc/_/proto/.../response.pb.cc.o` (spurious `_/`) and `module_dir`
became `maps/doc` — both diverging from the reference
`$(B)/maps/doc/proto/.../response.pb.cc.o`, `module_dir=maps/doc/proto/...`.

Evidence the arm was wrong: object naming co-locates the `.o` with the generated
`.pb.cc` and rebases relative to the OWN module dir. The GLOBAL namespace modules
(googleapis-common-protos, apache/orc-format — both with `namespace != instance.Path`)
already rely on `instance.Path` and match the reference `_/` paths. GLOBAL-ness only
affects include propagation, not object naming. The multi-seg non-GLOBAL arm had no
counterpart in the build until maps got a namespace, so it was untested dead logic.

Fix: `protoCPPModulePath` always equals `instance.Path`, so it was deleted and the
two `cppInstance.Path = protoCPPModulePath(...)` reassignments dropped (net-negative).
Verified: sg2..sg6 stay byte-exact, so the deletion changed nothing for existing
namespace modules.
