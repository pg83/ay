# T-131 plan: EV/PROTO_LIBRARY archive-member order parity

## Upstream mechanism

In a `PROTO_LIBRARY()`, both `.proto` and `.ev` are ordinary `_SRC` macros with
no statement-priority annotation:

- `build/ymake.core.conf:3315` — `_SRC("ev", …)` → `$EVLOG_CMD($SRC)` →
  `CPP_EV_CMDLINE` (proto.conf:473-482), emitting `${output}` `.ev.pb.{cc,h}`.
- `build/ymake.core.conf:3321` — `_SRC("proto", …)` → `$PROTO_CMD($SRC)` →
  `CPP_PROTO_CMDLINE`, emitting `${output}` `.pb.{cc,h}`.

`SRCS()` dispatches each source through `_SRC` in textual declaration order, and
each command's generated `.pb.cc` / `.ev.pb.cc` output is queued back into the
module source list (`module_builder.cpp` `QueueCommandOutputs()` →
`AddSource("SRCS", output)`). The queued generated compiles run in the next FIFO
round **in the order they were queued** — i.e. the SRCS textual order. Because
`.proto` and `.ev` share the same (default) statement priority, their generated
`.pb.cc` objects interleave by SRCS position. So `SRCS(a.proto e.ev b.proto)`
archives `a.pb.cc.o`, `e.ev.pb.cc.o`, `b.pb.cc.o`.

`.gztproto` is a deeper round: `_SRC("gztproto")` first generates a `.proto`,
which is then queued and (next round) generates a `.pb.cc`, whose compile runs
one FIFO round later than direct `.proto`/`.ev` pb.cc compiles. Its objects
therefore sort after all direct proto/ev objects.

## Defect (ours)

`emit_proto.go:emitCPPProtoSrcs` builds `codegenOutputs` in two grouped phases:

1. the `for _, src := range protoSrcs` loop appends every `.proto` (and
   gzt-generated proto) `.pb.cc`;
2. the `if len(evSrcs) > 0` block then appends every `.ev` `.ev.pb.cc`.

`codegenOutputs` order is the archive member order (the proto AR is built by
`emitARNode(... ccOutputs ...)` with no later `reorderARMembers` pass). So every
`.ev.pb.cc.o` lands at the tail after all `.pb.cc.o`, diverging from upstream's
SRCS-interleaved order. Representatives: `$(B)/search/idl/libsearch-idl.a`
(`bolver_events.ev`, `events.ev` appended late) and
`$(B)/extsearch/goods/plutometa/proto/libgoods-plutometa-proto.a`
(`events.ev` appended at tail instead of after `event_log_config.pb.cc.o`).

## Change

Reproduce the FIFO queue order: tag each `codegenOutput` with the declaration
index of its originating source and stable-sort `codegenOutputs` by that index
before the `emitCC` member loop.

- Build `srcDeclIdx[src.string()] = i` over `d.srcs`.
- Add a `declIdx int` field to the local `protoCodegenOutput`.
- Direct `.proto` and `.ev` outputs carry `srcDeclIdx[src]` (their textual SRCS
  position) → interleaved by SRCS order.
- Gzt-generated protos carry a tail key (`len(d.srcs) + gztOrdinal`) so they keep
  their existing after-direct-sources position (deeper FIFO round).
- `sort.SliceStable(codegenOutputs, by declIdx)` just before the emitCC loop.

Within one source, the multiple `orderedCC` members (`.pb.cc`, `.grpc.pb.cc`,
plugin `.cpp`) keep their relative append order (stable sort). No new ordering
concept is introduced — the sort key mirrors ymake's source FIFO position.
