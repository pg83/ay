# T-32 plan: model the C++ build-root half of EXPORT_YMAPS_PROTO's PROTO_ADDINCL

## Upstream mechanism

`/home/pg/monorepo/4/build/conf/proto.conf`:

```
macro PROTO_ADDINCL(GLOBAL?"GLOBAL":"", Path, WITH_GEN?"BUILD":"") {
    _ORDER_ADDINCL($WITH_GEN $GLOBAL FOR proto ${ARCADIA_BUILD_ROOT}/$Path SOURCE $GLOBAL FOR proto ${ARCADIA_ROOT}/$Path)
    ADDINCL($GLOBAL ${ARCADIA_BUILD_ROOT}/$Path)
}
macro PROTO_NAMESPACE(GLOBAL?"GLOBAL":"", WITH_GEN?"WITH_GEN":"", Namespace) {
    SET(PROTO_NAMESPACE $Namespace)
    PROTO_ADDINCL(GLOBAL $WITH_GEN $Namespace)   # literal GLOBAL
}
```

`EXPORT_YMAPS_PROTO()` -> `PROTO_NAMESPACE(maps/doc/proto)` -> `PROTO_ADDINCL(GLOBAL maps/doc/proto)`.

PROTO_ADDINCL has two distinct contributions:

1. `_ORDER_ADDINCL(... FOR proto ...)` — the **protoc include closure** (`_PROTO__INCLUDE`),
   rendered as `-I=$(S|B)/maps/doc/proto` in protoc commands. **T-30 already modeled
   the SOURCE arm** (`protoAddInclGlobal`), closing the `host_info.pb.cc`/`events.ev.pb.cc`
   protoc-node residual.
2. `ADDINCL($GLOBAL ${ARCADIA_BUILD_ROOT}/$Path)` — an **ordinary GLOBAL C++ ADDINCL**
   of the **build root** `$(B)/maps/doc/proto`. This rides the GLOBAL peer addincl
   closure (PropagateTo UserGlobal) and renders as `-I$(B)/maps/doc/proto` in the C++
   compile commands of every transitive consumer — including the generated-protobuf
   compile node `$(B)/extsearch/goods/plutometa/proto/host_info.pb.cc.o`.

T-30's `tokExportYmapsProto` deliberately modeled only contribution (1) and explicitly
left (2) out of scope. T-32 adds contribution (2).

## Our defect

`tokExportYmapsProto` (modules.go) appends only `mapsDocProto` (`source("maps/doc/proto")`)
to `d.protoAddInclGlobal`. It never contributes the build-root GLOBAL C++ ADDINCL, so the
`.pb.cc.o` C++ compile nodes of transitive consumers (representative:
`extsearch/goods/plutometa/proto/host_info.pb.cc.o`) omit `-I$(B)/maps/doc/proto` which
the reference carries — the build-root half of the same upstream PROTO_ADDINCL.

## Change

In `modules.go` `case tokExportYmapsProto`, additionally append the build-root path
`build("maps/doc/proto")` to `d.addInclGlobal` and `d.addInclUserGlobal`, mirroring the
existing `tokProtoNamespace` PROTO_LIBRARY branch (modules.go:1809-1811). `ADDINCL(GLOBAL ...)`
in ymake lands in the module's UserGlobal set and propagates to consumers; adding to both
fields matches the existing PROTO_NAMESPACE modeling and the upstream propagation order.

The build-root VFS is exempt from `filterExistingSourceDirs` (`shouldCheckSourceDir`
returns false for non-source paths), so it survives even though `$(B)/maps/doc/proto`
is not a source dir.

Source-root (`$(S)/maps/doc/proto`) is NOT added to any C++ addincl set — only the
build root — so there is no source-root C++ include leakage. protoc `-I=` rendering
is untouched.

### Files touched

1. `modules.go` — extend `tokExportYmapsProto` with the build-root GLOBAL C++ addincl.
2. `vfs_consts.go` — `ay refac consts` hoists the new `build("maps/doc/proto")` literal.
3. `emit_pb_test.go` — new T-32 regression; relax the T-30 leakage assertion to forbid
