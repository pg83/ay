# T-120 plan: cross-namespace generated `.pb.h` direct-import rooting

## Reproduced divergence (fact, not hypothesis)

Representative pair `$(B)/ads/bsyeti/libs/scatter/client.cpp.o`. Ref has three
ref-only CC inputs; ours does not:

```
+$(B)/yt/yt_proto/yt/orm/client/proto/object.pb.h
+$(B)/yt/yt_proto/yt/orm/data_model/controls.pb.h
+$(B)/yt/yt_proto/yt/orm/data_model/finalizers.pb.h
```

Crucially, our node DOES carry the source-`.proto` leaves
(`$(S)/.../object.proto`, `controls.proto`, `finalizers.proto`) — they ride as
the transitive-import `.proto` directives (emit_proto.go:419-432) — but NOT the
generated `.pb.h`. So the generated-from leaves arrive while the `.pb.h` include
chain is broken.

## Root cause (upstream mechanism)

The only path to these three YT-ORM (`PROTO_NAMESPACE(yt)`) headers in this
target is through `yp/yp_proto/yp/client/api/proto/data_model.proto`
(`PROTO_NAMESPACE(yp)`), which directly imports:

```
import "yt_proto/yt/orm/client/proto/object.proto";
import "yt_proto/yt/orm/data_model/controls.proto";
import "yt_proto/yt/orm/data_model/finalizers.proto";
```

A generated `<proto>.pb.h` registers its direct imports' `.pb.h` as parsed
includes via `protoDirectPbHIncludes` → `protoPbHIncludes`, which roots every
import header by **prefixing the IMPORTER's own `PROTO_NAMESPACE`**
(`protoOutputRel(cppOutRoot, …)`). protoc actually writes `#include
"<import>.pb.h"` verbatim and ymake resolves it against the GLOBAL
`PROTO_NAMESPACE` addincl of the *imported* library, so a generated header lives
under the **imported** proto's namespace root, not the importer's. For a
**same-namespace** import the two coincide and the directive is correct; for a
**cross-namespace** import (`yp` proto importing `yt_proto/…`) the directive is
mis-rooted to `yp/yt_proto/…orm/…object.pb.h`, which is not a registered output.

This mis-rooted directive only resolves when some unrelated header chain pulls
the same `.pb.h` (its correctly-rooted sibling chain). The YT-core headers
(`error.pb.h`, `attributes.pb.h`, …) survive that way. The YT-ORM headers are
reached ONLY through the cross-namespace `data_model`/`conditions` imports, so
they are lost.

## Fix

Root each direct-import `.pb.h` at the imported proto's ACTUAL location, found
through the same `-I` roots protoc/our proto scan already use
(`resolveProtoImportPath`), instead of blindly prefixing the importer's
`cppOutRoot`. The resulting directive is the full generated-output rel, so the
scanner's build-includer `lookupSTR` binds it **context-free** (no consumer
addincl involved — the resolution-cache invariant is preserved). For a
same-namespace import the resolved root equals `cppOutRoot`, so existing
(byte-exact) behavior is unchanged; cross-namespace imports are fixed.

Scope: PB direct-import path only (emit_proto.go:392). The EV/cfgproto/sproto/pr
callers of `protoDirectPbHIncludes` keep their current behavior (those buckets
are reserved for later tickets per T-113). Extract the import→induced-header
extension rule shared with the proto parser to avoid duplication.

## Files touched

- `parsers.go`: use `protoImportInducedHeader` in `parseProtoDirectiveSet`.
