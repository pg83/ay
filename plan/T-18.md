# T-18 plan: Cython pxd side-input closure for CYTHONIZE_PY .py sources

## Upstream mechanism (verified)

`build/plugins/pybuild.py::onpy_srcs`, in the cython emit loop, computes a
per-source `Dep` and passes it to the `_BUILDWITH_CYTHON_{C,CPP}{,_H,_API_H}`
macro as the second positional argument:

```python
dep = path
if path.endswith('.py'):
    pxd = '/'.join(mod.split('.')) + '.pxd'
    if unit.resolve_arc_path(pxd):
        dep = pxd
cython_args.append(dep)
```

`mod` is the python module name (the `=<mod>` override or the path-derived
name). For a `.py` source whose module has a sibling `<mod-as-path>.pxd`, `dep`
becomes that pxd; otherwise `dep == path` (the source itself).

The macros consume `Dep` as `${hide;input:Dep}` (ymake.core.conf:4027/4063/…):
a **hidden input** — it rides the CY node's input set but never appears on the
command line. ymake then scans that input with the registered `.pxd` parser
(cimport / `from … cimport` / `include` / `cdef extern from`) and pulls its
transitive source closure onto the CY node as further inputs.

`unit.resolve_arc_path(pxd)` is `ResolveSourcePath(pxd, Module.GetDir())`
(devtools/ymake/module_wrapper.cpp): resolves relative to the module dir, then
SRCDIRs, then arcadia root — returns empty when the file does not exist.

When `dep == path` (`.pyx` sources, or `.py` with no pxd) the hidden input is
the source, already an input, so the dedup makes it a no-op. Only a
CYTHONIZE_PY `.py` source with a resolving pxd adds anything — exactly this
ticket's scope.

### Representative (gevent)

`contrib/python/gevent/py3/ya.make`:
`PY_SRCS(TOP_LEVEL … CYTHON_C … CYTHONIZE_PY
gevent/_abstract_linkable.py=gevent._gevent_c_abstract_linkable …)`.
`mod = gevent._gevent_c_abstract_linkable` →
`pxd = gevent/_gevent_c_abstract_linkable.pxd`, resolving under the module dir
to `$(S)/contrib/python/gevent/py3/gevent/_gevent_c_abstract_linkable.pxd`.

That pxd `from gevent._gevent_c_greenlet_primitives cimport …`,
`from gevent._gevent_c_hub_local cimport …` and
`cdef extern from "greenlet/greenlet.h"`. Resolved through the cython scan
addincl (`FOR cython contrib/python/gevent/py3`, regular ADDINCL
`contrib/python/greenlet/py3`), the closure is exactly the four inputs the CY
node currently lacks (confirmed by `dump diff --pair`):

```
+$(S)/contrib/python/gevent/py3/gevent/_gevent_c_abstract_linkable.pxd
+$(S)/contrib/python/gevent/py3/gevent/_gevent_c_greenlet_primitives.pxd
+$(S)/contrib/python/gevent/py3/gevent/_gevent_c_hub_local.pxd
+$(S)/contrib/python/greenlet/py3/greenlet/greenlet.h
```

## Files to touch

- `modules.go`: in the CYTHONIZE_PY `.py` branch set a new
  `CythonStmt.Pxd` = `<modName-as-path>.pxd` (module-relative candidate).
- `emit_cython_cpp.go`: when `stmt.Pxd != ""`, resolve it against the module's
  source dirs (module dir → SRCDIRs → arcadia root); if it exists, walk its
  include closure with the existing cython scan cfg and merge the pxd + closure
  into the CY node's input set (toolInputs) deduped. No command-line change
  (`${hide}`), no effect on the generated `.c`'s own CC closure.
- `emit_cython_cpp_test.go`: regression test — a CYTHONIZE_PY `.py` with a
  matching module `.pxd` that cimports a second pxd and `cdef extern from` a
  header; assert the CY node inputs contain the pxd and its source closure
  exactly once, and that a `.py` with no pxd adds nothing.

## Expected effect on the gate

- sg2..sg6 stay byte-exact (no CYTHONIZE_PY-with-pxd pattern there).
- sg7: the `_abstract_linkable.py.c` pair stops reporting the four pxd-side
  inputs as ref-only; `matched` rises, `ref_only` falls. The `.py.c.pic.o`
  consumer stays paired (input set change does not alter CY output names). lxml
  `etree.c/.h/_api.h` outputs unaffected (those are `.pyx`, `dep == path`).
