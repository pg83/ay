# T-14 plan: brotli/snappy third-party include placement in yabs proto contour

## Symptom

`./dev/validate.py` sg7 reports a cmds-only diff on
`$(B)/yabs/server/libs/aggregated_ctv_socdem_profile/_/enums/socdem_type.h_serialized.cpp.o`
(`cmd_args multiset matches`, only order differs):

ours:  `… memfd_create, protobuf/src, abseil-tstring, abseil, brotli/c/include, snappy/include, re2 …`
ref:   `… memfd_create, brotli/c/include, snappy/include, protobuf/src, abseil-tstring, abseil, re2 …`

i.e. `-I$(S)/contrib/libs/brotli/c/include` and `-I$(S)/contrib/libs/snappy/include`
must precede `-I$(S)/contrib/libs/protobuf/src` and the abseil dirs.

The same signature appears at the producing PROTO_LIBRARY. For
`$(B)/yabs/proto/content_tags.pb.cc.o`:

ours:  `protobuf, abseil-tstring, abseil, brotli, snappy, re2, grpc`
ref:   `brotli, snappy, protobuf, abseil-tstring, abseil, re2, grpc`

socdem (a plain LIBRARY) inherits the order from peer `yabs/proto` via the
transitive `AddInclGlobal` (gen.go:1073 `dedupVFS(d.addInclGlobal,
peerContribs.addIncl)`), so fixing the PROTO_LIBRARY propagates downstream.

## Root cause (upstream mechanism)

The `-I` block order is the first-occurrence dedup of the transitive GLOBAL
ADDINCL closure, walked in peer order.

`yabs/proto` is a `PROTO_LIBRARY` with `CPP_EVLOG()`. Upstream's per-source proto
command for EVLOG (`build/conf/proto.conf` `_CPP_PROTO_EVLOG_CMD`) carries
`.PEERDIR=library/cpp/eventlog contrib/libs/protobuf`, inducing the plugin
runtime `library/cpp/eventlog` **ahead of** the declared `PEERDIR()` for
include-dir propagation. `library/cpp/eventlog` PEERDIRs `library/cpp/blockcodecs`
first (→ `codecs/brotli`, `codecs/snappy`, each `ADDINCL(GLOBAL …)`), then
`library/cpp/eventlog/proto` (→ protobuf, abseil) and `contrib/libs/re2`. So
eventlog's GLOBAL ADDINCL subtree is exactly `[brotli, snappy, protobuf,
abseil-tstring, abseil, re2]`, and leading with it reproduces ref byte-for-byte.

Our code registers `event2cpp DEPS library/cpp/eventlog` via `addCPPProtoPlugin`
(append to `d.peerdirs`) and protobuf via the `hasProto` line — both AFTER the
declared `PEERDIR()` block. Walking declared proto peers first reaches protobuf
(through their own closures) before eventlog, so brotli/snappy land late.

Contrast with a grpc PROTO_LIBRARY (`_CPP_PROTO_CMD .PEERDIR=contrib/libs/protobuf`
+ `GRPC()` → plugin DEPS `contrib/libs/grpc`): there ref leads with `grpc` and
trails with `protobuf`, and our baseline already matches. The base proto runtime
`contrib/libs/protobuf` must therefore NOT be hoisted — only the plugin DEPS
(plugin-runtime peers) lead. An earlier experiment that hoisted protobuf as well
regressed sg3/sg5/sg6 (grpc protos got protobuf-first).

## Change

ADDINCL-order only; declared `d.peerdirs` order still drives archive/global/link
closure (reordering it broke sg3/sg5/sg6 — proto runtimes link late, not early).

`modules.go`:

- `protoCmdPeers(d)`: the C++ proto plugin DEPS (plugin-runtime peers), in plugin
  order, deduped. NO protobuf. Stored on `ModuleData.protoCmdPeers` for
  `PROTO_LIBRARY` modules; `d.peerdirs` untouched.

`gen.go` (`walkPeersForGlobalAddIncl`):

- Track `firstPeerdirIdx` (start of declared-PEERDIR region in `resolved`) and a
  `peerdirFront` mask of which declared peers are `protoCmdPeers`.
- The ADDINCL pass alone iterates prefix (defaults/cppSelf/googleapis) → front
  proto-cmd peers → remaining declared peers. Every other pass (archive, global,
  proto-include, flags) keeps the original `resolved` order.

Generic: keys only off the proto plugin DEPS the module already records — no
brotli/snappy/eventlog special-casing, no per-input reordering.
