# T-14 — Duplicate PB producer for `ad_group_additional_targeting.pb.h`

## Symptom

Direct sg7 (`grut/libs/proto/public/auxiliary`, a `PROTO_LIBRARY`) aborts with:

```
CodegenRegistry: duplicate producer for $(B)/grut/libs/proto/public/auxiliary/ad_group_additional_targeting.pb.h (existing kind=PB, new kind=PB)
```

## Root cause (the real mechanism, not the ticket's hypothesis)

The ticket hypothesised LIST_PROTO + SRCS both declaring the same proto. That is
NOT the mechanism. `LIST_PROTO` is an acknowledged no-op macro (`UnknownStmt` →
`d.unhandledMacros`); it never adds to `d.srcs`.

The actual defect is **non-idempotent `SET_APPEND` across our probe / re-collect
double pass for `PROTO_LIBRARY`**:

`genModule` (gen.go) collects a `PROTO_LIBRARY` twice:

1. `env := buildIfEnv(instance)` then `collectModule(..., env)` — a *probe* to
   discover `moduleStmt.Name`. `collectModule` mutates `env` in place; the
   module's `SET_APPEND(PROTO_FILES …)` sets `PROTO_FILES = "<84 files>"`.
2. For a cpp `PROTO_LIBRARY`: `cppProtoEnv := env.clone()` — clones the
   **already-mutated** probe env (PROTO_FILES = 84 entries) — then
   `collectModule(..., cppProtoEnv)` re-runs every statement. `SET_APPEND`
   semantics are `VAR = "$VAR x"`, so it appends the 84 files *again*:
   `PROTO_FILES` becomes 168 entries (each proto twice).

`SRCS(${PROTO_FILES})` in the second pass therefore expands to 168 = 84×2, so
`emitCPPProtoSrcs` loops the same proto twice and calls `emitProtoPB` →
`register(pb.h)` twice → duplicate-producer abort.

Verified by instrumentation: two `SRCS` collections for the module, expanding to
84 then 168; `protoSrcs` had every proto exactly twice; `d.srcs` length 168.

Upstream ymake evaluates each module's statements once per module env, so
`SET_APPEND(PROTO_FILES)` is applied once → 84. Our specialization re-collect
must start from a clean module base env, not from the probe-mutated env's clone.

## Fix

In `genModule`, derive the `PROTO_LIBRARY` cpp/py re-collection env from a fresh
`buildIfEnv(instance)` (the pristine module base, identical to the probe's
starting state) instead of `env.clone()` (the probe-mutated env). This isolates
the specialized collection from the probe's accumulated mutations, so
`SET_APPEND` runs exactly once in the pass whose `d` we keep.

- `buildIfEnv(instance)` is deterministic and reproduces the exact pre-probe
  base; `collectModule` re-runs all statements, so nothing legitimate is lost.
- Runs only for `PROTO_LIBRARY` modules (small fraction); zero added work for
  every other module — no hot-path regression.

## Files touched

- `gen.go` — the two `PROTO_LIBRARY` re-collect branches: `env.clone()` →
  `buildIfEnv(instance)`.

## Test

`emit_proto_test.go` — new regression: a `PROTO_LIBRARY` with
`SET_APPEND(PROTO_FILES foo.proto)` + `SRCS(${PROTO_FILES})` +
`LIST_PROTO(${PROTO_FILES})`. Before the fix `testGen` panics with the
duplicate-producer error; after, the graph has exactly one PB producer for
`foo.pb.h` (and `foo.pb.cc`) and does not panic.
