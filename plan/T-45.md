# T-45 plan: yt/yt_proto DESC_PROTO output-root naming and include-span parity

This ticket implements the **T-39A residual only**: the DESC_PROTO descriptor
producer's `.proto.desc` output root for nested `PROTO_NAMESPACE(yt)` sources and
the missing `_PROTO__INCLUDE` span on the descriptor protoc command. Scope is
exactly the descriptor producer shape; `.rawproto` paths/md5 stems and the
T-37/T-42 builtin `.desc` roots stay untouched.

## Upstream mechanism

`build/conf/proto.conf:711` `_PROTO_DESC_RAWPROTO_CMD`:

```
... desc_rawproto_wrapper.py
    --desc-output ${output;suf=.desc:File}
    --rawproto-output ${norel;output;suf=.${_MODDIR_HASH}.rawproto:File}
    --proto-file ${rootrel;input:File}
 -- $PROTOC -I=./$PROTO_NAMESPACE -I=$ARCADIA_ROOT/$PROTO_NAMESPACE
    ${pre=-I=:_PROTO__INCLUDE} -I=$ARCADIA_BUILD_ROOT -I=$PROTOBUF_INCLUDE_PATH
    --include_source_info ...
```

Two facts drive the two defects:

1. **`${output;suf=.desc:File}` vs `${norel;output;...rawproto}`.** The desc
   output goes through ymake's output-name policy (the `_/` rebasing for a
   source that lives in a subdirectory of the declaring module), while the
   rawproto output carries `norel`, which suppresses that rebasing and keeps the
   natural source path. This is exactly the policy our `composeSrcDirOutputRel`
   (emit_cc.go) already reproduces for object outputs: a module-local source
   with a `/` in its module-relative path maps to `<module>/_/<subpath>`; a flat
   source keeps `<module>/<src>`. Hence:
   - `yt/yt_proto/yt/client` + `hive/proto/cluster_directory.proto`
     -> `$(B)/yt/yt_proto/yt/client/_/hive/proto/cluster_directory.proto.desc`
   - `yt/yt_proto/yt/orm/api` + `access_control_service.proto` (flat)
     -> `$(B)/yt/yt_proto/yt/orm/api/access_control_service.proto.desc` (no `_/`)

2. **The descriptor command uses the SAME `_PROTO__INCLUDE` span** as the
   cpp/py/go proto commands, but our `emitDescProtoSubmodule` currently passes
   `nil` and renders only `-I=./<ns> -I=$(S)/<ns> -I=$(B)
   -I=$(S)/contrib/libs/protobuf/src --include_source_info`. The reference (sg7,
   `yt/yt_proto/yt/client` descriptor) is:

   ```
   -I=./yt -I=$(S)/yt
   -I=$(B) -I=$(S) -I=$(S)/yt -I=$(S)/contrib/libs/protoc/src
   -I=$(S)/contrib/libs/protobuf/src -I=$(S)/yt
   -I=$(B) -I=$(S)/contrib/libs/protobuf/src --include_source_info
   ```

   The middle band is `_PROTO__INCLUDE`, structurally identical to what
   `composePBArgBlocks` emits: `-I=$(B) -I=$(S)` then `-I=$(S)/<cppOutRoot>`,
   then the DESC peer closure's proto addincl set (here protos_from_protoc ->
   `contrib/libs/protoc/src`, protos_from_protobuf -> `contrib/libs/protobuf/src`
   via NEED_GOOGLE_PROTO_PEERDIRS), then the bare-namespace tail from the
   declared proto peer `yt/yt_proto/yt/core` -> `$(S)/yt`.

   The DESC peer closure differs from the CPP one (it carries the
   NEED_GOOGLE_PROTO_PEERDIRS builtins protos_from_protoc/protos_from_protobuf),
   which is why protoc/src appears here but not on the cpp command.

## Files touched

- `emit_proto_desc.go`
  - new `descProtoDescOutput(moduleDir, protoRelPath)` helper: reuse
    `composeSrcDirOutputRel` for a module-local source, keep `build(rel+".desc")`
    for SRCDIR/non-local sources (so T-37/T-42 builtin `.desc` roots do NOT move).
  - `descPeerClosure` additionally aggregates each entered peer's transitive
    `ProtoAddInclGlobal` / `ProtoNamespaceTail` (read from the genModule result
    already computed — no new genModule call, no memo poisoning).
  - `emitDescProtoSubmodule` computes its own proto-namespace contribution
    (mirroring gen.go's GLOBAL-vs-tail split), threads the DESC peer span into
    `descProtocIncludes`, and returns populated `ProtoAddInclGlobal` /
    `ProtoNamespaceTail` so a parent DESC submodule aggregates transitively.
  - `descProtocIncludes(peerProtoAddIncl, cppOutRoot)` rebuilt to mirror
    `composePBArgBlocks`'s `_PROTO__INCLUDE` band: `-I=./<r> -I=$(S)/<r> -I=$(B)
    -I=$(S) [-I=$(S)/<r>] <peer span...> -I=$(B)
    -I=$(S)/contrib/libs/protobuf/src --include_source_info`.
- `emit_proto_desc_test.go`
  - nested `PROTO_NAMESPACE(yt)` source: `_/`-rebased desc output, rawproto
    unchanged, ordered include span including the peer `-I=$(S)/yt` tail.
  - root-level `PROTO_NAMESPACE(yt)` source: no `_/` segment.
  - existing builtin/no-namespace case (`TestEmitProtoDescriptions_PDProducerShape`)
    stays green (flat module-local source, no `_/`).

## Expected gate effect

- sg7: the three T-39 descriptor pair commands lose the reference-only include
  tokens (`-I=$(B)`, `-I=$(S)`, `-I=$(S)/yt`, `-I=$(S)/contrib/libs/protoc/src`,
  `-I=$(S)/contrib/libs/protobuf/src`); nested `yt/client` and `yt/core`
  `.proto.desc` left/right entries pair up; rawproto filenames unchanged.
  matched count rises, no unrelated residual grows.
- sg2..sg6 stay byte-exact: the desc output helper only rebases module-local
  nested sources (none of the gating descriptor cases use a nested
  module-local proto under a PROTO_NAMESPACE), and the include span change adds
  tokens that match upstream `_PROTO__INCLUDE` for those modules too.
