# T-45 plan: config-proto (.cfgproto) source induces codegen+protos+protobuf peers

## Divergence (reproduced)

`./dev/validate.py` sg7, target `yabs/server/daemons/bs_static`, the linked
PROGRAM `yabs-server` (`p=LD`). The `proto_config` archive members of the LD
link closure differ:

```
REF  : .../proto_config/defaults  types_nora  proto_config/codegen  proto_config  proto_config/protos
OURS : .../proto_config/defaults  types_nora                        proto_config  proto_config/protos
```

`library/cpp/proto_config/codegen/libcpp-proto_config-codegen.a` is **ref-only**
(missing in ours); the `library/cpp/proto_config/protos` archive sits in the
wrong global position because its induced sibling `codegen` is absent. This is
the residue T-42 reported (messages.txt: "_CPP_CFGPROTO_CMD .PEERDIR=…codegen
…protos induced-peer-leads class, not yet modeled").

## Upstream mechanism

`balancer/kernel/client_request` is a plain `LIBRARY()` whose `SRCS` include
`backend_config.cfgproto`. The `.cfgproto` extension is handled by
`_SRC("cfgproto")` (ymake.core.conf:3335) → `_CPP_CFGPROTO_CMD($SRC)`
(proto.conf:494), whose body carries:

```
.PEERDIR=library/cpp/proto_config/codegen library/cpp/proto_config/protos contrib/libs/protobuf
```

So any module with a `.cfgproto` source induces those three peers into its
PEERDIR closure (exactly as a `.ev` source induces `library/cpp/eventlog
contrib/libs/protobuf` via `_CPP_EVLOG_CMD .PEERDIR`). `protos` is also a plain
declared peerdir of many other modules (so it already reaches the closure in
ours), but `codegen` is induced ONLY through the cfgproto command, hence its
absence in ours.

We do not model `.cfgproto` at all today: `srcExtClassOf` returns
`srcExtRegular`, `emitOneSource` falls through to the `WarnUnsupportedSource`
branch, and `collectModule` adds no induced peerdir — so `codegen` never enters
the link closure.

## Change (mirror the `.ev` induced-peer path, no allowlist)

1. `str_intern.go`: add `srcExtCfgProto` class; `classifySrcExt` returns it for
   `.cfgproto`.
2. `modules.go` `collectModule`: in the per-source extension pass detect
   `hasCfgProto`; when set, append
   `library/cpp/proto_config/codegen`, `library/cpp/proto_config/protos`,
   `contrib/libs/protobuf` to `d.peerdirs` (the `_CPP_CFGPROTO_CMD .PEERDIR`
   order), mirroring the existing `hasEv`/`hasSc` arms. New interned consts for
   the two proto_config paths.
3. `emit_sources.go` `emitOneSource`: recognize `.cfgproto` and return `nil`
   without the unsupported-source warning. The `.cfgproto.pb.cc`/`.pb.h`
   codegen-output emission (and the cfgproto object in the owning module's own
   archive) is a separate node-count class, explicitly out of this ticket's
   scope (induced-peer reachability/ordering only).

This is the generic config-proto induced-peer mechanism: keyed on the
`.cfgproto` source extension, not on any module path.
