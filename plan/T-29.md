# T-29 plan: generated-code input closure for BASE_CODEGEN / SPLIT_CODEGEN consumers

## Upstream mechanism

`build/internal/conf/codegen.conf`:

```
macro BASE_CODEGEN(Tool, Prefix, Opts...) {
    .CMD=${tool:Tool} ${input;suf=.in:Prefix} ${noauto;output;suf=.cpp;nopath:Prefix} ${output;suf=.h;nopath:Prefix} $Opts ...
}
macro _SPLIT_CODEGEN_BASE(Tool, Prefix, OUT[], OPTS[], OUTPUT_INCLUDES[]) {
    .CMD=${tool:Tool} ${input;suf=.in:Prefix} ${hide;output:OUT} ${noauto;output;suf=.cpp;nopath:Prefix} ${output;suf=.h;nopath:Prefix} $OPTS ${hide;output_include:OUTPUT_INCLUDES} ...
}
```

Both producers take the `${prefix}.in` source as the single command input and emit
`${prefix}.cpp` + `${prefix}.h` (BC) / the numbered parts + `${prefix}.cpp` +
`${prefix}.h` (SC). ymake's flat-input model lists the producer-command source
closure on every CC node that compiles, or transitively includes, a generated
output. Concretely, in the post-T-22 sg7 reference:

* The generated header carries its generated-from sibling `.cpp` and the
  `${prefix}.in` source as bare closure members (a "generated-from" edge, not a
  C++ include). For SC the sibling is `${prefix}.0.cpp` (the first part); for BC
  it is `${prefix}.cpp`.
* The generated `.cpp` parts and the noauto `${prefix}.cpp` carry
  `${prefix}.0.cpp` + `${prefix}.in` in closure, and do NOT include
  `${prefix}.h`.

This is the same "non-expanded closure leaf" model we already use for
`COPY_FILE(TEXT)` and PB `.proto` generated-from edges (`GeneratedFileInfo.ClosureLeaves`,
spliced by the scanner for every `$(B)` window member). No new data path is
required â€” only correct registration of the existing closure-leaf / parsed-include
fields.
