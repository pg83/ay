# T-29 plan: proto source-root prefix for LIBRARY-hosted .proto sources

## Symptom (reproduced)

`$(B)/yt/yt/library/query/proto/query.pb.cc` protoc command differs from upstream:

- ours: `--cpp_out=:$(B)/`, `--cpp_styleguide_out=:$(B)/`, `-I=$(S)/`, `-I=./`
- ref:  `--cpp_out=:$(B)/yt`, `--cpp_styleguide_out=:$(B)/yt`, `-I=$(S)/yt`, `-I=./yt`

Additionally the import set: ours emits `-I=$(S)/yt` once, ref emits it three times
(explicit `-I=$ARCADIA_ROOT/$PROTO_NAMESPACE` prefix + two `_PROTO__INCLUDE` copies:
own PROTO_NAMESPACE + the peer `yt/yt/client`'s GLOBAL `FOR proto $(S)/yt`).

## Root cause

`yt/yt/library/query/proto` is a plain `LIBRARY()` carrying `PROTO_NAMESPACE(yt)` and
`SRCS(*.proto)`. Two code paths compile `.proto`:

- PROTO_LIBRARY / header-only path → `emitCPPProtoSrcs` builds
  `ProtoPBConfig{cppOutRoot: protoCPPOutRoot(d), duplicateOutputRootInclude: ...}`.
- plain LIBRARY (AR) path → `emitOneSource` → `emitLibraryProtoSource`
  (emit_proto.go:756), which passes an **empty** `ProtoPBConfig{}`, so
  `cppOutRoot=""` and `duplicateOutputRootInclude=false`.

A `.proto` must compile identically regardless of host module type; upstream's
`_CPP_PROTO_CMDLINE_BASE` (ymake.core.conf:612) roots `--cpp_out`, the styleguide
out, `-I=./` and `-I=$ARCADIA_ROOT/` at `$PROTO_NAMESPACE` for both. The empty
config is the defect.

## Upstream mechanism

`_CPP_PROTO_CMDLINE_BASE` = `$PROTOC -I=./$PROTO_NAMESPACE -I=$ARCADIA_ROOT/$PROTO_NAMESPACE
${pre=-I=:_PROTO__INCLUDE} -I=$ARCADIA_BUILD_ROOT -I=$PROTOBUF_INCLUDE_PATH
--cpp_out=${CPP_PROTO_PLUGINS}:$ARCADIA_BUILD_ROOT/$PROTO_NAMESPACE ... $PROTOC_STYLEGUIDE_OUT`.
`PROTO_NAMESPACE` is set by the `PROTO_NAMESPACE` macro for any module type (LIBRARY
or PROTO_LIBRARY). The double `_PROTO__INCLUDE` namespace copy arises when a peer also
declares the same namespace (`PROTO_NAMESPACE` always emits a GLOBAL `FOR proto`
source addincl that propagates) — our model collapses own+peer in `effectiveProtoInclude`
and re-expands the second copy via `duplicateOutputRootInclude`.

## Fix

`emitLibraryProtoSource` must compute the same config the PROTO_LIBRARY path does:

- `cppOutRoot = protoCPPOutRoot(d)`
- `duplicateOutputRootInclude` = a peer contributes the module's own namespace.

The peer-only proto-namespace signal is not present in `in ModuleCCInputs`
(`in.ProtoInclude` already unions own+peer). Compute it in gen.go's general module
path (where `peerProtoInclude` is in scope) and thread it via a new
`ModuleCCInputs.ProtoOwnNamespaceInPeers bool`, then consume it in
`emitLibraryProtoSource`.

Do NOT set `moduleTag` (the reference LIBRARY proto node has no module tag — the
pair already matches on target_properties; adding `tagCppProto` would regress it).
Leave `emitCPPProtoSrcs` untouched (PROTO_LIBRARY nodes already match in sg7).

## Files

- `emit_proto.go`: `emitLibraryProtoSource` — build proper `ProtoPBConfig`.
- `gen.go`: compute `protoOwnNamespaceInPeers` from `peerProtoInclude`; set on
  `moduleInputs`.
- `emit_cc.go`: add `ProtoOwnNamespaceInPeers bool` to `ModuleCCInputs`.
- `emit_pb_test.go`: regression — a LIBRARY+PROTO_NAMESPACE(yt) peering a
  LIBRARY+PROTO_NAMESPACE(yt) yields the rooted command with `-I=$(S)/yt` ×3;
  a top-level LIBRARY proto (no namespace) keeps `$(B)/`/`$(S)/`/`./` roots.

## Expected gate effect

- query.pb.cc/query.pb.h pair loses the four root-prefix cmd diffs and matches.
- sg2..sg6 stay byte-exact; sg7 matched increases; no new validate failure.
