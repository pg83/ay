## Root cause (verified by reading, not assumed)

Two emit paths exist for proto-family sources:

1. Specialized modules (`PROTO_LIBRARY`/`DLL`, `isSpecializedLibraryType`) — the
   header-only path at `gen.go:1125` calls `emitProtoSrcs` → `emitCPPProtoSrcs`,
   which fully handles `.gztproto` (GZ producer → `emitProtoPB` → compile →
   archive). This is what the existing `TestGen_GztProto_*` (PROTO_LIBRARY) covers.

2. Regular modules (`LIBRARY`, `PROGRAM`) — the SRCS loop at `gen.go:2089+`. Each
   source dispatches through `emitOneSource` (`emit_sources.go`). `.proto` is a
   first-class case there (`emitLibraryProtoSource`). `.gztproto` is **not**: it is
   absent from both `isCodegenProducingSrc` (`gen.go:3376`) and the `emitOneSource`
   switch (`emit_sources.go:27`), so it falls through to the
   `WarnUnsupportedSource` branch and is skipped entirely. `emitProtoSrcs` is never
   called for a `LIBRARY`, so nothing emits the synset chain.

Upstream `_SRC("gztproto")` (ymake.core.conf:3324) is a per-source macro, not a
module-type feature — it applies to any module with a `.gztproto` source. Our
specialized path already reproduces it; the regular path simply never wired it in.

## Change

Mirror the `.proto` treatment for `.gztproto` in the regular-module path. The SRCS
two-pass split in `genModule` gates on the **interned-id** classifier
`isCodegenProducingSrcID` (`str_intern.go`), via the memoized `srcExtClassOf`, NOT
the string `isCodegenProducingSrc` (`gen.go`). `classifySrcExt` mapped `.gztproto`
to `srcExtRegular`, so:

- Pass 1 (`gen.go:2138`, `if !isCodegenProducingSrcID(src) continue`) skipped it —
  the generated `.pb.h` was never registered in the codegen registry before sibling
  header closures were walked.
- Pass 2 (`gen.go:2234`, `if isCodegenProducingSrcID(src) continue`) emitted it as a
  NON-codegen object (`appendCC(..., generated=false)`), so `sortKey` archived its
  `.pb.cc.o` in SRCS declaration order instead of after the direct compiles.

For `kernel/synnorm` (`SRCS(synnorm.cpp synset.gztproto)`) the chain still paired by
coincidence — the `.gztproto` sits last, so its object naturally trailed
`synnorm.cpp.o`. With the `.gztproto` declared first the ordering inverts (codegen
object before direct object), which is the generic divergence from upstream.

1. `str_intern.go`: add a dedicated `srcExtGztProto` class, classify `.gztproto`
   into it, and include it in `isCodegenProducingSrcID`. A separate class (not
   reusing `srcExtProto`) keeps `.gztproto` out of `modules.go`'s `hasProto`
   protobuf-autopeer arm, so emission stays identical except for the codegen
   ordering. This is the path the SRCS pass actually consumes — pre-emit in pass 1
   (registry populated before header walks), archive the `.o` in the pass-2 codegen
   section (`Generated=true` → `sortKey` orders it after direct compiles).

2. `gen.go` `isCodegenProducingSrc` (string twin, used by `emit_cython_cpp.go`):
   keep `.gztproto` for parity with the id classifier.

3. `emit_sources.go` `emitOneSource`: add a `.gztproto` case dispatching to a new
   thin wrapper `emitLibraryGztProtoSource`-for-AR that:
   - runs the existing `emitLibraryGztProtoSource` (GZ producer + generated `.proto`
     registration), then
   - delegates the generated `.proto` to the existing `emitLibraryProtoSource`
     (PB node + `.pb.cc` compile → archived `SourceEmit`).

   This reuses both existing emitters; no new node-construction code, no new
   information-passing channel. The generated `$(B)/<base>.proto` is picked up by
   `emitProtoPB`'s existing codegen-registry `protoSrcOverride` lookup.

Net: ~1 line in `isCodegenProducingSrc`, ~2 lines in `emitOneSource`, one small
wrapper function. The specialized-module gztproto path is untouched.
