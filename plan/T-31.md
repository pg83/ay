# T-31 plan: attribute the BUNDLE MOVE_FILE fs_tools.py input to the consuming linkable module node

## Observed divergence (reproduced)

sg7, target `yabs/server/daemons/bs_static`, node
`$(B)/ads/argus/proto/profile/generated/codegen/codegen` (a C++ `PROGRAM` LD node
that declares `BUNDLE(... NAME proto.desc)` + `RESOURCE(proto.desc ...)`).

Reference (upstream) LD node `$(S)` script inputs include:

```
$(S)/build/scripts/fs_tools.py        <- BUNDLE MOVE_FILE
$(S)/build/scripts/objcopy.py         <- RESOURCE objcopy   (out of scope, broader sg7 diff)
$(S)/build/scripts/process_command_files.py
```

Ours lists only `process_command_files.py`. `fs_tools.py` is reference-only on
the LD node. (`objcopy.py`, `cpp_proto_wrapper.py`, `wrapcc.py` are part of the
broader sg7 include/objcopy-attribution diff and are explicitly NOT this ticket.)

After T-23 the objcopy/BN/PD chain is paired: the LD depends on
`objcopy_f1b4b3539b79a9a8a902252c63.o`, whose input is the BN build output
`$(B)/.../codegen/proto.desc`, produced by the BN node that renames
`proto-profile-descriptions.protodesc`. That chain must remain intact.

## Upstream mechanism

`_BUNDLE_TARGET` (build/ymake.core.conf:2881):

```
.CMD=$MOVE_FILE ${result:Target}$Suffix ${noauto;output:Destination} ...
```

with `MOVE_FILE = $FS_TOOLS rename` and
`FS_TOOLS = $YMAKE_PYTHON3 ${input:"build/scripts/fs_tools.py"} ${hide;input:"build/scripts/process_command_files.py"}`.

The `${input:"build/scripts/fs_tools.py"}` is a *non-hidden* input declared by a
macro invoked inside the linkable module. ymake attributes that input to the
module's executable-link (LD) node in addition to the BN bundle-producer command.
Our model splits the BN into its own node and already gives it the fs_tools
closure, but never attributes the same input to the consuming LD node — that is
the gap.

Empirical boundary (verified against sg7 ref): of 90 LD nodes, exactly one — the
codegen PROGRAM that declares BUNDLE — carries fs_tools.py; non-BUNDLE programs
do not. A BUNDLE-declaring *LIBRARY* (`grut/libs/misc/protobuf`, same
BUNDLE+RESOURCE shape) does NOT carry fs_tools.py on its per-module `.a` (AR)
node — only `link_lib.py`. So the attribution targets the executable link (LD)
node only, never the library AR. (An early attempt to also add it to the AR path
regressed sg7 parity by 2 nodes — the grut AR — confirming the asymmetry.)

(Normalize sorts+dedups a node's `inputs` set — `normSortedStrings` in dump.go —
so the *set* must match, not the position; `process_command_files.py` already
present via `link_exe.py` simply dedups.)

## Change (files touched)

- `emit_ld.go`: thread a `hasBundles bool` into `emitLD` and `composeLDInputs`;
  widen the existing `copyFsToolsVFS` gate from `!emitCopy` to
  `!emitCopy && !hasBundles`. Reuses the `ldScriptInputs` closure-expansion loop
  (`scripts[copyFsToolsVFS]` = `[fs_tools.py, process_command_files.py]`).
- `gen.go`: pass `len(d.bundles) > 0` at the program LD call site only. The
  library AR path is deliberately left untouched (see the empirical boundary
  above — upstream does not attribute fs_tools.py to a library `.a`).

No change to `emitBundles` (BN producer shape stays byte-identical), no objcopy /
Python-resource change, no path/output-name special-casing.

## Why no gating-case (sg2..sg6) regression

The gating cases are currently byte-exact PASS. A reachable BUNDLE-declaring
linkable module would already diverge today (our LD/AR missing fs_tools.py vs
upstream); since none do, the new `len(d.bundles) > 0` branch is unexercised by
sg2..sg6 and cannot change their bytes. It only adds the missing input on sg7.
