# T-56 — `_SRC("gperf")` generation + generated `.gperf.cpp` compilation

## Upstream mechanism (reproduce, do not invent)

`/home/pg/monorepo/yatool/build/ymake.core.conf:3277-3278`:

```
macro _SRC("gperf", SRC, SRCFLAGS...) {
    .CMD=$RUN_NO_SANITIZE ${tool:"contrib/tools/gperf"} $GP_FLAGS ${SRCFLAGS} \
         ${pre=-Nin_;suf=_set;nopath;noallext:SRC} ${input:SRC} \
         ${stdout;output;defext=.gperf.cpp;nopath;noext:SRC} \
         ${hide;kv:"p GP"} ${hide;kv:"pc yellow"}
}
```

- `GP_FLAGS` default (`ymake.core.conf:839`): `-CtTLANSI-C -Dk* -c` (three argv tokens).
- `RUN_NO_SANITIZE` is empty in this build config (only set under cross-sanitize) —
  the reference GP node has no `run_tool.py` wrapper. Confirmed against the raw
  reference node below.
- `${pre=-Nin_;suf=_set;nopath;noallext:SRC}` → `-Nin_<basename-all-ext-stripped>_set`
  (e.g. `tag.gperf` → `-Nin_tag_set`, `tags.gperf` → `-Nin_tags_set`).
- `${stdout;output;defext=.gperf.cpp;nopath;noext:SRC}` → stdout redirected to
  `$(B)/<module>/<basename>.gperf.cpp` (nopath ⇒ flat in module build dir; differs
  from bison/ragel `_/` subdir rebasing).
- `${input:SRC}` → the `.gperf` source is a direct input; the `.gperf` is parsed for
  C includes (`parsers_generated.go:67` already registers `CIncludeDirectiveParser`
  for `.gperf`), so its include closure rides as GP-node inputs.

## Our side (reuse existing machinery)

Closest analogs: `emit_r5.go` / `emit_r6.go` — single-pass: emit producer node,
register the generated output in the codegen registry, walk the closure, compile via
`emitCC`. gperf is structurally identical, with one difference: the producer-node
input closure comes from walking the **source** `.gperf` (its C includes), not the
generated cpp. No two-phase pre-pass is needed (the generated `.gperf.cpp` is an
implementation file siblings do not include — unlike a bison `.h`).

Diff is field-based (`dumpContentFields`: cmds, env, inputs, kv, outputs, platform,
requirements, target_properties); `inputs`/`cmds` compared as multisets, so input
order is irrelevant. `foreign_deps` is not a content field; the dep relation is
captured by the Merkle re-uid.
