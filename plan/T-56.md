# T-56 plan: auto RUN_PROGRAM/RUN_PYTHON assembler output is a module source

## Upstream mechanism

ymake auto outputs (`OUT`/`STDOUT`, as opposed to the `_NOAUTO` variants) of a
buildable extension are re-fed as **module sources** and compiled by their
per-extension rule. For an assembler extension (`.asm`, `.s`, `.S`) that rule is
the same one a declared assembler `SRC` uses: x86_64 `.asm` → `yasm`
(`emitASYasm`), otherwise the clang assembler path (`emitAS`). The resulting
object is archived into the module library, and the producing `RUN_PROGRAM`/
`RUN_PYTHON` node is a real build dependency of that compile.

Verified residual cause (corrects T-41's PIC/RESOURCE_LIB classification):

- `yabs/server/cs/libs/mkdb_info/builtin/ya.make` has
  `RUN_PROGRAM(yabs/server/cs/tools/dump_mkdb_info archive_asm zstd_9 STDOUT mkdb_info.asm)`.
- `STDOUT` is an auto output, so upstream assembles `mkdb_info.asm` →
  `mkdb_info.pic.o` (host/PIC platform) and archives it into
  `liblibs-mkdb_info-builtin.a`. That archive member edge keeps the host tool
  `dump_mkdb_info` and its whole closure — including the yweb icookie blacklist
  `LIBRARY` peers compiled `.cpp.pic.o` and archived into the duplicate
  `libicookie*.a` — reachable from the `sg7` `bs_static` target closure.
- Our graph registers generated assembler outputs for include resolution but
  `emitRunProgramsForAR` only compiles+archives auto OUT/STDOUT outputs whose
  extension passes `isCCSourceExt` (C/C++). Generated `.asm`/`.s`/`.S` outputs
  are never assembled, never archived, and the producer tool closure is pruned.

This is a missing host-tool closure edge, **not** a PIC/RESOURCE_LIB platform
transition. No new `Platform` and no `derivePeerInstance`/`instanceKey` change.

## Files to touch

- `emit_pr.go`: in `emitRunProgramsForAR`, replace the CC-only `continue` with a
  `switch` — `isCCSourceExt` → `emitPRDownstreamCC` (unchanged), `isAsmSourceExt`
  → `emitCodegenDownstreamAS`. Add `isAsmSourceExt` (`.asm`/`.s`/`.S`). Preserve
  the existing OUT/STDOUT vs OUT_NOAUTO/STDOUT_NOAUTO selection (only `OUTFiles`
  + `StdoutFile` are iterated; the `_NOAUTO` sets are not).
- `emit_codegen_cc.go`: add `emitCodegenDownstreamAS`, mirroring
  `emitCodegenDownstreamCC` and `emitLibraryAsmSource`'s per-platform dispatch
  (x86_64 `.asm` → `emitASYasm`; else `emitAS`), threading the producer
  `depRefs` (plus any resolved sibling codegen deps) as build deps.
- `emit_as.go`: `emitAS`/`emitASYasm` apply `in.ExtraDepRefs` to `node.DepRefs`
  (matching `emitCC`). Declared-SRC `.asm` leaves `ExtraDepRefs` nil → no-op.

`emit_ar.go` is untouched: it archives the member list it is handed; the
`.pic.o` suffix comes from the host platform's PIC flag, not an output-name
special case.

### STDOUT vs STDOUT_NOAUTO (reviewer-flagged noauto regression)

Upstream spells the two stdout outputs distinctly:
`${stdout;output:STDOUT}` (auto — a module source) versus
`${stdout;noauto;output:STDOUT_NOAUTO}` (noauto — NOT a module source), exactly
as `${hide;output:OUT}` versus `${hide;noauto;output:OUT_NOAUTO}`
(ymake.core.conf:4780,4832 for RUN_PROGRAM/RUN_PYTHON3). Our parser collapsed
both `STDOUT` and `STDOUT_NOAUTO` into the single `StdoutFile` field, so the
auto-output compile loop (the same one this ticket extends to assembler) would
incorrectly assemble+archive a `STDOUT_NOAUTO gen.asm`.

Fix: add a `StdoutNoAuto bool` to `RunProgramStmt`/`RunPythonStmt`, set it in the
parser when the section is `kwSTDOUT_NOAUTO`, and gate the auto-output compile
loops in `emitRunProgramsForAR`/`emitRunPythonForAR` on `!StdoutNoAuto`. The
noauto stdout remains a registered generated output, a node output, and the
command's stdout redirect — only its treatment as a downstream module source is
suppressed, matching upstream's `noauto` modifier. Files: `yamake.go`,
`emit_pr.go`, `emit_py_run.go`. Regressions: `STDOUT_NOAUTO gen.asm` for both PR
and PY assert no `gen.o`/archive member appears.

## Test-first regression (`emit_pr_test.go`)

A `LIBRARY` whose `RUN_PROGRAM(tool ... STDOUT gen.asm)` auto output is `.asm`,
peered by a consumer `PROGRAM`; the tool `PROGRAM` itself `PEERDIR`s a library
that is reachable only through the tool closure. Assert (fails on current code):
1. `gen.asm` is compiled to an object node, depending on the RUN_PROGRAM
   producer;
2. that object is a member of the module library `.a`;
3. the tool program and its peer library become reachable from the consumer
   program's target closure through that archive edge.
