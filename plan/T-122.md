# T-122 plan: serialized-enum / RUN_PROGRAM AR member-order

## Upstream mechanism

ymake processes a module's statements in `(StatementPriority, name)` order
(`devtools/ymake/module_loader.cpp:38`). Every non-temp command output is
re-queued back onto the module source list at its *declaring statement's*
processing point via `QueueCommandOutputs()` â†’
`AddSource("SRCS", output, nullptr)` (`module_builder.cpp:1167-1189`).
`RUN_PROGRAM` and `GENERATE_ENUM_SERIALIZATION` are both default-priority (2)
macros, so among the generated auxiliary members the archive order follows the
declaration sequence of their generating statement. For
`ads/bsyeti/libs/features`, `RUN_PROGRAM` (emits `formula.cpp`) is declared
before `GENERATE_ENUM_SERIALIZATION(formula.h)`, so `formula.cpp.o` precedes
`formula.h_serialized.cpp.o`.

Our emitter instead appends all EN generated CC members before all PR members
in `gen.go` `genCC`, and tags every generated member `SrcMeta{Prio:2,
Generated:true}` with `Seq:0`. They tie, so the stable sort in
`reorderARMembers` keeps append order â€” EN before PR â€” which is the divergence.

## Change

Carry the declaring statement's module-global declaration sequence into the
generated CC members so `reorderARMembers` sorts EN vs PR by real declaration
order instead of append order.

- `modules.go`: add `Seq int` to `RunProgramStmt` and
  `GenerateEnumSerializationStmt`; assign `d.nextDeclSeq()` when each is
  collected in `collectStmts` (the same in-order walk SRCS uses).
- `emit_pr.go` / `emit_en.go`: add a parallel `Seqs []int` to
  `RunProgramsForARResult` / `EnumSrcsResult`; record the statement's Seq for
  each emitted member.
- `gen.go`: in the EN and PR `genCC` blocks, set
  `arDeclMeta[out] = SrcMeta{Prio: stmtPrioDefault, Seq: <stmtSeq>, Generated: true}`.

Producer-before-EN *emission* order (so EN resolves the RUN_PROGRAM-generated
header) is unchanged â€” only the archive-member ordering metadata changes.
