# PERF.md — кандидаты на однопоточное ускорение генерации

База измерения (2026-06-09): 10 прогонов sg5 без `-G`, слитый CPU-профиль
`go tool pprof -proto` = **14 590 ms** сэмплов (~1459 ms CPU/прогон).
Тёплый прогон: **1.17s wall / 1.05s user / 0.25s sys** — фаза генерации CPU-bound,
профиль репрезентативен. Протокол верификации каждого пункта: 10x sg5 merged
pprof до/после + `./dev/validate.py` (байт-в-байт).

Снято на: `edd4ff6`. Профили: `.out/prof/sg5_merged.pb.gz` (+10 индивидуальных).

## Tier 1 — крупные (5–10% каждая)

### 1. Сабсампшен окон в сплайс-цикле (атака на 12.77x read amplification) — СДЕЛАНО

**Результат (10x sg5)**: поглощено 52.5% сплайсов (212,926 из ~405k;
`closureSubsumed` в perf stats); `dfs.func2` 2280→1360 ms (−40%), `IdSet.has`
1250→520 ms (−58%); тёплый прогон wall 1.14→1.09 s, user 1.02→0.90 s. Все
остальные счётчики сканера и гейт — байт-в-байт.

- **Где**: `scanner.go:489-497` (dfs pass 2), `tarjan_ctx.go:200-214` (strongconnect).
- **Доказательство**: `IdSet.has` = 1250 ms flat (8.6%); весь `dfs.func2` = 2280 ms
  (15.6%). ~405k сплайсов окон/прогон (closureHits target+host) × ~100 элементов
  ≈ 40M `has`-проверок.
- **Механизм**: окна транзитивно замкнуты ⇒ если `ch` уже в блоке, всё его окно —
  подмножество блока, цикл по нему лишний; при скипе не нужны даже
  `cachedClosure`/`closureWindow`.
- **Нюанс (верифицирован)**: листья `ClosureLeaves` (`scanner.go:505-509`)
  добавляются в окна без транзитивного замыкания — элемент, присутствующий как
  (в т.ч. вложенный) лист, не гарантирует поглощения своего окна.
- **Хранение (реализовано)**: глобальный `CodegenRegistry.leafEver BitSet` —
  бит ставится при регистрации листа (`Register`/`AddClosureLeaf`). Скип:
  `closure.has(ch) && !leafEver.has(ch)` (`scanCtx.windowSubsumed`, оба
  сплайс-цикла через `closureSink`). Корректность по монотонности: лист
  попадает в окно только из `ClosureLeaves()` ⇒ бит стоит строго до того, как
  листовой элемент может оказаться в дедуп-наборе; флаг только запрещает
  скипы. Консервативность: VFS двойной роли (лист и обычный include) теряет
  скип — листья это copy-tooling и COPY-источники, потерь не наблюдается.
  (Отвергнуты: пометка «пришёл сплайсом» — сплайс копирует вложенные листья
  чужих окон, для них поглощение не выполняется; `XFS = VFS<<1|leaf` в окнах —
  ломает zero-copy границу `walkClosureRoot → Node.Inputs`
  (`emit_sources.go:51-58`), конверсия = десятки MB churn на ~22.9k CC-узлов;
  два списка на окно — усложняет сплайс и всех потребителей.)
- **Верификация**: счётчик скипов в `YATOOL_PERF_STATS`; скип не переставляет
  элементы, только не добавляет дубликаты ⇒ гейт обязан остаться байт-в-байт.
- **Ожидание**: до ~10%.

### 2. Отключить GC на фазе генерации — СДЕЛАНО

**Результат**: `-j 0` (generate-only) — GC off целиком: sg5 user 0.95→0.77 s
(−19%), peak 496→681 MB; `-j N` (сборка, процесс живёт весь билд и делит RAM
с компиляторами) — GOGC=400 (`executorGCPercent`): скан 100..800/off показал
плато user с 400 и выше (0.83–0.90 s; разброс 400..800 — фаза последнего
марк-цикла, не тренд). Явный `GOGC` в окружении побеждает в обоих режимах.
Гейт байт-в-байт.

## Tier 2 — средние (2–3% каждая)

### 3. Сырой syscall-путь чтения файлов — СДЕЛАНО

**Результат (10x sg5)**: `osFS.Read` cum 2030→1730 ms (−15% пути чтения, −3%
профиля); тёплый wall 1.02→0.99-1.00 s. Реализация: per-platform
`readFileInto` — `fs_read_linux.go` (сырые open/fstat/read/close с
EINTR-ретраями, без `os.File`/финализатора/poll.FD) + `fs_read_other.go`
(`//go:build !linux`, прежний os.File-путь с `f.Stat()` вместо
Linux-специфичного `syscall.Fstat` — заодно убрал Stat_t из fs.go, darwin/
windows-совместимо). Счётчики идентичны, гейт байт-в-байт. dirfd-relative
openat (п.10) не делался — отдельная мелочь.

### 4. `hashScanContext` по интернированным id вместо байтов строк — СДЕЛАНО

**Результат (10x sg5)**: 580→<60 ms (ниже порога отсечки профиля); тёплый
прогон user 0.77→0.70-0.75 s. Реализация: цепочка `h = mix64(h ^
internTable.los[v.strID()])` (выделен `mix64` — финализатор splitmix64, его же
использует `splitMix64`) + length-prefix каждого слайса вместо сентинелов.
Счётчики сканера идентичны (мерджа конфигов нет), гейт байт-в-байт.

### 5. Мемоизация позиции `/*` в `parseCIncludes` — ОТВЕРГНУТО АНАЛИЗОМ

Гипотеза о ре-сканах неверна: covers-вызов случается только для line-leading
`#`, и оба его исхода двигают `clean` вперёд (`end` при покрытии, `next > hi`
после parseDirectiveInline); путь `q0 != hi` (mid-line `#`) до covers-вызова
не доходит. Окна `[clean, hi)` между вызовами дизъюнктны ⇒ скан `/*` уже
амортизированно линеен — одна доп. проходка по межддирективному тексту на
файл (того же порядка, что и `IndexByte('#')`-проход). Мемо нечего убирать;
сокращение возможно только слиянием проходов — несоразмерная сложность.

## Tier 3 — по ~1–1.5%

### 6. `Source()/Build()` без конкатенации — СДЕЛАНО

**Результат (10x sg5)**: `Source` cum 370→220 ms (остаток — xxh3+probe,
неустраним), `concatstrings` всего 370→150 ms, total samples −2%.
Реализация: `internPrefixed(prefix, rel)` — scratch `[]byte` + `internBytes`
(хит — ноль аллокаций, мисс — одна стабильная копия как раньше); root-бит
статический, ветка `full[2]=='B'` не нужна. Третья точка того же паттерна —
`recordContentHash`. Хранение полной строки в intern-таблице не менялось.
Счётчики идентичны, гейт байт-в-байт.

### 7. UID-слой (`writeVFSSlice`) — СДЕЛАНО (иначе, чем планировалось)

Слияние `{los, contentHash}` в AoS **отклонено** при детальном разборе: (а)
два лукапа в петле индексируются одним `s` независимо (не цепочкой) — OoO-ядро
параллелит промахи, латентность уже ~1×miss; (б) AoS ×2-страйдит `los` для
всех остальных читателей (интерн-верификация ×3 сайта + writeSTR каждого STR
в uid), пара же нужна только $(S)-inputs — асимметрия с профилем риска
битсета (п.9).

Реальное мясо нашлось в `canonBuf.writeUint64` (310 ms flat): форма
`PutUint64(stack)` + `append(b[:]...)` платила runtime-вызов memmove на
каждые 8 байт. Замена на прямой 8-байтовый append (инлайн-сторы), то же для
writeUint32; байтовая последовательность идентична ⇒ uid'ы не меняются, кеши
живы. **Результат (10x sg5)**: `writeUint64` 310→190 ms, `writeVFSSlice` cum
880→670 ms (−24%), user 0.71→0.69 s. Гейт байт-в-байт.

### 8. `Listdir`: `File.ReadDir(-1)` + pointerless-набор — ОТВЕРГНУТО ИЗМЕРЕНИЕМ

Сортировка в `os.ReadDir` = 20 ms на 10 прогонов (0.2%) — ниже порога
интереса. GC-составляющая (строки-ключи map под сканом) исчезла из гейта
вместе с отключением GC в `-j 0` (п.2); для `-j N` (GOGC=400) остаётся
теоретической — GC-циклы там редки, измерять надо на реальной сборке.

### 9. BitSet вместо IdSet для `tjc.closure` — ОТВЕРГНУТО ИЗМЕРЕНИЕМ

Реализован denseBitSet (1 бит/id, очистка по списку dirty-слов) и переключён
`tjc.closure`. 10x sg5: `dfs.func2` cum 1290→1730 ms (+34%), `has` 490→540,
`add` 120→300 (+2.5x), user 0.71→0.75 s. Гипотеза «`has` доминируют
LLC-промахи» опровергнута: id кластеризованы (intern в порядке обнаружения ⇒
общие заголовки = низкие id), рабочее множество IdSet уже кеш-резидентно;
битсет добавил инструкций (shift+and+test на пробе; RMW слова + dirty-ветка
на add против одного store эпохи). Откачено.

### 11. Reentry вместо предзнания языка входа в peer — СДЕЛАНО

Полка `derivePeerInstance → peerLanguageFor → moduleInfoForInstance` (880 ms,
9.6%: отдельный ParseFile+collectModule каждого peer'a до его genModule +
peerYaMakeExists на каждое ребро) удалена архитектурно: язык входа больше не
предвычисляется. `peerEntryLanguage` выбирает вариант только по потребителю
(py-ish родитель ⇒ запрос LangPy); genModule после первого `collectModule`
видит, что модуль не PROTO_LIBRARY (py-варианта нет), и **реентрит** себя как
LangCPP, алиася py-ключ memo на C++-результат. Это общая точка
reenter-with-corrected-parameters — будущий фикс DLL (reentry с PIC) ложится
сюда же; ограничение: решение о reentry ДО первой эмиссии (стриминговый
эмиттер не умеет отзывать ноды). Удалены moduleTypeCache/moduleTypeInfo
(включая мёртвую копию ExcludeTags). **Результат (10x sg5)**: полка 880→0 ms,
total −4.7%, user 0.65→0.61-0.64 s, wall 0.91→0.89-0.90 s. Счётчики
идентичны, гейт байт-в-байт (включая sg4 с py/proto-модулями).

### 12. Раунд micro-кандидатов A/B/C — НИЖЕ РАЗРЕШЕНИЯ ИЗМЕРЕНИЯ

После п.11 проверены три кандидата из свежего профиля; все три дали эффект
ниже шума батча (±100-200 ms на 10 прогонов):

- **A. `sourceUnderCache` хранит VFS вместо rel-строки** — хит больше не
  ре-интернирует через `Source(rel)`. Оставлено как cleanup (`4297456`).
- **B. bulk-сплайс первого окна без `has`** — стоимость переехала из
  `IdSet.has` (−250 ms) в `runtime.memmove` (+290 ms): копирование больших
  окон memory-bound, а has-цикл шёл по кеш-резидентному набору. Третье
  подтверждение урока п.9. Откачено (меняло содержимое окон без выигрыша).
- **C. дедуп `resolveSearchPath` по `out` вместо pooled-map** — set имел ≤3
  элемента на вызов; map/pool/clear/«B:»-конкаты убраны. Оставлено как
  cleanup (`64c79d4`).

Текущий floor (10x sg5 ≈ 8.8 s сэмплов, user ≈ 0.62 s): syscalls чтения
исходников 20% (потолок без п.10), интерн-пробы xxh3+IntMap ~7%, сплайс
окон ~12% (кеш-резидентный, нечего срезать), парсер ~9% (линейные проходы),
UID ~9%. Дальнейшие однопоточные выигрыши требуют архитектурных решений
(п.10 или сокращение числа читаемых файлов), не микрооптимизаций.

## Архитектурный рычаг (отдельное решение)

### 10. Персистентный кеш parse + content-hash по (mtime, size, inode)

- **Доказательство**: 44.6k чтений = 16.5% CPU (`readIntoRaw`), парсинг 6.6%
  (`parseCIncludes`), xxh3 контента ~2% — повторяются каждый прогон на
  неизменённых файлах.
- **Механизм**: кеш уровня upstream-ymake, ключ (mtime, size, inode).
- **Цена**: меняет операционную модель (доверие mtime); решение за владельцем.
- **Ожидание**: ~25% CPU тёплого прогона.

## Порядок исполнения

Предложение по соотношению эффект/строки: **2 → 1 → 4 → 3 → 5 → 6**, далее
7/8/9 по результатам повторного профиля. Каждый пункт — отдельный коммит с
10x-профилем до/после и зелёным гейтом.
