# PERF.md — кандидаты на однопоточное ускорение генерации

База измерения (2026-06-09): 10 прогонов sg5 без `-G`, слитый CPU-профиль
`go tool pprof -proto` = **14 590 ms** сэмплов (~1459 ms CPU/прогон).
Тёплый прогон: **1.17s wall / 1.05s user / 0.25s sys** — фаза генерации CPU-bound,
профиль репрезентативен. Протокол верификации каждого пункта: 10x sg5 merged
pprof до/после + `./dev/validate.py` (байт-в-байт).

Снято на: `edd4ff6`. Профили: `.out/prof/sg5_merged.pb.gz` (+10 индивидуальных).

## Tier 1 — крупные (5–10% каждая)

### 1. Сабсампшен окон в сплайс-цикле (атака на 12.77x read amplification) — СДЕЛАНО

**Результат (10x sg5)**: поглощено 52.5% сплайсов (212,926 из ~405k;
`closureSubsumed` в perf stats); `dfs.func2` 2280→1360 ms (−40%), `IdSet.has`
1250→520 ms (−58%); тёплый прогон wall 1.14→1.09 s, user 1.02→0.90 s. Все
остальные счётчики сканера и гейт — байт-в-байт.

- **Где**: `scanner.go:489-497` (dfs pass 2), `tarjan_ctx.go:200-214` (strongconnect).
- **Доказательство**: `IdSet.has` = 1250 ms flat (8.6%); весь `dfs.func2` = 2280 ms
  (15.6%). ~405k сплайсов окон/прогон (closureHits target+host) × ~100 элементов
  ≈ 40M `has`-проверок.
- **Механизм**: окна транзитивно замкнуты ⇒ если `ch` уже в блоке, всё его окно —
  подмножество блока, цикл по нему лишний; при скипе не нужны даже
  `cachedClosure`/`closureWindow`.
- **Нюанс (верифицирован)**: листья `ClosureLeaves` (`scanner.go:505-509`)
  добавляются в окна без транзитивного замыкания — элемент, присутствующий как
  (в т.ч. вложенный) лист, не гарантирует поглощения своего окна.
- **Хранение (реализовано)**: глобальный `CodegenRegistry.leafEver BitSet` —
  бит ставится при регистрации листа (`Register`/`AddClosureLeaf`). Скип:
  `closure.has(ch) && !leafEver.has(ch)` (`scanCtx.windowSubsumed`, оба
  сплайс-цикла через `closureSink`). Корректность по монотонности: лист
  попадает в окно только из `ClosureLeaves()` ⇒ бит стоит строго до того, как
  листовой элемент может оказаться в дедуп-наборе; флаг только запрещает
  скипы. Консервативность: VFS двойной роли (лист и обычный include) теряет
  скип — листья это copy-tooling и COPY-источники, потерь не наблюдается.
  (Отвергнуты: пометка «пришёл сплайсом» — сплайс копирует вложенные листья
  чужих окон, для них поглощение не выполняется; `XFS = VFS<<1|leaf` в окнах —
  ломает zero-copy границу `walkClosureRoot → Node.Inputs`
  (`emit_sources.go:51-58`), конверсия = десятки MB churn на ~22.9k CC-узлов;
  два списка на окно — усложняет сплайс и всех потребителей.)
- **Верификация**: счётчик скипов в `YATOOL_PERF_STATS`; скип не переставляет
  элементы, только не добавляет дубликаты ⇒ гейт обязан остаться байт-в-байт.
- **Ожидание**: до ~10%.

### 2. Отключить GC на фазе генерации — СДЕЛАНО

**Результат**: `-j 0` (generate-only) — GC off целиком: sg5 user 0.95→0.77 s
(−19%), peak 496→681 MB; `-j N` (сборка, процесс живёт весь билд и делит RAM
с компиляторами) — GOGC=400 (`executorGCPercent`): скан 100..800/off показал
плато user с 400 и выше (0.83–0.90 s; разброс 400..800 — фаза последнего
марк-цикла, не тренд). Явный `GOGC` в окружении побеждает в обоих режимах.
Гейт байт-в-байт.

## Tier 2 — средние (2–3% каждая)

### 3. Сырой syscall-путь чтения файлов

- **Где**: `fs.go:287` (`readIntoRaw`).
- **Доказательство**: `os.Open` = 1120 ms, из них сам openat-syscall 860 ms —
  остаток (~260 ms) это аллокация `os.File` + финализатор + `poll.FD`;
  `f.Read` через `ignoringEINTRIO` добавляет ~200 ms обвязки над read(2) (990 ms).
  44.6k чтений/прогон (parsedMisses).
- **Механизм**: `syscall.Openat(dirfd(srcRoot), rel)` + `Fstat` + `Read` +
  `Close` — без `os.File`, финализаторов и poll-обвязки; dirfd-relative open
  заодно срезает kernel path-walk префикса source root.
- **Ожидание**: ~3%.

### 4. `hashScanContext` по интернированным id вместо байтов строк

- **Где**: `scanner.go:299-321`; вызов на каждый source из `walkClosureRoot`
  (`sources.go:125`), хотя от source хеш не зависит.
- **Доказательство**: 400 ms (2.7%), из них 360 ms — FNV по байтам `v.Rel()`
  каждого элемента `PeerAddInclSet`.
- **Механизм**: ключ нужен только in-memory (`ctxNumByHash`,
  `resolveIndexByConfig`) ⇒ id стабильны в рамках прогона: миксовать `uint32(v)`
  через splitmix вместо побайтового FNV строк.
- **Ожидание**: ~2.5% (400 → ~30 ms).

### 5. Мемоизация позиции `/*` в `parseCIncludes`

- **Где**: `parsers.go:439` → `leadingBlockCoversHi` (parsers.go:549).
- **Доказательство**: 340 ms (2.3%) — на каждую директиву
  `bytes.Index(data[clean:hi], "/*")` ре-сканирует межддирективный текст.
- **Механизм**: помнить абсолютную позицию следующего `/*` (и его закрытия),
  инвалидировать только при проходе через неё — текст сканируется один раз
  вместо двух.
- **Ожидание**: ~1.5–2%.

## Tier 3 — по ~1–1.5%

### 6. `Source()/Build()` без конкатенации

- **Где**: `vfs.go:43,47`.
- **Доказательство**: `Source` = 500 ms cum (3.4%), из них concat-аллокация
  `"$(S)/" + rel` = 180 ms (лидер среди `concatstring2`) + GC-хвост.
- **Механизм**: многоразовый scratch `[]byte` (prefix+rel) + `internBytes` —
  без аллокации строки.
- **Ожидание**: ~1.2%.

### 7. Совмещённый массив `{los, contentHash}` по strID

- **Где**: `uid.go:117-126` (`writeVFSSlice`), `str_intern.go` (`los`),
  `fs.go` (`contentHashes`).
- **Доказательство**: `writeVFSSlice` = 940 ms (6.4%); цикл inputs делает два
  зависимых случайных доступа в два ~4MB массива (`los[s]`,
  `contentHashes[s]`) на каждый input-элемент.
- **Механизм**: оба массива индексируются одним strID — один массив структур
  `{lo, contentHash uint64}` = один кеш-промах вместо двух.
- **Ожидание**: ~1–1.5%.

### 8. `Listdir`: `File.ReadDir(-1)` + pointerless-набор

- **Где**: `fs.go:132-164`.
- **Доказательство**: `os.ReadDir` сортирует ~370k имён (6180 каталогов), map
  не нуждается в порядке; `map[string]bool` держит ~370k string-заголовков под
  GC-сканом (`scanobject` = 1200 ms суммарно по куче).
- **Механизм**: `File.ReadDir(-1)` (без сортировки); набор без указателей
  (например, open-addressing по xxh3 имени) вместо `map[string]bool`.
- **Ожидание**: ~0.5–1% (частично через снижение GC).

### 9. BitSet вместо IdSet для `tjc.closure`

- **Где**: `id_set.go`, использование в `scanner.go:473-497`,
  `tarjan_ctx.go:186-214`.
- **Доказательство**: `gen []uint32` × vfsBound = единицы MB ⇒ `has`-цикл бьёт
  в LLC; `IdSet.has` = 1250 ms flat преимущественно кеш-промахи.
- **Механизм**: BitSet в 32 раза плотнее (L2-resident) при очистке по списку
  dirty-слов вместо epoch.
- **Зависимость**: оценивать ПОСЛЕ №1 — тот сократит число `has` на порядок и
  может сделать пункт ненужным.
- **Ожидание**: 1–3% (до №1).

## Архитектурный рычаг (отдельное решение)

### 10. Персистентный кеш parse + content-hash по (mtime, size, inode)

- **Доказательство**: 44.6k чтений = 16.5% CPU (`readIntoRaw`), парсинг 6.6%
  (`parseCIncludes`), xxh3 контента ~2% — повторяются каждый прогон на
  неизменённых файлах.
- **Механизм**: кеш уровня upstream-ymake, ключ (mtime, size, inode).
- **Цена**: меняет операционную модель (доверие mtime); решение за владельцем.
- **Ожидание**: ~25% CPU тёплого прогона.

## Порядок исполнения

Предложение по соотношению эффект/строки: **2 → 1 → 4 → 3 → 5 → 6**, далее
7/8/9 по результатам повторного профиля. Каждый пункт — отдельный коммит с
10x-профилем до/после и зелёным гейтом.
